# 理财助手深度理解与提升设计稿

日期：2026-06-20  
状态：设计稿  
目标：让投资分析助手从“能调用多个分析功能”升级为“持续理解用户、形成可执行理财决策闭环”的个人理财协同系统。

## 一、结论先行

当前项目已经具备不错的基础：持仓、交易、现金、估值、关注列表、文章、RAG、Agent 编排、质量评估、异步任务和执行链路都已经存在。它的问题不是“缺一个 Agent”或“缺一个页面”，而是数据和能力还分散在多个入口里，尚未形成一个连续的理财决策系统。

最值得增强的方向是：

1. 建立“决策档案”：每一次 AI 建议都要沉淀为可追踪的建议、证据、适用条件、执行状态和复盘结果。
2. 升级“用户理解”：KYC 问卷、真实持仓、交易行为、反馈、对话偏好、风险承受变化要统一成可解释画像。
3. 建立“行动闭环”：把建议从 Markdown 文本变成行动清单，支持接受、暂缓、拒绝、提醒、执行、复盘。
4. 增强“证据可信度”：所有投资结论要带来源、时间、数据新鲜度、置信度、反面风险和缺失数据说明。
5. 治理“大文件和重复逻辑”：保持功能前进的同时，逐步拆分后端大路由和前端大页面。

推荐路线：先做“理财决策中枢”而不是继续堆单点分析功能。它会复用现有持仓、估值、RAG、Agent、质量系统，让已有能力真正串起来。

## 二、当前项目理解

### 2.1 已有能力

后端已经形成多层能力：

- 数据层：SQLite 记录指数估值、螺丝钉估值、文章、分析历史、持仓、交易、现金、关注列表、预警、异步任务、执行链路、工具审计、RAG 日志和用户记忆。
- 投资数据：支持估值百分位、基金净值、持仓盈亏、现金管理、债市温度、热点新闻、基金诊断、MCP 数据源。
- Agent 系统：`orchestrator.py` 负责需求澄清、专家路由、RAG 注入、KYC 注入、持仓上下文、交叉审阅和仲裁；`multi_agent.py` 负责专家工具调用。
- 个性化：已有 KYC 问卷、反馈学习、用户记忆、画像注入、RAG 个性化加权。
- 质量闭环：已有 LLM 输出评估、Bad Case、评测集、质量仪表盘、专家运行监控。
- 工程化：已有异步任务、SSE 恢复、trace、tool audit、token 预算。

前端已经形成主要工作台：

- 每日看板：低估指数、持仓健康、现金债券、热点、日报、全景诊断、推荐验证。
- AI 对话：SSE 流式、多专家结果、工具调用、执行链路、反馈、恢复。
- 持仓管理：持仓、交易、现金、基金详情、AI 分析、预警、关注列表。
- 估值数据：指数估值、趋势、DD 图片解析、增强策略、AI 深度分析。
- 知识库与进化系统：RAG 测试、质量仪表盘、Bad Case、评测集、Agent 管理。

### 2.2 已有强项

1. 数据底座比较完整  
   持仓、交易、现金、估值、文章和反馈都已在本地数据库中沉淀，具备做个人化理财分析的前提。

2. Agent 架构已经可扩展  
   专家 Agent 从数据库加载，prompt 可管理，工具列表可分配，不是完全硬编码。

3. RAG 不只是简单搜索  
   已有 FTS5、ChromaDB、RRF、同义词扩展、知识图谱、时效过滤和个性化加权。

4. 质量意识很强  
   已经有 LLM-as-judge、自动评估、Bad Case、评测集、反馈学习和运行监控。

5. 前端已经开始组件化治理  
   已有 `AgentResultCard`、`AIActionButton`、`AsyncStateBlock`、`StatusBadge`、`useAsyncTask` 等公共能力。

## 三、当前不好的点

### 3.1 产品层问题

1. 建议停留在“回答”，没有进入“行动”  
   AI 会给出建议，但建议没有统一转成行动卡片。用户无法稳定地记录“我接受了哪条建议、为什么、计划什么时候执行、后来结果怎样”。

2. 懂用户还不够深  
   KYC 问卷覆盖风险偏好、期限、资金规模、经验、亏损承受度和关注品种，但缺少更关键的理财约束：
   - 家庭/个人目标：养老、买房、教育、备用金、长期增值。
   - 资金分层：短期可动用、1-3 年、3 年以上。
   - 现金流：每月可投入金额、未来大额支出。
   - 执行偏好：喜欢定投、网格、一次性建仓、只看提醒。
   - 行为偏差：追涨、恐慌卖出、过度集中、频繁交易。

3. “质量评估”还不是“理财结果评估”  
   现有质量系统主要评回答的数据准确性、逻辑、可执行性、用户理解；但理财最重要的是：
   - 建议是否被采纳。
   - 用户为什么拒绝。
   - 执行后是否偏离计划。
   - 回看时是否避免了错误。
   - 是否改善了组合风险收益结构。

4. 看板信息多，但缺少一个总控问题  
   每日看板展示了估值、持仓、现金、热点、日报，但用户每天真正想知道的是：
   - 今天我需要做什么？
   - 哪些不用动？
   - 哪些只是观察？
   - 哪些数据过期不能信？
   - 哪条建议和我的长期计划有关？

5. 证据可追溯还不够显性  
   聊天里有工具调用和 RAG 来源，但最终建议未统一呈现“证据包”：数据来源、更新时间、新鲜度、置信度、缺失项、反向风险。

### 3.2 技术层问题

1. 核心文件过大  
   目前文件行数显示：
   - `frontend/src/components/PortfolioManagement.vue`：7652 行
   - `frontend/src/components/ValuationHistory.vue`：3442 行
   - `backend/routers/portfolio.py`：2452 行
   - `backend/agent/orchestrator.py`：1956 行
   - `frontend/src/api/index.js`：1603 行

   这些文件已经影响理解、测试和后续功能演进。

2. 同一类能力有多套入口  
   AI 分析、反馈、异步状态、质量评估、推荐验证分散在 Dashboard、Portfolio、Valuation、Chat 等页面，体验和数据模型不完全统一。

3. 配置治理不彻底  
   部分阈值已从 `system_config` 读取，但仍存在模块内阈值、策略、提示文案散落。比如主动预警的亏损承受度映射、估值极端区间、watchlist 巡检阈值仍可进一步配置化。

4. 记忆系统已有雏形，但缺少产品化校验  
   `user_memories`、反馈学习和 KYC 更新都存在，但用户看不到系统“记住了什么、为什么记住、是否可撤销”。这会影响信任。

5. Agent 路由偏任务分类，不够像理财顾问  
   Clarification 主要判断 chat/simple/medium/complex 和专家列表；面对“要不要买/加仓/减仓”时，系统应先判断目标、期限、现金、风险预算、已有仓位是否足够明确，必要时主动澄清。

## 四、方案取舍

### 方案 A：继续增强单点 Agent

做法：增加更多专家、更多工具、更多 prompt，例如基金经理专家、宏观专家、行为专家。

优点：
- 实施快。
- 对现有结构冲击小。
- 单个问题的回答可能更丰富。

缺点：
- 容易继续堆能力，用户仍然只得到一段回答。
- 不解决建议无法追踪、无法复盘的问题。
- 更容易增加 token 和延迟。

适合：短期补某个具体分析能力。

### 方案 B：先重构前后端大文件

做法：拆 `PortfolioManagement.vue`、`ValuationHistory.vue`、`portfolio.py`、`api/index.js`。

优点：
- 降低工程复杂度。
- 后续开发更稳。
- 测试更容易补。

缺点：
- 用户短期感知弱。
- 容易陷入重构，理财能力不一定提升。

适合：与新能力并行做小步治理，不适合作为唯一主线。

### 方案 C：建设“理财决策中枢”（推荐）

做法：以用户每日理财决策为中心，新增统一的决策档案、行动清单、画像证据、证据包和复盘机制，把现有 Agent/RAG/持仓/估值/反馈串起来。

优点：
- 直接服务“更懂我们、提升理财”。
- 能复用已有数据和 Agent，不需要推倒重来。
- 能把质量系统从“评回答”升级成“评决策闭环”。
- 可以分阶段落地，每阶段都有可见收益。

缺点：
- 需要定义新的核心数据模型。
- 需要前后端共同调整。
- 对一致性和追踪要求更高。

推荐选择方案 C，并配合必要的小步结构治理。

## 五、推荐设计：理财决策中枢

### 5.1 核心概念

新增一个跨页面的“理财决策中枢”，它不是替代现有页面，而是把现有页面产生的建议和信号统一收口。

它回答四个问题：

1. 我现在是谁：画像、目标、风险预算、现金约束、交易习惯。
2. 我现在拥有什么：持仓、现金、关注列表、估值、债市、热点、知识库。
3. 我现在该做什么：买入、卖出、加仓、减仓、持有、观察、补数据、复盘。
4. 我上次做得怎样：建议是否执行、是否有效、是否要调整画像或规则。

### 5.2 新增数据模型

#### 1. decision_records：决策档案

记录一次 AI 或规则生成的理财建议。

关键字段：

- `id`
- `user_id`
- `source_type`：chat / dashboard / portfolio / valuation / watchlist / alert / manual
- `source_id`：来源记录 ID
- `decision_type`：buy / sell / add / reduce / hold / watch / rebalance / collect_data
- `target_type`：fund / index / portfolio / cash / bond / market
- `target_code`
- `target_name`
- `summary`：一句话建议
- `rationale`：理由摘要
- `evidence_json`：证据包
- `risk_json`：风险和反证
- `suitability_json`：与用户画像的匹配情况
- `confidence`：high / medium / low
- `status`：proposed / accepted / rejected / deferred / executed / expired / reviewed
- `user_note`
- `created_at`
- `due_at`
- `review_at`
- `updated_at`

#### 2. decision_actions：行动清单

把一条建议拆成可执行动作。

关键字段：

- `id`
- `decision_id`
- `action_type`：buy / sell / set_alert / refresh_data / read_article / review / ask_clarification
- `title`
- `params_json`
- `status`：todo / doing / done / skipped
- `scheduled_at`
- `completed_at`

#### 3. profile_evidence：画像证据

让“系统为什么觉得我稳健/喜欢数据/讨厌泛泛建议”可解释。

关键字段：

- `id`
- `user_id`
- `dimension`
- `value`
- `source_type`：kyc / feedback / conversation / transaction / manual
- `source_id`
- `evidence_text`
- `confidence`
- `is_active`
- `created_at`

#### 4. decision_reviews：复盘记录

把质量闭环从“回答质量”延伸到“理财结果和行为改善”。

关键字段：

- `id`
- `decision_id`
- `review_type`：auto / user / ai
- `outcome`：helped / neutral / hurt / not_executed / insufficient_data
- `market_result_json`
- `portfolio_result_json`
- `behavior_note`
- `lesson`
- `created_at`

### 5.3 决策证据包格式

所有投资建议都应附带统一证据包，避免“AI 说了算”。

建议结构：

```json
{
  "data_points": [
    {
      "name": "沪深300 PE百分位",
      "value": "18%",
      "source": "index_valuations",
      "as_of": "2026-06-20",
      "freshness": "fresh"
    }
  ],
  "portfolio_context": {
    "holding_pct": "12%",
    "profit_rate": "-4.2%",
    "cash_ratio": "18%"
  },
  "knowledge_sources": [
    {
      "type": "book",
      "title": "聪明的投资者",
      "reason": "安全边际原则"
    }
  ],
  "missing_data": [
    "基金最新重仓股未获取到"
  ],
  "counter_arguments": [
    "低估可能来自盈利下修，而不是单纯价格便宜"
  ]
}
```

前端展示时不必把 JSON 暴露给用户，而是显示为：

- 依据
- 适合你的原因
- 风险与反证
- 数据缺口
- 来源与更新时间

### 5.4 用户画像升级

当前 KYC 保留，但升级为“静态画像 + 动态画像 + 可解释证据”。

#### 静态画像

来自问卷和手动编辑：

- 风险偏好
- 亏损承受度
- 投资期限
- 资金体量
- 投资经验
- 关注品种

#### 动态画像

从行为中学习：

- 是否偏好数据表格还是结论先行。
- 是否经常问“能不能买”，说明需要决策型建议。
- 是否对亏损敏感，来自下跌时的提问和交易。
- 是否频繁买卖，来自交易记录。
- 是否接受过定投/分批策略。
- 是否经常拒绝高风险建议。

#### 可解释画像面板

新增“我的画像”页面或弹窗模块：

- 当前画像
- 系统最近学到的偏好
- 每条偏好的证据来源
- 用户可确认、修改、删除
- 画像更新时间

这会显著提升信任感：系统不是神秘地“记住你”，而是让用户看见它如何理解自己。

### 5.5 Agent 编排升级

新增一个“决策守门员”阶段，放在现有 clarification 之后、专家执行之前。

职责：

1. 判断用户问题是否会产生交易/配置建议。
2. 检查关键约束是否足够：
   - 投资目标是否明确。
   - 资金期限是否明确。
   - 现金比例是否足够。
   - 当前仓位是否过高。
   - 数据是否新鲜。
   - 用户风险承受是否允许。
3. 若缺少关键条件，则先澄清，不直接给操作建议。
4. 若条件足够，则生成 `decision_context` 给专家。

示例：

用户问：“恒生科技能加仓吗？”

守门员应先检查：

- 用户是否持有恒生科技相关基金。
- 当前占总资产比例。
- 当前盈亏和最近买入价。
- 恒生科技估值是否新鲜。
- 用户亏损承受度和投资期限。
- 现金比例是否足够。

如果缺失现金或期限，应回复：

“可以分析，但加仓建议需要先确认这笔资金是否 3 年内不用，以及你是否愿意承受港股科技较大波动。你希望按长期定投还是短期交易来判断？”

### 5.6 行动清单体验

在每日看板新增“今日行动”模块。

动作分为四类：

1. 必须处理  
   例如数据过期、持仓触发高风险、现金低于安全垫。

2. 可执行  
   例如某指数进入计划内低估区，且现金充足、仓位未超限。

3. 观察  
   例如关注列表到达提醒线，但证据不足或波动较大。

4. 复盘  
   例如 30 天前建议已到复盘时间。

每条行动卡片包含：

- 建议标题
- 涉及资产
- 建议动作
- 为什么适合我
- 关键证据
- 风险提示
- 按钮：接受、暂缓、拒绝、查看证据、转入对话

接受后可以：

- 生成提醒。
- 创建交易草稿。
- 写入调仓计划。
- 进入复盘队列。

### 5.7 输出格式收敛

所有会影响投资行为的 AI 输出，都要求同时产出两层结果：

1. 用户可读 Markdown  
   用于聊天、卡片展示。

2. 结构化决策 JSON  
   用于入库、行动清单、复盘和质量评估。

如果模型 JSON 解析失败，则只展示 Markdown，不创建决策档案，并记录到 Bad Case。

## 六、后端设计

### 6.1 新增模块

建议新增：

- `backend/db/decisions.py`  
  决策档案、行动清单、画像证据、复盘 CRUD。

- `backend/agent/decision_guard.py`  
  决策守门员：判断是否可直接给建议、是否需要澄清、生成 decision_context。

- `backend/agent/decision_extractor.py`  
  从 AI 输出中提取结构化决策 JSON，并做 schema 校验。

- `backend/agent/decision_reviewer.py`  
  到期复盘：检查建议后的估值、价格、持仓、交易和用户反馈。

- `backend/routers/decisions.py`  
  决策档案和行动清单 API。

### 6.2 API 草案

```text
GET  /api/decisions
GET  /api/decisions/today
GET  /api/decisions/{id}
PUT  /api/decisions/{id}/status
POST /api/decisions/{id}/actions/{action_id}/complete
POST /api/decisions/{id}/review
GET  /api/profile/evidence
PUT  /api/profile/evidence/{id}
POST /api/profile/evidence/{id}/confirm
DELETE /api/profile/evidence/{id}
```

### 6.3 集成点

1. Chat SSE 完成后  
   如果回答包含投资建议，调用 `decision_extractor`，成功则创建 `decision_records`。

2. Dashboard 全景诊断完成后  
   将“今日行动”写入决策档案。

3. Portfolio AI 分析完成后  
   将加仓、减仓、持有、再平衡建议入库。

4. Watchlist 巡检触发后  
   将提醒转为观察类 action。

5. 质量评估后  
   低分输出不进入行动清单，或标记 `confidence=low`。

## 八、质量与测试

### 8.1 新增评估维度

在现有 LLM 评估之外，新增“决策质量”：

- 是否引用真实数据。
- 是否包含来源和更新时间。
- 是否说明适用条件。
- 是否符合用户画像。
- 是否给出反面风险。
- 是否可执行。
- 是否设置复盘条件。

### 8.2 回归测试重点

后端：

- 决策 JSON schema 校验。
- 无持仓时不得生成“你当前持有”类建议。
- 数据过期时建议必须降级为观察或补数据。
- 缺少期限/风险预算时必须走澄清。
- 用户拒绝建议后，画像证据应记录但不直接覆盖高置信画像。

前端：

- 今日行动卡片状态切换。
- 证据面板展示来源和更新时间。
- 画像证据可确认、修改、删除。
- 拒绝/暂缓/接受均有二次确认。

## 九、分阶段落地

### 第一阶段：决策档案 MVP

目标：先把 AI 建议沉淀下来。

范围：

- 新增 `decision_records` 和 `decision_actions`。
- 新增 `/api/decisions`。
- Chat 和 Dashboard 全景诊断完成后尝试提取决策。
- 每日看板展示“今日行动”。

验收：

- 用户能看到今天有哪些建议。
- 每条建议有来源、状态、证据摘要。
- 用户可接受、暂缓、拒绝。

### 第二阶段：画像证据产品化

目标：让“懂你”可见、可控。

范围：

- 新增 `profile_evidence`。
- 反馈学习和 KYC 更新写入证据。
- 新增画像证据面板。
- 用户可确认或删除系统学到的偏好。

验收：

- 用户能看到系统为什么认为自己偏好某类建议。
- 删除画像证据后，后续 prompt 不再注入该证据。

### 第三阶段：决策守门员

目标：减少不成熟、不适合用户的操作建议。

范围：

- 新增 `decision_guard.py`。
- 对买入、卖出、加仓、减仓、调仓类问题先检查约束。
- 缺少关键条件时主动澄清。
- 数据过期时提示补数据。

验收：

- “能买吗/加仓吗”类问题不再直接裸给建议。
- 输出会明确说明适用条件和缺失数据。

### 第四阶段：复盘闭环

目标：让系统知道建议后来怎样。

范围：

- 新增 `decision_reviews`。
- 对已接受或已执行建议设置复盘时间。
- 自动比较建议后的市场、估值、持仓、交易变化。
- 将复盘结果反馈给质量系统和用户画像。

验收：

- 用户能看到“上次建议后来如何”。
- 系统能识别经常失败的建议类型。

### 第五阶段：结构治理

目标：降低维护成本。

范围：

- 拆分 `PortfolioManagement.vue`。
- 拆分 `ValuationHistory.vue`。
- 拆分 `backend/routers/portfolio.py`。
- 拆分 `frontend/src/api/index.js` 为按域 API 文件。

验收：

- 页面行为不变。
- 核心大文件行数明显下降。
- 测试覆盖关键交互。

## 十、风险与缓解

| 风险 | 缓解 |
| --- | --- |
| AI 生成错误结构化决策 | schema 校验失败则不入库，只展示文本并记录 Bad Case |
| 用户误把建议当确定指令 | 决策卡片必须展示风险、适用条件、非投资承诺提示 |
| 行动清单过多造成噪音 | 只展示 must_do / executable / review，观察类默认折叠 |
| 画像学习过度 | 所有动态画像都写证据，低置信不直接覆盖主画像 |
| 工程范围变大 | 先做决策档案 MVP，不同时重构所有大文件 |

## 十一、待确认问题

1. “今日行动”是否放在每日看板首屏，还是新增独立“决策中枢”导航？
2. 用户是否希望系统生成“交易草稿”，还是只生成提醒和建议？
3. 复盘周期默认多久：7 天、30 天，还是按建议类型决定？
4. 画像证据是否需要区分“小鱼儿 / 花无缺”两个账户或用户？
5. AI 建议是否允许直接关联现有调仓配置，生成再平衡计划草稿？

## 十二、推荐下一步

下一步建议先写“决策档案 MVP 实施计划”，只覆盖：

- 数据表和 CRUD。
- Chat/Dashboard 两个入口的决策提取。
- 每日看板“今日行动”模块。
- 状态切换与反馈。

这个范围能最快验证核心价值：系统是否真的能从“回答问题”变成“陪你做连续理财决策”。
