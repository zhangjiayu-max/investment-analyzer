## 一、产品设计专家视角

> 评估方法论：用户旅程地图 → 信息架构 → 功能优先级 → 交互模式 → 信任建立

### 1.1 用户旅程：当前状态

```
用户来到系统
  │
  ├─ 首页仪表盘：估值、健康分、四笔钱、恐贪 ← 信息密度高但无优先级引导
  ├─ AI 对话：问什么答什么 ← 被动响应，缺主动引导
  ├─ 持仓管理：数据录入 + 分析入口 ← 功能堆砌，无主线
  ├─ 决策记录：历史决策列表 ← 归档性质，非行动导向
  └─ 各分析工具：费率/相关性/滚动收益 ← 散落在各处，用户不知道什么时候用
```

**问题诊断：用户来到系统后，不知道该先做什么、后做什么。功能很多但没有一条主线串起来。**

### 1.2 对标：成熟产品的信息架构

**有知有行：**
```
首页 = 市场温度计（一眼看大方向）
  → 有知（知识内容）← 不知道买什么时来学
  → 有行（交易记录）← 操作入口
  → 账本（四笔钱）  ← 资产全貌
四条主线清晰独立
```

**Betterment（美国最大智能投顾）：**
```
首页 = "你的目标是 87% 可达成"（一个数字概括一切）
  → 一键调仓（核心操作）
  → 目标进度（退休/买房/教育）
  → 税收优化（自动执行）
用户不需要学习，系统主动管理
```

**蚂蚁财富：**
```
首页 = 昨日收益 + 理财分
  → 帮你投（智能投顾）
  → 精选（推荐）
  → 持仓（管理）
  → 我的（设置）
标准 Tab 结构，每页有明确主操作
```

### 1.3 本系统的信息架构问题

**问题 1：首页信息过载，无主次**

当前首页仪表盘包含：估值卡片、健康分、市场热点、零钱配置、组合诊断、低估指数……用户一眼看去不知道该看哪个。一个设计原则：**首页应该只回答一个问题——"我现在该做什么？"**

**问题 2：AI 对话是孤岛**

AI 对话没有和系统其他部分联动。用户在对话中得到"建议加仓沪深 300"，但这条建议不会出现在持仓页、不会生成决策草案、不会触发定投优化。

**问题 3：分析工具散落，无引导**

费率分析、相关性矩阵、滚动收益、四笔钱——这些都是好工具，但用户不知道"什么时候该用哪个"。应该像体检一样：每月自动推荐"你该做费率检查了"。

### 1.4 推荐的信息架构

```
首页（每日必看，30秒）
  ├─ 市场温度计 + 恐贪指数（大方向）
  ├─ 我的健康分（一句话诊断）
  └─ 今天该做什么？（智能推荐 1-2 条操作）

持仓工作台（每周管理，5分钟）
  ├─ 持仓总览 + 盈亏
  ├─ 四笔钱配比
  ├─ 调仓建议（自动检测偏离）
  └─ 快捷操作：定投调整/费率检查/相关性复查

AI 顾问（有疑问时用）
  ├─ 自由提问
  ├─ 快捷问题（"现在适合加仓吗""我的持仓健康吗"）
  └─ 分析结果 → 一键保存为决策草案

决策中心（关键操作）
  ├─ 待确认的决策（AI 建议 + 分析结果自动沉淀）
  ├─ 到期复盘（3 个月前的决策，回顾判断是否正确）
  └─ 决策历史 + 准确率统计

工具箱（需要时用）
  费率分析 | 相关性矩阵 | 滚动收益 | 压力测试 | 策略回测
  → 每月自动推荐使用时机
```

### 1.5 交互设计问题

**问题：用户录入持仓太痛苦**

当前需要手动输入：基金代码、基金名称、份额、成本价、当前价。这是用户流失的第一大原因。

**对标解决方案：**
- 蚂蚁财富：授权后自动同步持仓（最优，但需要合规资质）
- 且慢：支持从天天基金/支付宝导出 CSV 导入（目前已支持 CSV）
- **可行方案：** 优化 CSV 导入体验 + 提供"快速录入"模式（只输入基金代码+金额，其他自动查）

**问题：AI 对话结果不落地**

当前对话结束就结束了，用户想回头找某条建议要翻聊天记录。

**解决方案：** 每条 AI 建议自动生成"决策卡片"，用户可：
- ✅ 接受 → 进入决策中心，生成交易草稿
- ❌ 拒绝 → 记录原因，反馈学习
- ⏰ 稍后决定 → 设置提醒，到期复盘

### 1.6 信任建立

金融产品最大的门槛是信任。用户不敢把真实持仓数据放进去。

**本系统现状：** 本地 SQLite + 自己的 API Key，数据完全自控。这是巨大的信任优势。

**应强化的信任信号：**
1. 首次使用时明确展示："你的数据存储在你自己的数据库中，API Key 只有你自己能访问"
2. AI 建议永远附带免责声明和置信度
3. 每条建议标注数据来源和新鲜度（"基于 6 月 24 日的估值数据"）
4. 决策回顾功能让用户看到"系统过去的建议准确率如何"

### 2.2 最严重的架构问题

**问题 1：无请求级追踪（Request Tracing）**

当前一个用户请求经过：`ChatView.vue → API → orchestrator.py → 3个专家 → 交叉审阅 → 仲裁 → 流式返回`。如果中间某步出错或超时，只能看日志猜测原因。没有请求 ID 贯穿全链路。

```python
# 应该在 API 入口生成 request_id，贯穿所有 LLM 调用
@router.post("/chat/stream")
async def chat_stream(data: dict):
    request_id = str(uuid.uuid4())
    # 注入到所有 downstream 调用
    orchestrator.orchestrate_stream(..., request_id=request_id)
```

**问题 2：SQLite 并发写入冲突**

当前多个专家并行执行时，各自调用 `_get_conn()` → 写入 → `conn.close()`。SQLite 默认是串行写入的，高并发下会有 `database is locked` 错误。

```python
# 解决方案：WAL 模式 + 超时重试
def _get_conn():
    conn = sqlite3.connect(DB_PATH, timeout=30)  # 30s 超时
    conn.execute("PRAGMA journal_mode=WAL")       # 并发读
    conn.execute("PRAGMA busy_timeout=5000")       # 5s 忙等待
    conn.row_factory = sqlite3.Row
    return conn
```

**问题 3：无熔断机制**

如果 akshare 挂了，系统会尝试每次调用都去请求，然后超时、报错。没有熔断器在连续失败 N 次后直接跳过。

```python
# 应该加熔断器
class CircuitBreaker:
    def __init__(self, failure_threshold=5, timeout=300):
        self.failures = 0
        self.last_failure = 0
        self.timeout = timeout
    
    def call(self, fn, *args, **kwargs):
        if self.failures >= 5 and time.time() - self.last_failure < self.timeout:
            raise CircuitOpenError("熔断器开启，跳过调用")
        try:
            result = fn(*args, **kwargs)
            self.failures = 0
            return result
        except Exception:
            self.failures += 1
            self.last_failure = time.time()
            raise
```

### 2.3 数据完整性风险

**风险 1：交易记录和持仓不同步**

`create_transaction` 和 `_recalculate_holding` 是两步操作，中间如果失败，交易记录写入了但持仓未更新。

```python
# 应该在事务中执行
def create_transaction(holding_id, ...):
    conn = _get_conn()
    try:
        conn.execute("BEGIN")
        conn.execute("INSERT INTO portfolio_transactions ...")
        _recalculate_holding(conn, holding_id)
        conn.execute("COMMIT")
    except Exception:
        conn.execute("ROLLBACK")
        raise
    finally:
        conn.close()
```

**风险 2：估值数据重复**

`index_valuations` 表有 `UNIQUE(index_code, snapshot_date, metric_type)` 约束，但 `save_valuation` 函数没有使用 `INSERT OR REPLACE`，可能导致 IntegrityError。

## 四、一句话总结

**产品维度：** 功能丰富度已超过大多数 C 端理财产品，但缺少一条串联所有功能的"用户主线"。信息架构需要从"功能展示型"转向"任务引导型"——用户来系统是要完成一个任务（"我该买什么""我的持仓健康吗"），而不是浏览功能列表。

**技术维度：** 核心架构思路清晰（Multi-Agent + 评测闭环），但缺少生产级基础设施（熔断、追踪、事务、备份）。作为个人工具已足够好，但如果未来要服务更多用户，需要补齐这层。

**一句话：** 这个系统最有价值的部分——让用户用数据替代情绪做投资决策——已经做到了 80%。剩下的 20% 是让用户更容易、更信任地使用这些数据。
