## 四、方案对比（3 个候选）

### 方案 1：分散实施（不推荐）
逐个实施已有 5 份设计稿 + 新写执行稳定性稿。
- **优点**：每份稿独立，改动隔离
- **缺点**：6 份稿并行实施，优先级冲突，A 类问题（用户最痛）可能排在后面，Token 浪费持续
- **风险**：实施周期长，用户痛点持续暴露

### 方案 2：统一蓝图 + 分批实施（推荐 ✅）
本设计稿作为**唯一实施蓝图**，把 19 个问题按优先级分成 P0/P1/P2 三批，引用已有稿的实施细节但不重复。
- **优点**：优先级清晰，P0 聚焦用户最痛的 8 个问题，1-2 周可见显著效果
- **缺点**：需要跨 5 份稿整合，工作量集中在规划阶段
- **风险**：低，每批独立可回滚

### 方案 3：全面重构（不推荐）
放弃已有稿，从 0 重写对话系统。
- **优点**：架构最干净
- **缺点**：丢弃已有 5 份设计成果，周期长，风险高
- **风险**：高，可能引入新 bug

**P0-1.1：取消语义重设计（修复 A1, A2）**

**问题根因**：
- [routers/conversations.py:167-196](file:///Users/xiaoyuer/projects/investment-analyzer/backend/routers/conversations.py) 的 cancel 接口只设置 `cancel_event`，但：
  - 前端刷新页面后重新加载对话，`_running_agents` 字典丢失，cancel_event 失效
  - 失败状态（`execution_status=failed`）被前端误判为"进行中"触发自动 resume
  - `_check_cancel()` 只在专家调度前检查，LLM 调用本身不可中断

**修复方案**：

```python
# 1. db/conversations.py 新增字段（ALTER TABLE）
# messages 表新增 execution_status 字段已存在，但需要新增 cancel_requested 字段
# conversations 表新增 last_message_status 字段，记录最终状态

# 2. 取消语义改为"持久化标记"
# routers/conversations.py cancel 接口增强：
@app.post("/api/conversations/{conv_id}/cancel")
async def cancel_conversation(conv_id: int):
    # 2.1 持久化取消标记到 DB（不依赖内存 cancel_event）
    mark_conversation_cancelled(conv_id)  # 写入 conversations.cancel_requested = 1
    # 2.2 设置内存 cancel_event（仍保留，用于正在运行的检查点）
    if conv_id in _running_agents:
        _running_agents[conv_id]["cancel_event"].set()
    # 2.3 立即把 streaming 消息状态改为 cancelled（已有）
    cancel_running_agents(conv_id)  # db/agents.py 已有
    # 2.4 新增：取消正在执行的 LLM 调用（通过 asyncio.Task.cancel）
    if conv_id in _running_agents:
        for task in _running_agents[conv_id].get("llm_tasks", []):
            task.cancel()  # 强制中断 LLM 调用
    return {"ok": True}
```

```python
# 3. 前端刷新时检查持久化取消标记
# frontend/src/views/Home.vue 或 ChatView.vue 的 onMounted 恢复逻辑：
async function resumeConversation(convId) {
    const conv = await api.getConversation(convId)
    // 3.1 如果 cancel_requested=true，不自动恢复
    if (conv.cancel_requested) {
        showCancelledToast()
        return  // 不调用 resume SSE

    // 3.2 如果 last_message_status=failed，提示用户手动重试，不自动恢复
    if (conv.last_message_status === 'failed') {
        showFailedRetryButton()
        return
    }
    // 3.3 只有 status=streaming 且无 cancel_requested 才恢复
    await resumeStreaming(convId)
}
```

**P0-1.2：失败状态显式化（修复 A3）**

**问题根因**：对话 #85 失败后，assistant 消息 metadata=`{"execution_status":"failed"}`，但前端没有处理 failed 状态，继续显示"⏳ 分析进行中"。

**修复方案**：

```python
# frontend/src/components/ChatView.vue 消息渲染逻辑：
function getMessageDisplayStatus(message) {
    const meta = message.metadata ? JSON.parse(message.metadata) : {}
    if (meta.execution_status === 'failed') {
        return { text: '❌ 分析失败，点击重试', action: 'retry', class: 'failed' }
    }
    if (meta.execution_status === 'cancelled') {
        return { text: '🚫 已取消', action: null, class: 'cancelled' }
    }
    if (meta.execution_status === 'streaming') {
        return { text: '⏳ 分析进行中...', action: 'cancel', class: 'streaming' }
    }
    return null  // 正常完成
}
```

# 1. 验证修复是否真的生效：grep -n "_schedule_auto_evaluation" orchestrator.py
#    确认 5 个 return 路径前都有调用

# 2. 评估逻辑本身可能有问题：_schedule_auto_evaluation 是异步触发，
#    但如果 conversation_evaluations 表写入失败（如 LLM 评估失败），
#    不会有任何记录。需要加 fallback：
def _schedule_auto_evaluation(message_id: int, ...):

    eval_id = create_evaluation_record(message_id, status='pending')
    # 2.2 异步触发 LLM 评估
    asyncio.create_task(_run_auto_evaluation(eval_id, message_id))

async def _run_auto_evaluation(eval_id: int, message_id: int):
    try:
        score = await llm_evaluate_message(message_id)
        update_evaluation(eval_id, status='completed', score=score)
    except Exception as e:
        # 2.3 失败也留记录，标注 failed，便于排查
        update_evaluation(eval_id, status='failed', error=str(e))
        logger.warning(f"[trace:{trace_id}] 自动评估失败: {e}")
```

**问题根因**：[2026-07-04 报告](../reports/2026-07-04-对话质量分析报告.md) 修复 1 改了 `should_run_cross_review()` 逻辑，但近 14 天 cross_review 仍然 0 次触发。说明：

# orchestrator.py should_run_cross_review() 最终逻辑：
def should_run_cross_review(complexity, needs_arbitration, predicted, conflicts):
    # 1. 实际冲突 > 预判 > complexity（已是 2026-07-04 修复逻辑）
    if conflicts and conflicts.get('detected') and conflicts.get('severity') in ('medium', 'high'):
        return True  # 实际检测到 medium+ 冲突，强制执行
    # 2. complex 对话 + 预判 True，即使无冲突也执行（用于发现隐性分歧）
    if complexity == 'complex' and needs_arbitration and predicted:
        return True
    return False

# 新增：complex 对话强制做冲突检测（即使预判 False）
def _stream_handle_no_tool_calls(...):
    if complexity == 'complex':
        conflicts = detect_conflicts_smart(specialist_results)  # 强制检测
        if should_run_cross_review(complexity, needs_arbitration, predicted, conflicts):
            run_cross_review(...)
```
