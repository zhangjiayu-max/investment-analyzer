# RAG 系统优化方案

## 一、背景与目标

### 1.1 现状评估

当前 RAG 系统评分：**5.5/10**

| 维度 | 评分 | 主要问题 |
|------|------|----------|
| 架构设计 | 7/10 | FTS5 + ChromaDB + RRF 架构合理，但实现有缺陷 |
| 数据覆盖 | 4/10 | articles/analysis_records 完全未索引 |
| 数据质量 | 6/10 | 已索引数据质量尚可，但估值数据时间跨度短 |
| 检索效果 | 5/10 | 用户反馈 60% 认为无用 |
| 反馈闭环 | 4/10 | 反馈数据仅 5 条，画像未生效 |

### 1.2 核心问题

1. **知识库覆盖严重不全**：369 篇 articles + 365 条 analysis_records 完全未索引
2. **检索精度不足**：FTS 分词效果存疑，向量搜索类型过滤不全
3. **反馈数据匮乏**：仅 5 条反馈，无法驱动优化
4. **时效性管理粗糙**：简单的月份过滤，无法区分内容重要性

### 1.3 优化目标

| 目标 | 当前 | 目标值 | 衡量指标 |
|------|------|--------|----------|
| 知识库覆盖率 | ~40% | 95% | 已索引数据/总数据 |
| 检索准确率 | ~40% | 75% | 用户反馈 helpful 比例 |
| 平均响应时间 | 未知 | <3秒 | API 响应时间 |
| 反馈数据量 | 5 条 | 100+ 条/月 | 月度反馈数 |

## 二、核心问题分析

### 2.1 是否需要自己训练数据？

**结论：不需要训练 embedding 模型，但需要优化知识库数据**

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 训练自定义 embedding | 领域适配性好 | 需要大量标注数据、计算资源、持续维护 | ⭐⭐ |
| 使用现有模型 + 优化知识库 | 成本低、见效快、易维护 | 依赖通用模型能力 | ⭐⭐⭐⭐⭐ |
| 微调 embedding 模型 | 平衡方案 | 需要一定标注数据 | ⭐⭐⭐ |

**推荐方案**：使用现有 `bge-small-zh-v1.5` 模型 + 优化知识库结构和内容

**理由**：
1. 投资领域的语义理解不需要特殊模型，通用中文 embedding 已经足够
2. 问题在于**知识库数据质量**，而非模型能力
3. 优化知识库的 ROI 远高于训练模型

### 2.2 知识库如何重构？

**核心思路**：分层知识库 + 结构化存储 + 智能索引

```
┌─────────────────────────────────────────────────────────┐
│                    知识库分层架构                         │
├─────────────────────────────────────────────────────────┤
│  L1: 实时数据层                                          │
│  ├── 估值数据 (index_valuations)                         │
│  ├── 行情数据 (akshare)                                  │
│  └── 持仓数据 (portfolio)                                │
├─────────────────────────────────────────────────────────┤
│  L2: 分析知识层                                          │
│  ├── 分析记录 (analysis_records)                         │
│  ├── 投资建议 (recommendations)                          │
│  └── 评测结果 (eval_runs)                                │
├─────────────────────────────────────────────────────────┤
│  L3: 文章知识层                                          │
│  ├── 作者文章 (author_articles)                          │
│  ├── 公众号文章 (articles)                               │
│  └── 个人文档 (linked_articles)                          │
├─────────────────────────────────────────────────────────┤
│  L4: 方法论层                                            │
│  ├── 技能知识 (skill_documents)                          │
│  ├── 投资策略 (investment_strategy)                      │
│  └── 债券知识 (bond_knowledge)                           │
└─────────────────────────────────────────────────────────┘
```

### 2.3 如何让 RAG 更强更准？

**三个维度优化**：

1. **数据维度**：补齐缺失索引，提升数据质量
2. **检索维度**：优化分词策略，修复多类型过滤
3. **反馈维度**：建立闭环机制，持续优化

```python
# 使用 jieba 作为 FTS5 的预处理，优化查询构建
def _build_fts_query(query: str) -> str:
    """将用户问题转为 FTS5 查询。"""
    tokens = _tokenize(query).split()
    tokens = [t for t in tokens if t and t not in _STOPWORDS]

    # 改进：对中文多字词使用 NEAR 查询而非短语匹配
    multi_char = [t for t in tokens if len(t) >= 2]
    single_char = [t for t in tokens if len(t) == 1]

    if multi_char:
        # 使用 AND 连接，允许词之间有间隔
        core = " AND ".join(multi_char)
        if single_char:
            return f"({core}) OR ({' OR '.join(single_char)})"
        return core
    return " OR ".join(single_char)
```

### 3.4 Phase 4: 高级优化（1-2周）

**目标**：将检索准确率从 65% 提升到 80%

#### 3.4.1 引入 Reranker

```python
# 使用 cross-encoder 进行重排序
from sentence_transformers import CrossEncoder

reranker = CrossEncoder("BAAI/bge-reranker-base")

def rerank_results(query: str, results: list[dict], top_k: int = 5):
    """对检索结果进行重排序。"""
    pairs = [(query, r["body"][:512]) for r in results]
    scores = reranker.predict(pairs)

    for i, score in enumerate(scores):
        results[i]["rerank_score"] = float(score)

    results.sort(key=lambda x: x["rerank_score"], reverse=True)
    return results[:top_k]
```

#### 3.4.2 实现 Query Rewrite

```python
# 使用 LLM 将口语化问题转换为检索查询
```

## 七、总结

### 7.1 核心结论

1. **不需要训练模型**：优化知识库比训练模型的 ROI 更高
2. **分层知识库架构**：实时数据 → 分析知识 → 文章知识 → 方法论
3. **反馈闭环是关键**：没有反馈数据，优化就是盲人摸象

### 7.2 推荐路径

```
Phase 1 (补齐索引) → Phase 2 (优化检索) → Phase 3 (反馈闭环) → Phase 4 (高级优化)
     ↓                    ↓                    ↓                    ↓
  覆盖率 90%          准确率 65%           反馈 100+条          准确率 80%
```

### 7.3 成功标准

- **短期**（1周）：知识库覆盖率 > 90%，检索准确率 > 60%
- **中期**（2周）：用户满意度 > 50%，反馈数据 > 100 条
- **长期**（1月）：检索准确率 > 80%，用户满意度 > 70%
