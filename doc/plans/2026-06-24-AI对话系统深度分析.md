# AI 对话系统深度分析 — 架构/执行/优化全链路

> 日期：2026-06-24  
> 范围：对话上下文构建 → 需求澄清 → 复杂度分级 → 专家路由 → 并行/串行执行 → 交叉审阅 → 仲裁判决  
> 原则：以"能否给用户最准确的投资建议"为唯一评判标准

---

## 一、完整执行链路

```
用户输入
  │
  ├─ 0. Token 预算检查（超限 → 直接拒绝）
  ├─ 0.3 链接检测 → 异步抓取文章 → 注入上下文
  ├─ 0.5 恢复模式检查（对话中断恢复：已完成专家跳过）
  │
  ├─ 1. 需求澄清
  │   ├─ @mention 指定专家 → 跳过路由
  │   ├─ 规则预判（chat/simple/complex）→ 零 LLM
  │   ├─ 边界情况（medium）→ LLM 确认
  │   └─ 回退：关键词匹配
  │
  ├─ 1.5 场景化 RAG（根据命中专家检索相关书籍）
  │
  ├─ 2. 上下文组装
  │   ├─ System prompt (Orchestrator 角色)
  │   ├─ RAG 知识库（token 感知截断）
  │   ├─ 用户偏好画像（feedback_learner 积累）
  │   ├─ KYC 理财画像
  │   ├─ 跨对话用户记忆
  │   ├─ 持仓上下文 + 估值上下文
  │   ├─ 债市数据 + 热点新闻
  │   └─ 语义压缩历史消息
  │
  ├─ 3. 多轮执行循环（MAX_TURNS=6, 保守模式=3）
  │   ├─ LLM 决策：需要调用哪些专家？
  │   ├─ 并行执行所有选中的专家 Agent
  │   ├─ 收集结果 → 追加到 LLM 消息
  │   └─ 下一轮：LLM 判断是否需要更多专家
  │
  ├─ 4. Phase B：交叉审阅（仅 complex + >=2专家 + 存在分歧）
  │   └─ 每个专家审阅其他专家的结论，互相验证
  │
  ├─ 5. Phase C：仲裁判决（高级模型最终裁决）
  │   └─ DeepSeek R1/V4-Pro 做最终判断
  │
  └─ 6. 流式输出
      ├─ reasoning_chunk（思考过程）
      └─ answer_chunk（最终答案）
```

## 二、各环节深度分析

### 2.1 需求澄清：规则+LLM 混合策略 ✅

**设计思路：**
- chat/simple/complex → 规则预判（零 LLM，零延迟）
- medium → LLM 确认（只有边界情况才调用 LLM）
- 节省 2-3 秒首响时间

**规则引擎质量分析：**

```
检测层数：
  第1层：闲聊检测（长度<10 + 闲聊关键词，无投资内容）
  第2层：极短消息（长度<=5，无疑问词，无投资词）
  第3层：Simple 检测（长度<30 + 简单关键词，排除复杂意图）
  第4层：Complex 检测（长度>100 + >=2复杂关键词，或多维组合）
  第5层：未命中 → 回退 medium（需 LLM 确认）
```

**⚠️ 问题 1：规则预判的边界模糊**

```python
# 这条消息会被判为 simple 还是 medium？
"沪深300估值多少，现在可以买吗"
# 前半段是 simple（单一查询），后半段是 buy_decision
# 但 length < 30 + has_simple → 进入 simple 分支
# 然后检测到 complex_markers（"可以买"）→ 可能判为 medium
# 实际走了 force_simple_prefixes 不匹配 → has_question → medium
# 结果是 medium → 调用 LLM → 额外 2-3 秒
```

**修复建议：** 增加一个"混合意图"检测——如果查询同时包含 simple_keywords 和 buy/sell 关键词，根据持仓情况直接判定复杂度。

---

### 2.2 上下文组装：七层注入 ✅

**当前注入层次：**

| 顺序 | 内容 | 来源 | Token 预算管理 |
|------|------|------|---------------|
| 1 | System prompt | 硬编码 + 数据库动态 | 固定 500 tokens |
| 2 | RAG 知识库 | ChromaDB + FTS5 | 动态截断 |
| 3 | 用户偏好画像 | feedback_learner | 固定 ~200 tokens |
| 4 | KYC 理财画像 | kyc.py | 固定 ~300 tokens |
| 5 | 跨对话记忆 | memory.py | 语义压缩 |
| 6 | 持仓+估值 | portfolio_context.py | token 感知截断 |
| 7 | 历史消息 | 对话表 | 语义压缩 + 近期原文 |

**⚠️ 问题 2：注入内容可能有冲突**

场景：用户偏好说"喜欢详细数据"，但 KYC 画像是"新手"。
→ feedback_learner 注入了"详细数据支撑"，KYC 注入了"新手偏好简洁"
→ 模型收到矛盾指令

**修复建议：** 优先级分层——KYC > 用户偏好 > RAG。KYC 画像作为基础约束层，用户偏好作为微调层。

### 2.3 并行执行模型 ✅

**当前设计：**

```python
if len(tool_tasks) == 1:
    # 单个专家：直接在当前线程执行
    result_str = _execute_specialist(...)
else:
    # 多个专家：ThreadPoolExecutor 并行
    with ThreadPoolExecutor(max_workers=len(tool_tasks)) as executor:
        for idx, tc in enumerate(tool_tasks):
            future = executor.submit(_execute_specialist, ...)
```

**并行策略分析：**

```
假设调用 3 个专家，每个 60s：
  串行：60 + 60 + 60 = 180s
  并行：max(60, 60, 60) = 60s
  加速比：3x ✅

但：
  ThreadPoolExecutor 中的每个线程会独立调用 LLM API
  → 3 个并行的 HTTP 连接
  → 如果 LLM API 有并发限制，可能被限流
  → 实际加速比 < 3x
```

**⚠️ 问题 3：没有并发限制**

当前 `max_workers=len(tool_tasks)`，如果 Orchestrator 决定调用 5 个专家，同时发起 5 个 LLM 请求。DeepSeek API 可能有并发限制。

**修复建议：** 限制最大并发数为 3（`max_workers=min(len(tool_tasks), 3)`），超出部分排队。

---

### 2.4 交叉审阅（Phase B）✅

**触发条件：**
```python
cross_review_enabled == "true"
AND not force_skip_cross_review
AND len(specialist_results) >= 2
AND (trigger == "always" OR (trigger == "disagreement" AND 存在分歧))
```

**执行方式：** 串行。每个专家依次审阅其他专家的结论。

**⚠️ 问题 4：交叉审阅是串行的**

```python
for sr in original_specialists:  # 串行循环
    cr_result = run_specialist_with_context(
        sr["agent_key"], refined_query, peer_analyses, ...
    )
```

3 个专家交叉审阅 = 3 次额外的 LLM 调用，每次 ~60s，总增加 ~180s。如果改成并行，只需 ~60s。

**修复建议：** 交叉审阅改为并行执行（每个专家独立审阅，无依赖关系）。

---

### 2.5 仲裁判决（Phase C）✅

**触发条件：**
```python
ARBITRATION_API_KEY 已配置
AND complexity >= "medium"
AND >=2 个专家参与
AND arbitration_enabled == "true"
```

**执行方式：** 单次调用高级推理模型（DeepSeek R1/V4-Pro）。

**⚠️ 问题 5：仲裁模型和专家模型可能是同一个**

如果 `ARBITRATION_MODEL` 和 `MODEL` 是同一个模型（如都用 DeepSeek V4-Pro），仲裁就是同一模型自己审自己——降低了仲裁的独立性。

**修复建议：** 仲裁强制使用不同模型（如专家用 V4-Flash，仲裁用 V4-Pro，或专家用 DeepSeek，仲裁用 MIMO）。

---

### 2.6 Token 预算管理 ✅

**三层管控：**

```
Layer 1: 每日总量限制（DAILY_TOKEN_LIMIT=500000）
  - < 80%: 正常模式
  - 80-100%: 保守模式（MAX_TURNS=3, 跳过交叉审阅）
  - > 100%: 拒绝服务

Layer 2: 上下文 Token 预算（按复杂度动态分配）
  - simple: 历史3条, RAG 800字, 1专家
  - medium: 历史5条, RAG 1500字, 2专家
  - complex: 历史10条, RAG 2500字, 5专家

Layer 3: 语义压缩
  - 超限历史 → LLM 摘要
  - 超限 RAG → 截断到最近的完整段落
```

**⚠️ 问题 6：Token 记录有时序问题**

```python
# 在函数末尾计算本次 token 用量：
row = conn.execute(
    "SELECT SUM(total_tokens) FROM token_usage "
    "WHERE created_at >= datetime('now', '-5 minutes')"
).fetchone()
```

如果 5 分钟内有其他对话也在调用 LLM，会把其他对话的 token 算进来。

**修复建议：** 每次 `_call_llm` 返回时记录 `call_id`，用 call_id 精确关联。

## 三、关键优化空间

### 🔴 P0：减少 LLM 调用次数

**当前完整链路（complex 场景）调用次数：**

```
复杂度澄清:        0 (规则命中) 或 1 (LLM 确认)
场景化 RAG:        1 (build_rag_context)
Orchestrator 决策: N 轮 × 1 (每轮判断调用哪些专家)
专家执行:          M 个专家 × 1 (每个专家内部有工具调用循环)
交叉审阅:          M 个专家 × 1 (串行)
仲裁:              1
最终综合:          1 (流式)
────────────────────────────────
最坏情况: 1 + 1 + 6 + 5 + 5 + 1 + 1 = 20 次 LLM 调用
```

**优化策略：**

1. **需求澄清缓存已做** ✅ — 相同查询 5 分钟内复用
2. **规则预判已做** ✅ — chat/simple/complex 跳过 LLM
3. **可优化：专家结果缓存** — 同一专家 + 同一上下文 → 5 分钟内复用
4. **可优化：跳过交叉审阅** — 方向一致时已跳过，但可增加"低冲突"阈值
5. **可优化：条件跳过仲裁** — 专家高度一致 + 置信度 > 0.9 → 跳过

---

### 🟡 P1：并行化交叉审阅

```python
# 当前：串行
for sr in specialists:
    run_specialist_with_context(sr, ...)  # 一次一个

# 优化：并行
with ThreadPoolExecutor(max_workers=len(specialists)) as executor:
    futures = {executor.submit(run_specialist_with_context, sr, ...): sr
               for sr in specialists}
    for future in as_completed(futures):
        results.append(future.result())
```

**预估收益：** complex 场景减少 60-120s 延迟。

### 🟡 P2：专家结果缓存

```python
# 同一专家 + 相同查询 + 5分钟内 → 直接复用
_expert_cache: dict[str, tuple[dict, float]] = {}

def _execute_specialist_cached(tool_name, query, prebuilt_context=""):
    cache_key = f"{tool_name}:{hash(query)}:{hash(prebuilt_context)}"
    if cache_key in _expert_cache:
        result, timestamp = _expert_cache[cache_key]
        if time.time() - timestamp < 300:
            return json.dumps(result)
    result_str = _execute_specialist(tool_name, query, prebuilt_context=prebuilt_context)
    _expert_cache[cache_key] = (json.loads(result_str), time.time())
    return result_str
```

ACTIONABILITY_CONSTRAINT = """

### 🟢 P4：上下文冲突检测

```python
def detect_context_conflicts(sections):
    """检测上下文中的矛盾指令。"""
    conflicts = []
    if "详细数据支撑" in sections.get("preferences", ""):
        if "新手" in sections.get("kyc", ""):
            conflicts.append("偏好详细 vs KYC新手 → 优先 KYC，简化输出")
    if "激进" in sections.get("kyc", ""):
        if "保守" in sections.get("strategy", ""):
            conflicts.append("激进画像 vs 保守策略 → 标记为需确认")
    return conflicts
```

## 五、优化优先级总结

| 优先级 | 优化项 | 收益 | 工作量 |
|--------|--------|------|--------|
| P0 | 减少 LLM 调用次数（专家缓存+跳过仲裁） | 节省 30-50% Token | 3h |
| P1 | 并行化交叉审阅 | 减少 60-120s 延迟 | 2h |
| P1 | 限制并发数(max_workers=3) | 避免 API 限流 | 0.5h |
| P2 | 专家结果缓存（5min TTL） | 重复查询即时响应 | 1h |
| P2 | 仲裁强制不同模型 | 增加独立性 | 0.5h |
| P3 | 分析模板增加可执行性约束 | 提升建议质量 | 0.5h |
| P3 | 上下文冲突检测 | 避免矛盾指令 | 1h |
| P3 | Token 记录精确关联 | 预算准确 | 1h |
