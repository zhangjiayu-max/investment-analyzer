# AI 对话系统深度分析 — 架构/执行/优化全链路

> 日期：2026-06-24  
> 范围：对话上下文构建 → 需求澄清 → 复杂度分级 → 专家路由 → 并行/串行执行 → 交叉审阅 → 仲裁判决  
> 原则：以"能否给用户最准确的投资建议"为唯一评判标准

---

## 一、完整执行链路

```
用户输入
  │
  ├─ 0. Token 预算检查（超限 → 直接拒绝）
  ├─ 0.3 链接检测 → 异步抓取文章 → 注入上下文
  ├─ 0.5 恢复模式检查（对话中断恢复：已完成专家跳过）
  │
  ├─ 1. 需求澄清
  │   ├─ @mention 指定专家 → 跳过路由
  │   ├─ 规则预判（chat/simple/complex）→ 零 LLM
  │   ├─ 边界情况（medium）→ LLM 确认
  │   └─ 回退：关键词匹配
  │
  ├─ 1.5 场景化 RAG（根据命中专家检索相关书籍）
  │
  ├─ 2. 上下文组装
  │   ├─ System prompt (Orchestrator 角色)
  │   ├─ RAG 知识库（token 感知截断）
  │   ├─ 用户偏好画像（feedback_learner 积累）
  │   ├─ KYC 理财画像
  │   ├─ 跨对话用户记忆
  │   ├─ 持仓上下文 + 估值上下文
  │   ├─ 债市数据 + 热点新闻
  │   └─ 语义压缩历史消息
  │
  ├─ 3. 多轮执行循环（MAX_TURNS=6, 保守模式=3）
  │   ├─ LLM 决策：需要调用哪些专家？
  │   ├─ 并行执行所有选中的专家 Agent
  │   ├─ 收集结果 → 追加到 LLM 消息
  │   └─ 下一轮：LLM 判断是否需要更多专家
  │
  ├─ 4. Phase B：交叉审阅（仅 complex + >=2专家 + 存在分歧）
  │   └─ 每个专家审阅其他专家的结论，互相验证
  │
  ├─ 5. Phase C：仲裁判决（高级模型最终裁决）
  │   └─ DeepSeek R1/V4-Pro 做最终判断
  │
  └─ 6. 流式输出
      ├─ reasoning_chunk（思考过程）
      └─ answer_chunk（最终答案）
```

## 二、各环节深度分析

### 2.1 需求澄清：规则+LLM 混合策略 ✅

**设计思路：**
- chat/simple/complex → 规则预判（零 LLM，零延迟）
- medium → LLM 确认（只有边界情况才调用 LLM）
- 节省 2-3 秒首响时间

**规则引擎质量分析：**

```
检测层数：
  第1层：闲聊检测（长度<10 + 闲聊关键词，无投资内容）
  第2层：极短消息（长度<=5，无疑问词，无投资词）
  第3层：Simple 检测（长度<30 + 简单关键词，排除复杂意图）
  第4层：Complex 检测（长度>100 + >=2复杂关键词，或多维组合）
  第5层：未命中 → 回退 medium（需 LLM 确认）
```

**⚠️ 问题 1：规则预判的边界模糊**

```python
# 这条消息会被判为 simple 还是 medium？
"沪深300估值多少，现在可以买吗"
# 前半段是 simple（单一查询），后半段是 buy_decision
# 但 length < 30 + has_simple → 进入 simple 分支
# 然后检测到 complex_markers（"可以买"）→ 可能判为 medium
# 实际走了 force_simple_prefixes 不匹配 → has_question → medium
# 结果是 medium → 调用 LLM → 额外 2-3 秒
```

**修复建议：** 增加一个"混合意图"检测——如果查询同时包含 simple_keywords 和 buy/sell 关键词，根据持仓情况直接判定复杂度。

### 2.2 上下文组装：七层注入 ✅

**当前注入层次：**

| 顺序 | 内容 | 来源 | Token 预算管理 |
|------|------|------|---------------|
| 1 | System prompt | 硬编码 + 数据库动态 | 固定 500 tokens |
| 2 | RAG 知识库 | ChromaDB + FTS5 | 动态截断 |
| 3 | 用户偏好画像 | feedback_learner | 固定 ~200 tokens |
| 4 | KYC 理财画像 | kyc.py | 固定 ~300 tokens |
| 5 | 跨对话记忆 | memory.py | 语义压缩 |
| 6 | 持仓+估值 | portfolio_context.py | token 感知截断 |
| 7 | 历史消息 | 对话表 | 语义压缩 + 近期原文 |

**⚠️ 问题 2：注入内容可能有冲突**

场景：用户偏好说"喜欢详细数据"，但 KYC 画像是"新手"。
→ feedback_learner 注入了"详细数据支撑"，KYC 注入了"新手偏好简洁"
→ 模型收到矛盾指令

**修复建议：** 优先级分层——KYC > 用户偏好 > RAG。KYC 画像作为基础约束层，用户偏好作为微调层。

### 2.3 并行执行模型 ✅

**当前设计：**

```python
if len(tool_tasks) == 1:
    # 单个专家：直接在当前线程执行
    result_str = _execute_specialist(...)
else:
    # 多个专家：ThreadPoolExecutor 并行
    with ThreadPoolExecutor(max_workers=len(tool_tasks)) as executor:
        for idx, tc in enumerate(tool_tasks):
            future = executor.submit(_execute_specialist, ...)
```

**并行策略分析：**

```
假设调用 3 个专家，每个 60s：
  串行：60 + 60 + 60 = 180s
  并行：max(60, 60, 60) = 60s
  加速比：3x ✅

但：
  ThreadPoolExecutor 中的每个线程会独立调用 LLM API
  → 3 个并行的 HTTP 连接
  → 如果 LLM API 有并发限制，可能被限流
  → 实际加速比 < 3x
```

**⚠️ 问题 3：没有并发限制**

当前 `max_workers=len(tool_tasks)`，如果 Orchestrator 决定调用 5 个专家，同时发起 5 个 LLM 请求。DeepSeek API 可能有并发限制。

**修复建议：** 限制最大并发数为 3（`max_workers=min(len(tool_tasks), 3)`），超出部分排队。

### 2.4 交叉审阅（Phase B）✅

**触发条件：**
```python
cross_review_enabled == "true"
AND not force_skip_cross_review
AND len(specialist_results) >= 2
AND (trigger == "always" OR (trigger == "disagreement" AND 存在分歧))
```

**执行方式：** 串行。每个专家依次审阅其他专家的结论。

**⚠️ 问题 4：交叉审阅是串行的**

```python
for sr in original_specialists:  # 串行循环
    cr_result = run_specialist_with_context(
        sr["agent_key"], refined_query, peer_analyses, ...
    )
```

3 个专家交叉审阅 = 3 次额外的 LLM 调用，每次 ~60s，总增加 ~180s。如果改成并行，只需 ~60s。

**修复建议：** 交叉审阅改为并行执行（每个专家独立审阅，无依赖关系）。

### 2.5 仲裁判决（Phase C）✅

**触发条件：**
```python
ARBITRATION_API_KEY 已配置
AND complexity >= "medium"
AND >=2 个专家参与
AND arbitration_enabled == "true"
```

**执行方式：** 单次调用高级推理模型（DeepSeek R1/V4-Pro）。

**⚠️ 问题 5：仲裁模型和专家模型可能是同一个**

如果 `ARBITRATION_MODEL` 和 `MODEL` 是同一个模型（如都用 DeepSeek V4-Pro），仲裁就是同一模型自己审自己——降低了仲裁的独立性。

**修复建议：** 仲裁强制使用不同模型（如专家用 V4-Flash，仲裁用 V4-Pro，或专家用 DeepSeek，仲裁用 MIMO）。

### 2.6 Token 预算管理 ✅

**三层管控：**

```
Layer 1: 每日总量限制（DAILY_TOKEN_LIMIT=500000）
  - < 80%: 正常模式
  - 80-100%: 保守模式（MAX_TURNS=3, 跳过交叉审阅）
  - > 100%: 拒绝服务

Layer 2: 上下文 Token 预算（按复杂度动态分配）
  - simple: 历史3条, RAG 800字, 1专家
  - medium: 历史5条, RAG 1500字, 2专家
  - complex: 历史10条, RAG 2500字, 5专家

Layer 3: 语义压缩
  - 超限历史 → LLM 摘要
  - 超限 RAG → 截断到最近的完整段落
```

**⚠️ 问题 6：Token 记录有时序问题**

```python
# 在函数末尾计算本次 token 用量：
row = conn.execute(
    "SELECT SUM(total_tokens) FROM token_usage "
    "WHERE created_at >= datetime('now', '-5 minutes')"
).fetchone()
```

如果 5 分钟内有其他对话也在调用 LLM，会把其他对话的 token 算进来。

**修复建议：** 每次 `_call_llm` 返回时记录 `call_id`，用 call_id 精确关联。

## 三、质量提升优化方案

> **设计原则：每次 LLM 调用必须有明确价值。不追求减少调用次数，追求每次调用产出更高质量。**

20 次调用的链路不是问题——只要每次调用都在提升分析质量。以下是确保"调用一次，提升一次"的优化。

---

### P0：并行化交叉审阅 — 同样时间，翻倍审阅深度

**当前：** 串行。M 个专家交叉审阅 = M 次 LLM 调用，每次 ~60s，总增加 M×60s。

**优化：** 并行。M 个专家同时审阅，总耗时 = max(60s) ≈ 60s。

```python
# orchestrator.py 中 _run_cross_review 改为：
import concurrent.futures

def _run_cross_review_parallel(specialist_results, refined_query, cancel_event, start_time):
    """并行交叉审阅：每个专家独立审阅其他专家的结论。"""
    cross_results = []
    
    def review_single(sr):
        _check_cancel(cancel_event)
        _check_timeout(start_time)
        peer_analyses = [
            r["analysis"] for r in specialist_results
            if r["agent_key"] != sr["agent_key"]
        ]
        return run_specialist_with_context(
            sr["agent_key"], refined_query,
            peer_analyses="\n---\n".join(peer_analyses),
            is_cross_review=True,
        )
    
    with concurrent.futures.ThreadPoolExecutor(
        max_workers=min(len(specialist_results), 3)  # 最多3并发
    ) as executor:
        futures = {executor.submit(review_single, sr): sr 
                   for sr in specialist_results}
        for future in concurrent.futures.as_completed(futures):
            try:
                result = future.result(timeout=300)
                if result:
                    cross_results.append(result)
            except Exception as e:
                logger.warning(f"交叉审阅超时或失败: {e}")
    
    return cross_results
```

**预期收益：** complex 场景减少 60-120s 延迟，审阅结论覆盖度不变。

### P0：仲裁模型差异化 — 真正的独立裁决

**当前问题：** 如果 `ARBITRATION_MODEL` == `MODEL`，仲裁是同一模型"自己审自己"。

**优化：强制不同模型链**

```python
ARBITRATION_REQUIRE_DIFFERENT_MODEL = os.getenv(
    "ARBITRATION_REQUIRE_DIFFERENT_MODEL", "1"
) == "1"

# orchestrator.py 在仲裁前检查
def _get_arbitration_model() -> str:
    """确保仲裁使用不同于专家的模型。"""
    from config import ARBITRATION_MODEL, MODEL, ARBITRATION_REQUIRE_DIFFERENT_MODEL
    
    arb_model = ARBITRATION_MODEL or MODEL
    
    if ARBITRATION_REQUIRE_DIFFERENT_MODEL and arb_model == MODEL:
        # 专家用了默认模型，仲裁切换到备选
        from llm_service import _fallback_model
        if _fallback_model and _fallback_model != MODEL:
            logger.info(f"仲裁模型切换: {MODEL} → {_fallback_model}")
            return _fallback_model
        else:
            logger.warning("无法保证仲裁模型独立性，仍使用同一模型")
    
    return arb_model
```

### P1：专家结果缓存 — 追问即时响应

用户追问时（"展开说说"、"那风险呢"），现有 2 个专家已完成分析，无需重新调用。

```python
# orchestrator.py 顶部
from functools import lru_cache
import hashlib

_expert_result_cache: dict[str, tuple[dict, float]] = {}
_EXPERT_CACHE_TTL = 300  # 5 分钟


def _cache_key(agent_key: str, query: str, context_hash: str) -> str:
    raw = f"{agent_key}:{query}:{context_hash}"
    return hashlib.md5(raw.encode()).hexdigest()


def _execute_specialist_cached(agent_key, query, 
                                prebuilt_context="", cancel_event=None, start_time=0):
    """带缓存的专家执行。同一专家+同一上下文 5 分钟内复用。"""
    context_hash = hashlib.md5(
        (prebuilt_context or "")[:500].encode()
    ).hexdigest()
    key = _cache_key(agent_key, query, context_hash)
    
    # 缓存命中且未过期
    if key in _expert_result_cache:
        result, timestamp = _expert_result_cache[key]
        if time.time() - timestamp < _EXPERT_CACHE_TTL:
            logger.info(f"专家缓存命中: {agent_key}")
            # 标记为缓存（非实时分析）
            result["cached"] = True
            return json.dumps(result, ensure_ascii=False)
    
    # 缓存未命中，正常执行
    result_str = _execute_specialist(
        agent_key, query, prebuilt_context=prebuilt_context,
        cancel_event=cancel_event, start_time=start_time
    )
    
    # 写入缓存
    try:
        result_obj = json.loads(result_str)
        _expert_result_cache[key] = (result_obj, time.time())
        # 控制缓存大小，清理最旧条目
        if len(_expert_result_cache) > 32:
            oldest = min(_expert_result_cache, key=lambda k: _expert_result_cache[k][1])
            del _expert_result_cache[oldest]
    except Exception:
        pass
    
    return result_str
```

**预期收益：** 追问场景延迟从 60s 降到 <1s。

### P1：分析模板增强 — 强制可执行输出

当前 `ANALYSIS_TEMPLATES` 约束了格式，但没约束"给出具体可执行的建议"。

```python
# multi_agent.py 中增加通用约束
_ACTIONABILITY_CONSTRAINT = """
## 🎯 可执行性要求（强制遵守）

你的分析结尾必须包含「具体操作建议」段落，格式：
```
## 💡 具体操作建议
1. **操作**：买/卖/持有/定投（选一）
   - **标的**：基金名称+代码
   - **金额或比例**：具体数字，如"2000元"或"现有仓位的10%"
   - **触发条件**：什么情况下执行，如"当PE分位降到20%时"
   - **理由**：一句话说明为什么
```

禁止出现：
- ❌ "建议根据自身情况决定"（等于没说）
- ❌ "可以考虑"（模棱两可）
- ❌ "适当调整"（没有具体比例）
"""

# 追加到所有专家 prompt 末尾
def inject_constraints(system_prompt: str) -> str:
    return system_prompt + _ACTIONABILITY_CONSTRAINT
```

**预期收益：** 每条 AI 建议都能直接落地执行，不需要用户再追问"那具体怎么做"。

### P1：上下文冲突检测 — 消除矛盾指令

```python
def detect_context_conflicts(sections: dict) -> list[str]:
    """检测上下文中的矛盾指令，按优先级排序解决。"""
    conflicts = []
    prefs = sections.get("user_profile_context", "")
    kyc = sections.get("kyc", "")
    strategy = sections.get("strategy", "")
    
    # 优先级：KYC > 策略 > 偏好
    if "详细数据支撑" in prefs and ("新手" in kyc or "保守" in kyc):
        conflicts.append("[KYC覆盖] 新手用户偏好详细数据 → 提供数据但用通俗语言解释")
    if "激进" in kyc and "保守" in strategy:
        conflicts.append("[策略冲突] 激进画像 vs 保守策略 → 提醒风险，让用户选择")
    if "长期" in kyc and "短期" in sections.get("decision_context", ""):
        conflicts.append("[期限冲突] 长期目标 vs 短期操作 → 标记为需确认")
    return conflicts
```

**集成到上下文：** 冲突列表作为 `[需确认]` 标记注入上下文，让模型在处理时显式提醒用户。

### P2：请求去重 + 执行状态机完善

**问题：** 用户快速点击"分析"按钮会重复调用相同专家。

```python
# 前端：pending state + disabled button
export function useExecuteGuard() {
  const executing = ref(false)
  async function execute(fn) {
    if (executing.value) return  // 已在执行，忽略
    executing.value = true
    try { return await fn() }
    finally { executing.value = false }
  }
  return { executing, execute }
}

# 后端：对话状态机已有 queued/streaming/completed/failed/cancelled
# 补充：同一个 conv_id 下不允许重复 running
if get_running_conversations(conv_id):
    return {"error": "已有分析任务进行中，请等待完成"}
```

**优化重点：** 目前对话状态机缺少 `overlapping_rejected` 状态的自描述反馈，需要前端展示"已有分析进行中"的提示。
