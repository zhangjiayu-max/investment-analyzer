# 投资分析助手 — 易用性与高可用性评估报告

> 日期：2026-06-24  
> 评估维度：易用性（8项）+ 高可用性（7项），每项按 10 分制评分

---

## 一、易用性评估

### 总分：48/80（60%）

| 维度 | 得分 | 问题 |
|------|------|------|
| 首次体验 | 3/10 | 需手动克隆/安装/配置，无引导 |
| 日常使用 | 6/10 | 可用但需要学习探索 |
| 任务完成 | 6/10 | 多数任务能完成但路径不够直接 |
| 学习曲线 | 4/10 | 功能散落各 Tab，无教程 |
| 操作反馈 | 5/10 | 错误提示不友好 |
| 移动体验 | 6/10 | 响应式适配但体验不比原生 |
| 数据录入 | 5/10 | 有快速录入但仍需手动 |
| 信息呈现 | 7/10 | 数据可视化优秀 |

---

### 1.1 首次体验：3/10 🔴

**当前流程：**
```
1. git clone 项目
2. pip install -r requirements.txt
3. 配置 .env（需要 DeepSeek/MIMO API Key）
4. 安装 akshare（数据源）
5. 配置 ChromaDB（向量检索）
6. 手动导入 CSV 或逐条录入手动持仓
7. 手动设置 KYC 画像
```

**问题：** 每一步都有摩擦。一个想立刻用的用户在第 3 步就放弃了。

**改进方向：**
- 提供 `docker-compose up` 一键启动
- 首次启动时自动弹出"快速设置向导"（API Key → KYC 画像 → CSV 导入）
- 提供 demo 数据模式（预置样例持仓，可以直接体验）

---

### 1.2 日常使用：6/10 🟡

**当前用户的一天：**

```
早：打开系统 → 看仪表盘（一堆卡片不知道该看哪个）
中：AI 问"帮我看看持仓" → 得到建议 → 建议在聊天里，不在持仓页
晚：想记录一笔交易 → 持仓页 → 找表单 → 填入 → 保存
```

**问题：** 功能之间是孤立的。AI 聊天的结果不会出现在持仓页，仪表盘的推荐不会变成交易草稿。

**改进方向：**
- 仪表盘首页简化为 3 个区域：今日温度计 + 我的健康分 + 今天该做什么
- AI 建议 → 自动出现在"待办决策"区
- 持仓页顶部加"最近变化"时间线（今天做了什么交易、AI 给了什么建议）

---

### 1.3 任务完成效率：6/10 🟡

| 常见任务 | 当前路径 | 步骤数 | 理想步骤 |
|---------|---------|--------|---------|
| 查看持仓健康度 | 仪表盘 → 找健康分卡片 | 2 | 首页直接看到 |
| 判断是否该加仓 | AI 对话 → 提问 → 等分析 | 3 | 首页温度计指示 |
| 记录一笔交易 | 持仓管理 → 找表单 → 填入 | 3 | 2（快速入口） |
| 检查费率 | 持仓管理 → 分析标签 → 费率分析 | 3 | AI 主动提醒 |
| 回顾上月决策 | 决策记录 → 翻列表 | 2 | 1（到期自动推送） |

**改进方向：** 高频操作（查健康分、记交易、看温度计）全部放在首页或一键入口。低频操作（费率分析、相关性检查）由系统主动推荐。

---

### 1.4 学习曲线：4/10 🔴

**问题：**

- 没有 onboarding 教程
- 功能命名偏技术化（"相关性矩阵"、"滚动收益"、"PANORAMA 分析"）
- 用户不知道每个工具什么时候用
- 没有任何引导提示

**改进方向：**
- 首次使用弹出 3 步引导（导入持仓 → 完善画像 → 查看健康分）
- 工具卡片增加"什么时候用"说明
- 改名为用户能懂的语言：
  - "相关性矩阵" → "你的基金重复买了吗？"
  - "滚动收益" → "这笔投资历史上赚钱概率多大？"
  - "PANORAMA 分析" → "我的持仓全面诊断"

---

### 1.5 操作反馈：5/10 🟡

**问题：**

- 网络错误时显示原始技术报错（"Connection refused"、"Timeout"）
- 表单提交后没有成功提示（静默刷新列表）
- 长时间分析（AI 对话）时，用户不知道系统在干嘛
- 删除操作无确认就直接删了

**已改进：** 流式输出已加上阶段标识（🔍/📊/🤝/⚡/⚖️）✅

**仍需改进：**
- 错误信息用用户能理解的语言重写
- 所有操作加 toast 提示（成功/失败/进行中）
- 危险操作（删除持仓、删除交易）加二次确认

---

### 1.6 移动端体验：6/10 🟡

**当前：** 响应式适配做了（480px/768px 断点），MobileApp.vue 有底部 Tab 栏。

**问题：**
- 表格在手机上横向滚动，体验差
- 图表在小屏幕上缩小后文字看不清
- 输入框在手机上不够大（触控区域 < 48px 的部分）
- 没有 PWA 离线缓存

**改进方向：**
- 表格改为卡片式布局（移动端）
- 最小触控区域统一 48px
- 添加 PWA manifest + service worker 离线缓存

---

### 1.7 数据录入：5/10 🟡

**已改进：** 快速录入模式（只输代码+金额，自动查名称+净值）✅

**仍存在问题：**
- 初始大量持仓导入仍然繁琐（CSV 格式要求严格）
- 没有从天天基金/支付宝导出并自动识别格式
- 持仓更新需要手动点击"刷新净值"

**改进方向：**
- 支持多种 CSV 格式自动识别（天天基金/支付宝/蛋卷/银行）
- 加"自动刷新净值"开关（定时自动更新）
- 支持截图识别持仓（OCR → 自动录入）

---

### 1.8 信息呈现：7/10 🟢

**优势：** ECharts 可视化、暗色模式、卡片式布局、趋势箭头动画。

**可优化：**
- 部分数据缺少时间标注（"估值数据来自哪天？"）
- 健康分缺少历史趋势对比（"比上周涨了还是跌了？"）
- 没有"一键分享"功能（生成图片分享持仓概览）

## 二、高可用性评估

### 总分：43/70（61%）

| 维度 | 得分 | 问题 |
|------|------|------|
| 部署方式 | 3/10 | 纯手动，无 Docker |
| 服务可用性 | 7/10 | 个人使用够用 |
| 数据安全 | 7/10 | 本地 SQLite + 备份 |
| 错误降级 | 7/10 | 熔断+降级策略完善 |
| 性能 | 6/10 | 首次加载慢，瓶颈在 akshare |
| 监控告警 | 5/10 | 有追踪无告警 |
| 恢复能力 | 5/10 | 手动恢复，无自动重启 |

---

### 2.1 部署方式：3/10 🔴

**当前：**
```bash
pip install -r requirements.txt
cp .env.example .env
# 手动编辑 .env
python app.py
```

**问题：**
- 没有 Docker 镜像 → 换一台机器要重新配环境
- 没有系统依赖说明（ChromaDB 需要特定版本的 onnxruntime 等）
- 端口号写死（多次在 GDM 的 8000 和某端口之间调整）

**改进方向：**
```dockerfile
# 增加 Dockerfile
FROM python:3.11-slim
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
ENV PORT=8000
CMD ["uvicorn", "app:app", "--host", "0.0.0.0", "--port", "8000"]
```

再加上 `docker-compose.yml` 一键启动。

---

### 2.2 服务可用性：7/10 🟢

**优势：**
- FastAPI 异步框架，并发能力好
- SSE 流式输出，长连接稳定
- 异常全局捕获，不会崩

**问题：**
- 无主进程守护（如果 Python 进程挂了，没人重启）
- 无端口冲突检测
- 启动时间较长（ChromaDB embedding 模型首次加载 2-5s）

**改进方向：**
- systemd service 或 supervisor 守护
- Docker 自动重启策略（`restart: always`）
- embedding 模型懒加载（首次查询时才初始化，不阻塞启动）

---

### 2.3 数据安全：7/10 🟢

**已改进：** 每日自动备份 + WAL 模式 + 事务包装 ✅

**仍存在问题：**
- 备份只在本地目录，没有异地备份
- 没有备份恢复测试（只备份了，没验证过能不能恢复）
- 没有数据导出功能（用户想迁移到其他工具怎么办）

**改进方向：**
- `backup_database()` 增加备份完整性校验（备份后 verify）
- 增加数据导出 API（导出全部持仓/交易/决策为 JSON/CSV）
- （可选）Git 自动提交备份（备份后 push 到私有仓库实现异地备份）

---

### 2.4 错误降级：7/10 🟢

**已改进：** 熔断器（akshare + LLM API）+ 多级降级策略 ✅

**仍存在问题：**
- akshare 熔断时，前端展示什么？目前可能显示空白或旧数据
- 部分功能无降级策略：估值查询失败的降级方案是什么？

**改进方向：**
- akshare 熔断时，前端明确提示"行情数据暂时不可用，显示的是最近一次缓存数据"
- 估值查询失败 → 展示上次成功获取的估值 + 标注数据日期
- 所有外部依赖都加降级 label

---

### 2.5 性能：6/10 🟡

**瓶颈分析：**

| 操作 | 耗时 | 瓶颈 |
|------|------|------|
| 首页加载 | 3-5s | akshare 行情+估值+新闻 3 个串行请求 |
| AI 对话(complex) | 60-180s | 多个专家 LLM 调用 |
| 持仓刷新净值 | 30-60s | 逐只查询 akshare 基金净值 |
| 相关性分析 | 20-60s | 逐只获取净值序列 + 计算 |

**已改进：** 并行交叉审阅 + 专家缓存 + 行情 5min 缓存 ✅

**仍需改进：**
- 首页启动时异步加载（先显示骨架屏，数据到了再填充）
- 持仓刷新改为并发（用 `ThreadPoolExecutor` 并行查询多只基金）
- 加数据预取策略（打开持仓页时预加载相关估值数据）

---

### 2.6 监控告警：5/10 🟡

**已改进：** Request Tracing（x-request-id 全链路）+ 健康检查 ✅

**缺失：**
- 无 LLM 调用异常告警（API Key 过期了不知道）
- 无 akshare 连续失败告警
- 无数据新鲜度告警（估值数据超过 7 天没更新）
- 无 Token 预算耗尽告警

**改进方向：**
- 每日 cron 任务：检查 akshare 可用性、估值数据新鲜度
- 关键异常写到一个 `alerts` 表，首页红点提示
- Token 用量接近上限时主动通知

---

### 2.7 恢复能力：5/10 🟡

**已改进：** 优雅关闭 + 备份 ✅

**缺失：**
- 备份恢复脚本（`backup_database` 做了，但没有对应的 `restore_database`）
- 异常退出后没有自动重启
- 没有数据一致性校验

**改进方向：**
- `restore_database(backup_path)` 函数
- Docker 自动重启策略
- 启动时自动检查数据库完整性（`PRAGMA integrity_check`）
