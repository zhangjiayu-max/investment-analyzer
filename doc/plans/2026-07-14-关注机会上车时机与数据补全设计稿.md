# 关注机会 — 上车时机与数据补全设计稿

> 日期：2026-07-14
> 组件：`backend/routers/portfolio/watchlist.py` + `frontend/src/components/market/EventRadarPage.vue`
> 目标：补全关注基金缺失数据、自动推算上车价位、显式呈现上车时机

## 一、问题诊断

### 数据缺失现状（5 只 watching 基金）

| 基金 | 目标价 | 目标分位 | 当前净值 | 当前估值分位 |
|------|--------|----------|----------|--------------|
| 嘉实资源精选C | None | None | None | None |
| 南方科创板芯片ETF C | None | None | None | None |
| 人工智能ETF | None | None | None | None |
| 易方达中证红利C | None | None | None | None |
| 国泰黄金ETF C | None | None | None | None |

5 只基金 4 项关键字段全为 None。前端卡片虽已支持显示，但无数据可展示。

### 根因

1. **巡检不刷净值**：`/api/watchlist/patrol` 只查估值，不刷新净值；净值依赖手动点"刷新净值"按钮
2. **估值匹配太严**：本地 DB 用 `search_indexes_by_keyword` 精确匹配，匹配不到就静默跳过；akshare 兜底缺失
3. **无自动上车价**：用户加关注时没填目标价，系统不会基于估值自动推算
4. **上车时机不显式**：信号灯只是个小标签，没有"距上车还差多少"、"建议上车价"等明确指引

## 二、改进方案

### A. 后端数据补全（手动触发，不自动）

原则：保持手动触发（用户点"扫描信号"或"刷新净值"按钮时才执行），避免后台静默调用 MCP。

#### A1. 巡检时强制刷新净值

`/api/watchlist/patrol` 对每只 watching 基金：
- 调用 `fetch_fund_nav(fund_code)` 获取最新净值
- 更新 `current_nav` 和 `nav_updated_at`
- 失败时保留旧值，不阻断巡检

#### A2. 估值查询增强（akshare 兜底）

现有逻辑：`search_indexes_by_keyword` 精确匹配 → 匹配不到就 None。

增强：本地 DB 匹配失败时，用 akshare 的 `stock_index_pe_lg` 或 `index_value_hist` 兜底：
- 从 `index_name` 提取关键词（去"指数"、"中证"、"上证"等前缀）
- 调用 akshare 查询指数历史估值
- 计算当前 PE/PB 在 10 年历史中的百分位
- 缓存到本地 DB `valuations` 表（避免重复调用）

#### A3. 自动推算上车价位

新增字段 `suggested_buy_price`（建议上车价）和 `buy_price_source`（推算依据），在巡检时自动计算：

```
规则（按估值分位分档）：
- 分位 ≤20%（低估区）：上车价 = 当前净值 × 1.00（已可上车）
- 分位 20-40%（偏低区）：上车价 = 当前净值 × 0.95（再跌5%）
- 分位 40-60%（中性区）：上车价 = 当前净值 × 0.90（再跌10%）
- 分位 >60%（偏高区）：上车价 = 当前净值 × 0.85（再跌15%）
- 分位缺失：上车价 = None（无法推算）

特殊处理：
- 用户已设 target_price：以用户设置为准，不覆盖
- buy_price_source 记录推算依据，如"低估区·当前净值"或"偏低区·-5%"
```

#### A4. 巡检响应字段扩展

`/api/watchlist/patrol` 的 `all_items` 每项新增：
- `suggested_buy_price`: float | None — 自动推算的上车价
- `buy_price_source`: str — 推算依据
- `distance_to_buy`: float | None — 当前净值距上车价的百分比（正数=高于上车价，负数=已跌破）

### B. 前端"上车时机"区块显式化

在每张关注基金卡片中，"目标价"行下方新增**"上车时机"区块**：

```
┌─────────────────────────────────────┐
│ 🟢 可上车                            │  ← 信号灯徽章（放大）
│ 建议上车价: 1.4520（低估区·当前净值） │
│ 距上车: -2.3%（已跌破上车价）        │
│ ─────────────────────────────────── │
│ 买入评分: 78 建议买入                │  ← 从底部移上来
│ [估值 85] [价格 70] [相关 80] [集中 100] │
└─────────────────────────────────────┘
```

#### B1. 信号灯徽章放大

现有：卡片右上角小标签。
改为：上车时机区块顶部的大徽章，4 档显示：
- 🟢 可上车（green）：当前净值 ≤ 建议上车价
- 🟡 接近上车（yellow）：距上车价 < 5%
- 🔴 等待中（red）：距上车价 ≥ 5%
- ⚪ 数据不足（gray）：净值或上车价缺失

#### B2. 顶部统计卡片调整

现有 6 张统计卡（关注总数/信号触发/低估数量/已设目标/目标到位/近7日新增）。
调整为 4 档上车状态统计：
- 🟢 可上车数量
- 🟡 接近上车数量
- 🔴 等待中数量
- ⚪ 数据不足数量
- 保留：关注总数、近7日新增

实际保留 6 张卡：关注总数 / 可上车 / 接近上车 / 等待中 / 数据不足 / 近7日新增。

## 三、数据流

```
用户点"扫描信号"按钮
  ↓
POST /api/watchlist/patrol
  ↓
对每只 watching 基金：
  1. fetch_fund_nav() → 更新 current_nav
  2. search_indexes_by_keyword() → 查本地估值
     失败 → akshare 兜底查询 → 计算百分位 → 缓存到 DB
  3. 更新 current_percentile
  4. 计算 suggested_buy_price（基于估值分位）
  5. 计算 signal_status（green/yellow/red/gray）
  6. 计算 distance_to_buy
  ↓
返回 all_items（含完整字段）
  ↓
前端渲染上车时机区块
```
