# 理财决策闭环与 AI 对话增强设计

> 日期：2026-06-23  
> 范围：持仓管理、AI 对话、多轮上下文、异步恢复、加减仓决策闭环、近期理财能力整合。  
> 原则：完整闭环设计，分期实施。每次大改先写设计稿，再拆实施计划。

---

## 1. 背景与目标

项目当前已经具备持仓、交易、现金、估值、关注列表、AI 持仓分析、多 Agent 对话、RAG、决策档案、复盘、KYC 画像和评测等能力。近期新增的滚动收益、四笔钱、定投优化、费率分析、相关性分析、基金经理变更等功能也进一步增强了理财分析基础。

主要问题不是能力不足，而是能力还没有稳定串成一条理财决策闭环：

```text
AI 对话/持仓分析
  -> 统一上下文包
  -> 结构化建议
  -> 决策草案
  -> 执行前检查
  -> 交易草稿/交易确认
  -> 到期复盘
  -> 用户画像/RAG/评测反哺
```

本设计目标：

1. 持仓管理下的 AI 分析、加减仓建议、交易记录、决策复盘形成闭环。
2. AI 对话支持长期多轮上下文，不漏持仓、盈亏、现金、近期决策。
3. 对话异步、中断、恢复、重试语义统一，切页或刷新不导致任务丢失。
4. 近期新增理财能力不再散落在功能 Tab 中，而是服务于决策、行动和复盘。
5. 建立项目级规则：涉及核心架构或大改时，先在 `doc/plans/` 写设计稿。

## 2. 总体架构

### 2.1 主线闭环

系统主线采用“完整决策闭环、分期实施”：

```text
AI 对话/持仓分析
  -> 统一上下文包
  -> 结构化建议
  -> 决策草案
  -> 执行前检查
  -> 交易草稿/交易确认
  -> 到期复盘
  -> 用户画像/RAG/评测反哺
```

### 2.2 核心原则

1. **AI 只产生建议，不直接改持仓**  
   加仓、减仓、调仓都先进入决策草案或交易草稿，必须由用户确认。

2. **所有高价值建议都可追溯**  
   每条建议保留来源：对话消息、持仓快照、估值快照、RAG 来源、Agent 输出、风险检查结果。

3. **对话和持仓共用同一个上下文构建器**  
   统一生成最近对话、压缩摘要、当前持仓/盈亏、现金、估值、关注列表、用户画像、历史决策。

4. **异步任务统一成“后台执行 + 状态落库”**  
   SSE 只负责展示实时过程；切页、刷新、断线后从数据库恢复，不依赖前端连接存活。

5. **每次大改先写设计稿**  
   涉及架构、核心数据流、Agent 流程、持仓交易、对话上下文的大改，都先在 `doc/plans/YYYY-MM-DD-主题.md` 写设计，再拆实施计划。

### 2.3 第一期 MVP

第一期只落地最关键的闭环地基：

- 统一 AI 对话上下文包，避免多轮对话漏持仓/盈亏。
- 修复 `chat/simple/complex` 三条对话分支的异步与恢复一致性。
- 支持把 AI 建议保存为决策草案，并生成执行前检查。
- 决策草案可生成交易草稿，但交易仍由用户二次确认。

## 3. 持仓管理与加减仓闭环

### 3.1 现有基础

可复用模块：

- `portfolio_holdings`：当前持仓、成本、市值、盈亏。
- `portfolio_transactions`：买入、卖出、转换、待确认、确认、到账。
- `portfolio_tx_audit_log`：交易审计。
- `rebalance_config` 与 `rebalancer.py`：目标配置、偏离度、现金目标、低估加仓建议。
- `portfolio_analysis_records`：AI 分析结果。
- `decision_records`、`decision_actions`、`decision_reviews`：决策草案、行动项、复盘。
- `watchlist`：关注基金，可承接“观察/等待触发”。

### 3.2 五步闭环

#### 1. 诊断

全景诊断、单基金分析、交易复盘、定投优化、四笔钱、调仓引擎都可以输出建议，但统一进入“建议候选池”或“保存为决策草案”入口。

#### 2. 结构化建议

AI 输出需要拆成结构化字段：

- `action_type`
- `target_code`
- `target_name`
- `suggested_amount`
- `suggested_ratio`
- `reason`
- `evidence`
- `risks`
- `confidence`
- `review_at`

支持动作：

- `buy`
- `add`
- `reduce`
- `sell`
- `hold`
- `watch`
- `rebalance`
- `dca`
- `cash_reserve`

#### 3. 决策草案

用户点击“保存为决策”后写入 `decision_records`。

决策草案应包含：

- 投资假设
- 证据快照
- 反方观点
- 执行条件
- 退出条件
- 复盘日期

#### 4. 执行前检查

执行前用规则检查：

- 是否超过单基金、单指数、单行业仓位上限。
- 是否导致现金低于目标备用金。
- 是否与当前调仓配置冲突。
- 估值或净值数据是否过期。
- 是否已有相同标的未完成交易或未复盘决策。
- 是否缺少反方观点或风险说明。

高风险时不阻止用户，但默认状态设为 `deferred` 或要求二次确认。

#### 5. 交易草稿与复盘

决策通过后生成 `portfolio_transactions(status='pending')` 草稿。

- 买入草稿使用金额。
- 卖出草稿使用份额或比例。
- 交易确认后关联回 `decision_actions`。
- 到 `review_at` 后生成复盘提醒。
- 复盘结果写入 `decision_reviews`，并反哺用户画像和评测集。

### 3.3 加减仓保护规则

- AI 不给“满仓/清仓”式单点建议，除非用户明确要求并通过风险检查。
- 默认使用分批执行，例如 `30% / 30% / 40%` 或按估值区间触发。
- 加仓必须说明资金来源、最大仓位、低估证据、失败条件。
- 减仓必须说明卖出理由、替代去向、是否只是情绪性止损。
- 所有操作建议都显示“这不是自动交易，需你确认”。

### 3.4 前端持仓页结构

持仓页建议分成三个区域：

1. **资产现状**
   - 持仓
   - 现金
   - 盈亏
   - 集中度
   - 配置偏离

2. **AI 分析**
   - 全景诊断
   - 单基金分析
   - 交易复盘
   - 定投优化
   - 四笔钱

3. **决策与行动**
   - 建议候选
   - 决策草案
   - 待确认交易
   - 到期复盘

## 4. AI 对话、多轮上下文与异步恢复

### 4.1 统一上下文包

新增或强化统一构建器，例如 `conversation_context.py`。

输入：

- `conversation_id`
- `current_user_message`
- `scenario_type`
- `agent_id`
- `token_budget`

输出：

- `conversation_summary`：历史压缩摘要。
- `recent_messages`：最近 N 轮完整消息。
- `portfolio_context`：当前持仓、市值、盈亏、现金、集中度。
- `valuation_context`：持仓相关指数估值。
- `decision_context`：近期决策、未执行决策、到期复盘。
- `watchlist_context`：关注标的和触发条件。
- `user_profile_context`：风险偏好、目标、行为偏差。
- `rag_context`：知识库、文章、历史经验。
- `missing_context`：缺失但影响建议的重要信息。

推荐拼接顺序：

```text
系统原则
  -> 用户画像
  -> 当前持仓/盈亏
  -> 历史摘要
  -> 最近对话
  -> RAG 证据
  -> 当前问题
```

### 4.2 对话异步状态统一

当前复杂任务已经接近目标，但 `chat/simple/complex` 分支还不一致。建议统一为：

1. 收到用户消息。
2. 幂等创建用户消息：同内容、同状态不重复写。
3. 立即创建 assistant 占位消息：
   - `queued`
   - `streaming`
   - `completed`
   - `failed`
   - `cancelled`
4. 后台任务执行，不依赖 SSE 连接。
5. SSE 只监听并转发事件。
6. 最终结果写入同一条 assistant 消息。
7. 页面恢复时只看 DB 状态，不猜前端状态。

刷新后前端判断：

- `queued` 或 `streaming`：调用 `/resume` 观察后台任务。
- `completed`：直接展示结果。
- `failed`：允许重试。
- `cancelled`：允许从原问题重新执行。

### 4.3 恢复、继续、重试

需要拆分三种动作：

- **恢复**：后台还在跑，只重新连接，不重新执行。
- **继续**：已有部分专家完成，跳过已完成专家，接着跑。
- **重试**：当前任务失败或取消，创建新的 assistant 消息，保留原失败记录。

前端按钮文案：

- `恢复连接`
- `继续分析`
- `重新生成`

### 4.4 多轮对话摘要

当 conversation 消息超过阈值，例如 20 条或 token 超过预算：

- 后台生成或更新 `conversation_summaries`。
- 摘要记录 `up_to_message_id`。
- 下一轮上下文使用“摘要 + 摘要后的最近消息”。

摘要必须保留：

- 用户长期目标和约束。
- 已经确认的事实。
- 未完成问题。
- 已提出但未执行的建议。
- 关键持仓和决策引用。
- 用户明确否定过的偏好。

### 4.5 投资建议场景模板

`clarify_requirement()` 已经有 `scenario_type`，建议进一步产品化：

- `buy_decision`
- `sell_decision`
- `portfolio_review`
- `decision_review`
- `article_check`
- `knowledge_qa`
- `general_analysis`

不同场景使用不同上下文、Agent、输出 schema。

`buy_decision` 必须输出：

- 是否值得行动。
- 建议仓位上限。
- 分批计划。
- 证据。
- 风险。
- 什么情况下不买。
- 是否建议保存为决策草案。

### 4.6 reasoning_content

MIMO thinking mode 的 `reasoning_content` 需要统一处理：

- `reasoning_content` 不展示为最终结论。
- 如果模型协议要求后续消息带回 reasoning，则在 `llm_service.py` 层统一处理。
- 落库保存可控摘要，不把长 reasoning 当普通对话上下文无限传递，避免污染后续轮次。

## 5. 理财工作台整合

### 5.1 页面结构

持仓管理保留明细表，上层改成四个工作区。

#### 1. 今日概览

- 总资产、持仓市值、现金、今日/昨日盈亏。
- 配置偏离度。
- 重要预警：跌幅、集中度、经理变更、待确认交易。
- 今日建议：只显示 3 条最高优先级行动。

#### 2. 组合诊断

- 全景诊断。
- 分散度分析。
- 相关性分析。
- 费率侵蚀。
- 基金经理变更。

输出统一进入建议候选池。

#### 3. 行动中心

- 决策草案。
- 待执行行动。
- 待确认交易。
- 到期复盘。
- 关注列表触发项。

这是“AI 建议变成真实理财动作”的主入口。

#### 4. 规划工具

- 四笔钱。
- 定投优化。
- 滚动收益。
- 调仓配置。
- 关注列表。

这些不直接产生交易，而是生成配置建议、观察条件或决策草案。

### 5.2 新增能力定位

#### 四笔钱

四笔钱用于判断资金用途和风险期限，输出应影响加减仓检查：

- 备用金不足时，拦截高波动加仓建议。
- 长期权益桶不足时，允许分批定投。
- 机会资金桶过高时，提示降风险。

#### 定投优化

定投优化生成 `dca` 类型决策草案：

- 标的。
- 月投金额。
- 情绪或估值调整倍率。
- 暂停条件。
- 复盘周期。

#### 滚动收益

滚动收益作为证据，不直接给交易指令：

- 展示持有 1/3/5 年胜率。
- 作为“是否适合长期桶”的依据。
- 避免只看短期涨跌。

#### 关注列表

关注列表承接“观察型决策”：

- AI 认为现在不适合买，则进入关注列表。
- 设置目标净值、目标估值百分位、触发备注。
- 触发后生成提醒或决策草案。

#### 交易复盘

交易复盘不只是报告，而是决策闭环的结果评估：

- 哪些买入后继续下跌。
- 哪些卖出后踏空。
- 哪些建议被执行。
- 哪些建议被用户拒绝但后来证明正确或错误。

#### 调仓配置

调仓配置成为执行前检查的规则来源：

- 基础目标配比。
- 估值调整。
- 现金目标。
- 偏离阈值。
- 低估加仓金额上限。

### 5.3 建议候选池

建议新增轻量概念：`recommendation_candidates`。第一期可以先不建表，用 `portfolio_analysis_records.result_data` 加前端按钮承接；中长期推荐建表。

建议字段：

- `source_type`
- `source_id`
- `scenario_type`
- `action_type`
- `target_code`
- `target_name`
- `suggested_amount`
- `confidence`
- `evidence_json`
- `risk_json`
- `status`: `new/saved/ignored/expired`

好处：

- 所有 AI 分析都能汇总到行动中心。
- 建议不会散落在各个 Tab 的 Markdown 里。
- 方便去重、忽略、延期、保存为决策。

### 5.4 体验原则

- 首页和持仓页不把大段 AI 长文作为第一屏重点。
- 长文报告保留，但默认先展示结论、行动建议、风险、是否保存为决策。
- 所有涉及交易的按钮必须二次确认。
- AI 分析按钮 tooltip 显示使用的 Agent、数据源、预计耗时。
- 分析中允许切页面，明确提示“后台继续执行”。

## 6. 分期实施

### Phase 1：闭环地基

目标：先让系统“不断线、不失忆、能沉淀决策”。

范围：

- 新增统一对话上下文构建器。
- 对 `chat/simple/complex` 对话分支做异步状态一致化。
- 强化 `/api/conversations/{id}/resume` 的恢复语义。
- AI 回复下方支持保存为决策草案。
- 决策草案支持执行前检查。
- 决策草案可生成交易草稿，但必须用户确认。

验收：

- 切换页面或刷新后，对话任务仍能恢复。
- 单个 conversation 多轮追问能带上持仓、盈亏、现金、近期决策。
- 同一条用户消息重试不重复写入。
- 失败任务能重新生成，历史失败记录保留。
- AI 建议能保存为 `decision_records`。
- 执行前检查能提示现金不足、仓位过高、数据过期等风险。

### Phase 2：建议结构化

目标：让各类 AI 分析不只输出 Markdown，而能产出可操作建议。

范围：

- 为全景诊断、单基金分析、交易复盘、定投优化增加结构化建议字段。
- 可选新增 `recommendation_candidates` 表。
- 行动中心展示建议候选、决策草案、待确认交易、到期复盘。
- 关注列表触发条件接入建议候选。
- 调仓建议可一键生成决策草案。

验收：

- 不同 AI 分析结果能统一汇总到行动中心。
- 用户可以忽略、保存、延期、执行建议。
- 每条建议能追溯来源和证据。
- 重复建议能识别并合并或提示。

### Phase 3：复盘与学习

目标：让系统越用越懂用户。

范围：

- 决策到期自动提醒复盘。
- 交易确认后关联决策行动项。
- 复盘结果更新用户画像。
- 坏建议和低分回答进入评测集。
- 生成个人决策统计：追涨、杀跌、踏空、有效加仓、无效减仓等。

验收：

- 到期决策能出现在复盘列表。
- 复盘能记录收益变化和经验教训。
- 用户画像能区分事实、推断、偏好。
- 评测集能覆盖历史低质量建议。

## 7. 测试策略

### 7.1 后端测试

- `db.decisions`：决策、行动、复盘、状态流转。
- `conversation_context`：摘要、最近消息、持仓、估值、决策上下文拼接。
- `routers.conversations`：`chat/simple/complex` 三分支都能创建 assistant 占位、完成落库、失败可重试。
- `decision_precheck`：现金不足、仓位过高、数据过期、重复交易。
- `portfolio_transactions`：交易草稿、确认、审计日志、决策关联。

### 7.2 前端测试

- `ChatView`：刷新恢复、重试、继续、重新生成按钮语义。
- `PortfolioManagement`：保存决策、执行前检查、生成交易草稿。
- `DecisionRecordsPage`：状态更新、行动完成、复盘。
- `useStreamingState`：多对话状态隔离、`done/error` 清理。

## 8. 设计稿规则

以后每次大改都先写设计稿，存放：

```text
doc/plans/YYYY-MM-DD-主题.md
```

适用范围：

- Agent 编排或 prompt 体系变化。
- 对话上下文、异步恢复、重试逻辑变化。
- 持仓、交易、调仓、决策表结构变化。
- 前端持仓、AI 对话、行动中心的信息架构变化。
- RAG、画像、评测闭环变化。

设计稿最少包含：

- 背景和目标。
- 当前问题。
- 方案边界。
- 数据流。
- API/DB 影响。
- 前端影响。
- 分期实施。
- 测试与验收。

## 9. 后续实施建议
