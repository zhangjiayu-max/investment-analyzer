# 多智能体对话降本增效设计（方案 B）

**目标**：在保持多轮对话体验和回答质量的前提下，降低多 Agent 对话的 LLM token 消耗。

**核心策略**：不在"加模型调用求质量"，而在"减少无效/重复调用 + 在关键环节做轻量质检"。

---

## 一、现状与问题

当前对话流程（简化）：

```
用户输入 → Orchestrator(多轮 tool_call 选专家) → Phase A(调用全部相关专家) → Phase B(交叉审阅) → Phase C(仲裁) → 最终答案
```

主要消耗点：

1. **Orchestrator 选专家成本高**：通过多轮 tool_call 试探，常走 2-3 轮才确定专家列表。
2. **专家调用过全**：相关专家常被全部召出，即使问题只需一个专家即可回答。
3. **专家结果无语义缓存**：用户换种问法就要重新调用专家。
4. **交叉审阅/仲裁常在不必要时触发**：即使专家结论一致，也可能进入 Phase B/C。
5. **缺少输出前质检**：低质量答案直接返回，用户可能要求重答，反而增加消耗。

---

## 二、设计目标

| 目标 | 指标 |
|------|------|
| 降低单次对话 token 消耗 | 较当前降低 25-40% |
| 不降低回答质量 | 用户满意度/自动评估分数不下降 |
| 保证多轮对话连贯性 | 历史压缩、上下文关联不受影响 |
| 可配置、可回退 | 所有新逻辑都有开关，异常时回退到原流程 |

---

## 三、总体架构

在现有流程前后各加一层"闸门"：

```
用户输入
  │
  ▼
Smart Router（智能路由） ──→ 规则优先，LLM 兜底
  │
  ▼
Semantic Cache（语义缓存） ──→ 命中则跳过专家调用
  │
  ▼
Phase A（调用必要专家）
  │
  ▼
Early Stop（早停检查） ──→ 无冲突则跳过 Phase B/C
  │
  ▼
Phase B（交叉审阅，仅冲突时）
  │
  ▼
Phase C（仲裁，仅冲突时）
  │
  ▼
Light Validator（轻量反思） ──→ 输出前质检，最多 1 次修复
  │
  ▼
最终答案
```

---

## 四、核心组件

### 4.1 Smart Router 智能路由

**目标**：用最小成本决定"需要哪些专家 + 复杂度等级"。

#### 输入

- `query`：用户当前问题
- `history_summary`：最近 3 条对话摘要
- `portfolio_snapshot`：持仓/估值极简概况（前 500 字）

#### 输出

```json
{
  "complexity": "simple|medium|complex",
  "specialists": ["valuation", "risk"],
  "reason": "用户问估值，且持仓有权益类基金，需评估估值+风险",
  "needs_arbitration": false,
  "route_by": "rule|llm_fallback|mention"
}
```

#### 实现

1. **规则路由（零 LLM 成本，默认）**

   基于关键词 + 专家职责表：

   | 关键词 | 触发专家 |
   |--------|----------|
   | 估值 / PE / PB / 百分位 / 低估 / 高估 | valuation |
   | 风险 / 回撤 / 止损 / 亏损 | risk |
   | 配置 / 仓位 / 股债 / 比例 | allocation |
   | 持仓 / 分散 / 穿透 | diversification, allocation |
   | 市场 / 大盘 / 行情 / 走势 | market |
   | 买 / 卖 / 操作 / 定投 / 止盈 | allocation, risk, valuation |
   | 文章 / 公众号 / 解读 | article_analysis |
   | 债券 / 利率 / 信用 | bond |

2. **LLM 兜底（可配置开关 `router.use_llm_fallback`，默认 true）**

   当规则无法匹配（如"帮我看看"这种模糊问题）时，用主模型做一次轻量判断：
   - prompt 约 800 tokens
   - 输出 JSON，约 200 tokens
   - 单次调用，不开 tool

3. **用户 @mention 覆盖**

   用户通过 `@估值专家` 等形式指定专家时，直接采用，跳过路由。

#### 成本收益

- 规则命中时：零新增 LLM 调用
- LLM 兜底时：约 1000 tokens，但可减少 Orchestrator 多轮试探（通常节省 2000-4000 tokens）

---

### 4.2 Semantic Cache 语义缓存

**目标**：让"意思相近但表述不同"的问题能复用专家结果。

#### 当前问题

`_execute_specialist_cached` 使用精确文本匹配作为 key，用户换一种说法就失效。

#### 改进方案

1. **缓存 key 结构**

   ```python
   cache_key = (
       agent_key,
       context_hash,          # 持仓/估值等上下文的哈希
       query_embedding,       # query 的 embedding 向量
   )
   ```

2. **命中条件**

   - 同一 `agent_key`
   - `context_hash` 一致
   - 新 query 与缓存 query 的 embedding 余弦相似度 ≥ 0.92

3. **缓存内容**

   只缓存成功执行的专家结果：
   - `analysis`
   - `tool_calls`
   - `duration_ms`
   - `model`

4. **TTL**

   默认 30 分钟，可配置 `cache.ttl_minutes`。

5. **embedding 来源**

   复用项目已有的 RAG embedding 模型（`moka-ai/m3e-base`），避免新增依赖。

#### 成本收益

- embedding 计算成本极低（本地模型）
- 对高频问题（如"今天大盘怎么看"、"白酒还能买吗"）命中率预计提升 2-3 倍

---

### 4.3 Early Stop 早停机制

**目标**：专家结论一致时，跳过交叉审阅和仲裁。

#### 当前问题

只要 ≥2 个专家就进入交叉审阅，即使结论一致。

#### 改进方案

在 Phase A 结束后执行"实质性冲突检测"：

1. **评级冲突**：买入 / 卖出 / 持有方向相反
2. **操作建议冲突**：同一标的给出相反操作（如一个建议买，一个建议卖）
3. **数据引用冲突**：对同一数据给出矛盾判断（如一个说高估，一个说低估同一指数）

如果 `detect_conflicts` 返回 `detected: false`，直接由 Orchestrator 综合答案，不进入 Phase B/C。

#### 成本收益

- 无冲突的简单问题可节省一次交叉审阅（N 个专家 × 约 3000 tokens）+ 可能的仲裁（约 4000 tokens）
- 预估占日常问题的 40-60%

---

### 4.4 Light Validator 轻量反思

**目标**：在最终答案输出前做低成本质检，拦截低质量答案，减少用户返工。

#### 定位

- 不生成新内容
- 只检查并标记问题
- 发现问题后，触发一次"修复"调用（最多 1 次）

#### 检查项

| 检查项 | 规则检查 | LLM 检查 | 说明 |
|--------|----------|----------|------|
| 数据真实性 | ✓ | ✓ | 数字是否来自上下文/工具结果 |
| 可执行性 | ✓ | - | 是否包含操作/标的/金额/触发条件 |
| 一致性 | - | ✓ | 最终答案是否与多数专家观点矛盾 |
| 幻觉标记 | ✓ | ✓ | 是否出现上下文未提及的基金代码 |

#### 实现

1. **规则检查（70%）**

   - 检查最终答案是否包含"操作建议"段落
   - 用正则匹配基金代码格式，与上下文中的代码比对
   - 检查是否包含"根据自身情况""仅供参考"等模糊表述

2. **轻量 LLM 检查（30%，约 500 tokens）**

   ```
   输入：原始问题 + 专家结论摘要 + 最终答案
   输出：{"passed": true/false, "issues": ["..."], "severity": "low|medium|high"}
   ```

3. **修复调用**

   如果 `severity == high`，把 `issues` 附加到 Orchestrator 消息中，请求修复，最多 1 次。

#### 成本收益

- 正常答案：仅增加约 500 tokens 检查成本
- 问题答案：增加约 500 + 1500 tokens 修复成本，但避免用户返工（通常返工需 3000-8000 tokens）

## 五、配置开关

所有新逻辑都可配置，默认开启但保守：

```python
# router
router.enabled = true
router.use_llm_fallback = true          # 规则未命中时用 LLM
router.default_complexity = "medium"

# cache
cache.semantic.enabled = true
cache.ttl_minutes = 30
cache.similarity_threshold = 0.92

# early stop
early_stop.enabled = true
early_stop.min_specialists = 2          # ≥2 个专家才检查早停
early_stop.conflict_threshold = "medium" # 冲突检测严格度

# validator
validator.enabled = true
validator.max_repair_attempts = 1
validator.llm_check_enabled = true
```

## 六、监控指标

新增以下指标用于评估效果：

| 指标 | 来源 | 目标 |
|------|------|------|
| 路由规则命中率 | Smart Router | > 70% |
| 路由准确率（与 @mention 重合度） | Smart Router | > 85% |
| 语义缓存命中率 | Semantic Cache | > 25% |
| 早停比例 | Early Stop | > 40% |
| 平均每次对话 token 数 | token_usage | 降低 25-40% |
| Validator 拦截率 | Light Validator | 5-15% |
| 用户返工率（重新提问同类问题） | 对话日志 | 下降 |
| 自动评估分数 | conversation_evaluator | 不下降 |

## 七、成本/收益估算

以一次典型的"3 专家中等问题"为例：

| 阶段 | 当前消耗 | 优化后消耗 | 说明 |
|------|----------|------------|------|
| Orchestrator 选专家 | 2000-4000 | 0-1000 | 规则命中时 0，LLM 兜底时 1000 |
| Phase A 专家调用 | 9000 | 6000-9000 | 路由减少专家数 + 缓存命中 |
| Phase B 交叉审阅 | 6000 | 0-6000 | 无冲突时跳过 |
| Phase C 仲裁 | 4000 | 0-4000 | 无冲突时跳过 |
| Validator | 0 | 500-2000 | 正常 500，修复时 2000 |
| **总计** | **21000-23000** | **6500-18000** | **平均降低约 30%** |

## 八、实施步骤

### Phase 1：基础优化（1-2 天）

1. 扩展 `_execute_specialist_cached`：支持语义缓存 + context hash + 30min TTL
2. 增强 `detect_conflicts`：从评级冲突扩展到操作建议/数据引用冲突
3. 改进早停逻辑：无冲突时跳过 Phase B/C

### Phase 2：智能路由（2-3 天）

1. 实现 `SmartRouter` 类：规则表 + LLM 兜底
2. 在 `orchestrate` / `orchestrate_stream` 入口调用 Smart Router
3. 根据路由结果决定 specialists 和 complexity
4. 保留 @mention 覆盖能力

### Phase 3：轻量反思（2 天）

1. 实现 `LightValidator`：规则检查 + LLM 检查
2. 在最终答案生成后调用 Validator
3. 实现最多 1 次修复调用

### Phase 4：监控与调优（1-2 天）

1. 新增配置项到 `db/config.py`
2. 新增监控指标记录
3. A/B 测试：对比开启前后的 token 消耗和评估分数

## 九、风险与回退

| 风险 | 缓解措施 |
|------|----------|
| 路由错误导致漏召专家 | @mention 可覆盖；配置开关可关闭；监控路由准确率 |
| 语义缓存命中但上下文已变 | context_hash 包含持仓/估值快照；TTL 30 分钟 |
| 早停漏掉潜在冲突 | 保守的冲突检测；复杂问题默认不走早停 |
| Validator 过度修复导致答案变差 | 最多 1 次修复；只修复 high severity 问题 |
| 整体质量下降 | 所有新逻辑都有开关；A/B 测试验证后再全量 |
