# 系统进化闭环设计方案

> 日期：2026-05-30 | 状态：待审批

## 一、目标

将现有分散的反馈收集、评测、Bad Case 看板串联成**完整的系统进化闭环**：

```
用户反馈 ──→ Bad Case 积累 ──→ 转化为 Eval 用例
                                    │
Prompt 改进 ←── 反馈模式分析 ←── 回归测试验证
     │
     └──→ 版本对比 ──→ 效果量化 ──→ 持续迭代
```

## 二、现状盘点

| 能力 | 状态 | 位置 |
|------|------|------|
| 点赞/点踩反馈 | ✅ 可用 | ChatView.vue / portfolio 分析结果 |
| 反馈学习 → 用户画像 | ✅ 可用 | `feedback_learner.py` → `user_profiles` 表 |
| Bad Case 看板 | ✅ 可用 | BadCasePage.vue（只读） |
| Prompt 版本管理 | ✅ 可用 | `agent_prompt_versions` 表 + 回滚 |
| Eval 用例管理 | ✅ 可用 | EvalSuitePage.vue |
| Eval 运行 | ⚠️ 无评分 | `run_eval_case_api` 不计算 score |
| Bad Case → Eval 转化 | ❌ 缺失 | 两个系统断开 |
| 反馈驱动 Prompt 优化 | ❌ 缺失 | AI 优化不参考 bad case |
| Prompt 变更回归测试 | ❌ 缺失 | 改 prompt 不自动跑 eval |
| 推荐自动验证 | ❌ 缺失 | `auto_verify_pending_recommendations` 无定时触发 |
| Prompt 性能指标 | ❌ 缺失 | 无版本维度的好评率/评分统计 |

## 三、功能设计

### 3.1 Eval 自动评分（LLM-as-Judge）

**思路**：eval run 完成后，用 LLM 对比 `expected_quality` 和实际输出，给出 1-5 分 + 评语。

**新增函数** `backend/agent/eval_scorer.py`：

```python
async def score_eval_result(expected_quality: str, actual_result: str,
                            analysis_type: str) -> tuple[float, str]:
    """返回 (score, reason)"""
    prompt = f"""你是投资分析质量评审专家。请对以下分析结果打分（1-5分）。

评分标准：
- 5分：专业、准确、可操作、覆盖全面
- 4分：较好，有小瑕疵但不影响核心判断
- 3分：合格，基本可用但有明显不足
- 2分：较差，存在事实错误或逻辑漏洞
- 1分：不可用，严重错误或完全偏题

分析类型：{analysis_type}
期望质量标准：{expected_quality}
实际输出：{actual_result[:3000]}

请严格输出JSON格式：
{{"score": 1-5的整数, "reason": "一句话评语"}}"""
    # 调用 LLM，解析 JSON 返回
```

**改造 `run_eval_case_api`**：

```python
# 在保存 run 之后，异步计算评分
score, reason = await score_eval_result(
    case["expected_quality"], result_data, analysis_type
)
update_eval_run(run_id, score=score, score_reason=reason)
```

**新增 DB 字段**：
- `eval_runs` 表加 `score_reason TEXT` 列

**前端改动**：
- EvalSuitePage.vue 的运行结果列表显示分数和评语
- Eval 统计卡片展示平均分

### 3.2 Bad Case → Eval Case 一键转化

**思路**：在 Bad Case 看板加「转为 Eval 用例」按钮，自动将反馈记录转为可回归测试的 eval case。

**后端新增** `POST /api/eval/cases/from-bad-case`：

```python
class BadCaseToEvalRequest(BaseModel):

# 1. 根据 source 查询原始 bad case 数据
# 2. 用 LLM 从 input + output + feedback_note 生成 expected_quality
```

- BadCasePage.vue 每条记录加「转为 Eval」按钮
- 点击后弹窗确认，可编辑 name 和 expected_quality
- 成功后跳转到 EvalSuitePage 查看

### 3.4 Prompt 变更自动回归测试

**思路**：当通过管理页面修改 agent prompt 后，自动运行该 agent 关联的所有 active eval cases，验证没有退步。

**流程**：

```
用户修改 prompt → 保存新版本 → 异步触发回归测试
                                    │
                        ┌───────────┴───────────┐
                        ↓                       ↓
                   所有 eval case 运行      计算新旧分数对比
                        │                       │
                        └───────────┬───────────┘
                                    ↓
                        生成回归报告（新分数 vs 旧分数）
```

**后端改动**：

`PUT /api/agents/{agent_id}` 保存后触发：

```python
# 异步执行回归测试
asyncio.create_task(run_regression_tests(agent_id, old_prompt_version))

async def run_regression_tests(agent_id: int, old_version: int):
    """运行该 agent 关联的所有 eval cases，对比新旧版本"""
    cases = list_eval_cases_for_agent(agent_id)  # 需新增关联逻辑
    for case in cases:
        run = await execute_eval_case(case)
        score_eval_result(run)
    # 对比新旧版本的平均分，生成报告
```

**新增关联机制**：

`eval_cases` 表加 `agent_id INTEGER` 列，表示该用例属于哪个 agent。在管理页面创建 eval case 时可选择关联 agent。

**前端改动**：
- AdminAgentsPage.vue 保存 prompt 后显示「回归测试中...」状态
- 测试完成后弹出报告：新旧分数对比、通过/退步的用例数
- 如果整体退步，提示「是否回滚到上一版本？」

### 3.5 推荐自动验证

**思路**：给现有的 `auto_verify_pending_recommendations` 加定时触发。

**实现方案**：在 `app.py` 的 startup 事件中加 `asyncio` 定时任务：

```python
@app.on_event("startup")
async def start_recommendation_verifier():
    async def verifier_loop():
        while True:
            try:
                # 每天早上 9 点执行
                now = datetime.now()
                if now.hour == 9:
                    await run_auto_verify()
                await asyncio.sleep(3600)  # 每小时检查一次
            except Exception as e:
                logger.error(f"推荐验证失败: {e}")
                await asyncio.sleep(3600)
    asyncio.create_task(verifier_loop())
```

**复用现有端点**：直接调用 `/api/dashboard/recommendations/auto-verify` 的逻辑（已有完整实现）。

## 五、实施顺序

### 第一批：让闭环能跑起来（核心）

1. **Eval 自动评分** — `eval_scorer.py` + 改造 `run_eval_case_api` + EvalSuitePage 显示分数
2. **Bad Case → Eval 转化** — 新端点 + BadCasePage 加按钮
3. **Prompt 变更回归测试** — agents router 保存后触发 + AdminAgentsPage 展示结果

### 第二批：让系统能自动改进

4. **反馈驱动 Prompt 优化** — `feedback_analyzer.py` + 改造 generate-prompt
5. **推荐自动验证** — startup 定时任务
6. **系统进化仪表盘** — EvolutionDashboard.vue + 聚合 API

## 七、与现有系统的关系

- **不改动** `feedback_learner.py`：用户画像学习保持原样，本方案是**系统层面**的进化，与用户画像学习互补
- **复用** `agent_prompt_versions`：回归测试对比的是版本间的分数差异
- **复用** `llm_feedback` / `portfolio_analysis_records`：bad case 数据源不变
- **复用** `auto_verify_pending_recommendations`：只加定时触发，不改验证逻辑
