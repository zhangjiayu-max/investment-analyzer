# AI 对话多智能体运行优化设计稿

> 日期：2026-06-28
> 背景：用户反馈 `messageId=256` 的 AI 对话一次运行了 8 个专家，感觉“AI 对话里的多智能体不太好”，希望先写设计稿再优化。

## 1. 结论摘要

`messageId=256` 确实运行了 8 个原始专家，不是前端误把交叉审阅算成专家。当前多智能体系统已有 Smart Router 和初始专家数量截断，但这个限制只作用在“初始专家名单”，后续 Orchestrator 的 tool-call 循环和动态追加专家仍然可以继续扩大专家数量，导致一次复杂对话被放大成“全团队会诊”。

建议本次优化不要继续堆更多专家，而是给 AI 对话增加一层明确的“运行预算闸门”：

1. 普通 AI 对话默认最多 2 个原始专家。
2. 复杂投资决策默认最多 3 个原始专家。
3. 只有用户显式选择“深度会诊”或系统触发高风险场景时，才允许 4 到 5 个专家。
4. 动态追加专家默认不直接执行，改为“建议追加”，由预算闸门或用户确认决定。
5. 前端展示区分“原始专家、动态追加、交叉审阅、仲裁”，避免用户看到一个总数后无法判断发生了什么。

## 2. 样本复盘：messageId=256

用户问题：

```text
详细分析下 我近期 有什么 可以加仓 或者建仓的吗 。分析下周有什么利好政策吗
```

数据库证据：

```sql
SELECT id, conversation_id, message_id, agent_key, agent_name, status, duration_ms, created_at
FROM agent_runs
WHERE message_id = 256
ORDER BY id;
```

实际运行的专家：

| agent_key | 专家 | 耗时 |
| --- | --- | ---: |
| valuation_expert | 估值分析师 | 63.2s |
| market_analyst | 择时分析师 | 33.9s |
| macro_strategist | 宏观策略师 | 34.9s |
| allocation_advisor | 资产配置师 | 32.3s |
| risk_assessor | 风险管理师 | 72.6s |
| wealth_advisor | 专属理财顾问 | 61.4s |
| behavior_coach | 行为金融辅导师 | 43.5s |
| counter_argument | 反方观点审查员 | 53.2s |

同一时间窗口的 token 使用：

| caller | 调用次数 | token |
| --- | ---: | ---: |
| specialist:market_analyst | 8 | 97,676 |
| specialist:allocation_advisor | 8 | 80,211 |
| specialist:risk_assessor | 7 | 74,682 |
| orchestrator | 3 | 48,594 |
| specialist:macro_strategist | 4 | 37,498 |
| specialist:valuation_expert | 4 | 34,635 |
| specialist:wealth_advisor | 3 | 29,152 |

| specialist:behavior_coach | 2 | 16,690 |
| embedding_query | 23 | 585 |
| **合计** | **64** | **437,316** |

这个样本说明两个问题：

1. `agent_runs` 看到的是 8 个专家，但 `token_usage` 显示每个专家内部还可能多轮调用 LLM 和工具，真实成本比“8 个专家”更高。
2. 当前系统对“复杂问题”的判断过于宽松，一旦进入 complex，Orchestrator 和动态追加会把用户的一个自然问题拆成过多子任务。

## 3. 当前链路中的放大点

当前流式对话主链路在 `backend/agent/orchestrator.py`：

```text
用户输入
  -> SmartRouter 路由
  -> context_config 限制初始 specialists 数量
  -> Orchestrator LLM tool-call
  -> 并行执行工具调用里的专家
  -> 专家结果返回给 Orchestrator
  -> 动态追加专家 _check_dynamic_spawn()
  -> 必要时交叉审阅
  -> 必要时仲裁
  -> 最终回答
```

当前已有保护：

1. `backend/agent/router.py` 的规则路由已把规则命中的专家截断到最多 3 个。
2. `route_to_specialists_by_keywords()` 已把关键词路由截断到最多 3 个。
3. `get_context_config()` 已对不同复杂度设置 `max_specialists`：simple=1，medium=2，complex=5。

保护失效点：

1. `specialists = specialists[:max_spec]` 只截断了前置变量，没有约束后续 `build_orchestrator_tools()` 暴露给 LLM 的所有专家工具。
2. Orchestrator 每轮仍能通过 tool-call 调用任何 `consult_*` 专家。
3. `MAX_TURNS` 正常模式是 6，复杂问题可能多轮追加。
4. `_check_dynamic_spawn()` 对每个专家结果扫描后可追加专家，且当前设计是自动执行，不需要用户确认。
5. `already_called` 只防止同一个专家重复动态追加，不防止总专家数超过预算。
6. 前端和消息 metadata 将所有专家合并展示，用户难以区分“初始专家、追加专家、复审、仲裁”。

## 4. 设计目标

| 目标 | 验收指标 |
| --- | --- |
| 控制专家数量 | 普通 AI 对话 P95 原始专家数 ≤ 3 |
| 降低成本 | 类似 `messageId=256` 的问题 token 降低 50% 以上 |
| 保留质量 | 自动评估分数不下降，关键决策问题仍能触发反方或风控 |
| 透明可解释 | 每次回答 metadata 明确记录为什么调用这些专家、为什么追加或拒绝追加 |
| 可回退 | 所有硬限制有配置开关，可临时恢复旧流程 |

非目标：

1. 不重写整个多 Agent 架构。
2. 不删除已有专家。
3. 不取消深度分析能力，而是把它从默认行为改成显式模式。
4. 不在本轮直接改代码，本设计稿确认后再进入实施。

## 5. 方案对比

### 方案 A：只关动态追加

做法：

1. 将 `dynamic_spawn_enabled` 默认改为 false。
2. Orchestrator 仍保留原来的 tool-call 自主选专家能力。

优点：

1. 改动最小。
2. 可以立刻减少理财顾问、行为教练、反方观点这类追加专家。

缺点：

1. 不能完全防止 Orchestrator 在多轮 tool-call 中调用过多专家。
2. 对 `messageId=256` 这种初始就复杂的问题，仍可能跑 4 到 5 个专家。

### 方案 B：专家预算闸门，推荐

做法：

1. 引入 `AgentRunBudget`，统一管理 `max_initial_specialists`、`max_total_specialists`、`max_turns`、`allow_dynamic_spawn`。
2. SmartRouter 输出的专家名单成为允许调用的工具白名单。
3. `build_orchestrator_tools()` 支持只暴露允许专家，而不是暴露所有专家。
4. tool-call 执行前做硬校验，超过预算的专家直接跳过并记录 `skipped_by_budget`。
5. 动态追加从“自动执行”改为“建议追加”，只有预算允许且场景匹配才执行。

优点：

1. 能从根上解决专家数失控。
2. 对普通对话、复杂决策、深度会诊可以使用不同预算。
3. 可观测性更强，方便后续调优。

缺点：

1. 需要改 Orchestrator 主链路和测试。
2. 初期可能有少数问题因为专家预算过紧，需要通过 @mention 或深度模式补充。

### 方案 C：场景工作流替代自由编排

做法：

1. 将“加仓建仓”“持仓体检”“文章解读”“政策分析”等场景做成固定 SOP。
2. 每个 SOP 固定专家顺序和输出 schema。
3. Orchestrator 只负责选择 SOP，不再自由 tool-call。

优点：

1. 稳定性最好。
2. 适合投资决策闭环和可审计输出。

缺点：

1. 工程量最大。
2. 对开放式聊天不够灵活。
3. 不适合作为本次快速优化的第一步。

推荐采用方案 B，并为高频决策场景逐步演进到方案 C。

## 6. 目标架构

```text
用户输入
  -> ChatModeClassifier
      - chat：直接回答
      - quick_analysis：最多 1 个专家
      - standard_analysis：最多 2 个专家
      - decision_analysis：最多 3 个专家
      - deep_consultation：最多 5 个专家，需要显式触发
  -> SmartRouter
  -> AgentRunBudget
  -> Tool Allowlist
  -> Orchestrator tool-call
  -> Budget Gate
      - 允许：执行专家
      - 超限：跳过并记录原因
  -> Dynamic Spawn Proposal
      - 默认只记录建议
      - 高风险或用户确认后才执行
  -> Cross Review Gate
  -> Arbitration Gate
  -> Answer + Run Summary
```

核心变化：

1. “路由推荐专家”升级为“本次可调用专家白名单”。
2. “复杂度”升级为“运行模式 + 专家预算”。
3. “动态追加”从执行路径改为提案路径。
4. “交叉审阅/仲裁”不再根据专家数量简单触发，必须存在实质冲突或高风险操作。

## 7. 运行模式设计

| mode | 适用场景 | 初始专家上限 | 总专家上限 | tool-call 轮数 | 动态追加 | 交叉审阅 | 仲裁 |
| --- | --- | ---: | ---: | ---: | --- | --- | --- |
| chat | 问候、解释概念、普通闲聊 | 0 | 0 | 0 | 禁止 | 禁止 | 禁止 |
| quick_analysis | 单一估值、单只基金、简单问题 | 1 | 1 | 1 | 禁止 | 禁止 | 禁止 |
| standard_analysis | 一般投资分析 | 2 | 2 | 2 | 只记录建议 | 有冲突才允许 | 禁止 |
| decision_analysis | 买卖、加仓、减仓、建仓 | 3 | 3 | 3 | 高风险白名单 | 有冲突才允许 | 仅高风险冲突 |
| deep_consultation | 用户明确要求“全面、深度、多专家会诊” | 5 | 5 | 4 | 预算内允许 | 有冲突才允许 | 允许 |

`messageId=256` 这类问题建议归入 `decision_analysis`：

1. 用户问“加仓或建仓”，属于行动建议。
2. 用户同时问“下周利好政策”，需要市场催化分析。
3. 推荐专家应控制在 3 个：`valuation_expert`、`allocation_advisor`、`market_analyst`。
4. `risk_assessor` 可以作为高风险替代或预算内第 3 个专家，不能在 3 个之外继续自动追加。
5. `behavior_coach`、`wealth_advisor`、`counter_argument` 应作为“可追加建议”展示，不默认执行。

## 8. 专家选择策略

### 8.1 默认专家组合

| 场景 | 默认专家 |
| --- | --- |
| 估值查询 | `valuation_expert` |
| 政策和行情 | `market_analyst` |
| 组合配置 | `allocation_advisor` |
| 加仓/建仓/减仓 | `valuation_expert` + `allocation_advisor` + `risk_assessor` |
| 持仓体检 | `allocation_advisor` + `risk_assessor` |
| 文章解读 | `article_expert` |
| 基金复盘 | `fund_analyst` + `risk_assessor` |

### 8.2 冲突时的专家优先级

当关键词命中超过预算时，按场景优先级截断：

1. 行动建议优先：`allocation_advisor`、`risk_assessor`、`valuation_expert`。
2. 政策行情优先：`market_analyst`、`macro_strategist`。
3. 持仓画像优先：`wealth_advisor` 只在用户问“适合我吗、家庭资产、长期规划”时启用。
4. 行为偏差优先：`behavior_coach` 只在用户表达恐慌、冲动、追涨杀跌时启用。
5. 反方观点优先：`counter_argument` 只在用户明确要求“反方、质疑、风险另一面”，或高风险大额操作时启用。

## 9. 动态追加设计

当前动态追加规则存在两个问题：

1. 规则基于专家输出关键词，容易被长文本误触发。
2. 一旦触发就直接执行，没有预算闸门和用户确认。

优化后改为三段式：

```python
proposal = build_spawn_proposal(result, already_called, budget)
decision = decide_spawn(proposal, budget, scenario, user_intent)
execute_spawn(decision)  # 只有 decision.approved == True 才执行
```

执行条件：

1. `budget.allow_dynamic_spawn == true`。
2. `current_total_specialists < budget.max_total_specialists`。
3. 追加专家在当前场景白名单中。
4. 触发原因达到 high priority。
5. 对 standard_analysis 默认不执行，只写入 `spawn_suggestions`。

建议新增 metadata：

```json
{
  "spawn_suggestions": [
    {
      "agent_key": "counter_argument",
      "reason": "检测到大额加仓建议，可补充反方审查",
      "priority": "high",
      "action": "skipped_by_budget"
    }
  ]
}
```

## 10. Orchestrator 工具白名单

当前 `build_orchestrator_tools()` 会把数据库里的所有专家都暴露给 Orchestrator。

建议改为：

```python
def build_orchestrator_tools(allowed_specialists: list[str] | None = None) -> list:
    specialists = load_specialist_agents()
    if allowed_specialists is not None:
        specialists = {k: v for k, v in specialists.items() if k in allowed_specialists}
    ...
```

调用方式：

```python
allowed_specialists = budget.allowed_specialists
tools = build_orchestrator_tools(allowed_specialists)
```

执行前再做一次硬校验：

```python
if agent_key not in budget.allowed_specialists:
    skipped_tasks.append({
        "agent_key": agent_key,
        "reason": "not_in_allowed_specialists"
    })
    continue

if budget.consumed_specialist_count >= budget.max_total_specialists:
    skipped_tasks.append({
        "agent_key": agent_key,
        "reason": "max_total_specialists_exceeded"
    })
    continue
```

这样即使 LLM 产生了额外 tool-call，也不会真的执行。

## 11. 交叉审阅与仲裁策略

当前配置：

```text
cross_review_enabled = true
cross_review_min_specialists = 2
cross_review_trigger = complex
arbitration_enabled = true
arbitration_complexity = medium
```

建议调整：

1. `cross_review_trigger` 统一使用 `disagreement`，不要使用 `complex` 或 `always`。
2. `arbitration_complexity` 默认改为 `complex`。
3. 仲裁必须满足“有实质冲突 + 高风险行动建议 + 至少 2 个原始专家”。
4. 交叉审阅结果必须标记 `is_cross_review=true`，前端单独展示。
5. 交叉审阅不计入“原始专家数”，但计入成本统计。

## 12. 前端展示设计

对用户来说，“跑了几个专家”比内部流程名更重要。前端应展示一份运行摘要：

```text
本次分析模式：决策分析
原始专家：3 位（估值分析师、资产配置师、风险管理师）
跳过专家：2 位（宏观策略师、行为金融辅导师，原因：超过本模式预算）
追加建议：1 位（反方观点审查员，可点击继续补充）
交叉审阅：未触发（未检测到实质冲突）
仲裁：未触发
```

建议事件字段：

```json
{
  "type": "run_summary",
  "mode": "decision_analysis",
  "initial_specialists": ["valuation_expert", "allocation_advisor", "risk_assessor"],
  "executed_specialists": ["valuation_expert", "allocation_advisor", "risk_assessor"],
  "skipped_specialists": [
    {"agent_key": "macro_strategist", "reason": "max_total_specialists_exceeded"}
  ],
  "spawn_suggestions": [],
  "cross_review_count": 0,
  "arbitration_done": false
}
```

## 15. 实施阶段

### Phase 0：设计确认

1. 确认默认运行模式和专家上限。
2. 确认是否允许短期先关闭 `dynamic_spawn_enabled`。
3. 确认前端是否需要展示“跳过专家”和“追加建议”。

### Phase 1：止血优化

1. 将动态追加改为受配置控制，默认关闭或 proposal_only。
2. 将 `cross_review_trigger` 调整为 `disagreement`。
3. 将 `arbitration_complexity` 调整为 `complex`。
4. 在 Orchestrator 执行 tool-call 前加总专家数硬上限。

### Phase 2：预算闸门

1. 新增 `AgentRunBudget` 数据结构。
2. 根据 query、complexity、scenario_type 计算运行模式。
3. 让 `build_orchestrator_tools()` 支持 allowed specialists。
4. 执行前校验 allowed list 和 max total。
5. 记录 skipped 和 spawn proposal 到 metadata。

### Phase 3：前端透明展示

1. ChatView 展示运行摘要。
2. 专家卡片按阶段分组：原始专家、追加专家、交叉审阅、仲裁。
3. 对 `spawn_suggestions` 提供“继续补充分析”入口。

### Phase 4：回归与调优

1. 针对 `messageId=256` 的同类问题建立回归测试。
2. 检查普通估值问题只调用 1 个专家。
3. 检查加仓/建仓问题最多调用 3 个专家。
4. 检查深度会诊显式触发时可调用 4 到 5 个专家。
5. 对比优化前后的 token、耗时、评估分数。

## 17. 风险与回退

| 风险 | 缓解 |
| --- | --- |
| 专家限制过严导致答案不完整 | 提供“继续补充专家分析”按钮，支持 @mention 指定专家 |
| Orchestrator 因工具白名单减少而回答变弱 | system prompt 明确当前预算和可用专家，让它在预算内综合 |
| 高风险操作缺少反方观点 | 对大额买卖、满仓、清仓等关键词允许 `counter_argument` 替换低优先级专家 |
| 用户想要深度分析但默认被限制 | 支持“深度会诊”显式模式 |
| 新预算逻辑影响已有恢复/重试 | `resume_from` 优先读取已完成专家，再用预算判断是否继续 |

回退方式：

```text
agent_budget.enabled = false
orchestrator.tools.use_allowlist = false
dynamic_spawn_enabled = true
```

## 18. 推荐决策

建议按“先止血，再结构化”的方式推进：

1. 先做 Phase 1：关闭或约束动态追加，交叉审阅和仲裁改为冲突触发。
2. 再做 Phase 2：加入 `AgentRunBudget` 和工具白名单，彻底防止专家数失控。
3. 最后做 Phase 3：前端展示运行摘要，让用户知道为什么是这些专家，而不是感觉系统在乱跑。

本设计确认后，再输出详细实施计划，并按测试驱动方式逐步修改代码。
