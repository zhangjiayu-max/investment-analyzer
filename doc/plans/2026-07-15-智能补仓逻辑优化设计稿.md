## 二、修复方案

### 修复1：rhythm_adjust 公式方向（严重）

**位置**：`smart_add_planner.py:411-414, 437-439`

**问题**：
```python
# 当前代码（错误）
if recovery["median_recovery_months"] > 24:
    rhythm_adjust = 0.5  # 注释说"双月投"，但下面是除法 → 金额翻倍
elif recovery["median_recovery_months"] < 12:
    rhythm_adjust = 1.5  # 注释说"半月投"，除以1.5 → 金额减少
final_monthly = base * multiplier * dca_mult * confidence / rhythm_adjust
```

长修复(>24月)时 rhythm=0.5，`base ÷ 0.5 = base × 2` —— 金额翻倍，与"拉长周期、减少月均"意图完全相反。

**修复**：将公式从 `÷ rhythm_adjust` 改为 `× rhythm_adjust`，并调整数值语义：
```python
if recovery["median_recovery_months"] > 24:
    rhythm_adjust = 0.5  # 长修复→月均减半（双月投的月均效果）
elif recovery["median_recovery_months"] < 12:
    rhythm_adjust = 1.5  # 快修复→加大月投
# 公式改为乘法
final_monthly = base_monthly * multiplier * type_strategy["dca_mult"] * confidence_mult * rhythm_adjust
```

同步更新 `engine1.formula` 字符串：`÷` 改为 `×`。

**影响**：机器人/红利/银行等未深套标的的月投从 5444 回落到 1361（合理），深套且长修复标的不再多投。

### 修复2：估值暂停机制（严重）

**位置**：`smart_add_planner.py:456-460`

**问题**：
```python
# 当前条件（错误）
if (
    cfg["pyramid_enabled"]
    and profit_rate_pct <= cfg["loss_threshold"]
    and (not valuation or (valuation.get("percentile") or 0) < cfg["valuation_pause_pct"])
):
```
- `not valuation` 为 True 时直接放行（无指数关联的基金）
- `percentile=None` 时 `(None or 0) < 60` = True 也放行

**修复**：估值数据缺失时**不触发**金字塔（保守原则），除非基金类型为债券（债券不走估值逻辑）：
```python
# 估值暂停判断：有估值数据时检查分位；无估值数据时按类型决定
valuation_ok = True
if valuation and valuation.get("percentile") is not None:
    # 有分位数据：超过暂停阈值则不触发
    valuation_ok = valuation["percentile"] < cfg["valuation_pause_pct"]
elif fund_type not in ("bond",):
    # 无估值数据且非债券：保守不触发（原逻辑是放行，导致高估值标的误补）
    valuation_ok = False
# bond 类型无指数估值是正常的，不因此阻止

if cfg["pyramid_enabled"] and profit_rate_pct <= cfg["loss_threshold"] and valuation_ok:
```

**影响**：8只触发标的中估值分位"无"的权益标的将不再触发金字塔（需先补全估值数据或确认是债券）。这避免了"高估值但数据缺失"的误补。

### 修复3：安全阀拦截（严重）

**位置**：`smart_add_planner.py:498, 465, 306-317`

**问题**：`can_add=False` 时仅设标志，`released_amount` 照常计算，假设交易照常落库。

**修复**：安全阀未通过时，金字塔释放额归零、假设交易不落库：
```python
# 引擎2计算后，安全阀拦截
if engine2 and not can_add:
    engine2["released_amount"] = 0
    engine2["blocked_reason"] = f"已达仓位上限 {max_position_pct:.1f}%，暂停补仓"
    engine2["triggered_tiers"] = 0  # 展示用：标记为未释放

# 反事实落库时检查
if snapshot_enabled:
    for p in plans:
        pyr = p.get("pyramid") or {}
        saf = p.get("safety") or {}
        suggested_amount = pyr.get("released_amount", 0) if pyr else 0
        # 安全阀拦截时不落库
        if suggested_amount > 0 and saf.get("can_add", True):
            ...
```

**影响**：013128/004643/013415/012725 四只达上限标的不再生成建议和假设交易。

### 修复5：小仓位标的补仓上限（严重）

**位置**：`smart_add_planner.py` 引擎2计算后

**问题**：014674(0.3%仓位)建议补 48995 元（原市值20倍）。

**修复**：新增"单标的补仓金额 ≤ 原市值 × 倍数"约束（倍数可配置，默认 2 倍）：
```python

max_add_amount = current_value * cfg["max_add_vs_position_mult"]
if engine2 and engine2["released_amount"] > max_add_amount:
    engine2["released_amount"] = round(max_add_amount, 2)
    engine2["capped_reason"] = f"受原市值{cfg['max_add_vs_position_mult']}倍上限约束"

**影响**：014674 的建议从 48995 降至 ~4800（原市值×2）；012725 从 18373 降至 ~11400。

**db/config.py 新增默认配置**：

```

### 修复6：资金池硬约束分配（严重）

**位置**：`smart_add_planner.py:239-280` 组合视角汇总

**问题**：各标的独立计算，需求 30.6 万 vs 池 12.2 万，超额 150% 但各标的不缩减。

**修复**：按优先级排序后，等比缩减各标的释放额至池总额内：
```python
# 1. 按优先级排序深套标的（已有排序逻辑）
# 2. 计算总需求
pool_demand = sum(p["pyramid"]["released_amount"] for p in deep_loss_plans)
# 3. 若超额，按优先级等比缩减（保留优先级高的相对占比）
if pool_demand > pool_total and pool_demand > 0:
    scale = pool_total / pool_demand
    for p in deep_loss_plans:
        pyr = p["pyramid"]
        original = pyr["released_amount"]
        pyr["released_amount"] = round(original * scale, 2)
        pyr["scaled_from"] = original  # 展示用：原始建议额
        pyr["scale_reason"] = f"资金池不足，按{scale:.0%}缩减"
```

**影响**：8只标的的释放额按优先级等比缩减至合计 12.2 万。前端可展示"原始建议/实际分配"对比。

### 修复8：反事实 improvement 口径（低）

**位置**：`counterfactual_verifier.py:226`

**问题**：对比"假设部分独立收益率" vs "真实组合整体收益率"，非真正组合对比。

**修复**：改为模拟"真实盈亏+假设盈亏"与"真实成本+假设投入"的对比：
```python
# 原口径（不严谨）
improvement = hypothetical_profit_rate - real_portfolio_profit_rate

# 新口径
combined_profit = real_profit_loss + hypothetical_profit_loss
combined_cost = real_total_cost + hypothetical_total_invested
combined_profit_rate = combined_profit / combined_cost if combined_cost else 0
improvement = combined_profit_rate - real_portfolio_profit_rate
```

**影响**：反事实面板的"改善幅度"更准确地反映"如果当时补了的整体效果"。

### 修复9：engine1 月投假设交易语义（低）

**位置**：`smart_add_planner.py:288-317` + `smart_add_snapshots.py`

**问题**：engine1 输出月投额（recurring），却落库为一笔一次性 buy。

**修复方案**：本期暂不改为"每月一笔"（会大幅增加假设交易量和复杂度），改为**仅金字塔触发的标的落库假设交易**，engine1 月投不落库：
```python
# 反事实落库条件收紧：仅金字塔释放额>0 且安全阀通过
if snapshot_enabled:
    for p in plans:
        pyr = p.get("pyramid") or {}
        saf = p.get("safety") or {}
        suggested_amount = pyr.get("released_amount", 0)
        # 仅金字塔建议落库，engine1 月投不再生成假设交易
        if suggested_amount > 0 and saf.get("can_add", True):
            create_snapshot_with_hypothetical(...)
```

**影响**：假设交易数量减少（仅深套+金字塔触发+安全阀通过的标的），语义更清晰——验证的是"金字塔补仓决策"的效果，而非定投。
