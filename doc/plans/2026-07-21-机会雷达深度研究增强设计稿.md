# 机会雷达深度研究增强设计稿

**日期**：2026-07-21
**背景**：调研发现机会雷达"上车"判断存在 3 个盲区：①政策解读是字符串匹配不理解实质；②无 LLM 深度推理无法权衡复杂情境；③回测验证偏宽松（15 交易日 ≥3%，牛市普涨无法区分信号有效）。

## 一、现状与问题

### 1.1 当前实现

机会雷达 [opportunity_engine.py:327-425](file:///Users/xiaoyuer/projects/investment-analyzer/backend/services/advisor/opportunity_engine.py) 的 `_score_theme` 是纯 if-else 规则系统：
- 9 维评分（新闻+政策+估值+持仓+可交易性+技术+资金+情绪+基础分）
- `score >= 75` → can_buy；`>= 50` → watch；否则 avoid
- 估值 >80% 一票否决、>60% 禁 can_buy

### 1.2 三个核心盲区

**盲区 1：政策解读是字符串匹配**

[opportunity_engine.py:356-358](file:///Users/xiaoyuer/projects/investment-analyzer/backend/services/advisor/opportunity_engine.py)：
```python
policy_text = " ".join(f"{n.get('title','')} {n.get('summary','')}" for n in news_hits)
score += 12 if _contains_any(policy_text, theme_rule.get("policy_terms", [])) else 5
```

问题：新闻里出现"半导体"就加分，但不理解政策实质利好还是利空、力度强弱、落地概率。

**盲区 2：无 LLM 深度推理**

遇到"政策利好但估值已高 + 资金流出"时，规则只能机械加减分。无法像专家那样权衡"这个政策利好是否足以覆盖估值风险"。

**盲区 3：回测验证偏宽松**

[db/opportunities.py:556-569](file:///Users/xiaoyuer/projects/investment-analyzer/backend/db/opportunities.py) 的 `list_pending_backtests` + `review_opportunity_backtests`：15 交易日 ≥3% 视为命中。在牛市里几乎所有信号都能命中，无法区分"信号有效"和"大盘普涨"。

## 二、修复方案

### 2.1 策略：三层增强 + 严格回测

- **L1 政策解读 LLM 化**：对 can_buy/watch 候选机会，调用 LLM 做政策实质解读
- **L2 深度推理评审**：对 can_buy 候选，调用 LLM 做多维度权衡评审
- **L3 回测基准化**：回测时引入大盘基准，超额收益才算命中
- **全部独立开关**：LLM 相关开关默认 false，符合硬约束

### 2.2 L1 — 政策解读 LLM 化（opportunity_engine.py）

**新增函数**：`_llm_policy_analysis(theme_rule, news_hits) -> dict`

对 `score >= 50`（watch/can_buy 候选）的机会，调用 LLM 做政策解读：
- 输入：主题关键词 + 命中的新闻标题/摘要
- 输出 JSON：
  ```json
  {
    "policy_substance": "strong|weak|neutral",  // 政策实质力度
    "beneficiary_alignment": "high|medium|low",  // 与主题受益契合度
    "implementation_probability": "high|medium|low",  // 落地概率
    "key_risk": "估值过高/业绩兑现存疑/...",
    "reasoning": "30-80字解读"
  }
  ```
- 评分调整：policy_substance=strong 且 beneficiary_alignment=high → +8 分；weak/low → -5 分
- 数据源：`services.llm_service._call_llm`，model 用 `deepseek-v4-flash`（文本类用便宜模型，符合 cost_routing 约束）

**开关**：`opportunity.llm_policy_analysis_enabled`（默认 false）

### 2.3 L2 — 深度推理评审（opportunity_engine.py）

**新增函数**：`_llm_deep_review(item, valuation, tech_signal, capital_signal, sentiment_signal) -> dict`

对 `verdict == "can_buy"` 的候选，调用 LLM 做最终深度评审：
- 输入：主题 + 估值分位 + 技术信号 + 资金信号 + 情绪信号 + 持仓重叠
- 输出 JSON：
  ```json
  {
    "final_verdict": "can_buy|watch|avoid",  // LLM 评审结论
    "confidence": "high|medium|low",
    "key_pros": ["..."],  // 看多理由
    "key_cons": ["..."],  // 看空理由
    "net_assessment": "50-150字综合权衡",
    "timing_note": "最佳入场时机判断"
  }
  ```
- 评审结论可降级（can_buy → watch/avoid），不可升级（避免 LLM 过度乐观）
- 评审理由写入机会卡的 `llm_review` 字段，前端展示

**开关**：`opportunity.llm_deep_review_enabled`（默认 false）

### 2.4 L3 — 回测基准化（db/opportunities.py + opportunity_engine.py）

**问题**：原逻辑"15 交易日 ≥3% 视为命中"在牛市普涨时几乎所有信号都命中。

**修复**：引入大盘基准（沪深300），超额收益才算命中。

- `review_opportunity_backtests` 修改：
  - 原逻辑：`change_pct >= 3` → hit=1
  - 新逻辑：`change_pct - benchmark_pct >= 2` → hit=1（超额收益 ≥2%）
  - benchmark_pct 从 akshare 取沪深300同期涨幅
  - 新增字段：`benchmark_pct`、`excess_return`（change_pct - benchmark_pct）

**回测表结构变更**（`theme_opportunity_backtests`）：
```sql
ALTER TABLE theme_opportunity_backtests ADD COLUMN benchmark_pct REAL;
ALTER TABLE theme_opportunity_backtests ADD COLUMN excess_return REAL;
```

**开关**：`opportunity.benchmark_backtest_enabled`（默认 true，非 LLM 相关）

### 2.5 评分维度调整

L1/L2 的 LLM 评分调整是叠加在原 9 维评分之上的，不替换原规则：
- 原 9 维评分仍然计算（保证基线稳定）
- L1 政策解读：±5~8 分微调
- L2 深度评审：可降级 verdict，不改变 score
- L3 回测基准化：只影响历史命中率反哺，不影响当前评分

## 四、风险评估

### 4.1 成本控制

- L1 政策解读：每次扫描最多 8 个机会卡 × 1 次 LLM 调用 = 8 次/天
- L2 深度评审：仅 can_buy 候选（通常 1-2 个）× 1 次 LLM 调用 = 1-2 次/天
- 模型用 `deepseek-v4-flash`（文本类便宜模型），单次成本极低
- **默认关闭**：不开启时零成本

### 4.2 兼容性

- 原 9 维评分逻辑完全保留，L1/L2 是叠加微调
- L3 回测基准化有开关，可回退原逻辑
- 数据库字段新增用 ALTER TABLE，不破坏现有数据
- 前端 `llm_review` 字段缺失时不显示评审区块

### 4.3 回归风险

- **风险点 1**：LLM 政策解读可能误判（把利空当利好）
  - **缓解**：L1 只做 ±5~8 分微调，不直接决定 verdict；L2 只能降级不能升级
- **风险点 2**：LLM 调用增加机会雷达延迟
  - **缓解**：L1/L2 都有超时和失败兜底，失败时保留原规则结果
- **风险点 3**：回测基准化使命中率下降
  - **这是期望行为**：原逻辑牛市普涨假命中是 bug
