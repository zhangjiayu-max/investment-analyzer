# 防退步机制增强设计稿

**日期**: 2026-07-16
**目标**: 补齐 370 式退步的根因——评估结果不回流路由、回归测试不阻断上线
**原则**: ③每次升级是增强非退步

## P0-1: 路由接入 eval 分数（低分专家降权）

### 问题
router.py 是纯关键词路由，grep `eval|score|weight` 零命中。专家连续低分不影响路由权重，照常被调用。

### 设计
**数据来源**: `agent_runs` 表 (agent_key + message_id) JOIN `conversation_evaluations` 表 (message_id + auto_score)

**新增函数** `db/eval.py: get_agent_eval_scores(days=7) -> dict`
```sql
SELECT ar.agent_key,
       AVG(ce.auto_score) as avg_score,
       COUNT(ce.auto_score) as eval_count
FROM agent_runs ar
JOIN conversation_evaluations ce ON ar.message_id = ce.message_id
WHERE ar.status = 'completed'
  AND ce.auto_score IS NOT NULL
  AND ce.created_at >= datetime('now', '-{days} days')
GROUP BY ar.agent_key
HAVING eval_count >= 3  -- 样本不足不参与降权
```

**路由层改造** `router.py: SmartRouter._adjust_by_eval_scores`
- 规则路由产出 specialists 后调用
- avg_score < 60 且 eval_count >= 3 的专家：降级（移到列表末尾，超限时优先淘汰）
- 不直接删除（避免误杀），仅调优先级
- 新增开关 `router.eval_aware_enabled` 默认 true（非 LLM 开关，是路由逻辑）
- 新增阈值 `router.eval_low_score_threshold` 默认 60

**降权策略**（温和，不硬删）:
```
specialists = [s for s in specialists if not (eval_scores.get(s, {}).avg < threshold and eval_scores.get(s, {}).count >= 3)]
+ 追加被降权专家到末尾（如果未超 max_specialists 上限）
```
实际实现：排序时低分专家 priority +100（排到最后），截断时自然被淘汰。

### 影响面
- router.py: 新增 `_adjust_by_eval_scores` 方法，在 `route()` 中规则路由后调用
- db/eval.py: 新增 `get_agent_eval_scores` 函数
- 不影响现有路由缓存（eval 分数独立查询，结果合并到路由决策）

## P0-2: 回归测试持久化 + 阻断门禁

### 问题
regression.py:11 `_regression_results` 仅存内存，重启清空；只报告 improved/degraded，不阻止上线。

### 设计

**1. 持久化表** `db/eval.py: init_eval_tables` 新增:
```sql
CREATE TABLE IF NOT EXISTS prompt_regression_results (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    agent_id INTEGER NOT NULL,
    agent_type TEXT DEFAULT 'conversation',
    status TEXT NOT NULL,           -- completed / error
    total_cases INTEGER DEFAULT 0,
    improved INTEGER DEFAULT 0,
    degraded INTEGER DEFAULT 0,
    unchanged INTEGER DEFAULT 0,
    cases_detail TEXT DEFAULT '[]', -- JSON: 每个case的详情
    created_at TEXT DEFAULT (datetime('now','localtime'))
)
```

**2. regression.py 改造**:
- `set_regression_result`: 同时写内存缓存 + DB 持久化
- `get_regression_result`: 优先读内存，fallback 读 DB 最近一条
- 新增 `get_latest_regression_from_db(agent_id, agent_type)`: 从 DB 取最近结果

**3. 阻断门禁** `routers/admin/agents.py`:
- prompt 变更触发回归测试后，**不阻断本次更新**（prompt 已生效，阻断会造成体验断裂）
- 但在 API 返回中附加 `regression_warning` 字段，前端展示警示
- 回归测试完成后（异步），若 degraded > 0 且 degraded/total > 30%，自动创建 `improvement_tasks` 记录，标记需人工确认
- 不硬阻断原因：回归测试依赖 eval cases，若 cases 不足或 LLM 评分有偏差，硬阻断会误伤。警示 + 任务跟踪是更合理的平衡。

### 影响面
- db/eval.py: 新增表 + `save_regression_result` / `get_latest_regression_result` 函数
- agent/eval/regression.py: `set/get_regression_result` 增加 DB 持久化
- routers/admin/agents.py: API 返回附加 regression 状态
