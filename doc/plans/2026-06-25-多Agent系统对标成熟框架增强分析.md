## 二、当前架构优势（不宜改动）

1. **Orchestrator 中心协调** — 星型拓扑在投资分析场景天然合理。投资建议需要统一裁决，不应该让 Agent 自己开会决定。AutoGen 的 group chat 模式适合创意讨论，不适合金融决策。

2. **规则+LLM 混合路由** — 比 LangGraph 的纯 conditional edge 和 AutoGen 的纯 LLM speaker selection 都更高效。70% 查询跳过 LLM。

3. **交叉审阅+仲裁** — 这是本系统独有的，LangGraph/CrewAI/AutoGen 都没有三层质量保障。

4. **7 层上下文注入** — CrewAI 的 memory 体系更细粒度，但本系统的上下文注入在投资场景下已经覆盖了关键信息。

## 三、值得吸收的 6 个增强点

### 增强 1：状态机检查点（LangGraph 风格）🟡 P1

**当前问题：** complex 场景（5 专家 + 交叉审阅 + 仲裁）耗时 3-5 分钟。如果中途浏览器刷新、网络断开、或 LLM 超时，整个分析丢失，需要从头开始。

**LangGraph 的做法：** 每个节点执行后自动 checkpoint 状态（消息列表 + 中间结果），恢复时从 checkpoint 继续。

**可吸收的方案（轻量版）：**

```python
# orchestrator.py 内联，不引入外部依赖

class OrchestrationState:
    """编排状态检查点"""
    def __init__(self, conv_id, message_id):
        self.conv_id = conv_id
        self.message_id = message_id
        self.phase = "init"           # init/routing/experts/cross_review/arbitrate/final
        self.completed_experts = []   # [{agent_key, analysis}]
        self.pending_experts = []     # 尚未执行的专家列表
        self.cross_review_results = []
        self.arbitration_done = False
    
    def checkpoint(self):
        """保存状态到 DB"""
        conn = _get_conn()
        conn.execute("""
            INSERT OR REPLACE INTO orchestration_checkpoints
            (conv_id, message_id, phase, state_json, created_at)
            VALUES (?, ?, ?, ?, datetime('now','localtime'))
        """, (self.conv_id, self.message_id, self.phase, 
              json.dumps(self.to_dict(), ensure_ascii=False)))
        conn.commit()
        conn.close()

    @classmethod
    def restore(cls, conv_id, message_id):
        """从 DB 恢复状态"""
        conn = _get_conn()
        row = conn.execute("""
            SELECT state_json FROM orchestration_checkpoints
            WHERE conv_id=? AND message_id=?
            ORDER BY created_at DESC LIMIT 1
        """, (conv_id, message_id)).fetchone()
        conn.close()
        if row:
            return cls.from_dict(json.loads(row["state_json"]))
        return None
```

**关键检查点位置：**
- Phase A 专家全完成 → checkpoint
- Phase B 交叉审阅完成 → checkpoint
- Phase A 进行中、单个专家完成 → 可选 checkpoint（权衡 DB 写入开销）

**收益：** 复杂分析中断后可恢复到最近检查点，不需要重新调用已完成的专家（省 60-180s + 省 Token）。

**实现代价：** 3h，新增一张 `orchestration_checkpoints` 表。

### 增强 2：动态 Agent 选择（AutoGen 风格）🟡 P1

**当前问题：** Orchestrator 在第一轮就确定需要哪些专家，之后不再调整。如果估值专家分析过程中发现"需要看看风险维度"，无法发起新的专家调用。

**可吸收的方案：**

```python
# 扩展 orchestrate() 的多轮循环

for turn in range(MAX_TURNS):
    response = _call_llm(messages=llm_messages, tools=TOOLS, ...)
    msg = response.choices[0].message
    
    if msg.tool_calls:
        # 现有多轮逻辑
        for tc in msg.tool_calls:
            if tc.function.name.startswith("consult_"):
                # 执行专家
                result = _execute_specialist(tc.function.name, query, ...)
                specialist_results.append(result)
                # 🆕 动态检测：是否需要更多专家？
                if _should_spawn_more_experts(result):
                    # 在 assistant 消息中追加提示
                    suggestion = _suggest_additional_experts(result)
                    llm_messages.append({
                        "role": "system",
                        "content": f"[自动检测] 前一专家分析发现可能需要: {suggestion}"
                    })
```

**何时要动态新增：**
- 估值专家分析结果中出现"风险"关键词超过 3 次 → 自动提议调用 risk_assessor
- 配置专家发现仓位偏离 > 15% → 自动提议调用 allocation_advisor
- 分析涉及多市场（A 股+H 股+美股）→ 自动提议调用 market_analyst

**实现代价：** 2h，在 orchestrator 的多轮循环末尾加入检测逻辑。

### 增强 3：人在回路（LangGraph 风格）🟢 P2

**当前问题：** Orchestrator 完全自主决策。如果它判断"复杂度 = complex"但实际只需要 medium，用户无法干预。如果它调用了 5 个专家但用户只想快点看结果，无法打断。

**可吸收的方案（关键决策点暂停）：**

```python
# 在以下时机暂停，等待用户确认：
HUMAN_IN_LOOP_CHECKPOINTS = {
    "routing": {
        "condition": lambda: complexity in ("medium", "complex") and confidence < 0.8,
        "question": "系统判断此问题复杂度为{complexity}（置信度{confidence}），
                     计划调用{len(specialists)}个专家。是否继续？[继续/简化为快速分析]",
    },
    "action_proposal": {
        "condition": lambda: analysis_contains_buy_sell_recommendation,
        "question": "分析结果包含买卖建议。是否需要先保存为决策草案再继续？[保存并继续/直接继续]",
    },
}
```

**前端实现：** SSE 流中新增 `"type": "confirm"` 事件，前端渲染确认按钮。用户 30s 不响应 → 默认继续。

**收益：** 给用户掌控感，尤其在涉及具体买卖建议时。

**实现代价：** 3h（后端暂停机制 + 前端确认 UI）。

### 增强 4：分析模板 SOP（MetaGPT 风格）🟡 P1

**当前问题：** 虽然 `ANALYSIS_TEMPLATES` 约束了输出格式，但缺少**预定义的流程模板**。用户问"帮我做完整的持仓诊断"，Orchestrator 需要自己决定先做什么后做什么，可能顺序不对。

**MetaGPT 的做法：** 预定义 SOP（Standard Operating Procedure），如 PM → Architect → Engineer → QA。

**可吸收的方案（投资分析领域 SOP）：**

```python
SOP_TEMPLATES = {
    "full_portfolio_diagnosis": {
        "name": "完整持仓诊断",
        "steps": [
            {"agent": "valuation_expert", "task": "评估所有持仓当前估值水位", "order": 1},
            {"agent": "risk_assessor", "task": "基于估值结果评估风险暴露", "order": 2},
            {"agent": "allocation_advisor", "task": "基于估值和风险评估配置合理性", "order": 3},
            {"agent": "behavioral_coach", "task": "从行为金融角度审视当前配置", "order": 4},
        ],
        "cross_review": True,
        "arbitration": True,
    },
    "buy_decision_checklist": {
        "name": "买入决策检查清单",
        "steps": [
            {"agent": "valuation_expert", "task": "判断标的估值是否合理", "order": 1},
            {"agent": "risk_assessor", "task": "评估买入后组合风险变化", "order": 1},  # 并行
            {"agent": "allocation_advisor", "task": "计算买入后仓位是否超限", "order": 2},  # 依赖前两个
        ],
        "parallel_groups": [[0, 1], [2]],  # 第1组并行(估值+风险)，第2组串行(配置)
        "cross_review": False,
        "arbitration": False,
    },
}
```

**触发方式：**
- 用户明确说"完整诊断"/"全面分析" → 匹配 SOP
- Orchestrator 检测到问题复杂度 + 已有持仓 → 自动选择 SOP
- @mention 指定特定 SOP（`@diagnosis`）

**收益：** 重要决策场景有固定流程保证，不会遗漏关键步骤。

**实现代价：** 3h，新增 `SOP_ENGINE` + 集成到 orchestrator。

### 增强 5：实体记忆（CrewAI 风格）🟢 P2

**当前问题：** 对话记忆是逐条消息的。用户上次分析了沪深 300，下次再问时 AI 不记得"你上次说沪深 300 PE 分位 25%，现在涨到 30% 了"。

**CrewAI 的做法：** 三种独立记忆——短期(最近消息)、长期(跨对话摘要)、实体(具体标的的属性变化)。

**可吸收的方案（轻量版，利用现有 entity_mentions 表）：**

```python
# db/knowledge_graph.py 已有 add_entity_mention
# 改造为自动记录关键实体

def record_entity_snapshot(agent_key, analysis_text):
    """从分析结果中提取实体并记录属性快照。"""
    entities = _extract_entities_from_text(analysis_text)
    for entity in entities:
        # 记录说到"沪深300 PE 25%"这个事实
        add_entity_mention(
            entity_name=entity["name"],
            entity_type="index" | "fund" | "sector",
            mention_type="valuation" | "risk" | "recommendation",
            snapshot={"pe_percentile": entity.get("pe"), 
                      "date": today, 
                      "source": agent_key},
        )
```

**使用时：**
```python
# build_conversation_context() 新增
def _build_entity_memory_context(limit=5):
    """获取近期讨论过的实体及其属性变化。"""
    mentions = get_recent_entity_mentions(days=30, limit=limit)
    lines = ["### 近期讨论的标的"]
    for m in mentions:
        lines.append(f"- {m['entity_name']}: "
                     f"上次分析时{m['snapshot']} "
                     f"（{m['created_at'][:10]}）")
    return "\n".join(lines)
```

**收益：** AI 能说"你上次分析沪深 300 时 PE 分位是 25%，现在是 30%，估值略有回升但仍低于历史中位数"——这种上下文感知是用户体验的分水岭。

**实现代价：** 2h，改造 `entity_mentions` 表 + 上下文注入。

### 增强 6：成本感知路由 🔵 P3

**当前问题：** 所有专家都用 `MODEL`（DeepSeek V4-Pro），即使是简单的"查询估值数据"也要用大模型。每次 complex 分析烧掉大量 Token。

**可吸收的方案：**

```python
# 按专家类型分配合适的模型
AGENT_MODEL_MAP = {
    "valuation_expert": "deepseek-v4-flash",   # 估值计算类，flash 够用
    "risk_assessor": "deepseek-v4-flash",      # 风险评估类
    "allocation_advisor": "deepseek-v4-flash", # 配置计算类
    "market_analyst": "deepseek-v4-pro",       # 市场解读需要深度推理
    "behavioral_coach": "deepseek-v4-pro",     # 行为金融需要深度推理
}

# Orchestrator（路由）: deepseek-v4-pro
# 仲裁: deepseek-v4-pro  # 已通过 arbitration.model 配置
# 交叉审阅: deepseek-v4-flash  # 审阅不需要深度推理
```

**收益：** complex 场景 Token 成本降低 40-60%，同时保持关键步骤的质量。

**实现代价：** 1h，修改 `_execute_specialist` 传递 model 参数。

## 四、优先级排序

```
收益
 高  │  ① 状态检查点      ② 动态选角  
     │  (防丢失)          (自适应)
     │  
     │  ④ 分析SOP         ⑤ 实体记忆
     │  (流程标准化)      (跨对话记忆)
 中  │  
     │  ③ 人在回路        ⑥ 成本路由
     │  (用户掌控感)      (省钱)
 低  │
     └──────────────────────────────
        易 ──── 实现难度 ────→ 难
```

| 优先级 | 增强 | 收益 | 工作量 | 依赖 |
|--------|------|------|--------|------|
| **P1** | 状态检查点 | 中断恢复，复杂分析不丢失 | 3h | 无 |
| **P1** | 动态 Agent 选择 | 分析自适应，不遗漏关键维度 | 2h | 无 |
| **P1** | 分析 SOP 模板 | 重要决策有固定流程 | 3h | 无 |
| **P2** | 实体记忆 | 跨对话记住标的属性变化 | 2h | knowledge_graph |
| **P2** | 人在回路 | 关键决策用户确认 | 3h | 前端 SSE |
| **P3** | 成本感知路由 | Token 成本降 40-60% | 1h | 无 |

## 五、一句话

本系统的多 Agent 架构在**质量保障**（交叉审阅+仲裁）和**效率**（规则路由+并行）上已经超过大多数框架的原生实现。差距主要在**鲁棒性**（无检查点，中断丢失）和**自适应性**（静态路由，无法动态调整）。前 3 项增强（检查点+动态选角+SOP）补上后，就是一套能和 LangGraph+AutoGen 正面比较的多 Agent 系统。
