> 日期：2026-07-14
> 目标：按功能模块重组前后端目录，提升可维护性

## 一、现状问题

### 后端
| 目录 | 文件数 | 问题 |
|------|--------|------|
| `routers/` 根 | 41 个 | 过于扁平，难以导航 |
| `services/` | 48 个 | 过于扁平，难以定位服务 |
| `agent/` | 44 个 | 过于扁平，职责边界模糊 |
| `db/` | 38 个 | 基本合理但部分可分组 |

### 前端
| 目录 | 文件数 | 问题 |
|------|--------|------|
| `components/` 根 | 53 个 .vue | 过于扁平，已有6个子目录但大部分组件未归类 |

## 三、实施策略

### 兼容性保障
1. 每个重组目录的 `__init__.py` 保持向后兼容的重导出
2. 旧的 `from services.llm_service import xxx` 仍可用
3. 新的 `from services.llm.llm_service import xxx` 也可用
4. 逐步迁移 import，不强制一刀切

### 实施顺序
1. **Phase 1（前端）**：风险最低，改完即可验证
2. **Phase 2（routers）**：中等风险，需更新路由注册
3. **Phase 3（services）**：中等风险，靠 `__init__.py` 兼容
4. **Phase 4（agent）**：最高风险，最后实施

### 验证方式
- 每个 Phase 完成后启动后端 + 构建前端，确认无报错
- 运行 pytest 确认测试通过
- 访问关键页面确认功能正常
