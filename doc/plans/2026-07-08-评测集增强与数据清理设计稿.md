## 二、设计方案

### 2.1 样式层升级（解决 S1-S10）

#### 2.1.1 统一按钮体系

**目标**：消除三套按钮体系，统一为编辑式金融终端风格。

**方案**：
- ConversationEvalCard 的 BEM 按钮（`__btn-eval`/`__btn-llm`/`__btn-submit`）→ 改用全局 `btn-primary`/`btn-secondary` + `btn-sm`
- 移除硬编码颜色 `#1890ff`/`#667eea`/`#764ba2`，改用 `var(--color-primary)` / `var(--color-primary-600)`
- BadCasePage 的 `.btn-link` → 改用全局 `btn-ghost btn-sm`
- 删除两个页面重复定义的 `.btn-sm`/`.btn-xs`，统一用全局尺寸

**按钮视觉层级规范**：
| 层级 | 用途 | 类名 | 数量限制 |
|------|------|------|----------|
| 主操作 | 运行、保存、提交 | `btn-primary` | 每个区块 ≤1 个 |
| 次操作 | 编辑、详情、分析 | `btn-secondary` | 不限 |
| 幽灵操作 | 刷新、关闭、取消 | `btn-ghost` | 不限 |
| 危险操作 | 删除 | `btn-danger` | 独立确认 |

### 2.2 功能层升级（解决 F1-F8，增强 agent 能力）

#### 2.2.1 统一评测入口与评分尺度（F1+F2）

**现状**：三套入口、三种分制（10/25/10-4维度），数据不可比。

**方案**：统一为 **10 分制 4 维度**，归一化存储。

```
eval_runs 表新增字段：
  score_data_accuracy   REAL    -- 数据准确性 0-10
  score_logic           REAL    -- 逻辑性 0-10
  score_actionability   REAL    -- 可操作性 0-10
  score_risk_awareness  REAL    -- 风险意识 0-10
  score_normalized      REAL    -- 归一化总分 0-100（四维度平均 ×10）
```

**迁移**：
- 现有 `score` 字段保留为兼容字段，新数据同时写 `score` 和 `score_normalized`
- 旧数据按 `score_normalized = score * 4.17`（24→100）批量回填
- `conversation_evaluations.auto_score` 已是 0-100，无需转换
- `user_score` 保持 1-5 制，前端展示时 ×20 归一化到 0-100

**统一入口**：废弃 `/api/eval-system/run`（同步模拟），所有评测统一走 `/api/eval/cases/{id}/run`（异步真实）+ `/api/eval/auto-regression`（批量真实）。`eval_system` 路由仅保留 A/B 对比和 Prompt 版本管理。

#### 2.2.2 引入 eval_suites 测试套件（F3）

**现状**：用例只能按 analysis_type 隐式分组。

**方案**：新增 `eval_suites` 表 + `eval_suite_cases` 关联表。

```sql
CREATE TABLE eval_suites (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,              -- 套件名（如"估值分析回归集"）
    description TEXT,
    suite_type TEXT DEFAULT 'regression',  -- regression/smoke/ab_test
    is_active INTEGER DEFAULT 1,
    created_at TEXT,
    updated_at TEXT
);

CREATE TABLE eval_suite_cases (
    suite_id INTEGER REFERENCES eval_suites(id) ON DELETE CASCADE,
    case_id INTEGER REFERENCES eval_cases(id) ON DELETE CASCADE,
    sort_order INTEGER DEFAULT 0,
    PRIMARY KEY (suite_id, case_id)
);
```

**API**：
- `POST /api/eval/suites` — 创建套件
- `GET /api/eval/suites` — 列出套件
- `POST /api/eval/suites/{id}/cases` — 添加用例到套件
- `DELETE /api/eval/suites/{id}/cases/{case_id}` — 移除用例
- `POST /api/eval/suites/{id}/run` — 运行整个套件

**前端**：EvalSuitePage 新增"套件"tab，可创建套件、拖拽用例进套件、一键运行套件。

#### 2.2.3 分页能力（F4）

**现状**：所有 list 接口只有 limit。

**方案**：统一加 `page` + `page_size` 参数，返回 `{items, total, page, page_size}`。

涉及接口：
- `GET /api/eval/cases?page=1&page_size=20`
- `GET /api/eval/runs?page=1&page_size=20`
- `GET /api/eval/conversation-list?page=1&page_size=20`
- `GET /api/eval/suggestions?page=1&page_size=20`

前端列表底部加分页器（上一页/下一页/页码）。

#### 2.2.4 BadCase 根因闭环（F5）

**现状**：`suggestion` 存了但没消费，闭环断裂。

**方案**：新增"根因 → 改进任务"自动转化。

```sql
CREATE TABLE improvement_tasks (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_type TEXT NOT NULL,       -- bad_case / root_cause / eval_degradation
    source_id INTEGER,
    agent_type TEXT,                 -- 需要改进的 agent
    root_cause TEXT,
    suggestion TEXT,                 -- 来自根因分析
    prompt_diff TEXT,                -- 建议的 prompt 修改（LLM 生成）
    status TEXT DEFAULT 'pending',   -- pending/applied/rejected
    created_at TEXT,
    applied_at TEXT
);
```

**流程**：
1. 根因分析产出 `suggestion` → 自动写入 `improvement_tasks`
2. 高严重度（critical/high）根因 → LLM 生成 `prompt_diff`（建议修改）
3. 前端新增"改进任务"列表，用户可查看 suggestion + prompt_diff
4. 用户点击"应用" → 复制到 Prompt 版本管理，创建新版本 → A/B 测试验证 → 上线

**API**：
- `GET /api/eval/improvement-tasks` — 列出改进任务
- `POST /api/eval/improvement-tasks/{id}/apply` — 应用（创建 prompt 版本）
- `POST /api/eval/improvement-tasks/{id}/reject` — 拒绝

#### 2.2.5 评测稳定性提升（F8）

**现状**：同一用例跑 9 次得分 4.0→20.0，波动大。

**方案**：
1. **多次采样取中位数**：`auto-regression` 对每条用例跑 3 次，取中位数而非单次
2. **温度参数固定**：评测时 LLM `temperature=0`（已部分实现，确认所有路径）
3. **缓存 RAG 上下文**：同一用例的 RAG 检索结果缓存 1 小时，减少检索波动

### 2.3 数据层清理（解决 D1-D7）

#### 2.3.1 用例去重（D1）

**策略**：按 `input_params` 去重，保留每组最早的一条。

```sql
-- 识别重复（68 组重复，124 → 68 条）
SELECT input_params, COUNT(*) as cnt, MIN(id) as keep_id, MAX(id) as max_id
FROM eval_cases
GROUP BY input_params
HAVING cnt > 1
ORDER BY cnt DESC;

-- 删除重复（保留 keep_id）
DELETE FROM eval_cases
WHERE id NOT IN (
    SELECT MIN(id) FROM eval_cases GROUP BY input_params
);
```

**特殊处理**：
- `✅ quality_evaluator-高分(8.3)` 重复 18 次 → 保留 1 条
- BadCase 重复 3 次 → 保留 1 条
- 同时清理 `eval_runs` 中 `case_id` 指向被删用例的记录

**预期结果**：124 条 → 约 68 条

#### 2.3.2 BadCase 补充期望输出（D2）

**策略**：53 条 `❌ BadCase` 的 `expected_quality` 为空，用 LLM 批量生成。

```python
# 伪代码
for case in bad_cases_without_expected:
    prompt = f"""
    这是一个 Bad Case（用户反馈不好的对话）。
    输入：{case.input_params}
    实际输出：{case.description}
    用户反馈：{case.name}
    
    请生成"期望质量标准"——即这个输入应该得到什么样的优质回答。
    要求：200字以内，包含数据准确性、逻辑性、可操作性三个维度的期望。
    """
    case.expected_quality = llm.generate(prompt)
    update_eval_case(case.id, expected_quality=case.expected_quality)
```

**开关**：`llm_cost.fill_bad_case_expected`（默认 false，按需开启）

#### 2.3.3 补全 agent_type（D3）

**策略**：从 `analysis_type` 推导 `agent_type`。

```python
MAPPING = {
    'valuation_expert': 'valuation_expert',
    'specialist:valuation_expert': 'valuation_expert',
    'risk_assessor': 'risk_assessor',
    'specialist:risk_assessor': 'risk_assessor',
    'allocation_advisor': 'allocation_advisor',
    'fund_analyst': 'fund_analyst',
    'market_analyst': 'market_analyst',
    'specialist:arbitrator': 'arbitrator',
    # 其他类型留空
}
for case in all_cases:
    if not case.agent_type:
        case.agent_type = MAPPING.get(case.analysis_type, '')
        update_eval_case(case.id, agent_type=case.agent_type)
```

#### 2.3.4 清理无效数据（D4-D7）

| 清理项 | 操作 | 影响范围 |
|--------|------|----------|
| D4: duration/token 为 0 的运行 | 保留（历史数据，不补） | 仅影响统计准确性 |
| D5: 重复告警 | `DELETE FROM expert_performance_alerts WHERE rowid NOT IN (SELECT MIN(rowid) FROM expert_performance_alerts GROUP BY conversation_id, agent_key, alert_type)` | 去重 |
| D6: 空表 | 保留（功能待启用） | 无影响 |
| D7: 4 条错误运行 | `DELETE FROM eval_runs WHERE error_msg IN ('id', 'name \'cancel_event\' is not defined')` | 清理无效记录 |

#### 2.3.5 数据清理脚本

创建 `backend/scripts/cleanup_eval_data.py`，提供 dry-run 模式：

```bash
# 预览（不执行）
python3 scripts/cleanup_eval_data.py --dry-run

# 执行
python3 scripts/cleanup_eval_data.py --execute

# 仅去重
python3 scripts/cleanup_eval_data.py --dedup-only

# 仅补 expected_quality（需开启 LLM 开关）
python3 scripts/cleanup_eval_data.py --fill-expected
```

**安全措施**：
- 执行前自动备份 `eval_cases` 和 `eval_runs` 表到 `backups/eval_cleanup_YYYYMMDD.db`
- dry-run 输出详细影响范围（删多少条、保留多少条）
- 执行后输出统计报告

## 四、风险与约束

1. **数据清理不可逆**：必须先备份，提供 dry-run 预览
2. **LLM 补 expected_quality 消耗 token**：默认关闭，按需开启，53 条预计消耗约 5 万 token
3. **评分尺度迁移**：旧数据回填可能不精确，仅做近似归一化
4. **移动端折叠菜单**：增加一层交互，需确保用户能发现操作入口
5. **eval_suites 新表**：需要数据迁移逻辑（现有用例按 analysis_type 自动归类到默认套件）

---

## 五、不做的事（YAGNI）

- 不做自动化 prompt 修改应用（只生成建议，不自动应用）
- 不做评测结果的可视化图表增强（现有 sparkline 够用）
- 不做评测用例的版本管理（用 is_active 软删除即可）
- 不做多用户评测权限隔离（单用户场景）
- 不做实时评测流式输出（评测是后台任务，无需流式）
