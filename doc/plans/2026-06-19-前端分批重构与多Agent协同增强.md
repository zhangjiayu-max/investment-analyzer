# 前端分批重构与多 Agent 协同增强设计

日期：2026-06-19

## 背景

当前前端已经开始做第一阶段基础治理：`navigation.js` 已抽出左侧导航配置，`Sidebar.vue` 从统一配置读取导航；同时已有 `AIActionButton`、`AgentResultCard`、`StatusBadge` 等基础组件。下一步不适合一次性全量改造，应按左侧导航和业务风险分批推进。

主要问题：

1. `PortfolioManagement.vue` 体量过大，持仓表、交易弹窗、图表、AI 分析、关注列表、基金详情等职责混在一起。
2. `ValuationHistory.vue` 同样混合估值趋势、指数表、DD 估值、增强策略、AI 分析等多个模块。
3. AI 结果展示和异步任务体验分散在多个页面，状态、反馈、复制、重试、失败态不统一。
4. Agent 能力仍偏“多个入口调用多个 Agent”，尚未沉淀为跨页面可复用的理财协同层。
5. 质量闭环入口分散，LLM 输出的反馈、bad case、评测回归需要统一接入。

## 目标

采用小步、安全、可验证的重构方式，优先沉淀公共能力，再拆大页面，最后增强多 Agent 协同。

阶段目标：

1. 完善公共 AI / 异步状态组件，避免后续页面拆分继续复制旧逻辑。
2. 拆分持仓管理页面，保持业务行为不变，降低单文件复杂度。
3. 拆分估值数据页面，复用统一 AI 结果展示和异步状态组件。
4. 新增“组合健康巡检”总入口，形成跨页面理财协同工作流。
5. 统一质量闭环入口，让每个 Agent 输出可反馈、可沉淀 bad case、可进入评测。

## 非目标

本轮不做以下事项：

1. 不重写前端路由体系，仍沿用 `activePage + v-if / KeepAlive` 模式。
2. 不引入 Pinia/Vuex，继续使用 Composition API 和组件本地状态。
3. 不一次性重写 `PortfolioManagement.vue` 或 `ValuationHistory.vue` 的所有业务逻辑。
4. 不修改金融计算公式、估值阈值或投资建议规则；如后续涉及，必须从配置或数据库读取，并注明数据来源和更新时间。
5. 不删除用户数据，不做破坏性数据库操作。

## 推荐实施路线

### 阶段 1：公共 AI / 异步状态组件增强

先补齐跨页面共用组件：

- `AgentResultCard`
  - 支持 `running / done / error / empty` 状态。
  - 支持 Markdown 渲染。
  - 支持复制结果。
  - 支持重试事件。
  - 支持展开 / 折叠。
  - 支持 agent 名称、token、更新时间、来源说明。
  - 支持点赞 / 点踩反馈入口，满足项目“所有 LLM 生成内容必须有反馈按钮”的规范。
- `AIActionButton`
  - 统一 AI 触发按钮视觉和文案。
  - 支持 `idle / submitting / queued / running / error` 等状态映射。
  - 保留二次确认接入能力，具体确认弹窗由调用方或包装层控制。
- 新增 `AsyncStateBlock`
  - 统一 loading、empty、error、retry 展示。
  - 用于表格、卡片、异步任务和分析面板。
- 完善 `StatusBadge`
  - 用于任务状态、风险等级、估值状态、Agent 状态。

验收标准：

- 公共组件具备测试覆盖。
- 不改变现有页面业务行为。
- 至少迁移一个低风险调用点验证组件 API 可用。
- 前端测试通过。

### 阶段 2：持仓管理拆分

保持业务行为不变，先拆结构：

- `portfolio/HoldingsTable.vue`：持仓列表、排序、选择、基础操作入口。
- `portfolio/TransactionModal.vue`：买入、卖出、分红、现金变动等交易表单。
- `portfolio/AnalysisPanel.vue`：持仓 AI 分析、后台分析状态、结果展示。
- `portfolio/WatchlistPanel.vue`：关注列表、提醒、候选基金。
- `portfolio/FundDetailDrawer.vue`：基金详情、净值、历史走势、相关操作。
- `portfolio/CashPositionCard.vue`：现金仓位、资产分布摘要。

拆分原则：

- 先搬模板和局部逻辑，不改变接口字段和业务流程。
- 父组件继续作为容器，负责数据加载和跨模块协调。
- 子组件通过 props / emits 通信，不直接互相调用。
- 交易、删除、AI 分析触发等操作继续使用二次确认。
- 所有 AI 输出统一走 `AgentResultCard`。

验收标准：

- `PortfolioManagement.vue` 行数明显下降。
- 持仓展示、交易录入、关注列表、AI 分析、基金详情行为保持一致。
- 关键交互测试覆盖或更新。
- 前端测试通过，构建部署成功。

### 阶段 3：估值数据拆分

拆分 `ValuationHistory.vue`：

- `valuation/ValuationTrendPanel.vue`：趋势图、筛选、区间展示。
- `valuation/IndexValuationTable.vue`：指数估值表、估值状态、来源和更新时间。
- `valuation/DDValuationTable.vue`：DD 估值相关数据表。
- `valuation/ValuationAnalysisPanel.vue`：AI 深度分析和策略解释。
- `valuation/EnhancedStrategyPanel.vue`：增强策略展示。

拆分原则：

- 金融数据必须继续显示来源和更新时间。
- 估值状态、风险状态使用统一 `StatusBadge`，但阈值不硬编码在组件中。
- AI 分析统一走 `AgentResultCard`。
- 筛选条件变更时，按 `doc/checklists/frontend-state.md` 检查关联数据同步刷新。

验收标准：

- 页面行为与当前一致。
- 估值图、表格、DD 数据、AI 分析可独立维护。
- 相关测试更新并通过。

### 阶段 4：组合健康巡检多 Agent 入口

新增一个跨页面理财工作流，暂定名称“组合健康巡检”。入口可放在每日看板和持仓管理页。

输入数据：

- 当前持仓和资产分布。
- 现金仓位。
- 估值数据和更新时间。
- 债券市场温度。
- 市场热点。
- 关注列表。
- 最近文章或知识库摘要。

输出结构：

- 今日组合健康评分或状态标签。
- 主要风险点。
- 可执行行动清单。
- 需要继续观察的资产 / 指数 / 基金。
- 引用的数据来源和更新时间。
- 风险提示。

前端体验：

- 使用统一 AI 按钮触发。
- 提交后显示“已提交后台分析”。
- 支持进度轮询、失败重试、历史入口。
- 结果使用 `AgentResultCard` 展示，并接入反馈。

后端约束：

- 编排逻辑应复用现有 Agent / task 能力。
- prompt 应通过 `analysis_agents` 表配置管理，避免硬编码。
- 金融结论不得给出绝对化收益承诺，必须包含风险提示。

验收标准：

- 用户能一键生成跨持仓、估值、债券、市场热点的行动清单。
- 输出包含数据来源、更新时间和风险提示。
- 失败时有明确错误态和重试入口。
- 反馈入口可用。

### 阶段 5：质量闭环统一

目标是让每个 Agent 输出都能进入持续改进链路。

统一入口：

- 点赞 / 点踩。
- bad case 备注。
- 复制结果。
- 标记为需要评测回归。
- 跳转到质量仪表盘或 bad case 页面。

约束：

- 不在展示组件中硬编码具体业务评价维度。
- 反馈 API 由调用方传入上下文，如页面、agent、recordId、任务 id。
- 对话、持仓、估值、Dashboard 逐步迁移，不一次性全改。

## 数据流设计

### 公共 AI 结果展示数据流

1. 页面或业务组件触发 AI 操作。
2. 操作前通过 `ConfirmDialog` 二次确认。
3. 调用 API 创建后台任务或直接发起分析。
4. 页面维护任务状态，或通过轮询获取状态。
5. 将状态、内容、agent、token、recordId、更新时间传给 `AgentResultCard`。
6. 用户可复制、重试、点赞、点踩或展开详情。
7. 反馈事件由页面层处理，调用对应 API。

### 页面拆分数据流

- 页面容器负责：数据加载、API 调用、跨模块状态、全局错误处理。
- 子组件负责：展示、局部交互、emit 用户动作。
- 公共组件负责：状态和视觉一致性，不持有业务规则。

## 错误处理

1. 异步加载失败统一用 `AsyncStateBlock` 展示。
2. AI 分析失败统一显示失败原因和重试按钮。
3. 金融数据缺少来源或更新时间时，不展示为可靠结论，应显示“来源缺失 / 更新时间未知”。
4. 后台任务提交成功但结果未完成时，明确提示用户可切换页面。
5. 删除、提交、重新生成、AI 分析触发等操作必须二次确认。

## 测试策略

每阶段都需要运行：

```bash
cd frontend && npm test
```

前端 Vue/JS 改动后还需要：

```bash
cd frontend && npm run build:deploy
```

重点测试：

- `AIActionButton` 状态和点击行为。
- `AgentResultCard` 状态、复制、反馈、重试、折叠。
- `AsyncStateBlock` loading / empty / error / retry。
- `navigation.test.js` 确保导航配置和页面注册一致。
- 持仓拆分后关键交易和 AI 分析交互。
- 估值拆分后筛选联动和 AI 分析交互。

## 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 大文件拆分时引入行为回归 | 先结构拆分，不改业务逻辑；每次只迁移一个区域 |
| 公共组件设计过度 | 只实现已知跨页面需要的 props / events |
| AI 反馈上下文不一致 | 反馈事件统一包含 page、agent、recordId、taskId |
| 金融状态被组件硬编码 | 状态组件只接收展示结果，不内置金融阈值 |
| 前端构建产物未更新 | 每次 Vue/JS 改动后执行 `npm run build:deploy` |

## 第一批实施建议

第一批只做“公共 AI / 异步状态组件增强”，范围如下：

1. 增强 `AgentResultCard`。
2. 增强或保持兼容 `AIActionButton`。
3. 新增 `AsyncStateBlock`。
4. 补充或更新组件测试。
5. 选择一个低风险页面调用点接入，验证组件 API。
6. 跑前端测试和构建部署。

完成后再进入 `PortfolioManagement.vue` 拆分。这样能避免持仓页拆分时继续复制旧的 AI 状态和结果展示逻辑。
