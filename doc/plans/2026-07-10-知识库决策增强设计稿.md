# 知识库决策增强设计稿

> 基于 2026-07-10 知识库全链路审计的实测数据，针对性解决「数据书籍很多但发不出力气」的系统性衰减问题。

## 背景与数据基线（审计实测）

| 指标 | 实测值 | 判断 |
|---|---|---|
| 书籍知识点 | 8109 条（占知识库 93%） | 书籍主导向量空间 |
| book 检索命中率 | 64.8%（107/165 rag_logs） | 检索阶段大量召回 |
| 专家引用率（`根据《`） | 10.1%（13/129 回答） | **召回多但引用少** |
| 提及「知识库」 | 15.5%（20/129） | 略高于引用 |
| 检索分数范围 | _score 0.014-0.125（9 倍波动） | 信号丢失严重 |
| rag_low_quality | 只追加文本，不改变行为 | 装饰件 |
| P4 半修 | pipeline 移除 book，orchestrator + multi_agent 仍含 book | 名存实亡 |
| recommendations 归因 | 无 referenced_books 字段，data_basis 硬编码 | 无法回答「蒸馏有没有用」 |

**核心矛盾**：书籍占 93%、命中 64.8%，但专家只在 10-15% 回答里引用。**不是数据不够，是数据到决策的传导链路有 4 层衰减**。

## 改造目标

1. 消除 P4 半修，让书籍该挡的挡住、该留的留
2. 补可信度信号，让专家知道哪条该信
3. 补归因字段，让胜率可追踪
4. 修审计发现的去重 bug 和意图映射缺口

## 改造范围（8 个任务，4 个阶段）

### Phase A: 修 P4 半修（让书籍该挡的挡住、该留的留）

#### Task 1: orchestrator.py SCENARIO_RAG_MAP 精细化

**问题**：[orchestrator.py:1923-1948](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/orchestrator.py#L1923-L1948) 的 `SCENARIO_RAG_MAP` 中 6 个专家场景全部含 book，与 [pipeline.py:815-821](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py#L815-L821) 的 P4 修复（portfolio/strategy 移除 book）冲突。

**设计决策**：不是「全移除 book」，而是「持仓类专家移除、估值/分析类专家保留」。理由：13 条引用证据显示 book 在估值判断/周期分析/行为偏差场景有价值（引用了《股市进阶之道》《周期、估值与人性》《资产配置的艺术》等），但持仓类场景（allocation_advisor/risk_assessor）书籍相关性差。

**改造**：
- `allocation_advisor`（资产配置/持仓）：移除 book → `["valuation", "analysis"]`
- `risk_assessor`（风险/持仓）：移除 book → `["valuation", "analysis"]`
- `fund_analyst`（基金选择）：移除 book → `["analysis", "valuation"]`（基金选择靠工具数据而非书籍）
- `valuation_expert`（估值判断）：**保留 book** → `["book", "valuation", "analysis"]`（引用证据支持）
- `market_analyst`（市场周期）：**保留 book** → `["book", "article", "author_article", "valuation"]`（周期类书籍价值高）
- `article_expert`（文章解读）：**保留 book** → `["article", "author_article", "book"]`（书籍作为解读参照）

**文件**：`backend/agent/orchestrator.py:1923-1948`

#### Task 2: multi_agent.py _ACTIVE_RETRIEVAL_INSTRUCTION 精细化

**问题**：[multi_agent.py:178-190](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/multi_agent.py#L178-L190) 的 `_ACTIVE_RETRIEVAL_INSTRUCTION` 强制要求所有专家在 6 类场景下用 `content_types=["book"]` 检索书籍，覆盖估值/买卖/情绪/企业质量/资产配置/周期。这导致即使 pipeline 挡住了 book，专家执行阶段又捡回来。

**改造**：从「6 类场景全部强制检索 book」改为「3 类场景才检索 book」：
- 保留：估值判断、市场情绪/心理偏差、周期位置（这 3 类有引用证据）
- 移除：买卖操作（靠工具数据）、企业质量（靠工具数据）、资产配置（持仓类场景书籍相关性差）

同时把 `content_types=["book"]` 改为 `content_types=["book", "analysis"]`，让专家检索时混合书籍和分析记录，避免纯书籍召回噪音。

**文件**：`backend/agent/multi_agent.py:178-190`

### Phase B: 修审计发现的 bug

#### Task 3: pipeline.py 去重 key 改二元组 + text 意图映射补全

**问题 1**：[pipeline.py:713](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py#L713) 去重 key 是 `content_type:reference_id:title` 三元组，而 [rag.py:2161](file:///Users/xiaoyuer/projects/investment-analyzer/backend/services/rag.py#L2161) 内部去重用 `content_type:reference_id` 二元组。三元组会因 title 微小差异（如 `_inject_valuation_data` 生成的 title 与 FTS 索引中的 title 不同）导致去重失效。

**改造**：pipeline.py:713 的 key 去掉 title 维度，改为 `f"{r.get('content_type','')}:{r.get('reference_id','')}"`，与 rag.py 一致。

**问题 2**：[pipeline.py:815-821](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py#L815-L821) 的 `_INTENT_TO_CONTENT_TYPES` 缺少 `text` 意图映射。纯文章解读类问题（`needed_info=['text']`）会命中 `.get(info, [])` 返回空，最终返回 None → 全类型检索，可能召回 book/valuation 等无关类型。

**改造**：补充 `"text": ["article", "author_article"]` 映射。

**文件**：`backend/agent/pipeline.py:713, 815-821`

### Phase C: 补可信度信号（让专家知道哪条该信）

#### Task 4: _build_rag_context_from_results 加相关度标签

**问题**：[pipeline.py:842-861](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py#L842-L861) 的 `_build_rag_context_from_results` 只拼 `[label] title [来源: source] \n body`，`_score` 字段只用于排序（pipeline.py:719），从未进入 rag_context 文本。专家无法区分「0.125 高分精确命中」和「0.014 阈值边缘擦过」。

**改造**：在每条结果前加 `[相关度:高/中/低]` 标签，阈值基于实测数据分布：
- 高：`_score >= 0.08`（实测最高 0.125，这是真正相关的命中）
- 中：`_score >= 0.03`（中等相关）
- 低：`_score < 0.03`（边缘命中，专家应谨慎参考）

同时改造 [rag.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/services/rag.py) 的 `build_rag_context_with_details`（单子查询路径，pipeline.py:723 用到的 `rag_data.get("context")`），让单子查询路径也带相关度标签。具体做法：在 rag.py 构建上下文的函数里（构建 `[{label}] {title}` 的位置，约 rag.py:2455-2463），同样加入相关度标签。

**文件**：
- `backend/agent/pipeline.py:842-861`
- `backend/services/rag.py`（构建上下文的函数，需定位具体行号）

## 不做的事（明确排除）

- **不重建 ChromaDB collection**：书籍单独 collection 风险高、收益不确定，先做 Phase A-C 看效果
- **不改蒸馏脚本**：蒸馏产物质量（352 字平均长度、importance 8.1）达标，问题在链路不在数据
- **不改 RRF 融合权重**：融合算法本身没问题，问题在信号丢失而非融合不准
- **不改压缩阈值**：800 token 压缩是 token 预算约束，改大会影响其他上下文层

## 验证方案

### 改造前基线（已采集）
- 专家引用率：10.1%（13/129）
- book 命中率：64.8%（107/165）
- rag_low_quality：无统计（未写日志）

### 改造后预期
- 专家引用率：目标 20%+（补可信度信号后，专家知道哪条该信）
- book 命中率：持仓类专家不检索 book，整体命中率应下降到 40-50%
- rag_low_quality：可统计低质命中频率
- recommendations.referenced_books：非空率应与引用率一致

### 验证 SQL（改造后一周执行）
```sql
-- 引用率变化
SELECT
  COUNT(*) AS total,
  SUM(CASE WHEN content LIKE '%根据《%' THEN 1 ELSE 0 END) AS ref_book,
  ROUND(100.0 * SUM(CASE WHEN content LIKE '%根据《%' THEN 1 ELSE 0 END) / COUNT(*), 1) AS ref_rate
FROM messages WHERE role='assistant' AND created_at >= date('now', '-7 days');

-- recommendations 归因
SELECT
  CASE WHEN referenced_books IS NOT NULL AND referenced_books != '[]' THEN '引用书籍' ELSE '未引用' END AS grp,
  COUNT(*) AS total,
  SUM(CASE WHEN status='correct' THEN 1 ELSE 0 END) AS correct,
  SUM(CASE WHEN status='wrong' THEN 1 ELSE 0 END) AS wrong
FROM recommendations WHERE status != 'pending' GROUP BY grp;

-- data_basis 真实分布
SELECT data_basis, COUNT(*) FROM analysis_conclusions
WHERE created_at >= date('now', '-7 days') GROUP BY data_basis;
```

## 风险评估

| 风险 | 等级 | 缓解 |
|---|---|---|
| orchestrator 移除 book 后持仓类专家引用率下降 | 低 | 持仓类专家本来引用率就低，移除 book 是预期行为 |
| 相关度标签阈值（0.08/0.03）不准 | 中 | 基于实测分数分布（0.014-0.125），可后续调整 |
| referenced_books 正则提取漏掉非 `根据《` 格式的引用 | 中 | 先用最明确的格式，后续可扩展正则 |
| data_basis 改造影响下游消费 | 低 | data_basis 是 TEXT 字段，下游只做 LIKE 查询 |
