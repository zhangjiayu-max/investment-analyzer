# 持仓管理优化设计稿

> **日期**：2026-07-07
> **范围**：持仓管理功能全链路优化（加减仓操作 + 交易流程 + 分析可视化 + 系统联动）
> **策略**：分批实施（P0 核心痛点 → P1 分析可视化 → P2 生态联动）

---

## 1. 背景与目标

### 1.1 现状问题

持仓管理已有较完整的基础功能（CRUD + 交易状态机 + 风险预警 + AI 分析），但存在以下痛点：

| # | 问题 | 影响 |
|---|------|------|
| 1 | 加仓无建议金额，用户随意输入 | 操作非理性，未参考4%定投法和持仓盈亏 |
| 2 | 减仓无约束提示 | 对话系统有减仓约束（单基金≤20%/总减仓≤10%/单次≤¥50,000），但持仓UI不校验不提示 |
| 3 | 确认流程繁琐 | 每笔交易需手动填净值，未自动获取T+1净值 |
| 4 | pending交易撤销入口缺失 | 后端有DELETE接口，前端无入口 |
| 5 | 减仓盈亏预览缺失 | 卖出时不显示预计盈亏/到账金额 |
| 6 | 批量操作缺失 | 无批量确认、批量加减仓 |
| 7 | 交易记录筛选弱 | 无按日期/基金/类型筛选 |
| 8 | 持仓分析可视化不足 | 缺少配置分布、盈亏分布、趋势图表 |
| 9 | 与对话/决策系统割裂 | 持仓操作无法直接发起对话分析，决策无法一键落地为交易 |

### 1.2 设计目标

- **加减仓智能化**：软提示方式（警告但放行），让操作更理性但不限制自主权
- **交易流程体验**：减少手动操作，自动获取净值，提供撤销和筛选
- **持仓分析可视化**：图表化展示配置分布、盈亏分布、持仓趋势
- **系统联动**：持仓操作 ↔ 对话决策双向打通

### 1.3 设计原则

- **软提示不硬约束**：所有智能化建议以提示形式呈现，用户可自由选择是否采纳
- **复用现有能力**：4%定投法、减仓约束、净值刷新逻辑已在对话系统实现，复用而非重建
- **分批实施**：P0解决最痛问题，P1/P2增量增强，每批可独立交付
- **向后兼容**：所有API和UI改造不破坏现有功能

## 2. 现状架构

### 2.1 数据模型

**portfolio_holdings 表**（持仓主表）：
- `id, fund_code, fund_name, shares, cost_price, current_price`
- `total_cost, current_value, profit_loss, profit_rate`（自动计算）
- `account, category, buy_date, notes, index_code, index_name`

**portfolio_transactions 表**（交易记录）：
- `id, holding_id, fund_code, fund_name, transaction_type`（buy/sell/dividend/convert）
- `amount, shares, price`（实际值）
- `submitted_shares, submitted_amount`（pending时提交值）
- `status`（pending/confirmed/settled）
- `transaction_date, transaction_time, expected_confirm_date`
- `fee, account, is_system`

**portfolio_alerts 表**（风险预警）：
- 估值百分位、回撤、集中度、现金闲置预警

**portfolio_cash 表**（现金账户）：
- 余额、日利息、最后计息日期

**加仓流程**（[submitAddPurchase](file:///Users/xiaoyuer/projects/investment-analyzer/frontend/src/components/PortfolioManagement.vue#L2938)）：

3. 创建 pending 交易（submitted_shares = 份额）
4. T+1 后用户手动确认

## 3. P0 设计：加减仓智能化 + 交易流程优化

### 3.1 加仓智能化

#### 3.1.1 新增后端 API：`GET /api/portfolio/{holding_id}/dca-suggestion`

**功能**：返回某持仓的加仓建议

**响应**：
```json
{
  "fund_code": "110022",
  "fund_name": "易方达消费",
  "current_profit_rate": -0.085,
  "current_shares": 1000,
  "current_value": 9150,
  "suggestion": {
    "recommended_amount": 1000,
    "tier": 2,
    "max_tiers": 3,
    "rule": "基础¥500/档，每跌4%加一档，当前亏损8.5%对应第2档",
    "already_added_today": false,
    "recent_add_count_30d": 1
  },
  "advice": "continue"  // "continue" | "watch" | "pause"
}
```

**建议逻辑**：
- 盈利状态（profit_rate > 0）→ `advice: "watch"`，建议金额 = 0，提示"当前盈利，建议观望"
- 亏损 0~4% → 第1档 ¥500
- 亏损 4~8% → 第2档 ¥1000
- 亏损 8~12% → 第3档 ¥1500
- 亏损 >12% → 第3档 ¥1500（封顶）+ 提示"已超过定投法覆盖范围，建议结合基本面分析"
- 当日已加仓 → `already_added_today: true`，提示"今日已加仓"
- 30天内加仓次数 ≥ 3 → `advice: "pause"`，提示"近期加仓频繁，建议放缓节奏"

**实现位置**：`backend/routers/portfolio.py` 新增路由，逻辑复用 orchestrator 的定投法规则

#### 3.1.2 前端改造：加仓弹窗增加建议卡片

**位置**：[PortfolioManagement.vue](file:///Users/xiaoyuer/projects/investment-analyzer/frontend/src/components/PortfolioManagement.vue#L2926) `openAddPurchase` / `submitAddPurchase`

**改动**：
1. `openAddPurchase` 时调用 `GET /api/portfolio/{holding_id}/dca-suggestion` 获取建议
2. 弹窗顶部增加"加仓建议"卡片：
   - 显示当前盈亏率、已加仓次数
   - 建议金额 + 一键填入按钮
   - advice 提示文案（黄/绿色提示框）
3. 用户仍可自由输入金额（软提示不强制）

### 3.2 减仓智能化

#### 3.2.1 新增后端 API：`POST /api/portfolio/{holding_id}/sell-preview`

**功能**：减仓前预览预计盈亏和约束警告

**请求**：
```json
{ "shares": 500 }
```

**响应**：
```json
{
  "fund_code": "110022",
  "shares_to_sell": 500,
  "current_price": 1.2345,
  "expected_proceeds": 617.25,
  "cost_basis": 1.5000,
  "expected_profit_loss": -132.75,
  "expected_profit_rate": -0.177,
  "remaining_shares": 500,
  "remaining_value": 617.25,
  "total_assets_after": 50000,
  "concentration_after": 0.0123,
  "warnings": [
    {
      "type": "concentration",
      "level": "info",
      "message": "减仓后该基金占总资产1.23%，集中度合理"
    },
    {
      "type": "batch_suggestion",
      "level": "warning",
      "message": "本次减仓金额¥617较小，无分批必要"
    }
  ]
}
```

**约束检测逻辑**（软提示，不拦截）：
- `concentration`：减仓后该基金占比仍 >20% → warning
- `total_reduction`：本次减仓 >总资产10% → warning"建议分批减仓"
- `single_amount`：本次减仓 >¥50,000 → warning"金额较大，建议分批"
- `profit_warning`：亏损状态下减仓 → info"当前亏损X%，确认是否止损"

#### 3.2.2 前端改造：减仓弹窗增加预览区域

**位置**：[submitSell](file:///Users/xiaoyuer/projects/investment-analyzer/frontend/src/components/PortfolioManagement.vue#L2982)

**改动**：
1. 减仓弹窗输入份额后，防抖调用 `POST /api/portfolio/{holding_id}/sell-preview`
2. 弹窗增加"减仓预览"区域：
   - 预计到账金额（大字突出）
   - 预计盈亏 + 盈亏率（红绿色）
   - 减仓后剩余份额、剩余市值
   - 约束警告列表（黄/橙色提示框）
3. 警告以提示框形式显示，不影响提交按钮（软提示放行）

### 3.3 交易流程优化

## 4. P1 设计：持仓分析与可视化

### 4.1 持仓快照

**新增表**：`portfolio_snapshots`
```sql
CREATE TABLE IF NOT EXISTS portfolio_snapshots (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  snapshot_date TEXT NOT NULL,       -- YYYY-MM-DD
  total_cost REAL,
  total_value REAL,
  total_profit_loss REAL,
  total_profit_rate REAL,
  cash_balance REAL,
  total_assets REAL,                -- total_value + cash_balance
  holdings_json TEXT,               -- 当日各持仓快照（基金代码/份额/净值/市值）
  created_at TEXT DEFAULT (datetime('now','localtime')),
  UNIQUE(snapshot_date)
);
```

**新增后端**：
- `db/portfolio.py`：`create_snapshot()`、`list_snapshots(start_date, end_date)`
- `routers/portfolio.py`：`POST /api/portfolio/snapshot`（手动触发）、`GET /api/portfolio/snapshots`
- 定时任务：每日收盘后自动记录快照（复用现有定时任务框架）

### 4.2 分析 API

| API | 功能 |
|-----|------|
| `GET /api/portfolio/analysis/distribution` | 配置分布（按基金类型/账户/指数） |
| `GET /api/portfolio/analysis/profit-by-fund` | 分基金盈亏对比 |
| `GET /api/portfolio/analysis/profit-trend?days=30` | 盈亏趋势（基于快照） |
| `GET /api/portfolio/analysis/concentration` | 集中度分析（单基金占比/行业占比） |

### 4.3 前端可视化组件

**新增组件**：`PortfolioCharts.vue`（持仓管理页增加"分析"Tab）

- **配置分布饼图**：按基金类型/账户展示资产占比
- **盈亏分布柱图**：各基金浮动盈亏对比（红绿配色）
- **持仓趋势折线图**：总市值/总成本/盈亏率随时间变化
- **集中度分析**：单基金占比表格 + 行业占比饼图

**技术选型**：复用项目已有图表能力（若有 ECharts 则用，否则用轻量 SVG 实现或引入 ECharts）

## 10. 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| 净值获取失败（akshare 限流） | auto-confirm 失败时降级为手动填入，不阻塞流程 |
| 快照定时任务失败 | 快照可手动触发，且失败不影响主流程 |
| 前端跳转预填跨组件通信 | 用 Home.vue 中介，避免组件直接耦合 |
| 减仓预览实时计算频繁 | 前端防抖（300ms），后端查询轻量 |
