# AI 对话与持仓管理增强设计

> 日期:2026-06-24
> 范围:AI 对话、持仓管理、每日持仓提示、加减仓建议、定投/补仓规则、决策闭环。
> 目标:让系统每天基于真实持仓主动给出可解释、可追踪、可执行但不自动交易的加减仓提示。

---

## 1. 背景与现状

当前项目已经具备较完整的投资助手底座:

- AI 对话:`backend/routers/conversations.py` 提供多 Agent 对话、恢复、SSE、RAG 注入。
- 统一上下文:`backend/conversation_context.py` 已聚合对话摘要、最近消息、持仓、估值、决策、关注列表、用户画像。
- 持仓管理:`backend/db/portfolio.py` 已覆盖持仓、交易、现金、快照、预警、分析记录。
- 决策闭环:`backend/db/decisions.py` 已有 `recommendation_candidates`、`decision_records`、`decision_actions`、交易草稿和复盘。
- 前端承载:`frontend/src/components/PortfolioManagement.vue` 已有预警面板、建议候选、决策、交易草稿、关注列表、定投优化等入口。

主要短板不是缺功能,而是"主动性"和"闭环感"不够:

1. AI 对话可以回答问题,但每日不会主动告诉用户当前持仓该关注什么。
2. 持仓预警已有风险扫描,但偏"风险提醒",缺少"今日行动建议"。
3. 4% 定投法、连续下跌、估值低估、仓位上限、现金约束等规则散落在不同 prompt 或功能中,尚未形成统一信号。
4. 用户看到预警后,还需要自己去 AI 对话追问、再手动保存决策或创建交易草稿。

本设计要把"AI 对话"和"持仓管理"两块功能升级为同一条每日理财工作流:

```text
每日行情/净值刷新
  -> 当前持仓信号计算
  -> 规则引擎生成提示
  -> AI 解释与风险复核
  -> 今日提示卡片
  -> 强建议进入候选池
  -> 用户保存为决策
  -> 执行前检查
  -> 交易草稿
  -> 到期复盘
```

## 2. 设计目标

### 2.1 产品目标

1. **每天给用户一页可行动提示**  
   **主入口是首页（Dashboard）**：用户打开系统第一眼就看到"今日应该看什么"。AI 对话和持仓页作为追问入口，点击"问 AI"或"查看详情"跳转到对应页面。不做三个入口的平行设计，避免用户困惑"我该去哪看"。

2. **针对已持仓给出加减仓提示**
   只对用户真实持仓、现金、历史买入价、估值、仓位占比做建议,不做脱离组合的泛泛推荐。

3. **规则可解释,建议不过度激进**
   连续下跌和 4% 定投法只是触发条件之一,必须叠加估值、仓位、现金、风险画像、数据新鲜度。

4. **用户确认后才进入交易链路**
   AI 可以建议"加仓 500 元"或"暂缓加仓",但不得直接改变持仓。所有交易都必须经过决策草案和交易草稿。

5. **对话和持仓页互相打通**
   AI 对话给出的强建议能出现在持仓页行动工作台;持仓页每日提示可以一键带上下文追问 AI。

### 2.2 技术目标

1. 新增独立的每日持仓提示引擎,不把规则硬塞进 prompt。
2. 复用 `portfolio_alerts` 展示轻量风险,复用 `recommendation_candidates` 承接强行动建议。
3. 扩展 `conversation_context.py`,把每日提示作为对话上下文的一部分。
4. 接入已有 `decision_records` 和 `create_transaction_draft_from_decision()`,保持交易闭环一致。
5. 每日任务可手动触发、自动触发、幂等去重、可回溯。

## 3. 总体方案

新增一层"每日持仓提示引擎",负责从真实持仓和行情数据里生成结构化信号。

建议模块名:

- `backend/daily_position_advisor.py`
- `backend/db/daily_advice.py`
- `backend/routers/daily_advice.py`

整体链路:

```text
portfolio_holdings
portfolio_transactions
portfolio_cash
portfolio_snapshots
index_valuations
user_profile
decision_records
recommendation_candidates
        │
        ▼
daily_position_advisor.py
        │
        ├─ 计算每只持仓的价格/估值/仓位/现金/交易状态
        ├─ 生成 add/reduce/hold/watch/rebalance/dca 信号
        ├─ 通过组合约束做风险降级
        ├─ 写入 daily_advice_runs 和 daily_position_signals
        ├─ 轻量提醒写入 portfolio_alerts
        └─ 强建议写入 recommendation_candidates
        │
        ▼
前端展示
  每日看板:今日持仓提示
  持仓管理:行动工作台
  AI 对话:上下文注入 + 可追问
```

核心原则：

- **规则先行，AI 解释**：规则引擎负责触发和金额边界，LLM 只负责解释、补充反方观点和优化表达。
- **与现有告警系统分工明确**：`wealth_advisor.py` 负责**被动告警**（已经发生的风险：亏损超承受度、估值进入极端区间），`daily_position_advisor.py` 负责**主动提示**（可能的机会：低估+下跌=定投时机、连续下跌=关注时点）。二者独立运行，互不干扰。
- **建议分级**:不是所有提示都进入候选池,只有达到行动阈值的信号才形成决策候选。
- **一日一信号**:同一用户、同一基金、同一信号类型每天最多生成一次,避免刷屏。
- **买入从小额开始**:连续下跌触发的加仓默认只建议基础定投金额的一档,不因跌幅变大而无限加码。

## 4. 每日持仓提示引擎

### 4.1 输入数据

每次运行读取:

- `list_holdings(user_id)`:持仓代码、名称、份额、成本、当前净值、市值、盈亏率、最近买入价、最近买入日期。
- `list_transactions()`:近 90 天交易,用于识别频繁交易、连续补仓、未确认交易。
- `get_cash_balance(user_id)`:现金余额。
- `get_portfolio_summary(user_id)`:总资产、总市值、现金占比。
- `get_latest_valuation(index_code/index_name)`:估值百分位。
- `list_portfolio_snapshots(user_id)`:组合级近几日变化。
- `decision_records`:未完成决策、到期复盘、重复建议。
- `recommendation_candidates`:已有未处理建议。
- `user_profile`:风险偏好、目标权益仓位、单标的上限、备用金、月结余。

数据新鲜度要求:

- 当前净值超过 `alert.stale_days` 未更新时，不生成买卖金额，只生成"数据过期，先刷新"的提示。净值新鲜度通过 `portfolio_holdings.updated_at` 或 `portfolio_snapshots` 表的最近记录判定；估值新鲜度通过 `list_index_freshness()` 的 `stale_days` 字段判定。
- 估值缺失时允许生成价格类提醒，但行动等级最多为 `watch`。
- **最近买入价获取**：`last_buy_price` 不在 `portfolio_holdings` 表中，需从 `portfolio_transactions` 表查询最近一条 `transaction_type='buy' AND status='confirmed'` 记录的 `price` 字段。如果用户从未买入（仅有初始建仓），回退使用 `cost_price`。
- 未设置现金或用户画像时允许生成观察提示,但强加仓建议必须降级。

### 4.2 输出结构

每只持仓输出一个或多个信号:

```json
{
  "signal_type": "dca_add",
  "action_type": "dca",
  "target_type": "fund",
  "target_code": "110020",
  "target_name": "易方达沪深300ETF联接",
  "severity": "actionable",
  "score": 78,
  "suggested_amount": 500,
  "suggested_ratio": 0.02,
  "summary": "连续下跌且低于最近买入价 4.6%,可考虑小额定投加仓",
  "rationale": "近 4 个交易日下跌,当前估值处于低估区,现金充足,单标的仓位未超过上限。",
  "evidence": {
    "consecutive_down_days": 4,
    "drop_from_last_buy_pct": 4.6,
    "valuation_percentile": 24,
    "position_pct": 8.5,
    "cash_pct": 12.0
  },
  "risks": {
    "notes": [
      "连续下跌可能来自基本面恶化,不能只按跌幅机械加仓",
      "若后续再跌 4%,仍需检查仓位上限和现金比例"
    ]
  },
  "next_step": "create_candidate",
  "dedupe_key": "daily:2026-06-24:dca_add:fund:110020"
}
```

信号评分公式（`score` 字段，0-100 分）：

```python
def _calc_signal_score(drop_pct, valuation_pct, position_pct, cash_pct, consecutive_days):
    """信号评分：用于排序、截断和优先级判断。"""
    score = 0
    # 1. 下跌幅度（0-40分）：跌幅越大分越高
    score += min(drop_pct / 0.12 * 40, 40) if drop_pct else 0
    # 2. 估值优势（0-30分）：分位越低分越高，>35% 不计分
    if valuation_pct is not None and valuation_pct <= 35:
        score += max(0, (35 - valuation_pct) / 35 * 30)
    # 3. 仓位空间（0-20分）：仓位越低分越高，>15% 不计分
    score += max(0, (15 - position_pct) / 15 * 20) if position_pct else 10
    # 4. 流动性（0-10分）：现金越多分越高
    score += min(cash_pct / 15 * 10, 10)
    return min(100, round(score))
```

- `score >= 60` → 可升级为 `actionable`（前提：无 blocker）
- `score 40-59` → 保持 `watch`
- `score < 40` → 降为 `info`

信号分四级：

| 等级 | 含义 | 落库动作 |
|---|---|---|
| `info` | 提醒关注，不建议行动 | 只写 `daily_position_signals` |
| `watch` | 建议观察或刷新数据 | 写 `daily_position_signals`，必要时写 `portfolio_alerts` |
| `actionable` | 可以进入候选池 | 写 `daily_position_signals` + `recommendation_candidates` |
| `blocked` | 理论触发但被风险条件拦截 | 写 `daily_position_signals` + 风险提示 |

## 5. 加减仓规则设计

### 5.1 规则不是"跌了就买"

每日提示引擎采用"触发条件 + 过滤条件 + 金额约束 + 风险降级"的四层模型。

```text
触发条件:连续下跌、跌破最近买入价 4%、估值低估、仓位偏离
过滤条件:数据新鲜、现金充足、估值不高、无重复交易
金额约束:基础定投金额、现金上限、单标的上限、组合权益上限
风险降级:高估、仓位过重、连续补仓、用户保守、待复盘决策
```

### 5.2 4% 定投法

4% 定投法适合作为"分批补仓触发器",不适合作为独立买入理由。

默认规则:

- 参考价:优先使用 `last_buy_price`,没有则使用持仓成本 `cost_price`。
- 触发:当前净值较参考价下跌达到 `daily_advice.dca_drop_step_pct`,默认 4%。
- 加仓档位:每多下跌一个 4% 增加一档,但最多 3 档。
- 基础金额:取 `daily_advice.base_dca_amount`,默认 500 元;未配置时取月结余的 10%,再限制在 300-1000 元之间。
- 单次上限:不超过可用现金的 `daily_advice.max_cash_use_pct_per_signal`,默认 10%。
- 单标的上限:加仓后目标标的占总资产不得超过用户 `max_single_position_pct`,没有画像时默认 15%。

金额示例:

| 条件 | 档位 | 建议 |
|---|---:|---|
| 跌破最近买入价 4%-7.99% | 1 档 | 基础金额 1 倍 |
| 跌破最近买入价 8%-11.99% | 2 档 | 基础金额 1.5 倍 |
| 跌破最近买入价 12% 以上 | 3 档 | 基础金额 2 倍,但必须提示风险 |

降级条件:

- 估值百分位 > 70%:从 `actionable` 降为 `watch`,提示"价格跌但估值不便宜"。
- 过去 10 个交易日已买入 2 次以上:降为 `watch`,提示"避免过密补仓"。
- 当前现金占比 < 5%:降为 `blocked`,提示先保留流动性。
- 该基金仓位已超过上限:降为 `blocked`,建议只观察或调其他仓位。
- 有未确认交易或未复盘相同标的决策:降为 `blocked`,避免重复操作。

### 5.3 连续下跌提示

连续下跌规则用于识别"可以关注"或"可以小额定投"的时点。

默认规则:

- 连续下跌 3 天:生成 `watch` 提示。
- 连续下跌 5 天且估值低估:可升级为 `actionable`。
- 单日跌幅超过 3%:生成 `risk_watch`,不直接加仓。
- 连续下跌但估值高、仓位高或数据缺失:只提示风险,不建议买入。

输出文案必须区分两类情况:

- "下跌带来更好买点":低估、仓位不高、现金充足、长期逻辑未破坏。
- "下跌暴露风险":高估回落、主题基金波动、仓位集中、基本面或政策风险。

### 5.4 加仓信号

加仓建议必须同时满足:

1. 估值百分位低于 `daily_advice.add_valuation_max_percentile`,默认 35%,或用户明确配置该标的可定投。
2. 当前标的仓位低于单标的上限。
3. 现金占比高于最低保留比例。
4. 最近没有同标的待确认交易。
5. 没有相同标的未复盘的高风险决策。

可触发场景:

- `dca_add`:4% 定投法触发。
- `valuation_add`:估值低于 20%-30% 且仓位不足。
- `rebalance_add`:目标配置偏离,需要补足权益或某类资产。
- `cash_deploy`:现金闲置且组合低估机会明确。

### 5.5 减仓信号

减仓建议比加仓更保守,默认优先提示"复核"而不是直接卖出。

触发场景:

- 估值百分位高于 `daily_advice.reduce_valuation_min_percentile`,默认 80%。
- 单标的占比超过上限。
- 组合权益仓位超过目标上限。
- 持仓盈利较高且偏离目标配置。
- 基金风格漂移、经理变更、费率过高等已有模块发现重大问题。

减仓输出必须包含:

- 减仓理由:估值、仓位、风险、替代去向。
- 建议比例:优先用 10%-25% 的份额区间,而不是"一次卖完"。
- 不减仓条件:如果用户长期目标明确、估值未高、仓位合理,则提示继续持有。

禁止默认生成"清仓"建议。只有当用户明确要求、或标的风险信号极强,并通过二次确认时,才允许作为决策草案。

### 5.6 持有与观察信号

多数日子系统应给出"持有/观察",这不是能力不足,而是投资纪律。

持有信号包括:

- 估值合理、仓位合理、无连续异常波动。
- 已触发下跌但现金不足或仓位已高。
- 已有待确认交易,先等待确认。
- 近期刚执行过决策,等待复盘。

观察信号包括:

- 数据过期,先刷新。
- 估值接近低估线或高估线。
- 连续下跌但缺少估值证据。
- 净值接近用户关注列表目标价。

## 6. 数据模型

### 6.1 新增 `daily_advice_runs`

记录每日提示引擎运行情况:

```sql
CREATE TABLE IF NOT EXISTS daily_advice_runs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT DEFAULT 'default',
    run_date TEXT NOT NULL,
    trigger_type TEXT NOT NULL,
    status TEXT DEFAULT 'running',
    summary TEXT DEFAULT '',
    stats_json TEXT DEFAULT '{}',
    error TEXT DEFAULT '',
    created_at TEXT DEFAULT (datetime('now','localtime')),
    updated_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(user_id, run_date, trigger_type)
);
```

`trigger_type`:

- `auto_nav_refresh`:每日净值刷新后自动运行。
- `manual`:用户在前端点击"生成今日提示"。
- `chat_request`:用户在对话里问"今天该不该加仓"。

### 6.2 新增 `daily_position_signals`

记录每只持仓的信号:

```sql
CREATE TABLE IF NOT EXISTS daily_position_signals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id INTEGER NOT NULL REFERENCES daily_advice_runs(id) ON DELETE CASCADE,
    user_id TEXT DEFAULT 'default',
    signal_date TEXT NOT NULL,
    signal_type TEXT NOT NULL,
    action_type TEXT NOT NULL,
    target_type TEXT DEFAULT 'fund',
    target_code TEXT DEFAULT '',
    target_name TEXT DEFAULT '',
    severity TEXT DEFAULT 'info',
    score INTEGER DEFAULT 0,
    suggested_amount REAL,
    suggested_ratio REAL,
    summary TEXT NOT NULL,
    rationale TEXT DEFAULT '',
    evidence_json TEXT DEFAULT '{}',
    risk_json TEXT DEFAULT '{}',
    source_snapshot_json TEXT DEFAULT '{}',
    candidate_id INTEGER,
    alert_id INTEGER,
    status TEXT DEFAULT 'new',
    dedupe_key TEXT DEFAULT '',
    created_at TEXT DEFAULT (datetime('now','localtime')),
    updated_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(user_id, signal_date, dedupe_key)
);
```

状态:

- `new`:新生成。
- `read`:用户已读。
- `promoted`:已转为建议候选。
- `ignored`:用户忽略。
- `expired`:过期。

### 6.3 复用现有表

强建议写入 `recommendation_candidates`:

- `source_type = "daily_advice"`
- `source_id = daily_position_signals.id`
- `scenario_type = signal_type`
- `action_type = add / reduce / sell / hold / watch / rebalance / dca`
- `dedupe_key = daily_signal.dedupe_key`
- `source_snapshot_json` 保存完整信号和组合快照。

风险类轻提醒写入 `portfolio_alerts`:

- `alert_type = "daily_advice_signal"` 或细分为 `dca_add_alert`、`reduce_watch_alert`。
- `source = "daily_advice"`
- `related_fund_code`、`related_fund_name` 关联持仓。

```text
## 今日持仓提示
- 沪深300:连续下跌 4 天,低于最近买入价 4.6%,可考虑一档定投 500 元;风险:现金使用后仍需保留 6 个月备用金。
- 中证医疗:连续下跌 5 天,但仓位已接近上限,建议暂缓加仓。
- 债基A:估值偏高且占比过高,建议复核是否减仓 10%-15%。
```

拼接顺序放在"当前持仓与盈亏"之后、"近期决策"之前。

AI 对话新增场景识别:

- "今天要不要加仓"
- "哪些持仓该补"
- "有没有减仓提醒"
- "按 4% 定投法看一下"
- "连续跌了要不要买"

这些问题走 `scenario_type = "daily_position_advice"`,优先读取今日信号;没有今日信号时触发一次 `daily_advice.run(force=false)`。

### 7.3 与自动任务集成

在 `backend/app.py` 的 `_auto_refresh_nav()` 净值刷新和 `scan_portfolio_alerts()` 之后调用:

```python
from daily_position_advisor import run_daily_position_advice

run_daily_position_advice(
    user_id="default",
    trigger_type="auto_nav_refresh",
    force=False,
)
```

自动任务失败不影响净值刷新主流程,只记录日志和 `daily_advice_runs.status = failed`。

## 8. AI 对话增强

### 8.1 回答形态

涉及加减仓的 AI 回复必须固定包含:

1. 今日结论:加仓、减仓、持有、观察、先刷新数据。
2. 证据:跌幅、连续下跌天数、估值、仓位、现金、历史交易。
3. 反方观点:为什么这次可能不该买/不该卖。
4. 建议动作:保存为决策、生成交易草稿、延期观察、刷新数据。
5. 风险提示:不构成投资建议,交易需用户确认。

示例:

```text
今日不建议机械加仓。沪深300虽然低于最近买入价 4.2%,触发一档 4% 定投条件,
但你过去 10 天已经补过 2 次,现金占比降到 4.8%,系统把信号降级为观察。

更稳妥的动作是:
1. 先不下单,等待下一次净值刷新。
2. 如果现金回到 8% 以上,且估值仍低于 30%,再考虑 500 元一档定投。
3. 可以保存为"观察决策",3 个交易日后复盘。
```

### 8.2 工具与 Agent

新增工具定义到 `backend/tools/__init__.py`:

- `get_daily_position_advice`
- `run_daily_position_advice`
- `create_candidate_from_daily_signal`
- `explain_daily_signal`

建议新增或配置一个专家 Agent:

- `position_advisor`
- 名称:每日持仓行动顾问
- 职责:解释每日信号、结合用户画像判断是否行动、输出反方观点。
- 工具:`get_daily_position_advice`、`get_portfolio_summary`、`get_valuation`、`create_recommendation_candidate`。

prompt 仍通过 `analysis_agents` 表管理,不硬编码。

### 8.3 建议结构化

对话中如果 AI 生成明确加减仓建议,继续沿用当前 `create_chat_decision_draft()` 和建议候选机制,但补充每日信号引用:

```json
{
  "source_type": "chat",
  "scenario_type": "daily_position_advice",
  "source_snapshot": {
    "daily_signal_id": 88,
    "run_id": 12,
    "evidence": {}
  }
}
```

这样后续复盘可以知道该决策来自"每日提示 + AI 对话确认"。

## 9. 持仓管理增强

### 9.1 页面结构

`PortfolioManagement.vue` 保留现有功能,但把首屏调整为"今日行动工作台":

  ├─ 可行动卡片:加仓/减仓/定投/再平衡
  ├─ 风险拦截卡片:为什么不能买/不能卖
  ├─ 观察卡片:连续下跌、估值接近阈值、数据过期

### 9.2 今日提示卡片

卡片字段:

- 标的名称和代码。
- 建议标签:一档定投、暂缓加仓、减仓复核、继续持有。
- 建议等级:可行动、观察、风险拦截。
- 建议金额或比例。
- 证据 chips:连续下跌、跌破买入价、估值百分位、仓位占比、现金占比。
- 反方风险。
- 操作按钮。

- 今日整体状态:偏加仓、偏观望、偏减仓、需刷新数据。
- Top 3 提示。
- "查看持仓行动工作台"入口。

### 9.4 AI 对话入口

AI 对话页增加快捷 prompt:

## 10. 风险控制

### 10.1 投资风险边界

系统必须遵守:

- 不自动交易。
- 不承诺收益。
- 不把连续下跌直接等同于买入机会。
- 不在数据过期时给金额建议。
- 不建议满仓或清仓。
- 不建议突破用户单标的上限、现金底线、目标权益仓位。
- 对主题基金、行业基金、窄基指数默认提高风险等级：
  - 单标的上限从 15% 降至 8%
  - 最高加仓金额为宽基指数的 50%
  - 加仓信号的估值要求更严格（`add_valuation_max_percentile` 从 35% 降至 25%）
  - 减仓信号更敏感（`reduce_valuation_min_percentile` 从 80% 降至 70%）
  - 基金类型判定：名称包含"主题"/"行业"/"赛道"或 QDII/商品类，或 fund_category 为窄基/行业指基

### 10.2 回退策略

如果新引擎上线后出现异常（如全部持仓生成"减仓"、信号计算逻辑错误等）：

1. **立即关闭**：设置 `daily_advice.enabled = false`（`db/config.py`），引擎不再执行。
2. **保留历史**：已生成的 `daily_advice_runs` 和 `daily_position_signals` 不删除。
3. **隔离影响**：已写 `recommendation_candidates` 的建议不自动失效（用户可能已采纳），但新 `candidates` 不再生成。
4. **回滚验证**：修复后先在 `force=true` 模式手动运行一次，验证输出正确后再开启自动运行。

### 10.3 幂等与去重

`dedupe_key` 规则:

```text
daily:{run_date}:{signal_type}:{target_type}:{target_code}
```

同一天重复运行:

- 默认返回已有 run 和 signals。
- `force=true` 时重新计算,但旧信号标记为 `expired`,不删除。
- 已保存为决策的 candidate 不重复生成。

### 10.3 数据缺失降级

| 缺失项 | 降级策略 |
|---|---|
| 当前净值缺失 | 不生成买卖建议,只提示刷新 |
| 估值缺失 | 最多 `watch`,不生成加仓金额 |
| 现金缺失 | 不生成加仓金额 |
| 用户画像缺失 | 使用保守默认值,提示完善画像 |
| 历史交易缺失 | 可用成本价替代最近买入价,但解释中说明 |

## 12. 分期落地

### Phase 1:规则引擎与数据闭环

目标:不依赖 LLM,先稳定产出每日信号。

改动:

- 新增 `daily_position_advisor.py`。
- 新增 `db/daily_advice.py` 和两张表。
- 新增 `/api/daily-advice/run`、`/today`、`/signals`。
- 支持 4% 定投、连续下跌、估值低估、仓位上限、现金约束。
- 强建议写入 `recommendation_candidates`。
- 自动任务在净值刷新后运行。

验收:

- 手动运行可生成今日信号。
- 重复运行不重复生成候选。
- 数据过期时不生成金额建议。
- 现金不足或仓位超限时加仓信号被拦截。

### Phase 2:前端行动工作台

目标:用户能在持仓页直接看到今日该关注什么。

改动:

- `api/index.js` 增加 daily advice API。
- `PortfolioManagement.vue` 新增"今日持仓提示"区域。
- 支持保存为候选、问 AI、延期、忽略、刷新数据。
- `Dashboard.vue` 增加 Top 3 摘要。

验收:

- 今日提示卡片可读、可操作。
- `保存为决策` 能流转到现有决策档案。
- 不出现直接交易按钮。

### Phase 3:AI 对话深度接入

目标:AI 对话能解释每日信号并生成结构化决策。

改动:

- `conversation_context.py` 注入 `daily_advice_context`。
- `tools/__init__.py` 增加每日提示工具。
- `analysis_agents` 增加"每日持仓行动顾问"。
- 对话场景识别 `daily_position_advice`。
- AI 回复引用 signal id,支持保存决策。

验收:

- 用户问"今天要不要加仓"时,优先引用今日信号。
- 没有今日信号时自动触发一次生成。
- AI 回复包含证据、反方观点、建议动作。

### Phase 4:复盘与策略优化

目标:让每日提示可评估、可迭代。

改动:

- 决策复盘关联 `daily_position_signals`。
- 统计不同规则的命中率、用户采纳率、复盘结果。
- 根据用户反馈调整 base amount、冷静期、阈值。
- 引入策略沙盒回测 4% 定投法和估值加权定投。

验收:

- 能看到"过去 30 天每日提示采纳率/忽略率/复盘收益变化"。
- 能识别哪些规则对用户有效,哪些经常被忽略。

## 14. 成功指标

产品指标:

- 用户打开持仓页后 10 秒内能知道今日是否需要行动。
- 每日提示中至少 80% 能说明"为什么行动/为什么不行动"。
- 强建议到决策草案的转化路径不超过 2 次点击。

质量指标:

- 重复候选率接近 0。
- 数据过期时金额建议为 0。
- 有 blocker 的信号不会生成交易草稿。
- 对话回答能引用今日信号证据,而不是泛泛建议。

长期指标:

- 每日提示采纳率、忽略率、复盘结果可统计。
- 根据复盘结果能调整用户个性化阈值。
