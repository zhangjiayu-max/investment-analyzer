# 三大核心功能深度优化设计稿

> 创建日期：2026-07-22
> 涉及功能：机会雷达 / 智能补仓 / 全账户诊断
> 基于：对 30+ 核心文件的逐行代码研读
> 原则：所有改动都是为了增强，不合理的地方先反馈再编码

---

## 一、总体诊断结论

三大功能均存在"**设计稿承诺未落地**"或"**数据模型缺失**"的结构性问题。核心矛盾：

| 功能 | 核心矛盾 | 严重度 |
|------|---------|--------|
| 机会雷达 | 三套系统（主题机会/事件雷达/watchlist信号）割裂，主题机会引擎无自动调度、仅5个硬编码主题 | P0 |
| 智能补仓 | `smart_add_plans`/`smart_add_signals` 表不存在，假设交易污染冷却期 | P0 |
| 全账户诊断 | 设计稿承诺"复用既有引擎"未落地，5维度评分是简化重写，比旧版V1更弱 | P0 |

本次优化共识别 **15 个 P0/P1 优化点**，分三阶段实施。

## 二、机会雷达优化（5项）

### 现状诊断

机会雷达实际上是三套独立系统：

```
A. 主题机会引擎 (opportunity_engine.py)
   - 5个硬编码主题（红利/AI/半导体/机器人/新能源）
   - 仅手动触发，无自动调度
   - 9维评分（含akshare技术指标，但不稳定）
   - 15交易日后回测

B. 前瞻事件雷达 (event_radar.py, 2500+行)
   - 三源新闻采集 + LLM提取事件
   - 每日11:30/14:30/20:00自动调度
   - 18个板块覆盖
   - T+3落地验证

C. 关注列表信号 (watchlist_multidim.py)
   - 估值信号灯 + 多维信号调整
   - 30分钟缓存
   - 15交易日后回测
```

**问题**：三套系统在前端整合为一个页面，但后端完全割裂，主题机会引擎与事件雷达无任何数据关联。

### O-1: 主题机会引擎添加自动调度 [P0]

**根因**：`app.py` 中只有事件雷达和回测的自动调度，没有主题机会的每日扫描调度。用户不手动点"扫描"按钮就不会生成新机会卡。

**修复方案**：
- 在 `app.py` 新增 `_auto_daily_opportunity_scan` 异步任务
- 每日盘后 16:00 自动调用 `scan_daily_opportunities`
- 新闻源复用事件雷达的 `_collect_news()` 三源采集结果（避免重复采集）
- 新增开关 `opportunity.auto_daily_scan_enabled`（默认 true）
- 触发后自动创建 backtest 记录（+15交易日）

**预期收益**：形成"每日生成→15日回测→命中率反馈"闭环，不再依赖手动触发。

### O-2: THEME_RULES 配置化 + 板块对齐 [P0]

**根因**：5个主题硬编码在 `opportunity_engine.py:31-107`，每个主题仅1只基金。新增主题需改代码。事件雷达有18个板块，主题机会仅5个，覆盖严重不对齐。

**修复方案**：
- 新建 `theme_rules` 表（theme/keywords/policy_terms/funds/future_direction/active）
- 新增 `db/theme_rules.py` CRUD 模块
- `opportunity_engine.py` 启动时从DB加载 THEME_RULES（保留硬编码作为兜底）
- 将事件雷达的 `SECTOR_TO_INDEX` 18板块中与主题相关的映射同步到 theme_rules 表
- 新增后台管理API `/api/admin/theme-rules` 支持增删改

**预期收益**：支持后台动态增删主题，无需改代码；板块覆盖与事件雷达对齐。

### O-3: 技术指标数据源容错 + 资金流向板块级化 [P0]

**根因**：
1. `_get_technical_score` 调用 `ak.stock_zh_index_daily`，sina API历史曾全部404
2. `_get_capital_flow_score` 获取的是**全市场北向资金**，所有主题共享同一信号，无法区分板块差异

**修复方案**：
- 技术指标优先查本地 `index_price_history` 表（已有数据），akshare 作为兜底
- 复用 watchlist_multidim 的5分钟缓存机制
- 资金流向改为板块级：用 `ak.stock_sector_fund_flow_rank` 获取主题对应板块的资金净流入
- 新增 `_get_sector_capital_flow(theme_rule)` 替代 `_get_capital_flow_score`
- 数据失败时返回 neutral 而非 0，避免误判

**预期收益**：技术维度和资金维度评分准确性提升，不再因数据源失败而退化。

### O-4: 回测失败原因分析 + 反馈闭环 [P1]

**根因**：`review_opportunity_backtests` 只计算 hit/miss，不分析 miss 原因，也不将命中率反哺到评分权重。系统无法从失败中学习。

**修复方案**：
- `theme_opportunity_backtests` 表新增 `miss_reason` 字段（TEXT）
- miss 时记录：估值分位/技术信号/新闻是否利空/持仓重叠度
- 新增 `get_backtest_stats(by_verdict=True)` 按 verdict 分组统计命中率
- 命中率反哺：若某主题连续3次 miss，自动降低该主题的评分权重系数
- 前端 EventRadarPage 新增"命中率分析"Tab，展示按 verdict/theme 分组的统计

**预期收益**：系统从失败中学习，相同误判不再重复；命中率统计可直接验证评分有效性。

### O-5: 三套系统机会统一视图 [P1]

**根因**：主题机会卡在 Dashboard 的 HotspotsCard 展示，事件雷达在 EventRadarPage Tab1，watchlist信号在 Tab2。用户在三个地方看机会信息，体验割裂。

**修复方案**：
- 后端新增 `/api/opportunities/unified` 接口
- 合并：主题机会卡（A）+ 事件雷达opportunity级事件（B）+ watchlist green信号（C）
- 按 `opportunity_score` 统一排序，去重（同标的取最高分）
- 前端 EventRadarPage 新增"统一机会"首Tab，展示合并后的机会列表
- 每个机会卡标注来源（主题/事件/关注）

**预期收益**：用户在一个页面看到所有机会，信息层次清晰。

## 三、智能补仓优化（5项）

### 现状诊断

智能补仓功能完整度高（双引擎+6类信号+六维仓位决策+反事实验证），但存在两个P0级隐患：

1. **数据模型缺失**：用户期望的 `smart_add_plans`/`smart_add_signals` 表不存在，计划与信号是即时计算的内存对象，仅 `smart_add_snapshots` 持久化（且只覆盖金字塔信号）
2. **假设交易污染冷却期**：假设交易写入 `portfolio_transactions` 表且 `status='confirmed'`，冷却期SQL未排除 `is_hypothetical=1`，导致真实补仓被误拦截

### S-1: 新建 smart_add_plans 表持久化计划与信号 [P0]

**根因**：计划与信号无法持久化，无法查询历史计划，无法做"计划vs实际"对比，反事实验证只覆盖金字塔信号。

**修复方案**：
- 新建 `smart_add_plans` 表：
  ```sql
  CREATE TABLE IF NOT EXISTS smart_add_plans (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT NOT NULL DEFAULT 'default',
    fund_code TEXT NOT NULL,
    fund_name TEXT,
    snapshot_date TEXT NOT NULL,          -- YYYY-MM-DD
    triggered_signals TEXT,              -- JSON: [A/B/C/D/E]
    exit_signals TEXT,                    -- JSON: [F子信号]
    total_suggested REAL,                 -- 信号触发金额汇总
    final_suggested_amount REAL,          -- 多维度最终金额
    safety_status TEXT,                   -- can_add/blocked/reason
    valuation_percentile REAL,
    profit_rate_pct REAL,
    position_pct REAL,
    plan_detail TEXT,                     -- JSON: 完整plan对象
    created_at TEXT NOT NULL,
    UNIQUE(user_id, fund_code, snapshot_date)
  );
  ```
- 新建 `db/smart_add_plans.py` CRUD 模块
- `generate_smart_add_plan` 末尾调用 `save_smart_add_plan` 持久化
- 新增 API `GET /api/smart-add/history?fund_code=&days=30` 查询历史计划
- 新增 API `GET /api/smart-add/plan-vs-actual?fund_code=` 对比计划与实际交易

**预期收益**：支持历史回溯、计划vs实际对比、所有信号的反事实验证。

### S-2: 修复假设交易对冷却期的污染 [P0]

**根因**：`_check_cooldown` 的 SQL `status IN ('confirmed','pending','submitted')` 未排除 `is_hypothetical=1` 的假设交易，导致冷却期计数虚高，真实补仓被误拦截。

**修复方案**：
- 方案A（最小改动）：`_check_cooldown` 的 SQL 增加 `AND is_hypothetical = 0`
- 方案B（彻底隔离）：假设交易改用独立表 `smart_add_hypothetical_txs`，与真实交易表物理隔离
- **推荐方案A**（改动小，风险低），同时方案B作为后续优化

**预期收益**：冷却期计数准确，真实补仓不被误拦截。

### S-3: 信号B（趋势加仓）默认开关调整 + 风险提示增强 [P0]

**根因**：`_detect_trend_signal` 在"近期涨势好"时建议加仓，与系统整体"低买高卖"理念冲突（代码注释line 1324自己也承认）。默认 `trend_signal_enabled=true`，保守用户可能误操作追涨。

**修复方案**：
- `trend_signal_enabled` 默认改为 false（需用户显式开启）
- 或增加条件：仅在用户风险偏好为 `aggressive/radical` 时才启用
- 信号B触发时增加明显的风险提示："此为趋势追涨信号，与价值投资理念不同，请谨慎评估"
- 前端配置面板增加"趋势加仓"的醒目说明

**预期收益**：避免保守用户被推荐追涨操作。

### S-4: 估值数据缺失时的降级策略 [P1]

**根因**：`_generate_single_plan` 无估值数据且非债类时 `valuation_ok=False`，金字塔/信号B/C/E都不触发。用户可能误以为"系统没建议=不需要补"，实际是数据缺失。

**修复方案**：
- 无估值数据时，基于基金净值历史计算"相对历史净值的分位"作为估值代理
- 复用 watchlist 的 `_resolve_watchlist_percentile` fallback 逻辑
- 仍无法获取时，在 plan 中明确标注 `valuation_status: "data_missing"`
- 前端展示"数据缺失"提示，而非静默跳过

**预期收益**：避免大量标的被静默跳过，用户明确知道数据状态。

### S-5: 信号叠加规则明确化 + 组合再平衡支持自定义权重 [P1]

**根因**：
1. 同一标的可能同时触发多个信号，`total_suggested` 累加后金额可能过大
2. 组合再平衡 `_calc_rebalance_suggestion` 硬编码等权，无法按用户自定义目标权重

**修复方案**：
- 定义明确的叠加规则：
  - A+C 互斥（已有）
  - A+B 取大值（不累加）
  - B+C 取大值（不累加）
  - D/E 独立计算，与A/B/C叠加时取max
- 单标的总建议金额硬上限 = `effective_base × max_add_vs_position_mult`
- 新建 `target_allocation` 表存储用户目标权重（fund_type/fund_code/weight）
- `_calc_rebalance_suggestion` 优先用自定义权重，无则用等权
- 与"四罐"家庭财务模块打通，支持按罐分配

**预期收益**：避免多信号累加导致金额过大，再平衡建议更贴合用户实际需求。

## 四、全账户诊断优化（5项）

### 现状诊断

全账户诊断 V2 的核心问题是"**设计稿承诺未落地**"：

> 设计稿第13节明确要求"必须复用现有数据源，避免引入独立计算导致数值不一致"
> 但实际代码中 `portfolio_intelligence`、`alerts`、`behavior_diagnosis`、`fear_greed` 均未 import

5维度评分是轻量重写，比旧版V1更弱：
- 选品质量：V1用akshare查同类排名，V2只看盈亏+费率（费率数据还拿不到）
- 持有行为：V1有追涨杀跌检测+持有时长，V2仅看交易次数
- 风控纪律：V1有高估持仓占比+止损意识，V2仅看单只仓位

### H-1: 真正复用既有引擎 [P0]

**根因**：`health_v2_service.py` 顶部注释声明"复用：health_score、four_pots、portfolio_intelligence、smart_add_planner、alerts、decisions"，但实际代码未import使用。5维度评分函数全部是轻量重写。

**修复方案**：
- **选品质量维度**：复用 `services.fund.fund_analysis` 和 `services.fund.fund_manager`
  - 引入同类排名、经理稳定性、规模健康度、夏普比率
  - 费率数据通过 `_fetch_fund_fee` 实时查询并缓存7天
- **持有行为维度**：复用 `services.portfolio.behavior_diagnosis`（如存在）
  - 处置效应/锚定效应/羊群效应/过度交易检测
  - 持有时长分析
- **风控纪律维度**：复用 `services.portfolio.portfolio_intelligence`（如存在）
  - VaR/CVaR/组合波动率/最大回撤
- **行动清单**：接入 `smart_add_planner` 的 `exit_signals`（止盈/止损/暂停）

**预期收益**：V2 与 V1 的同维度分数可比，评分维度真正全面，不再退化为简化版。

### H-2: 修复行动项重复沉淀 + 引入定时调度 [P0]

**根因**：
1. `action_id` 含日期 `datetime.now().strftime('%Y%m%d')`，跨日会重复创建候选
2. 无自动调度，快照仅在用户打开页面时写入，`score_change_7d` 计算失真

**修复方案**：
- `action_id` 去掉日期后缀，改为 `smart_add:{fund_code}` 等稳定ID
- `_persist_actions` 前查询 `health_action_tracking` 已存在的 pending/accepted 行动项，跳过已活跃项
- 新增 `_auto_daily_health_diagnosis` 异步任务，每日18:00自动调用 `get_health_v2_dashboard(force_refresh=True)`
- 新增开关 `health_v2.auto_daily_diagnosis_enabled`（默认 true）

**预期收益**：`recommendation_candidates` 表不膨胀，历史快照连续，`score_change_7d` 计算准确。

### H-3: 接入Agent系统 + Dashboard卡片联动 [P1]

**根因**：
1. `router_config.yaml` 的 `diagnosis` 路由未接入 `health_v2_service`
2. `PortfolioHealthCard.vue` 未跳转到 `health-v2` 页面，未展示 Top3 行动项

**修复方案**：
- 在 `tools/__init__.py` 注册 `health_v2_diagnosis` 工具，调用 `get_health_v2_dashboard`
- 修改 `router_config.yaml` 的 `diagnosis` 路由，增加 `health_v2_expert` 或直接调用工具
- `PortfolioHealthCard.vue` 增加"查看全账户诊断"按钮，跳转 `health-v2` 页面
- 卡片底部展示 `actions[:3]` 行动项预览

**预期收益**：用户在AI对话中说"给我做个全账户诊断"能正确路由；Dashboard与全账户诊断打通。

### H-4: 补全保险分类 + 费率数据 + LLM解读 [P1]

**根因**：
1. `four_pots.classify_fund` 无"保险"关键词，`insurance` 恒为0
2. `portfolio_holdings` 表无 `mgmt_fee` 字段，费率加减分永远不触发
3. V2未调用 `_call_llm`，所有 `top_issue` 均为硬编码模板

**修复方案**：
- `classify_fund` 的 `NAME_KEYWORDS` 增加 `["保险", "重疾", "医疗", "寿险", "意外险"]`
- `TYPE_MAP` 增加 `"保险型": "保险保障"`
- 持仓表增加 `mgmt_fee` 字段，或诊断时通过 `_fetch_fund_fee` 实时查询并缓存
- 在 `get_health_v2_dashboard` 末尾调用 `_call_llm`，基于5维度得分+top_issue 生成个性化解读
- 受 `llm_cost.page_llm_summary` 开关控制（与V1一致）

**预期收益**：四笔钱诊断完整，选品质量维度恢复费率检查，报告有个性化解读。

### H-5: 评分基线统一 + 行为/风控维度增加加分项 [P1]

**根因**：
1. 无持仓时各维度默认分不一致（quality 100、diversification 100、valuation 100、behavior 130、risk 130）
2. 行为/风控维度基础150只减不加，满分200但实际最高150，总分天花板约900而非1000

**修复方案**：
- 无持仓时所有维度统一返回100（中性），总分500（"一般"）
- 行为维度增加加分：持有时长>365天 +30；冷却期内无频繁交易 +20
- 风控维度增加加分：设置了止损计划 +30；现金占比在3-20%区间 +20
- 确保各维度真正可达200分，总分天花板恢复为1000

**预期收益**：评分基线一致，分数天花板正确，用户有明确的上限目标。

## 六、新增配置开关清单

所有新增功能均通过独立开关控制，默认按以下原则：

## 八、风险与注意事项

1. **H-1依赖确认**：`portfolio_intelligence`、`behavior_diagnosis` 模块需确认存在且可复用。若不存在，需先实现或从V1迁移。
2. **O-2数据迁移**：THEME_RULES配置化时，需将现有5个主题的数据迁移到新表，保留硬编码作为兜底。
3. **S-1表设计**：`smart_add_plans` 表的 `plan_detail` JSON 字段可能较大，需评估存储和查询性能。
4. **H-2定时调度**：每日18:00自动诊断会调用智能补仓引擎，增加计算负载。需评估是否影响其他任务。
5. **开关默认值**：所有LLM相关开关默认false，非LLM相关开关默认true（遵循项目硬约束）。
6. **向后兼容**：所有新表新字段用 `CREATE TABLE IF NOT EXISTS` 和 `_add_column_if_not_exists`，不影响现有数据。
