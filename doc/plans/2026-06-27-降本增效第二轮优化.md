# 降本增效第二轮优化设计稿

> 日期：2026-06-27
> 背景：第一轮已落地 8 个 `llm_cost.*` 开关（自动日报/评测/摘要/Shadow/Judge/根因/页面总结等），关闭了显式自动触发链路。本轮通过**全量代码走读**，定位第一轮未覆盖的"隐藏消耗点"和"浪费型 bug"，进一步压降 LLM 成本。
> 原则：先修 bug（白丢的 token），再加开关（隐藏自动触发），最后上缓存（用户主动但可复用）。

第一轮只覆盖了"显式后台任务 + 页面总结"，**没有覆盖以下三类隐藏消耗**：

1. **对话链路中夹带的隐式 LLM 调用**（kyc_learner / feedback_learner / memory 压缩 / daily_advisor 行业评分）——每条用户消息或每次反馈都可能触发，单次量小但频次高。

3. **用户主动分析无缓存**——同参数重复点击就重复调 LLM。

## 二、走读结论：可优化点清单

### 2.1 P0 — 浪费型 Bug（修了直接省）

#### Bug 1：`rag.py:403` Query Rewrite 参数错位

```python
# 现状（rag.py:403）
rewritten = _call_llm(prompt, temperature=..., max_tokens=...)
```

`_call_llm` 签名是 `_call_llm(caller: str = "", trace_id: str = "", **kwargs)`。当前把**整段 prompt 字符串当作 caller** 传入，且 `messages` 参数缺失。后果：
- `client.chat.completions.create(**kwargs)` 因缺 `messages` 抛错
- 被 `_llm_retry` 重试 4 次后抛出，被外层 `try/except` 吞掉，返回原 query
- **每次 RAG 查询都走这条失败链路**，token 可能没消耗（请求未发出），但 caller 字段会被污染成一长串 prompt 文本，污染 token 审计数据

**修复方案**：要么修 bug 让 query rewrite 真正工作（但会增加 token 消耗），要么直接关闭走 jieba 分词兜底。鉴于 RAG 已有 jieba 分词 + FTS5，**推荐直接关闭 query rewrite**，省心且不丢检索质量。

```python
# 方案 A（推荐）：关闭 LLM query rewrite，走 jieba
def rewrite_query(query: str) -> str:
    if get_config("llm_cost.rag_query_rewrite", "false") != "true":
        return _tokenize(query)  # 走 jieba 分词
    # ... 原有 LLM 逻辑（修复 caller/messages 后保留）
```

#### Bug 2：`correlation.py:261` response 当 dict 处理

```python
# 现状（correlation.py:261）
llm_result = await asyncio.to_thread(lambda: _call_llm(...))
llm_text = llm_result.get("content", "") if isinstance(llm_result, dict) else str(llm_result)
```

`_call_llm` 返回 OpenAI response 对象，不是 dict。后果：
- `isinstance(llm_result, dict)` 为 False
- `llm_text = str(llm_result)` → 把整个 response 对象 repr 当成总结文本
- 这段垃圾文本被拼进 `final_text` 展示给用户，**LLM token 消耗了但生成内容完全没用上**

`rolling_return.py`、`fee_analyzer.py`、`four_pots.py`、`health_score.py` 需要同步检查是否有相同 bug 模式。

**修复方案**：

```python
llm_result = await asyncio.to_thread(lambda: _call_llm(...))
# response 对象取 content
llm_text = llm_result.choices[0].message.content or ""
```

#### 开关 11：`llm_cost.memory_summarize` — 对话历史压缩

**现状**（`agent/memory.py:90`）：
- `_compress_with_summary()` 在对话历史过长时调用 `_generate_summary(old_messages)`
- **这是 `auto_conversation_summary` 开关没覆盖的盲区**——auto_conversation_summary 只管 conversations.py:1273 的"对话结束摘要"，不管 memory.py 的"历史压缩摘要"
- 长对话每轮都可能触发压缩，隐藏消耗

**改动**：

```python
# agent/memory.py _compress_with_summary() 或 _generate_summary()
def _generate_summary(messages: list) -> str:
    if get_config("llm_cost.memory_summarize", "false") != "true":
        # 回退：简单截取，不调 LLM
        return _fallback_summary(messages)
    ...  # 原有 LLM 逻辑
```

> 注意：`auto_conversation_summary` 和 `memory_summarize` 是两个不同场景，前者是对话结束存档摘要，后者是上下文窗口压缩。两者都应可独立关闭。

#### 开关 12：`llm_cost.daily_advisor_llm` — 持仓信号 LLM 行业评分

**现状**（`daily_position_advisor.py:924`）：
- 模块顶部注释宣称"纯规则零 LLM"，但 `_calc_industry_score()` 内部**有 _call_llm 调用**做行业前景评分
- 每次生成持仓信号时，若信号涉及 `index_name` 就调一次 LLM（max_tokens=200）
- 持仓信号每日刷新后批量生成，可能一次刷新触发多次

**改动**：

```python
# daily_position_advisor.py _calc_industry_score()
if index_name and get_config("llm_cost.daily_advisor_llm", "false") == "true":
    try:
        resp = _call_llm(caller="daily_advisor_industry", ...)
        ...
```

> 这个开关默认 false 后，行业评分纯走 akshare 涨跌幅规则，质量略降但零 LLM 消耗。需要时手动开启。

### 2.3 P2 — 用户主动分析加缓存

#### 现状

用户主动分析（portfolio_ai / fund_analysis / correlation / trade_review / index_analysis / what_if / hotspots）每次点击都调 LLM，**同参数重复点击重复消耗**。第一轮 `page_llm_summary` 开关是"全关"，对用户主动分析太粗暴。

#### 缓存方案

新增 `analysis_cache` 表（或复用现有 `analysis_records`）：

```sql
CREATE TABLE IF NOT EXISTS analysis_cache (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    cache_key TEXT UNIQUE,          -- sha256(分析类型 + 参数 JSON)
    analysis_type TEXT,             -- portfolio_ai / fund_analysis / ...
    result_text TEXT,
    tokens_used INTEGER,
    created_at TEXT,
    expires_at TEXT                 -- 按类型配置 TTL
);
```

**缓存命中规则**（按类型差异化）：

| 分析类型 | 缓存键 | TTL | 失效条件 |
|---------|--------|-----|---------|
| portfolio_ai | sha256(holdings_hash + user_question) | 1 天 | 持仓变动 |
| fund_analysis | sha256(fund_code + question) | 1 天 | 基金净值更新 |
| correlation | sha256(holdings_hash + lookback_days) | 1 天 | 持仓变动 |
| trade_review | sha256(trade_ids) | 永久 | 交易不变 |
| hotspots | sha256(date) | 当天 | 跨日 |
| what_if | sha256(params) | 不缓存 | — |

**改动模式**（以 portfolio_ai 为例）：

```python
# routers/analysis/portfolio_ai.py
async def _run_portfolio_ai_async(task_id, holdings, user_question, ...):
    # 1. 计算缓存键
    cache_key = _build_cache_key("portfolio_ai", holdings, user_question)
    
    # 2. 查缓存
    cached = get_analysis_cache(cache_key)
    if cached and not _is_cache_expired(cached):
        update_analysis_record(record_id, status="done", result=cached["result_text"])
        return
    
    # 3. 缓存未命中，调 LLM
    response = await asyncio.to_thread(lambda: _call_llm(...))
    result_text = response.choices[0].message.content or ""
    
    # 4. 写缓存
    save_analysis_cache(cache_key, "portfolio_ai", result_text, tokens)
```

新增开关 `llm_cost.analysis_cache`（默认 true，因为是纯优化无副作用）。

## 五、保留与回退

### 5.1 保留能力

- 所有开关默认 false，需要时改 `system_config` 表一键恢复，不改代码
- 缓存开关默认 true，副作用仅为偶尔展示稍旧结果（用户可手动刷新触发重算）
- jieba 分词作为 RAG query rewrite 的兜底，关闭 LLM 重写不丢检索能力

### 5.2 回退方案

- 任意开关回退：`UPDATE system_config SET value='true' WHERE key='llm_cost.xxx'`
- 缓存回退：`llm_cost.analysis_cache=false` 或 `DELETE FROM analysis_cache`
- Bug 修复回退：git revert 对应 commit
