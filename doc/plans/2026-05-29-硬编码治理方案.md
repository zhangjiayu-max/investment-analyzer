# 硬编码治理方案

## Context

审计发现项目中存在 ~150 处硬编码，其中金融阈值硬编码最为严重——同一套估值百分位分界线在 8 个位置独立定义且存在不一致（20/80 vs 30/70）。作为理财投资系统，这些不一致直接影响用户资金决策的准确性。

**核心原则**：一个配置值，一个来源，多处引用。

---

## 架构设计

### 新增 `app_config` 表（通用配置中心）

```sql
CREATE TABLE IF NOT EXISTS app_config (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    description TEXT,
    updated_at TEXT DEFAULT (datetime('now','localtime'))
)
```

复用项目已有的模式：`_get_conn()` 连接 + `init_db()` 建表 + CRUD 函数。

### 配置读取桥接函数

参考已有的 `get_rebalance_config()` 模式（config.py:170），在 `config.py` 中新增：

```python
def get_config(key: str, default=None):
    """优先读 DB，fallback 到 env var，最后用 default"""
    val = db.get_app_config(key)
    if val is not None:
        return val
    return os.getenv(key.upper(), default)
```

所有硬编码处统一改为调用 `get_config()`。

## 分阶段实施

### Phase 1：紧急修复 — 消除不一致性（1 天）

**目标**：解决 5 处已确认的冲突，零架构变更。

| 冲突项 | 修复方式 |
|--------|---------|
| 估值百分位 20/80 vs 30/70 | `portfolio_context.py` 改读 `config.py` 的 `VALUATION_PERCENTILES` |
| 单仓上限 20% vs 30% | 统一到 `config.py`，prompt 中引用变量 |
| 温度阈值 30/50/70 vs 20/35/50/70 | 统一到 `config.py` 的 `BOND_TEMPERATURE_THRESHOLDS` |
| `REBALANCE_BASE_allocation` 重复赋值 | 删除重复行（config.py:124-125） |

**涉及文件**：`config.py`, `portfolio_context.py`, `app.py`, `db.py`

### Phase 2：金融阈值外部化（3 天）

**目标**：所有金融阈值统一从 `app_config` 表读取，代码中不再有金融数值字面量。

#### 2.1 建表 + CRUD

- `db/core.py` init_db 加 `app_config` 表
- `db/config.py` 新增：`get_app_config()`, `set_app_config()`, `get_all_configs()`
- 预置默认值（从现有硬编码中提取）

#### 2.2 配置分组与迁移

| 配置组 | key 前缀 | 包含项 |
|--------|----------|--------|
| 估值阈值 | `valuation.` | percentiles, pe_ratio_thresholds, pb_ratio_thresholds |
| 评分体系 | `scoring.stock.` / `scoring.fund.` | 各档阈值和权重 |
| 集中度分析 | `concentration.` | 单基金/行业/相关性阈值 |
| 温度计 | `bond.temperature.` | 温度区间和配比 |
| 定投法 | `dca.` | 下跌比例、份数、仓位上限 |
| 风险评估 | `risk.` | CV 阈值、z-score 阈值 |
| 年龄法则 | `age_rule.` | 权益下限、各类比例 |
| 调仓建议 | `rebalance.` | 金额比例、最低变动阈值 |

#### 2.3 Prompt 中的阈值处理

**方案 A（推荐）：Prompt 模板化**
- Prompt 中用 `{valuation.percentile_low}` 占位符
- 运行时从 `app_config` 读取并 format
- 好处：阈值改一处，所有 prompt 自动更新

**方案 B：Prompt 运行时拼接**
- 把阈值作为额外 context 注入，不改 prompt 本身
- 好处：prompt 更干净，但需要改调用逻辑

**涉及文件**：`db/core.py`, `db/config.py`(新), `config.py`, `valuation.py`, `portfolio_context.py`, `db.py`(prompt 定义), `multi_agent.py`, `llm_service.py`, `orchestrator.py`, `image_parser.py`, `tools/__init__.py`

## 验证方式

1. **一致性验证**：修改一个配置值后，检查所有引用处是否生效（写脚本 grep 确认无残留硬编码）
2. **回归验证**：原有估值分析、评分、调仓建议功能输出不变
3. **配置管理验证**：通过 API 修改配置 → 重启后仍生效 → 回滚到旧版本
4. **Prompt 验证**：检查 LLM 收到的 prompt 中阈值与配置表一致
