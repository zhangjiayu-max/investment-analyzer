**日期**：2026-07-10
**前置**：基于已完成的 [2026-07-09-智能补仓计划器设计稿.md](./2026-07-09-智能补仓计划器设计稿.md) + [2026-07-09-智能补仓增强算法设计稿.md](./2026-07-09-智能补仓增强算法设计稿.md)
**目标**：从"相对市值公式"升级为"目标仓位驱动 + 多维约束"的金额计算体系，解决"加一点点无感、跌时恐慌、金额算不出"三大痛点

## 一、核心理念：原始投入驱动（关键修正）

### 1.1 问题：当前市值驱动的金额计算失真

现有所有金额公式都是基于"当前市值 × 比例"，对深套标的严重失真：

| 场景 | 原投入 | 当前市值 | 亏损 | 当前公式建议 | 问题 |
|------|--------|---------|------|------------|------|
| 白酒A | 5万 | 2万 | -60% | 2万×5%=1000元 | 永远补不起来 |
| 白酒B | 3万 | 1万 | -67% | 1万×5%=500元 | 越亏越补不动 |
| 医药C | 1万 | 0.5万 | -50% | 0.5万×5%=250元 | 仓位误判 |

### 1.2 修正：用 total_cost（已投入本金）替代 current_value

**核心原则**：所有仓位判断与金额计算基准，统一使用 `max(total_cost, current_value)`：

```python
# 旧：仓位判断基于当前市值
position_pct = current_value / total_assets  # 深套时严重失真

# 新：仓位判断基于"已投入本金"（亏损时不缩水）
effective_base = max(total_cost, current_value)  # 盈利时不虚增，亏损时不缩水
position_pct = effective_base / total_assets
```

**为什么用 max 而不是只用 total_cost？**
- 亏损时（current_value < total_cost）：用 total_cost → 不因亏损缩水误判低仓位
- 盈利时（current_value > total_cost）：用 current_value → 反映真实暴露，避免过度加仓

### 1.3 影响范围（所有金额公式同步修正）

| 模块 | 旧基准 | 新基准 |
|------|--------|--------|
| 引擎2金字塔 | current_value × add_ratio | max(total_cost, current_value) × add_ratio |
| 信号B趋势 | current_value × trend_base_ratio | max(total_cost, current_value) × trend_base_ratio |
| 信号C大跌 | current_value × dip_base_ratio | max(total_cost, current_value) × dip_base_ratio |
| 安全阀仓位 | current_value / total_assets | max(total_cost, current_value) / total_assets |
| 维度1目标缺口 | current_value vs 目标 | max(total_cost, current_value) vs 目标 |
| 维度3首次仓 | current_value vs 标准仓 | total_cost vs 标准仓（首次只看投入） |

**摊薄成本计算不变**：`new_cost = (total_cost + 补仓额) / (份额 + 补仓份额)`，这里 total_cost 本就是原始投入。

## 二、六维金额计算体系

### 维度1：目标仓位锚定（核心，解决"加一点点"）

**引入"目标仓位"概念**，所有加仓金额基于"当前仓位 vs 目标仓位"的缺口。

#### 1.1 目标仓位计算

```python
target_position_pct = min(
    kelly_limit_pct × valuation_coeff,  # L3凯利 × 估值系数
    type_hard_cap_pct,                  # L2类型硬上限
    user_config_max_pct,                # 用户配置 25%
    index_exposure_room_pct,            # 维度2穿透余量
    cash_constraint_pct,                # 维度6资金约束

# 估值系数（估值越低目标越高）
valuation_coeff = {
    分位 0-20%:  1.0,   # 极低估，满配
    分位 20-40%: 0.7,
    分位 40-60%: 0.5,
    分位 60-80%: 0.3,
    分位 80%+:   0.1,   # 高估，接近清仓
}[当前分位区间]
```

#### 1.2 动态调整周期（用户确认：动态）

```python

adjust_months = {
    分位 0-20%:  3,    # 极低估，3个月快速补足
    分位 20-40%: 6,    # 低估，6个月
    分位 40-60%: 9,    # 合理，9个月
    分位 60-80%: 12,   # 偏高，12个月慢慢减
```

#### 1.3 金额计算

```python
# 基于"已投入本金"计算缺口（核心修正）
effective_base = max(total_cost, current_value)
current_position_pct = effective_base / total_assets
gap_pct = target_position_pct - current_position_pct

# 仅在缺口为正时计算加仓金额
if gap_pct > 0:
    target_add_total = total_assets × gap_pct            # 总缺口金额
    monthly_add = target_add_total / adjust_months        # 月度建议
else:
    # 缺口为负：触发减仓（维度5）
    monthly_reduce = total_assets × abs(gap_pct) / adjust_months
```

#### 1.4 举例（白酒场景）

| 项目 | 旧公式 | 新公式 |
|------|--------|--------|
| 原投入 | 5万 | 5万 |
| 当前市值 | 2万 | 2万 |
| 亏损 | -60% | -60% |
| 当前仓位（旧） | 2万/80万=2.5% | max(5,2)/80=6.25% |
| 凯利上限 | 40% | 40% |
| 估值分位 | 10%（极低估） | 10%（极低估） |
| 估值系数 | — | ×1.0 |
| 目标仓位 | — | min(40%×1.0, 25%, 25%, 穿透余量, 资金) |
| 调整周期 | — | 3个月（动态，极低估） |
| **月度建议** | 2万×5%=1000元 | (25%-6.25%)×80万/3 = **5000元/月** |

**对比效果**：新公式月度建议是旧公式的5倍，能在3个月内把白酒从6.25%补到25%目标仓。

### 维度2：穿透指数集中度（解决"同类重复"）

**用户确认：软提示**（给警告但允许用户手动确认）。

#### 2.1 穿透计算

```python
def _calc_index_exposure(holdings: list, total_assets: float) -> dict:
    """计算每个指数的穿透暴露占比。
    
    数据源：fund_metadata.tracking_index（已验证存在）
    返回：{index_code: {"total_pct": 12.5, "funds": [...], "limit_pct": 30, "exceeded": bool}}
    """
    exposure = {}
    for h in holdings:
        index_code = h.get("index_code") or _lookup_tracking_index(h["fund_code"])
        if not index_code:
            continue
        # 基于原始投入穿透（避免亏损标的被低估暴露）
        effective_base = max(h.get("total_cost", 0), h.get("current_value", 0))
        pct = effective_base / total_assets * 100
        exposure.setdefault(index_code, {"total_pct": 0, "funds": []})
        exposure[index_code]["total_pct"] += pct
        exposure[index_code]["funds"].append(h["fund_code"])
    return exposure
```

```python
INDEX_EXPOSURE_LIMITS = {
```

```python
def _check_index_exposure_warning(exposure: dict, index_code: str, fund_type: str) -> dict:
    """检查当前指数穿透是否超限，返回软提示。"""
    current = exposure.get(index_code, {}).get("total_pct", 0)
    limit = INDEX_EXPOSURE_LIMITS.get(fund_type, 30)
    if current >= limit:
        return {
            "level": "warning",
            "message": f"该指数穿透仓位{current:.1f}%已超{fund_type}上限{limit}%，建议优先补仓其他低估品种",
            "exceeded": True,
            # 软提示：不拦截，仅警告
        }
    # 穿透余量（供维度1目标仓位约束用）
    room_pct = limit - current
    return {"level": "ok", "room_pct": room_pct, "exceeded": False}
```

#### 3.2 仓位不足判定（基于原始投入，核心修正）

```python
def _check_first_position_needed(holding: dict, fund_type: str, total_assets: float) -> dict:
    """判断是否需要首次仓补足。
    
    关键：基于 total_cost 判断，而非 current_value
    原因：首次建仓只关心"投入了多少"，不关心"现在值多少"
    """
    first_pct = FIRST_POSITION_PCT.get(fund_type, 5)
    total_cost = holding.get("total_cost", 0)
    current_value = holding.get("current_value", 0)
    
    # 基于原始投入判断
    cost_position_pct = total_cost / total_assets * 100
    
    if cost_position_pct < first_pct * 0.5:
        # 仓位严重不足（< 标准仓50%）
        target_add = total_assets * first_pct / 100 - total_cost  # 补足至标准仓
        monthly = target_add / 3   # 分3个月
        return {
            "needed": True,
            "level": "critical",
            "first_pct": first_pct,
            "current_cost_pct": round(cost_position_pct, 2),
            "target_add_total": round(target_add, 2),
            "monthly_add": round(monthly, 2),
            "period_months": 3,
            "reason": f"原投入{cost_position_pct:.1f}% < 首次标准仓{first_pct}%的50%，分3个月补足",
        }
    return {"needed": False}
```

#### 3.3 与维度1的关系

```python
# 最终月度加仓 = max(目标缺口驱动, 首次仓补足)
target_driven_monthly = (维度1) gap_pct × total_assets / adjust_months
first_position_monthly = (维度3) target_add / 3
final_monthly_add = max(target_driven_monthly, first_position_monthly)
```

#### 3.4 举例

```
持仓：白酒C 原投入 0.8万（1%），当前市值 0.5万
- 基于当前市值：0.5/80=0.625% < 2.5%（标准仓5%的50%）→ 触发补足
- 基于原始投入：0.8/80=1% < 2.5% → 同样触发补足（一致）

补足金额 = 80万 × 5% - 0.8万 = 3.2万，分3个月 = 1.07万/月
对比旧公式：0.5万 × 5% = 250元/月（差40倍）
```

### 维度4：收益弹性预警（解决"涨了无感"）

#### 4.1 弹性计算

```python
def _calc_return_elasticity(holding: dict, valuation: dict, total_assets: float) -> dict:
    """计算持仓收益弹性。
    
    弹性 = 仓位 × 指数预期涨幅（基于估值分位的历史均值）
    """
    # 基于原始投入计算仓位（避免深套被低估）
    effective_base = max(holding.get("total_cost", 0), holding.get("current_value", 0))
    position_pct = effective_base / total_assets
    
    # 估值预期涨幅（基于 L5 胜率数据）
    percentile = valuation.get("percentile", 50)
    if percentile < 20:
        expected_return_pct = 30   # 极低估，历史均值+30%
    elif percentile < 40:
        expected_return_pct = 15
    elif percentile < 60:
        expected_return_pct = 5
    else:
        expected_return_pct = -10  # 高估，预期下跌
    
    elasticity = position_pct * expected_return_pct / 100  # 组合收益贡献(%)
    
    return {
        "position_pct": round(position_pct, 2),
        "expected_return_pct": expected_return_pct,
        "elasticity": round(elasticity, 2),  # 组合收益贡献
        "level": "low" if position_pct < 3 and expected_return_pct > 10
                 else ("high_risk" if position_pct > 30 and expected_return_pct < 0
                       else "normal"),
    }
```

#### 4.2 预警文案

```python
# 低弹性预警
if elasticity["level"] == "low":
    msg = (f"仓位{position_pct:.1f}%，即使涨{expected_return_pct}%"
           f"也只贡献组合{elasticity:.2f}%收益。"
           f"建议补至5%+，让涨{expected_return_pct}%能贡献{5*expected_return_pct/100:.2f}%组合收益")

# 高弹性风险
if elasticity["level"] == "high_risk":
    msg = (f"仓位{position_pct:.1f}%，跌{abs(expected_return_pct)}%"
           f"会损失组合{abs(elasticity):.2f}%收益。建议减仓至15%")
```

#### 4.3 作用

仅在 UI 展示橙色/红色预警条，不直接修改金额。作为维度1目标仓位的"放大器"：低弹性标的可适当上调目标仓位。

### 3.3 现有公式基准修正（必做）

## 四、决策树与金额公式

### 4.1 单标的决策树

```
对每个持仓：

0. 计算原始投入基准
   effective_base = max(total_cost, current_value)
   position_pct = effective_base / total_assets

1. 维度2：穿透集中度检查
   ├─ 超限 → 软提示 warning（不拦截），目标仓位受余量约束
   └─ 未超限 → 继续

2. 维度5：减仓信号检查（优先级最高）
   ├─ 估值>80% OR 盈利>止盈线×1.5 OR 亏损>止损线(估值非低估) → 触发减仓
   │   ├─ 计算减仓金额（维度5公式）
   │   ├─ 加仓金额 = 0
   │   └─ return 减仓建议
   └─ 未触发 → 继续加仓逻辑

3. 维度1：目标仓位计算
   target_pct = min(凯利×估值系数, 类型硬上限, 25%, 穿透余量, 资金余量)
   gap_pct = target_pct - position_pct

   ├─ gap_pct < 0（超配）→ 持有，发维度4高弹性预警
   ├─ gap_pct = 0（达标）→ 持有
   └─ gap_pct > 0（低配）→ 计算目标驱动金额

4. 维度3：首次仓补足检查
   if cost_position_pct < first_pct × 0.5:
       first_position_monthly = (总资产×first_pct - total_cost) / 3

5. 维度4：收益弹性预警
   if position_pct < 3% and 估值低估:
       发低弹性预警（UI提示，不直接修改金额）

6. 金额汇总
   target_driven_monthly = max(维度1缺口÷周期, 维度3首次仓÷3)
   signal_driven_amount = max(引擎1, 引擎2, 信号B, 信号C, 信号D, 信号E)
   base_amount = max(target_driven_monthly, signal_driven_amount)

7. 多维约束
   final_amount = min(base_amount, 资金约束, 安全阀)
   if 穿透超限: final_amount × 0.5（软惩罚，不完全截断）+ UI警告

8. 返回
   {
     "target_position_pct": ...,
     "current_position_pct": ...,
     "gap_pct": ...,
     "target_driven_amount": ...,
     "signal_driven_amount": ...,
     "final_amount": ...,
     "index_exposure": {...},  # 维度2
     "first_position": {...},  # 维度3
     "elasticity": {...},      # 维度4
     "exit_signals": [...],    # 维度5
     "cash_warning": "...",    # 维度6
   }
```

### 4.2 最终金额公式汇总

```
加仓金额 = min(
  max(
    目标缺口驱动 = (目标仓位 - 当前仓位) × 总资产 ÷ 调整周期,
    首次仓补足 = (首次标准仓 - 原始投入仓位) × 总资产 ÷ 3,
    信号触发金额 = max(定投, 金字塔, 趋势, 大跌, 价值平均, 网格)
  ),
  资金约束上限,
  安全阀上限,
  穿透余量约束（软，×0.5 惩罚）
)

减仓金额 = 维度5触发金额（基于当前市值，卖出按现价）
```

## 九、风险与注意事项

1. **穿透映射依赖**：维度2依赖 `fund_metadata.tracking_index`，映射不全会有漏算
   - 缓解：复用 `services/index_fund_mapper.py` 已有映射逻辑，启动时回填

2. **目标仓位过激进**：凯利数据不足时默认25%，所有低估标的都目标25%
   - 缓解：加用户风险偏好调节（保守×0.6/稳健×0.8/平衡×1.0/进取×1.2）

3. **资金约束可能导致建议过小**：资金不足时所有建议按比例缩减
   - 缓解：UI 明确展示"资金不足，建议延长周期或追加资金"

4. **首次仓分3个月可能买在半山腰**：一次性 vs 分批的权衡
   - 缓解：分3个月，配合金字塔补仓（继续下跌时自动加码）

5. **减仓资金转移**：减仓释放资金需手动再投资，不自动转入其他标的
   - 缓解：UI 提示"减仓释放X元，可用于补仓其他低估品种"
