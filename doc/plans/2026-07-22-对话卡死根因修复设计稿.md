# 对话卡死根因修复设计稿

## 日期：2026-07-22

## 一、问题现象

conv#132、conv#133 第一次请求均出现 6~40 分钟无日志卡死，模式完全一致：
1. clarify LLM 调用完成，打印"专家配额补充"日志
2. 之后日志完全空白（conv#132 空白 6 分 22 秒，conv#133 空白 40 分 30 秒）
3. 用户重发后第二次请求正常
4. 事件循环未卡死（alert_scanner 仍在运行）

## 二、根因定位

### 2.1 卡死点

`_stream_build_context`（orchestrator.py:4496）中的 `build_bond_fund_holdings_context()`（portfolio_context.py:4577）调用。

### 2.2 完整因果链

```
_producer 线程
  → orchestrate_stream (orchestrator.py:5294)
    → _stream_build_context (orchestrator.py:5450) — 同步函数，无 yield
      → build_bond_fund_holdings_context (portfolio_context.py:209)
        → with ThreadPoolExecutor(max_workers=1) as executor:       ← L256
          → executor.submit(get_fund_holdings, fund_code)
            → get_fund_holdings (db/portfolio.py:2187)
              → _akshare_fund_holdings (db/portfolio.py:2234)
                → ak.fund_portfolio_bond_hold_em                    ← akshare HTTP 无超时，卡死
          → future.result(timeout=5)                                ← 超时返回 None，但线程仍在运行
        → with 退出 → __exit__ → shutdown(wait=True)               ← 无限等待 akshare zombie 线程！
      → build_bond_fund_holdings_context 无限阻塞
    → _stream_build_context 无限阻塞
  → orchestrate_stream 在 yield 第一个事件之前无限阻塞
→ for event in orchestrate_stream(...) 循环体不执行
→ 超时检查(warn_at_sec=5min / abort_at_sec=8min)在循环体内，永远不触发
→ 40 分钟无任何日志
```

### 2.3 核心缺陷

`concurrent.futures.ThreadPoolExecutor` 的 `with` 语句在退出时调用 `shutdown(wait=True)`，会**无限等待所有已提交的任务完成**。`future.result(timeout=5)` 超时后只是停止等待结果，但底层线程仍在运行。如果 akshare HTTP 请求挂起（无超时），`with` 退出会永远阻塞。

### 2.4 为什么 conv#132/133 都触发

两个问题都涉及"博时恒乐债券"（债券基金），触发 `build_bond_fund_holdings_context` 扫描债券基金底层持仓，akshare `fund_portfolio_bond_hold_em` 接口不稳定。

### 2.5 为什么 8 分钟超时不生效

`_producer` 的超时检查（warn_at_sec/abort_at_sec）在 `for event in orchestrate_stream(...)` **循环体内**。如果 `orchestrate_stream` 在 yield 第一个事件之前卡住，循环体不会执行，超时检查永远不触发。

## 三、修复方案

### 3.1 P0 核心修复：build_bond_fund_holdings_context — 消除 with 阻塞

**文件**：`services/portfolio/portfolio_context.py` L254-263

**修改前**：
```python
import concurrent.futures
with concurrent.futures.ThreadPoolExecutor(max_workers=1) as executor:
    future = executor.submit(get_fund_holdings, fund_code)
    try:
        fund_data = future.result(timeout=5)
    except concurrent.futures.TimeoutError:
        fund_data = None
```

**修改后**：
```python
import concurrent.futures
# 不使用 with 语句：with 退出时 shutdown(wait=True) 会无限等待 akshare zombie 线程
# 改为手动管理，超时后不等待线程退出（线程为 daemon，进程退出时自动回收）
executor = concurrent.futures.ThreadPoolExecutor(max_workers=1)
future = executor.submit(get_fund_holdings, fund_code)
try:
    fund_data = future.result(timeout=5)
except concurrent.futures.TimeoutError:
    fund_data = None
    logger.warning(f"[bond_holdings] {fund_code} 底层持仓获取超时 5s，跳过")
except Exception as e:
    fund_data = None
    logger.warning(f"[bond_holdings] {fund_code} 底层持仓获取失败: {e}")
finally:
    # shutdown(wait=False) 不等待线程完成，立即返回
    # cancel_futures=True 尝试取消尚未开始的任务（Python 3.9+）
    executor.shutdown(wait=False, cancel_futures=True)
```

### 3.2 P1 深层修复：_stream_build_context 超时保护

**文件**：`agent/core/orchestrator.py` L5450-5453

`_stream_build_context` 是同步函数，在 `_producer` 线程中执行。当前无超时保护，内部有多个外部 I/O 调用。由于它不是 async 函数，无法直接用 `asyncio.wait_for`。

**方案**：将 `_stream_build_context` 用 `ThreadPoolExecutor` + `future.result(timeout=60)` 包装，超时后跳过。

**修改前**（orchestrator.py:5450-5453）：
```python
ctx_data = _stream_build_context(refined_query, rag_context, complexity,
                                 context_config, token_budget, history)
```

**修改后**：
```python
# _stream_build_context 内部有多个外部 I/O（akshare/HTTP），用超时包装避免卡死
# 案例：conv#132/133 因 build_bond_fund_holdings_context 内 akshare 卡死导致 40min 无日志
import concurrent.futures
_ctx_executor = concurrent.futures.ThreadPoolExecutor(max_workers=1)
_ctx_future = _ctx_executor.submit(
    _stream_build_context, refined_query, rag_context, complexity,
    context_config, token_budget, history
)
try:
    ctx_data = _ctx_future.result(timeout=60)
except concurrent.futures.TimeoutError:
    logger.warning(f"[trace:{trace_id}] _stream_build_context 超时 60s，使用空上下文降级")
    ctx_data = {"llm_messages": [], "prebuilt_context": "", "system_content": build_orchestrator_system_prompt()}
except Exception as e:
    logger.warning(f"[trace:{trace_id}] _stream_build_context 失败: {e}，使用空上下文降级")
    ctx_data = {"llm_messages": [], "prebuilt_context": "", "system_content": build_orchestrator_system_prompt()}
finally:
    _ctx_executor.shutdown(wait=False, cancel_futures=True)
```

### 3.3 P1 超时保护前移：_producer 首事件超时守卫

**文件**：`routers/conversation/conversations.py` L1672

当前 `_producer` 的超时检查在 `for event in orchestrate_stream(...)` 循环体内，如果生成器在 yield 第一个事件之前卡死，超时检查永远不触发。

**方案**：在 `for` 循环之前加一个"首事件超时"守卫，用 `concurrent.futures` 包装 `orchestrate_stream` 的第一个 `next()` 调用。

**修改前**（conversations.py:1672）：
```python
for event in orchestrate_stream(effective_query, msg_list, orchestrator_context, cancel_event=cancel_event, conversation_id=conv_id, message_id=stream_msg_id, trace_id=trace_id, target_specialists=req.target_specialists):
```

**修改后**：
```python
# 创建生成器
_orch_gen = orchestrate_stream(effective_query, msg_list, orchestrator_context, cancel_event=cancel_event, conversation_id=conv_id, message_id=stream_msg_id, trace_id=trace_id, target_specialists=req.target_specialists)

# P0 修复：首事件超时守卫
# orchestrate_stream 在 yield 第一个事件前可能卡在 _stream_build_context（akshare 等）
# 此时 for 循环体不执行，循环内超时检查不触发，导致无限卡死
# 用 ThreadPoolExecutor 包装 next() 调用，超时后降级
import concurrent.futures as _cf
_first_event_executor = _cf.ThreadPoolExecutor(max_workers=1)
_first_event_future = _first_event_executor.submit(next, _orch_gen)
try:
    _first_event = _first_event_future.result(timeout=90)  # 首事件 90s 超时
except _cf.TimeoutError:
    logger.error(f"[trace:{trace_id}] orchestrate_stream 首事件超时 90s，可能卡在上下文构建（akshare/HTTP），强制终止")
    _first_event_executor.shutdown(wait=False, cancel_futures=True)
    q.put({"type": "error", "message": "上下文构建超时，请重试。可能因外部数据源（akshare）响应缓慢。"})
    _running_agents.pop(f"prod_{conv_id}", None)
    q.put(None)
    return
except StopIteration:
    _first_event_executor.shutdown(wait=False, cancel_futures=True)
    _running_agents.pop(f"prod_{conv_id}", None)
    q.put(None)
    return
finally:
    _first_event_executor.shutdown(wait=False, cancel_futures=True)

# 处理首事件，然后进入正常循环
_events_iter = _orch_gen

# 内联处理首事件的逻辑（与循环体相同）
# ... [首事件处理代码]

# 正常循环处理后续事件
for event in _events_iter:
    # ... [原有循环体]
```

**简化实现**（避免重复代码）：将首事件放入列表，然后 `itertools.chain([first_event], orch_gen)` 与循环体统一处理。

```python
import itertools

_orch_gen = orchestrate_stream(...)
# 首事件超时守卫
import concurrent.futures as _cf
_executor = _cf.ThreadPoolExecutor(max_workers=1)
try:
    _first_future = _executor.submit(next, _orch_gen)
    _first_event = _first_future.result(timeout=90)
    _event_chain = itertools.chain([_first_event], _orch_gen)
except _cf.TimeoutError:
    logger.error(f"[trace:{trace_id}] orchestrate_stream 首事件超时 90s")
    q.put({"type": "error", "message": "上下文构建超时（akshare/HTTP 可能卡住），请重试"})
    _executor.shutdown(wait=False, cancel_futures=True)
    _running_agents.pop(f"prod_{conv_id}", None)
    q.put(None)
    return
except StopIteration:
    _executor.shutdown(wait=False, cancel_futures=True)
    _running_agents.pop(f"prod_{conv_id}", None)
    q.put(None)
    return
finally:
    _executor.shutdown(wait=False, cancel_futures=True)

for event in _event_chain:
    # ... [原有循环体，不变]
```

### 3.4 P2 akshare HTTP 超时（可选，后续优化）

akshare 部分接口无 HTTP 超时设置。可在 `db/portfolio.py` 的 `_akshare_fund_holdings` 中给 akshare 调用加全局超时（如通过 `signal.alarm` 或 `threading.Timer` + 异常注入）。此为可选项，P0-P1 修复已能避免卡死。

## 四、影响范围

| 修改 | 文件 | 影响面 |
|------|------|--------|
| P0: ThreadPoolExecutor 不用 with | portfolio_context.py | 仅影响债券基金底层持仓获取，超时后跳过 |
| P1: _stream_build_context 超时 | orchestrator.py | 上下文构建超时后降级为空上下文，不影响主流程 |
| P1: 首事件超时守卫 | conversations.py | 首事件超时后返回错误提示，用户可重试 |
