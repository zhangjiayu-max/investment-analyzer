# 2026-07-21 关注机会上车信号 P0 优化设计稿

> 调研报告:见同日调研输出。本设计稿仅覆盖 P0 三项,聚焦"上车判断多维度化、退出信号闭环、上车信号回测"。

## 一、目标

1. **P0-1**:关注机会(`watchlist.py`)上车判断接入技术/资金/情绪多维信号,引入 `signal_confidence` 字段。
2. **P0-2**:退出信号/波动预警开关默认开启;`alert_scanner` 新增 `scan_watchlist_exit_signals()` 扫描器。
3. **P0-3**:新建 `watchlist_signal_backtests` 表,signal_status 由非 green 变 green 时自动插入回测记录;15 交易日后回测命中率;前端顶部统计卡新增"信号命中率"chip。

## 二、P0-1 多维信号接入 + 信号置信度

### 2.1 现状

[`watchlist.py:618-663 _compute_signal_status`](../../backend/routers/portfolio/watchlist.py) 仅看估值/回撤/净值分位,与 [`opportunity_engine._score_theme`](../../backend/services/advisor/opportunity_engine.py) 多维评分体系割裂。

### 2.2 改造方案

**新增函数** `_compute_multidim_signals(item, index_name, index_code) -> dict`:

```python
{
    "tech_signal": "bull|bear|neutral",
    "tech_score": 0-15,
    "capital_signal": "inflow|outflow|neutral",
    "capital_score": -5~+10,
    "sentiment_signal": "fear|greed|neutral",
    "sentiment_score": -5~+10,
    "reasons": ["技术面MACD多头+5", "北向资金净流出-2", ...]
}
```

**实现策略**(避免性能崩盘):
- 仅当 `signal_status` 已是 `green/yellow` 时触发多维查询(red/gray 不查,降本)
- 复用 `opportunity_engine._get_technical_score / _get_capital_flow_score / _get_sentiment_score`,但传入"虚拟 theme_rule"(用 `index_name` 反查 `_THEME_INDEX_CODES` 或回退到 `index_code`)
- akshare 调用加 5s 超时(用 `asyncio.wait_for` 在 to_thread 中执行)
- 多维结果 5 分钟缓存(同 `_patrol_cache` 周期)

**信号灯降级规则**:
- green + tech_signal=bear → 降级 yellow,reason 拼接"⚠️ 技术面看空"
- green + capital_signal=outflow → 维持 green,reason 拼接"⚠️ 资金流出"
- yellow + tech_signal=bull → 升级 green,reason 拼接"✓ 技术面确认"
- 其他维持原判

**signal_confidence 计算**(0-100):
- 估值偏离度 40%:green=80、yellow=55、red=20、gray=0
- 技术面 20%:bull=100、neutral=50、bear=10
- 资金面 20%:inflow=90、neutral=50、outflow=20
- 情绪面 20%:fear=80(逆向)、neutral=50、greed=30
- 无多维数据时仅按估值偏离度计算,confidence 上限 60(强制提示数据不足)

### 2.3 持久化字段

## 三、P0-2 退出信号默认开启 + alert_scanner 扫描器

### 3.1 现状

[`watchlist.py:307,343`](../../backend/routers/portfolio/watchlist.py) `exit_signal_enabled` 和 `volatility_alert_enabled` 默认 False,上车后无人看管。

### 3.2 改造方案

#### 3.2.1 默认值调整

- `watchlist.exit_signal_enabled`:默认 **true**
- `watchlist.volatility_alert_enabled`:默认 **true**
- 已通过 `get_config_bool(key, default)` 读取的,只需修改默认参数

#### 3.2.2 alert_scanner 新增扫描器

[`alert_scanner.py:1062 run_periodic_scan`](../../backend/services/advisor/alert_scanner.py) 结果 dict 新增 `watchlist_exit`:

```python
def scan_watchlist_exit_signals() -> dict:
    """扫描关注列表已上车基金的止盈/止损触发,生成 alert。
    
    与 watchlist_patrol_api 区别:
    - patrol 是用户主动调用,5分钟缓存
    - 这里是定时扫描(每30分钟),触发后立即生成 portfolio_alerts
    - 24h 去重(fund_code + alert_type)
    """
```

- 仅扫描 `status='bought'` 且 `entry_price` 非空的项
- 计算 pnl_pct,触发 profit_target/stop_loss 时:
  - `alert_type = "watchlist_profit_target"` / `"watchlist_stop_loss"`
  - `severity = "info"` / `"warning"`
  - 24h 去重(复用 `_is_alert_recently_created`)
- 开关:`alerts.watchlist_exit_scan_enabled` 默认 true

### 3.3 影响面

- 不影响 `watchlist_patrol_api` 的现有逻辑,只是在 `run_periodic_scan` 加一个扫描器
- `_is_alert_recently_created` 已修复(2026-07-20),可直接复用

## 四、P0-3 watchlist 上车信号回测闭环

### 4.1 表结构

新建表 `watchlist_signal_backtests`(在 `db/__init__.py` 的 `init_db` 中):

```sql
CREATE TABLE IF NOT EXISTS watchlist_signal_backtests (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    watchlist_id INTEGER NOT NULL,
    fund_code TEXT NOT NULL,
    fund_name TEXT,
    signal_date TEXT NOT NULL,         -- 信号触发日(当日)
    signal_status TEXT NOT NULL,       -- green(可上车)/yellow(接近,暂不回测)
    entry_nav REAL,                    -- 信号触发日基金净值
    entry_percentile REAL,             -- 信号触发日估值分位
    review_date TEXT NOT NULL,         -- signal_date + 15 交易日
    review_nav REAL,
    change_pct REAL,
    hit INTEGER,                       -- 1=命中(change_pct>=3%) / 0=未命中
    signal_confidence REAL,            -- 信号置信度(P0-1 新增)
    created_at TEXT DEFAULT (datetime('now','localtime')),
    reviewed_at TEXT,
    FOREIGN KEY (watchlist_id) REFERENCES watchlist(id) ON DELETE CASCADE
);
CREATE INDEX IF NOT EXISTS idx_wl_bt_review ON watchlist_signal_backtests(review_date, hit);
CREATE INDEX IF NOT EXISTS idx_wl_bt_fund ON watchlist_signal_backtests(fund_code);
```

### 4.2 CRUD(`db/watchlist.py` 新增)

- `create_signal_backtest(item: dict) -> int`
- `list_pending_signal_backtests() -> list[dict]`(filter: hit IS NULL AND review_date <= today)
- `update_signal_backtest(bt_id: int, **fields) -> bool`
- `get_signal_backtest_stats(fund_code: str = None) -> dict` 返回 `{total, reviewed, hit, miss, hit_rate}`

### 4.3 触发逻辑(在 `watchlist_patrol_api` 中)

每次巡检时,对每只 watching 基金:
1. 读取上一次 `signal_status`(从 watchlist 表新增 `last_signal_status` 字段)
2. 若新 `signal_status == 'green'` 且 `last_signal_status != 'green'`:
   - 调用 `create_signal_backtest({watchlist_id, fund_code, signal_date: today, signal_status: 'green', entry_nav: current_nav, entry_percentile: current_pct, review_date: today + 15 交易日, signal_confidence})`
3. 更新 watchlist.`last_signal_status`

**注意**:同一天内仅插入一条(去重:`SELECT 1 FROM watchlist_signal_backtests WHERE watchlist_id=? AND signal_date=?`)

### 4.4 回测逻辑(`review_watchlist_signal_backtests`)

在 `backend/services/advisor/opportunity_engine.py` 中新增函数(或独立模块 `services/advisor/watchlist_backtest.py`):

```python
def review_watchlist_signal_backtests() -> dict:
    """批量回测已到期的 watchlist 信号记录。
    
    命中定义:15 交易日后涨幅 >= 3%(考虑申购费+赎回费)。
    
    review_nav 来源:
    1. 优先 fund_nav_history 表(本地)
    2. 兜底 fetch_fund_nav(实时拉取)
    """
```

### 4.5 定时回测

- 每日 09:35 自动执行(比 `_auto_opportunity_backtest` 晚 5 分钟避免抢资源)
- 开关:`alerts.watchlist_backtest_enabled` 默认 true

## 六、配置项变更

| key | 默认值 | 说明 |
|-----|--------|------|
| `watchlist.multidim_signal_enabled` | true | P0-1 多维信号开关 |
| `watchlist.multidim_signal_timeout_seconds` | 5 | akshare 调用超时 |
| `watchlist.exit_signal_enabled` | **true**(原 false) | P0-2 退出信号 |
| `watchlist.volatility_alert_enabled` | **true**(原 false) | P0-2 波动预警 |

## 七、风险与降级

1. **akshare 多维查询性能**:5 分钟缓存 + 5s 超时 + 仅 green/yellow 触发,失败时降级到纯估值判断(只填 `signal_confidence` 上限 60)
2. **回测表膨胀**:每只基金每次信号触发一条,假设 50 只基金每月触发 2 次,年增量 1200 条,可接受
3. **历史数据缺失**:P0-3 上线后新数据开始累积,1 个月后才有首个命中率,前端显示"--"占位
4. **退出信号误报**:`profit_target` 默认 30%、`stop_loss` 默认 10%,用户可在上车设置弹窗修改

## 八、实施顺序

1. P0-2(最简单,先打通闭环)→ 默认值 + alert_scanner 新扫描器
2. P0-3(建表 + CRUD + 触发 + 回测 + 前端)
3. P0-1(多维信号 + confidence)→ 最后做,避免与 P0-3 调试冲突
