## 1. 家庭财务统一仪表盘

### 目标

把净值、现金流、负债、目标进度、资金桶、配置偏离、压力测试结果放到同一个视图里，让投资建议不再只围绕某只基金或某个指数，而是回到完整家庭资产负债表。

### 核心能力

- 净值总览：总资产、现金、持仓市值、总成本、浮盈亏、收益率。
- 现金流：月收入、月支出、月结余、结余率、备用金覆盖月数。
- 负债：负债摘要、月供压力、负债收入比。
- 目标进度：资金桶目标金额、当前金额、进度、优先级。
- 投资状态：当前配置、目标配置、最大偏离、调仓建议。
- 风险视图：压力测试损失、备用金缓冲、组合回撤承受度。

### 建议数据来源

- `user_profiles`：收入、支出、结余、备用金、目标、负债、行为偏差。
- `goal_buckets`：目标账户和资金桶。
- `portfolio_holdings` / `portfolio_cash`：持仓和现金。
- `/api/portfolio/allocation-dashboard`：配置偏离。
- `/api/portfolio/stress-test`：压力测试。

### 初版页面

新增页面：`FamilyFinanceDashboard.vue`

导航建议：

- 名称：`财务总览`
- 位置：放在 `每日看板` 后、`资金桶` 前。

### 验收标准

- 用户能在一个页面看到家庭财务健康状态。
- 页面能明确提示：备用金是否达标、月结余是否支持新增投资、组合是否明显偏离目标。
- 页面不生成交易指令，只生成检查提示和后续行动建议。

## 2. 压力测试前端可视化入口

### 目标

当前已经有确定性压力测试 API，但还没有前端入口。后续需要把压力测试接到“配置偏离”或“持仓管理”页面，让用户在决策前直观看到组合在不同冲击下可能损失多少。

### 核心能力

- 场景选择：
  - 市场下跌 20%
  - 利率上行
  - 流动性冲击
- 结果展示：
  - 当前总资产
  - 压力后总资产
  - 预计损失金额
  - 预计损失比例
  - 风险等级
  - 各资产类别损失贡献
  - 备用金提示

### 建议接口

已完成：

```http
POST /api/portfolio/stress-test
```

请求：

```json
{ "scenario": "market_drop_20" }
```

### 初版交互

- 在“配置偏离”页面右侧增加“压力测试”面板。
- 使用 segmented control 切换场景。
- 使用条形表展示各资产类别冲击结果。
- 如果备用金不足，用 warning 状态提示。

### 验收标准

- 用户点击场景后无需 LLM 即可得到结果。
- 结果能解释“损失主要来自哪些资产类别”。
- 若备用金桶未达标，页面必须明确提示。

## 3. 策略沙盒和历史回测

### 目标

让用户可以测试定投、再平衡、估值分位买卖规则在历史数据中的表现，避免仅凭主观感觉调整策略。

### 初版策略

1. 固定金额定投
2. 低估多投、高估少投
3. 估值分位买入 / 止盈
4. 定期再平衡
5. 偏离阈值再平衡

### 输入参数

- 标的：指数代码或基金代码。
- 时间区间：开始日期、结束日期。
- 初始资金。
- 每月投入金额。
- 买入规则。
- 卖出 / 止盈规则。
- 再平衡频率或偏离阈值。

### 输出结果

- 总投入。
- 期末资产。
- 累计收益率。
- 年化收益率。
- 最大回撤。
- 波动率。
- 交易次数。
- 现金闲置比例。
- 与买入持有对比。

### 建议模块

- 后端新增：`backend/strategy_sandbox.py`
- API 新增：
  - `POST /api/strategy-sandbox/backtest`
  - `GET /api/strategy-sandbox/presets`
- 前端新增：`StrategySandboxPage.vue`

### 验收标准

- 至少支持一个指数的定投回测。
- 回测结果能和买入持有进行对比。
- 回测明确标注：历史不代表未来。

## 4. 多模型 / 多 Prompt 评审

### 目标

关键投资决策由不同模型或不同 prompt 进行交叉评审，降低单一模型偏差，特别适用于买入、卖出、重仓、清仓、资产配置大调整等高风险场景。

### 触发条件

- 决策类型为：买入、加仓、卖出、减仓、清仓、再平衡。
- 决策金额或仓位超过用户设定阈值。
- 风险 Agent 或反方观点 Agent 给出 blocker。
- 用户主动点击“多模型评审”。

### 评审维度

- 适当性：是否匹配资金用途、期限、风险承受。
- 证据质量：数据是否新鲜、是否有来源、是否过度依赖单一证据。
- 反方观点：是否充分说明不该做的理由。
- 行动可执行性：是否有仓位、节奏、退出条件、复盘日期。
- 过度自信：是否把不确定判断说成确定结论。

### 数据结构建议

新增表：`decision_peer_reviews`

字段建议：

- `id`
- `decision_id`
- `reviewer_type`
- `model_name`
- `prompt_version`
- `verdict`
- `score_json`
- `concerns_json`
- `suggestions_json`
- `created_at`

### API 建议

- `POST /api/decisions/{decision_id}/peer-review`
- `GET /api/decisions/{decision_id}/peer-reviews`

### 验收标准

- 同一决策可以生成至少两种独立评审意见。
- 评审结果可回写决策前检查。
- 如果多个评审都给出高风险结论，决策状态建议降级为 `deferred`。

## 5. RAG 个性化排序观测页

### 目标

当前 RAG 已经支持个性化加权，但用户和开发者还看不到“为什么这条证据排在前面”。后续需要提供观测页，帮助调试和建立信任。

### 核心展示

- 查询文本。
- 原始命中结果。
- 最终排序结果。
- `_score`
- `personal_boost`
- `personal_reasons`
- 命中的画像字段：
  - 关注资产
  - 资金用途
  - 行为偏差
  - 知识原子类型
  - 反方观点 / 限制条件

### 页面建议

可扩展现有 `RagTestPage.vue`：

- 增加“个性化解释”列。
- 增加“使用当前画像重排”开关。
- 增加“只看用户经验 / 反方观点 / checklist”过滤器。

### 验收标准

- 同一个 query 可以看到个性化排序前后的变化。
- 每条结果能解释加权原因。
- 支持开发者快速发现错误加权或噪声知识。

## 6. 后续开发顺序建议

1. 压力测试前端入口  
   已有 API，开发成本低，能立刻进入决策前检查流程。

2. 家庭财务统一仪表盘  
   当前数据底座已经具备，适合做成新的核心首页。

3. RAG 个性化排序观测页  
   帮助后续调试“越用越懂我”的效果。

4. 策略沙盒和历史回测  
   工程量较大，需要认真处理历史数据、交易成本、缺失值和基准对比。

5. 多模型 / 多 Prompt 评审  
   适合在决策中枢稳定后接入，避免过早增加 token 成本和流程复杂度。
