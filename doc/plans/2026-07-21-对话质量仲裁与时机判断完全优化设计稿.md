# 对话质量仲裁与时机判断完全优化设计稿

**日期**：2026-07-21
**背景**：conv#131 暴露系统在仲裁一致性、时机判断、止盈不止损原则三大维度存在严重缺陷
**目标**：通过代码层硬兜底+prompt 增强+工具结果注入，彻底修复"仲裁裁决被架空、亏损标的被建议清仓、时机判断缺失"等问题

---

## 一、conv#131 暴露的核心问题

### 问题 1：仲裁裁决与综合报告自相矛盾（无一致性校验）

| 环节 | 结论 |
|---|---|
| 仲裁 Agent 裁决 | `verdict="建议持有/观望"`, `confidence="high"` |
| 综合报告操作建议 | "立即卖出恒生科技 2 只基金"、"3 个月内清理 10 只小仓位" |

**根因**：
- 仲裁模块 `arbitration.py:19-145` 输出 `verdict`/`confidence`/`reasoning`
- 综合阶段 `pipeline.py:2087-2107` / `orchestrator.py:5017-5028` 仅通过 prompt 软约束"必须与仲裁一致"
- **没有任何代码层校验**：LLM 可以完全无视仲裁，自由生成冲突的操作建议
- 现有 `_enforce_risk_veto`（`pipeline.py:2142-2172`）只处理 `blackboard.get_vetoes()`，不处理仲裁 `verdict`

### 问题 2：时机判断完全缺失

**用户原话**："需要清仓一些，是不是不太好**现在**"——问的是**时机**

**AI 回答**：
- "未来一周内择机执行"清仓恒生科技
- "3 个月内清理完毕"小仓位
- "立即行动——整合持仓"

**根因**：
- 综合报告 prompt（`orchestrator.py:633-678`）没有"时机判断"显式逻辑
- 时机判断仅在专家层 `_build_dca_rules`（`orchestrator.py:575-608`）注入，且只约束估值分析师/资产配置师
- 综合阶段可以建议"立即卖出"而无需查询当前估值分位
- 风险管理师查到白酒 PE 0.45% 分位、恒生科技 -19.5% 亏损，但综合报告完全没用这些数据

### 问题 3："止盈不止损"原则未实现

**综合报告行为**：
- 恒生科技 -19.5% 亏损 → 建议"全部卖出"（锁定亏损）
- 白酒 PE 0.45% 分位 → 建议"暂停买入"（错过底部）

**根因**：
- 综合报告 prompt **没有**"止盈不止损"明文原则
- `pipeline.py:2030` 是"止盈+止损"对等提示，不是"止盈不止损"
- `_build_dca_rules:605` 有"禁止建议减仓亏损基金（除非风险止损场景）"，但只在专家层约束，不约束综合报告
- `smart_add_planner.py:669-670` 有"低估时不止损"，但不在综合报告路径上

### 问题 4：query_smart_add_plan 工具广播字段提取缺失

**项目规范要求**："综合报告需充分利用 query_smart_add_plan 返回的档位数据作为建议金额依据"

**现状**：
- `tool_broadcast.py:169-272` 的 `_extract_key_fields` 没有为 `query_smart_add_plan` 实现专门分支
- 工具调用后，黑板广播只有"查询「default」完成"——档位数据完全丢失
- 综合报告 prompt 没有强制引用档位数据的约束
- 专家建议金额不一致时，综合报告无法引用档位数据作为依据

### 问题 5：交叉审阅结果未记录（conv#131 中 `cross_review_results: []`）

**调研发现**：
- 串行黑板最后位专家魔鬼代言人注入（`orchestrator.py:5664-5674`）：已实现
- disagreements 为空时二次提示强制反驳（`multi_agent.py:1345-1412`）：已实现
- 但 conv#131 中 `cross_review_results` 为空，说明该路径未触发或未记录

**根因**（待进一步确认）：
- Pipeline 路径的交叉审阅触发逻辑可能不同于流式路径
- 或 `should_run_cross_review` 的 `min_specialists=1` 条件未满足
- 或触发了但 `cross_review_results` 没写入 metadata

## 二、设计原则

1. **代码层硬兜底优先于 prompt 软约束**：关键约束必须代码校验，不能只靠 prompt
2. **所有新开关默认关闭**：符合项目规范"新 LLM 相关开关默认 false"
3. **不破坏现有路径**：只增加约束层，不修改核心流程
4. **可观测性**：所有校验失败要记录日志，便于后续优化
5. **不引入新 LLM 调用**：所有硬校验用纯 Python 关键词/正则实现，控制成本

## 三、任务分解

### P0-A：仲裁-综合一致性硬校验【核心】

**目标**：仲裁裁决为 hold/watch 时，综合报告操作建议出现"卖出/减仓/清仓"时，必须降级或警告

**新建文件**：`backend/agent/safety/arbitration_guard.py`

**核心函数**：
```python
def enforce_arbitration_consistency(
    answer: str,
    arbitration_summary: dict | None,
    trace_id: str = "",
) -> tuple[str, list[str]]:
    """
    仲裁-综合一致性硬校验。
    
    Returns:
        (修正后的 answer, warnings 列表)
    """
    # 1. 从 arbitration_summary 提取 verdict → stance
    # 2. 从 answer 中提取操作建议（正则匹配"卖出/减仓/清仓/买入/加仓"等关键词）
    # 3. 检测方向冲突：
    #    - verdict=hold/watch 但 answer 有"卖出/减仓/清仓" → 冲突
    #    - verdict=buy 但 answer 有"卖出/减仓/清仓" → 冲突
    #    - verdict=sell 但 answer 有"买入/加仓" → 冲突
    # 4. 冲突时在 answer 末尾追加警告区块：
    #    "⚠️ 系统提示：本回答的操作建议与仲裁裁决（{verdict}）方向不一致，请审慎参考。"
    #    "原仲裁裁决：{reasoning}"
    # 5. 记录 warnings 日志
```

**stance 关键词映射**：
```python
_STANCE_KEYWORDS = {
    "sell": ["卖出", "减仓", "清仓", "止损", "退出", "赎回", "全部卖", "立即卖"],
    "buy":  ["买入", "加仓", "建仓", "上车", "增持", "补仓", "定投"],
    "hold": ["持有", "观望", "保持", "暂不", "等待"],
}
```

**冲突检测规则**：
| 仲裁 verdict | answer 中检测到 | 是否冲突 | 处理方式 |
|---|---|---|---|
| 建议持有/观望 (hold) | sell 关键词 | ✅ 冲突 | 追加警告 |
| 建议持有/观望 (hold) | buy 关键词 | ✅ 冲突 | 追加警告 |
| 建议买入/加仓 (buy) | sell 关键词 | ✅ 冲突 | 追加警告 |
| 建议卖出/减仓 (sell) | buy 关键词 | ✅ 冲突 | 追加警告 |
| 建议观望等待 (watch) | sell/buy 关键词 | ✅ 冲突 | 追加警告 |

**集成点**：
- `pipeline.py:_synthesize_multiple_specialists` 返回前（L2139 附近）
- `orchestrator.py:_stream_final_synthesis` 验证后（L5074 附近）

**开关**：`agent.arbitration_consistency_guard_enabled`（默认 false）

**注意事项**：
- 只检测"操作建议"段（第 4 段），不检测"风险提示"段（避免误报）
- 冲突不强制修改 answer，只追加警告（保持 LLM 输出完整性）
- 如果 arbitration_summary 为 None，跳过校验

### P0-B：时机判断强制注入

**目标**：涉及买卖操作时，综合报告必须显式回答"现在是否适合执行"

**修改文件**：`backend/agent/core/orchestrator.py` 的 `_build_final_synthesis_prompt`（L613-823）

**新增 prompt 段**（在第 4 段"操作建议"内）：
```
### 时机判断强制要求（如涉及买卖操作）

每条操作建议必须显式回答以下 3 个问题：
1. **估值时机**：标的当前 PE/PB 百分位是多少？属于"低估（<30%）/合理（30-70%）/高估（>70%）"？
2. **盈亏时机**：标的当前盈亏率是多少？属于"深度亏损（<-15%）/浅度亏损/微利/盈利（>15%）"？
3. **执行时机**：当前是否适合执行？为什么？（如"估值处于历史 0.45% 分位，此时清仓等于高买低卖"）

禁止规则：
- 禁止建议"立即清仓"亏损标的（profit_rate < 0）且估值分位 < 30%（底部割肉）
- 禁止建议"立即减仓"亏损标的（除非风险止损场景：单只亏损 > 20% 且趋势确认向下）
- 禁止建议"立即买入"高估标的（估值分位 > 70%）
- 禁止仅因"功能重叠/持仓过多"就建议清仓亏损标的，必须先判断时机
```

**开关**：`agent.timing_judgment_enforced`（默认 false）

**配套修改**：
- 综合阶段 `pipeline.py:2018-2031` 的 `build_portfolio_summary_line` 已注入持仓摘要
- 需额外注入"持仓盈亏+估值分位"对照表，便于 LLM 引用
- 新建辅助函数 `build_portfolio_timing_summary(user_id) -> str`，返回：

### P0-C："止盈不止损"原则注入

**目标**：综合报告对亏损标的的建议必须遵循"止盈不止损"原则

**修改文件**：`backend/agent/core/orchestrator.py` 的 `_build_final_synthesis_prompt`

**新增 prompt 段**（在"通用要求"部分）：
```
### 止盈不止损原则（违反时必须显式说明理由）

- **盈利标的（profit_rate > 0）**：可建议部分止盈（如减仓 50%）或全部止盈
- **亏损标的（profit_rate < 0）**：禁止建议清仓/止损，除非满足以下任一条件：
  1. 标的估值分位 ≥ 70%（高估）
  2. 标的基本面恶化（如基金规模骤降、经理变更、策略失效）
  3. 风险止损场景（单只亏损 > 20% 且趋势确认向下）
  4. 功能重叠整合（合并同类项，不改变总敞口，如 3 只恒生科技合并为 1 只）

- **亏损标的整合建议应为**：
  * "持有等反弹"（估值低位时）
  * "合并同类项"（不改变总敞口，只减数量不减仓位）
  * "分批减仓"（估值修复到中位后逐步减）

- **禁止行为**：
  * 禁止仅因"功能重叠/持仓过多"就建议清仓亏损标的
  * 禁止"先减后加"逻辑漏洞（低位割肉再低位买入=无意义操作）
  * 禁止用"未来一周内择机执行"等模糊时机表述
```

**开关**：`agent.profit_not_loss_principle_enabled`（默认 false）

### P1-E：卖出操作时机守卫

**目标**：检测到卖出建议时，强制要求 answer 包含估值分位和盈亏状态

**修改文件**：`backend/agent/core/orchestrator.py` 的 `_check_human_in_loop_trade`（L375-396）

**新增逻辑**：
```python
def _check_sell_timing_guard(answer: str, trace_id: str = "") -> tuple[str, list[str]]:
    """
    卖出时机守卫：检测卖出建议时，强制要求 answer 包含估值分位和盈亏状态。
    """
    sell_kws = ["建议卖出", "应该卖出", "建议减仓", "建议止损", "建议止盈", 
                "立即卖出", "全部卖出", "清仓", "赎回"]
    has_sell = any(kw in answer for kw in sell_kws)
    if not has_sell:
        return answer, []
    
    # 检测是否包含估值分位信息
    has_percentile = bool(re.search(r'(百分位|分位|PE|PB|市盈率|市净率).{0,30}(\d+(\.\d+)?%)', answer))
    # 检测是否包含盈亏状态
    has_profit = bool(re.search(r'(盈亏|亏损|盈利|profit).{0,30}(-?\d+(\.\d+)?%)', answer))
    
    warnings = []
    if not has_percentile:
        answer += "\n\n⚠️ **时机守卫提示**：本回答建议卖出操作但未查询标的当前估值分位，建议先调用 query_valuation 确认时机。"
        warnings.append("sell_without_percentile")
    if not has_profit:
        answer += "\n\n⚠️ **时机守卫提示**：本回答建议卖出操作但未说明标的当前盈亏状态，无法判断是止盈还是止损。"
        warnings.append("sell_without_profit")
    return answer, warnings
```

**集成点**：
- `pipeline.py:_synthesize_multiple_specialists` 返回前
- `orchestrator.py:_stream_final_synthesis` 验证后

**开关**：`agent.sell_timing_guard_enabled`（默认 false）

## 九、风险与权衡

### 风险1：硬校验可能误报
**场景**：answer 在"风险提示"段提到"如果继续下跌可考虑卖出"，被误判为操作建议
**缓解**：只检测"第 4 段：操作建议"内的关键词，不检测"第 5 段：风险提示"

### 风险2：prompt 过长导致 LLM 注意力分散
**场景**：新增的时机判断+止盈不止损 prompt 约 500 字，可能稀释其他约束
**缓解**：
- 用"禁止规则"形式而非"建议规则"，LLM 对禁止性约束更敏感
- 关键约束放在 prompt 末尾（近因效应）

### 风险3：开关默认关闭，修复不生效
**场景**：用户不知道有新开关，conv#131 重放时仍出现原问题
**缓解**：
- 设计稿审批后，建议用户手动开启 5 个开关
- 或在 `project_memory.md` 记录"对话质量优化开关需手动开启"

### 风险4：query_smart_add_plan 字段提取依赖返回结构
**场景**：smart_add_planner.py 返回结构变更，字段提取失效
**缓解**：
- 使用 `.get()` 防御性访问
- 字段缺失时返回空 dict，不抛异常
- 单元测试覆盖空数据场景
