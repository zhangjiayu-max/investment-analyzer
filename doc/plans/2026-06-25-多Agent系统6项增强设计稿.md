## 1. 增强 1：状态机检查点（P1）

### 1.1 目标

complex 场景分析耗时 3-5 分钟，中途刷新/断网/超时后可从最近检查点恢复，不重新调用已完成的专家。

### 1.2 数据模型

```sql
CREATE TABLE IF NOT EXISTS orchestration_checkpoints (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    conv_id INTEGER NOT NULL,
    message_id INTEGER,
    phase TEXT NOT NULL,
    state_json TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(conv_id, message_id, phase)
);
```

### 1.3 状态结构

```python
class OrchestrationState:
    phase: str          # "routing" | "experts" | "cross_review" | "arbitrate" | "final"
    completed_experts: list[dict]   # 已完成的专家结果
    pending_experts: list[str]      # 尚未执行的专家 key
    specialist_results: list[dict]  # 所有完成的分析结果
    cross_review_done: bool
    arbitration_done: bool
    turn_count: int
    budget_mode: str                # "normal" | "conservative"
```

### 1.4 检查点位置

```
Phase A 所有专家执行完成  → checkpoint("experts")
Phase B 交叉审阅完成      → checkpoint("cross_review")
Phase B 仲裁完成          → checkpoint("arbitrate")
```

**不在单个专家完成时存档**（5 个专家 = 5 次 DB 写入，性价比低）。

### 1.5 恢复流程

```python
def orchestrate_with_checkpoint(query, history, rag_context, cancel_event, conv_id, message_id):
    # 1. 尝试恢复
    state = OrchestrationState.restore(conv_id, message_id)
    
    if state and state.phase == "cross_review":
        # 专家已完成，直接跳转到交叉审阅
        specialist_results = state.completed_experts
        # ⚡ 不需要重新跑 Phase A
        return _run_cross_review_and_finalize(specialist_results, ...)
    
    if state and state.phase == "arbitrate":
        return _run_arbitrate_and_finalize(specialist_results, ...)
    
    # 2. 全新执行
    return _run_full_pipeline(query, history, ...)
```

### 1.6 前端改动

- SSE 流中新增 `"type": "checkpoint"` 事件，前端静默忽略
- 刷新页面后，若该消息的 assistant 内容为空但后台有 running 任务 → 自动调用 `resume` API
- ChatMessage 中显示"恢复中..."状态

## 2. 增强 2：动态 Agent 选择（P1）

### 2.1 目标

Orchestrator 在分析过程中自动检测是否需要追加专家，不在第一轮就固定专家列表。

### 2.2 触发规则

```python
DYNAMIC_SPAWN_RULES = [
    {
        "detect": lambda result: _count_keyword(result["analysis"], ["风险", "回撤", "波动"]) > 3,
        "suggest": "risk_assessor",
        "reason": "前一专家多次提及风险，建议追加风险评估",
        "priority": "high",
    },
    {
        "detect": lambda result: result.get("confidence", 1.0) < 0.6,
        "suggest": "behavioral_coach",
        "reason": "分析置信度较低，建议从行为金融角度补充",
        "priority": "medium",
    },
    {
        "detect": lambda result: any(kw in result["analysis"] for kw in ["港股", "美股", "QDII"]),
        "suggest": "market_analyst",
        "reason": "涉及海外市场，建议追加市场分析",
        "priority": "medium",
    },
    {
        "detect": lambda result: _detect_position_over_concentration(result),
        "suggest": "allocation_advisor",
        "reason": "检测到持仓集中度问题，建议追加配置分析",
        "priority": "high",
    },
]
```

### 2.3 集成位置

在 `orchestrate()` 的多轮循环中，每轮工具调用完成后：

```python
for turn in range(MAX_TURNS):
    response = _call_llm(...)
    msg = response.choices[0].message
    
    if msg.tool_calls:
        for tc in msg.tool_calls:
            if tc.function.name.startswith("consult_"):
                result = _execute_specialist(tc.function.name, ...)
                specialist_results.append(result)
                
                # 🆕 动态检测：是否需要追加专家
                suggestions = _check_dynamic_spawn(result, existing_experts)
                for s in suggestions:
                    if s["agent_key"] not in already_called:
                        # 追加到下一轮可用的工具列表中
                        pending_experts.append(s["agent_key"])
```

**防无限循环：** 每个专家最多被调用 1 次。`already_called` 集合记录已调用的专家。

### 2.4 前端改动

SSE 流中新增事件：

```json
{"type": "dynamic_spawn", "agent_key": "risk_assessor", "reason": "估值分析多次提及风险，追加风险评估"}
```

前端展示为"🔄 自动追加：风险评估专家"。

### 3.3 SOP 引擎

```python
def match_sop_template(query: str, has_portfolio: bool) -> dict | None:
    """根据用户查询匹配合适的 SOP 模板。"""
    # 1. 关键词匹配
    for key, template in SOP_TEMPLATES.items():
        if any(kw in query for kw in template["trigger_keywords"]):
            return template
    
    # 2. 场景推断
    if has_portfolio and _is_complex_query(query):
        return SOP_TEMPLATES["full_diagnosis"]
    
    return None  # 无匹配 → 走原有 LLM 路由

def execute_sop(template: dict, query: str, prebuilt_context: str, cancel_event, start_time):
    """按 SOP 模板执行分析。"""
    # 按 group 分组，组内并行，组间串行
    groups = {}
    for step in template["steps"]:
        groups.setdefault(step["group"], []).append(step)
    
    results = []
    for group_id in sorted(groups.keys()):
        group_steps = groups[group_id]
        if len(group_steps) == 1:
            # 单步：直接执行
            step = group_steps[0]
            result = _execute_specialist(step["agent"], step["task"], ...)
            results.append(result)
        else:
            # 多步：并行执行
            with ThreadPoolExecutor(max_workers=len(group_steps)) as executor:
                futures = {executor.submit(_execute_specialist, s["agent"], s["task"]): s
                          for s in group_steps}
                for f in as_completed(futures):
                    results.append(f.result())
    
    # 交叉审阅 + 仲裁（按模板配置）
    if template.get("cross_review") and len(results) >= 2:
        results = _run_cross_review(results, ...)
    if template.get("arbitration"):
        results = _run_arbitration(results, ...)
    
    return _synthesize_final(results, template.get("final_agent"))
```

## 4. 增强 4：实体记忆（P2）

### 4.1 目标

跨对话记住关键标的的属性变化。下次用户问起时，AI 能说"你上次分析时 PE 分位是 25%，现在涨到 30% 了"。

### 4.2 数据模型

```sql
CREATE TABLE IF NOT EXISTS entity_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_name TEXT NOT NULL,
    entity_type TEXT NOT NULL,        -- "index" | "fund" | "sector" | "strategy"
    entity_code TEXT DEFAULT '',
    attribute TEXT NOT NULL,          -- "pe_percentile" | "valuation_level" | "risk_level"
    old_value TEXT DEFAULT '',
    new_value TEXT NOT NULL,
    source TEXT DEFAULT 'analysis',   -- "analysis" | "daily_advice" | "user_input"
    source_id INTEGER,                -- conversation_id or run_id
    snapshot_date TEXT NOT NULL,
    created_at TEXT DEFAULT (datetime('now','localtime'))
);

CREATE INDEX IF NOT EXISTS idx_entity_memory_name ON entity_memory(entity_name);
CREATE INDEX IF NOT EXISTS idx_entity_memory_date ON entity_memory(snapshot_date);
```

### 4.3 自动记录

在每个专家分析完成、交叉审阅完成、和每日提示生成后调用：

```python
def record_entity_snapshots(analysis_text: str, source: str, source_id: int):
    """从分析文本中提取实体属性并记录。"""
    entities = _extract_entities(analysis_text)
    for e in entities:
        # 查询上次记录
        last = get_last_snapshot(e["name"], e["attribute"])
        old_value = last["new_value"] if last else ""
        
        if e["new_value"] != old_value:  # 属性变化了才记录
            save_entity_memory(
                entity_name=e["name"],
                entity_type=e["type"],
                entity_code=e.get("code", ""),
                attribute=e["attribute"],
                old_value=old_value,
                new_value=e["new_value"],
                source=source,
                source_id=source_id,
                snapshot_date=datetime.now().strftime("%Y-%m-%d"),
            )
```

### 4.4 上下文注入

在 `conversation_context.py` 新增：

```python
def _build_entity_memory_context(limit: int = 5) -> str:
    """获取近期实体属性变化。"""
    changes = get_recent_entity_changes(days=30, limit=limit)
    if not changes:
        return ""
    
    lines = ["### 近期标的属性变化"]
    # 按实体聚合
    entities = {}
    for c in changes:
        key = c["entity_name"]
        entities.setdefault(key, []).append(c)
    
    for name, attr_changes in entities.items():
        latest = attr_changes[0]
        lines.append(f"- **{name}**：{latest['attribute']} {latest['old_value']} → {latest['new_value']} "
                     f"（{latest['snapshot_date']}）")
    
    return "\n".join(lines)
```

## 6. 增强 6：成本感知路由（P3）

### 6.1 目标

不同步骤用不同成本的模型，关键推理用 pro，计算/审阅用 flash。

### 6.2 模型分配

```python
AGENT_MODEL_MAP = {
    # 计算类 → flash（便宜、快）
    "valuation_expert": "deepseek-v4-flash",
    "allocation_advisor": "deepseek-v4-flash",
    "fund_analyst": "deepseek-v4-flash",
    
    # 推理类 → pro（深度、贵）
    "market_analyst": "deepseek-v4-pro",
    "behavioral_coach": "deepseek-v4-pro",
    "risk_assessor": "deepseek-v4-pro",
    
    # 编排类 → pro
    "orchestrator": "deepseek-v4-pro",
    "arbitrator": "deepseek-v4-pro",     # 仲裁必须 pro
    
    # 审阅类 → flash
    "cross_review": "deepseek-v4-flash",
}

# 保守模式全部用 flash
CONSERVATIVE_MODEL = "deepseek-v4-flash"
```

### 6.3 实现

在 `_execute_specialist` 中：

```python
def _get_model_for_agent(agent_key: str, budget_mode: str = "normal") -> str:
    if budget_mode == "conservative":
        return "deepseek-v4-flash"
    return AGENT_MODEL_MAP.get(agent_key, MODEL)

def _execute_specialist(agent_key, query, cancel_event=None, prebuilt_context=""):
    model = _get_model_for_agent(agent_key, budget_mode)
    # 传给 _call_llm
    response = _call_llm(
        caller=f"specialist_{agent_key}",
        model=model,
        messages=llm_messages,
        ...
    )
```

### 6.4 成本预估

```
当前 complex 场景（5专家+审阅+仲裁）全部用 pro：
  估值(prompt 4K) + 风控(3K) + 配置(3K) + 市场(4K) + 行为(3K)
  + 审阅(5K×5) + 仲裁(6K) = ~50K prompt tokens ≈ $0.07

改为成本路由后：
  flash: 估值+配置+审阅(5次) = ~30K tokens ≈ $0.003
  pro: 风控+市场+行为+仲裁 = ~20K tokens ≈ $0.06
  总计：≈ $0.063
  
节省：~10%（Token 数不变，但 flash 更便宜）
更关键的收益：flash 响应更快，减少总体延迟。
```
