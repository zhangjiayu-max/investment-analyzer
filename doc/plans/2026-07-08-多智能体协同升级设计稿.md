# 多智能体协同升级设计稿

> 日期：2026-07-08
> 范围：在 11→7 专家精简基础上，进一步对齐成熟多智能体框架（LangGraph / AutoGen / CrewAI / MetaGPT），修复可观测性缺口，引入反思机制与统一黑板。
> 原则：每项改动必须可回滚、有明确收益、不破坏现有 SSE 流式契约。

---

## 一、现状与差距

### 1.1 当前协同架构

```
用户查询
  ├─ 简单路径 → 单 LLM 直答（max_tokens=200）
  └─ 复杂路径 → SmartRouter 路由
       ├─ ReAct 模式 → 专家独立执行 + 单轮交叉评审（peer dict）
       └─ Pipeline 模式 → 6 阶段顺序执行 + Blackboard 共享
```

### 1.2 五个关键差距

| # | 差距 | 现状代码位置 | 成熟框架做法 |
|---|---|---|---|
| G1 | **可观测性断裂** | `pipeline.py:_phase_execution` (L433) / `_fallback_execution` (L598) / `multi_agent.py:run_cross_review_opinion` (L982) / `run_specialist_with_context` (L749) 均未调用 `create_agent_run()` | LangSmith / OpenTelemetry 全链路 trace |
| G2 | **双轨黑板** | ReAct 用 `orchestrator.py:L4760` 的 `shared_blackboard: dict[str, dict]` + `_format_blackboard_summary`；Pipeline 用 `blackboard.py` 的 `Blackboard` 类，两者不互通 | 统一 State 对象（LangGraph `TypedDict`） |
| G3 | **执行顺序串行** | `pipeline.py:L473-590` 的 `for step in steps:` 顺序循环，无 `asyncio.gather` | 独立节点并行 + 依赖图 |
| G4 | **无反思机制** | Pipeline 是 Phase 0-5，无自评节点；ReAct 是 MAX_TURNS=3 + 单轮 cross_review，无自评 | Reflection 节点：自评 → 修订 → 再评，最多 N 轮 |
| G5 | **路由依赖 LLM 兜底** | `router.py:_llm_fallback_route` (L119) 在规则未命中时调用 LLM | 声明式条件边（DSL / 图） |

### 1.3 升级前效果（昨日精简版）

- Token 单次调用下降 21%（10,901 → 8,613）— 可信（token_usage 表完整）
- arbitrator 完全移除 — 可信
- agent_runs 平均调用 2.62 → 1.5 — **不可信**（G1 缺口导致 Pipeline/cross_review 路径不写日志）

## 二、目标设计

### 2.1 总体目标

1. **可观测性优先**：所有 LLM 调用路径写入 `agent_runs`，让"质量"可量化
2. **统一黑板**：ReAct 与 Pipeline 共享同一个 `Blackboard` 类，状态可追溯
3. **并行执行**：Pipeline Phase 3 独立专家并发，延迟下降 30%+
4. **反思机制**：综合阶段前增加 Reflection 节点，质量可量化提升
5. **声明式路由**：用 YAML 配置替换 LLM fallback，路由可解释、可测

### 2.2 非目标（避免过度设计）

- ❌ 不引入图执行引擎（LangGraph 风格的 DAG）— 改造成本过高
- ❌ 不做多轮辩论（多轮 cross_review）— token 成本不可控
- ❌ 不引入 A2A 协议（消息总线）— 当前 7 个专家不需要
- ❌ 不做 Agent 评估器自动迭代 — eval_suite 已覆盖

## 三、实施计划（P0 → P4）

### P0：修复 agent_runs 日志缺口（必做，0.5 天）

**问题**：4 个执行路径不写 `agent_runs`，导致：
- 数据库统计严重少计（"2.62→1.5 调用"不可信）
- risk_assessor 升级后的反方观点/行为偏差质量无法验证
- eval_suite 评测无法关联到具体 run

**改造点**：

| 文件 | 函数 | 改动 |
|---|---|---|
| `backend/agent/pipeline.py` | `_phase_execution` (L433) | 在 `run_specialist()` 调用后，调用 `create_agent_run()` 记录结果 |
| `backend/agent/pipeline.py` | `_fallback_execution` (L598) | 同上 |
| `backend/agent/multi_agent.py` | `run_specialist_with_context` (L749) | 函数签名增加 `conversation_id` / `message_id` 可选参数，写入 `agent_runs`（run_phase="cross_review"） |
| `backend/agent/multi_agent.py` | `run_cross_review_opinion` (L982) | 同上（run_phase="cross_review"） |
| `backend/agent/orchestrator.py` | 调用上述函数的位置 | 传入 `conversation_id` / `message_id` |

**关键决策**：
- `conversation_id` / `message_id` 通过 `PipelineState` 传递（Pipeline 路径）
- ReAct 路径在 `orchestrator.py` 已有这两个 ID，直接传入
- `run_phase` 字段已存在（primary/cross_review/arbitration），复用即可
- 不破坏现有 `create_agent_run()` 签名，新增可选参数

**验证**：
- 触发一次 Pipeline 路径对话，确认 `agent_runs` 表有对应记录
- 触发一次 cross_review，确认 `run_phase="cross_review"` 的记录存在
- 对比 token_usage 总量与 agent_runs 记录数，差距应 < 10%

**回滚**：纯日志写入，失败不影响主流程（try/except 包裹）

### P1：统一黑板（1 天）

**问题**：双轨黑板导致：
- ReAct 专家看到的是 `_format_blackboard_summary` 的简陋文本
- Pipeline 专家看到的是 `Blackboard.to_context_text()` 的结构化文本
- 两者数据结构不互通，无法共享冲突检测、动作信号等能力

**改造点**：

| 文件 | 改动 |
|---|---|
| `backend/agent/orchestrator.py` L4760 | 删除 `shared_blackboard: dict[str, dict] = {}`，改为 `from agent.blackboard import Blackboard, extract_entry_from_result` + `bb = Blackboard(max_entries=6)` |
| `backend/agent/orchestrator.py` L4765 | `_format_blackboard_summary(shared_blackboard)` → `bb.to_context_text(exclude_agent=agent_key)` |
| `backend/agent/orchestrator.py` L4778 | `shared_blackboard[agent_key] = conclusion` → `bb.write(extract_entry_from_result(agent_key, agent_name, result))` |
| `backend/agent/orchestrator.py` | 删除 `_format_blackboard_summary` 和 `_extract_structured_conclusion` 函数（已被 Blackboard 取代） |

**关键决策**：
- ReAct 路径的 `Blackboard` 实例在 `orchestrate()` 入口创建，传递给执行逻辑
- 全局单例 `get_global_blackboard()` 已存在，但**不强制使用**（避免跨对话污染）
- 保留 `shared_blackboard_enabled` 配置开关

**验证**：
- ReAct 路径执行 2-3 专家，确认第二个专家能看到第一个的结构化结论
- 触发 `find_conflicts()`，确认买/卖冲突能被检测
- 黑板日志 `[blackboard] {agent_name} 写入黑板` 出现

**回滚**：保留 `_format_blackboard_summary` 函数代码（注释），必要时一键恢复

### P2：Phase 3 并行化（1 天）

**问题**：`pipeline.py:L473` 的 `for step in steps:` 顺序执行，3 个专家串行需 15-20 秒

**改造点**：

| 文件 | 函数 | 改动 |
|---|---|---|
| `backend/agent/pipeline.py` | `_phase_execution` (L433) | 按 `depends_on` 分组，组内并行执行 |

**新逻辑**：
```
1. 拓扑分组：按 step.depends_on 把 steps 分成若干 "wave"
2. 每个 wave 内：用 asyncio.gather 并行执行（专家无依赖）
3. wave 之间：串行（保证依赖顺序）
4. 黑板写入：wave 完成后统一写入（避免并发写竞争）
5. SSE 事件：维持顺序（按 wave 内完成时间 yield specialist_done）
```

**关键决策**：
- `run_specialist` 是同步函数，用 `asyncio.to_thread()` 包裹实现并发
- 黑板写入加 `threading.Lock`（Blackboard 类已有，但需暴露）
- 单 wave 内最多 3 个并发（避免 token 突刺）
- 取消事件检查移到 wave 级别

**验证**：
- 3 专家无依赖场景，延迟从 ~15s 降到 ~6s
- 有依赖场景（step B depends_on step A），仍按顺序执行
- SSE 事件顺序正确（specialist_start 不重复）

**回滚**：保留原顺序循环代码作为 fallback，配置开关 `pipeline.parallel_execution` 默认 true，失败时降级

### P3：加 Reflection 节点（1.5 天）

**问题**：当前综合阶段（`_phase_synthesis`）直接拼接专家结论，无自评/修订

**改造点**：

| 文件 | 函数 | 改动 |
|---|---|---|
| `backend/agent/pipeline.py` | 在 Phase 4 (SYNTHESIS) 之前插入 Phase 3.5 (REFLECTION) | 新增 `_phase_reflection` 函数 |
| `backend/agent/pipeline.py` | `PipelinePhase` 枚举 | 新增 `REFLECTION = "reflection"` |
| `backend/agent/pipeline.py` | `EVENT_*` 常量 | 新增 `EVENT_REFLECTION_DONE` |

**Reflection 逻辑**：
```
输入：所有专家的 BlackboardEntry + 冲突列表
处理：
  1. 调用 1 次 LLM（caller="reflection"），输入黑板摘要 + 冲突
  2. LLM 输出 JSON：{
       "quality_issues": ["估值分析师未考虑宏观利率", "..."],
       "missing_perspectives": ["未覆盖债券市场"],
       "conflict_resolutions": [{"target": "中证500", "resolution": "..."}],
       "confidence_adjustment": -0.1
     }
  3. 将 quality_issues 注入综合阶段的 prompt
  4. 若 confidence_adjustment < -0.2，触发"重做"标记
输出：reflection_result dict
```

**关键决策**：
- Reflection 只跑 1 次 LLM，成本可控（~500 token）
- 配置开关 `pipeline.reflection_enabled` 默认 **false**（先观察再启用）
- Reflection 不直接修改专家结论，只影响综合阶段
- 不做"自评→修订→再评"多轮（避免 token 爆炸）

**验证**：
- 启用后，综合阶段 prompt 包含 `quality_issues`
- token_usage 多一条 `caller=reflection` 记录
- eval_suite 跑 10 个用例，对比启用前后的 score_normalized

**回滚**：配置开关关闭即回滚

### P4：声明式路由（2 天）

**问题**：`router.py:_llm_fallback_route` (L119) 在规则未命中时调用 LLM，每次 ~200 token

**改造点**：

| 文件 | 改动 |
|---|---|
| `backend/agent/router_config.yaml`（新增） | 声明式路由配置：intent pattern → experts |
| `backend/agent/router.py` | 加载 YAML，扩展 `_KEYWORD_ROUTES` 为分层规则 |
| `backend/agent/router.py` | `_llm_fallback_route` 保留但默认禁用（`router.use_llm_fallback` 改为 "false"） |

**新路由配置**：
```yaml
# router_config.yaml
routes:
  - name: valuation
    patterns: ["估值", "PE", "PB", "百分位", "低估", "高估"]
    experts: [valuation_expert]
  - name: risk
    patterns: ["风险", "回撤", "止损", "亏损"]
    experts: [risk_assessor]
  - name: allocation
    patterns: ["配置", "仓位", "股债", "比例"]
    experts: [allocation_advisor]
  # ...
  
# 兜底规则（替代 LLM fallback）
fallback:
  # 按查询类型匹配
  - patterns: ["买", "卖", "操作"]
    experts: [allocation_advisor, risk_assessor, valuation_expert]
  - patterns: ["怎么", "如何", "建议"]
    experts: [allocation_advisor]
  # 最终兜底
  default: [allocation_advisor, valuation_expert]
```

**关键决策**：
- YAML 配置可热更新（无需重启）
- LLM fallback 保留代码，但默认关闭（`router.use_llm_fallback=false`）
- 路由命中时记录 `route_by="declarative"`，未命中走 `default`
- 支持正则模式（`patterns` 支持 `re.search`）

**验证**：
- 10 个测试查询，路由命中率 > 90%
- LLM fallback 调用次数下降 90%+
- 路由日志可追溯（`route_by` 字段）

**回滚**：`router.use_llm_fallback` 改回 "true" 即恢复

## 四、实施顺序与依赖

```
P0 (日志) ──┐
            ├─→ P1 (黑板) ──→ P2 (并行) ──→ P3 (反思)
            │
            └─→ P4 (路由) [独立]
```

- P0 必须先做（否则后续改造效果无法度量）
- P1 是 P2 的前置（统一黑板后并行写入才安全）
- P4 与 P0-P3 独立，可并行开发
- 每完成一项即提交，便于回滚
