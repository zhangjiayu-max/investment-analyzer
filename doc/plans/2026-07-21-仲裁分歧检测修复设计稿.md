# 仲裁分歧检测修复设计稿

**日期**：2026-07-21
**背景**：基于近 11 次对话分析发现，仲裁系统在 5 次实战中 `key_conflicts` 恒为空，`verdict` 恒为"持有/观望"。导致 conv#131 修复的 P0-A 仲裁守卫在实战中永远不会触发（仲裁 stance 永远 hold，综合报告只要不说"买入"就不会冲突）。

## 一、问题定位

### 1.1 根因链路

```
专家不输出 stance 字段（multi_agent.py:906-917 无 stance 字段）
    ↓
仲裁用 _detect_stance(analysis) 关键词猜测
    ↓
_detect_stance 优先级：hold > buy > sell
hold 关键词含"风险""谨慎""持有"等几乎所有专家都会用的词
    ↓
所有专家被识别为 hold（即使专家明确说"建议卖出"）
    ↓
stance_groups = {"buy": [], "sell": [], "hold": [全部]}
    ↓
key_conflicts 检测条件不满足（要求 buy vs sell/hold 或 sell vs hold）
    ↓
key_conflicts 恒为空
    ↓
arbitration_mode = "consensus"（因为没冲突）
    ↓
5 次实战裁决全是"持有/观望"
```

### 1.2 三个具体根因

**根因 1：专家不输出 stance 字段**

`backend/agent/core/multi_agent.py:906-917` 专家返回字典只有 `agent_key/agent/icon/analysis/status/structured/tool_calls/...`，没有 `stance` 字段。仲裁只能用 `_detect_stance(analysis)` 猜。

**根因 2：`_detect_stance` 关键词优先级倒置**

`backend/agent/core/arbitration.py:8-16`：

```python
def _detect_stance(text: str) -> str:
    text = (text or "").lower()
    if any(kw in text for kw in ["不建议", "观望", "持有", "风险", "谨慎", "先别", "暂停"]):
        return "hold"    # ← 先判断 hold
    if any(kw in text for kw in ["建议买", "买入", "加仓", "建仓", "上车", "分批买"]):
        return "buy"
    if any(kw in text for kw in ["卖出", "减仓", "止盈", "止损", "退出"]):
        return "sell"
    return "unknown"
```

致命问题：
- hold 关键词含"风险""谨慎" — 任何负责任的专家分析都会提到风险，导致几乎所有专家都被识别为 hold
- 即使专家明确说"建议卖出"，只要文中也提到"注意风险"，hold 先匹配 → 识别为 hold
- sell 关键词"止盈"其实是看多信号（盈利才止盈），却被归为 sell

**根因 3：hold 优先级压倒一切**

`backend/agent/core/arbitration.py:36-39`：

```python
counts = Counter(s for s in stances if s != "unknown")
if counts["hold"] > 0:
    final_stance = "hold"
    arbitration_mode = "conflict" if counts["buy"] or counts["sell"] else "consensus"
```

只要 1 个专家被识别为 hold，final_stance 就是 hold。配合根因 2，导致**几乎永远输出 hold**。

### 1.3 实战数据验证

5 次有仲裁的对话，专家分析都含"风险""谨慎"等词：
- conv#127："风险暴露""风险缓冲""避免情绪化"
- conv#128："风险极高""流动性风险""风险否决"
- conv#129："信用风险""久期风险""利率风险"
- conv#130："追高风险极大""估值回调风险""风险收益比"
- conv#131："过度分散风险""管理成本高昂"

**每篇专家分析都含"风险"二字 → 全部识别为 hold → key_conflicts 恒空**。

### 1.4 影响范围

这个 bug 导致的连锁后果：
1. **仲裁裁决永远 hold** → P0-A 仲裁守卫的 stance 提取永远拿到 hold，无法检测"hold vs sell"冲突（因为仲裁永远是 hold）
2. **key_conflicts 恒空** → 综合报告的"分歧反驳"段永远无内容可写
3. **arbitration_mode 永远 consensus** → conv#130 修复的"四段式 reasoning"永远输出"专家立场一致"
4. **conv#130 的 cross_review 排查日志**没有解决根因，只是加了日志

**这意味着 conv#131 修复的 P0-A 仲裁守卫，在实战中永远不会触发** — 因为仲裁 stance 永远是 hold，综合报告只要不明确说"买入"就不会冲突。

## 二、修复方案

### 2.1 修复策略：两层修复 + 验证

- **L1（立即生效）**：修复 `_detect_stance` 关键词优先级和 hold 门槛（低风险，立即生效）
- **L2（结构化）**：让专家显式输出 stance 字段（需要改 prompt + 解析逻辑，中等成本）
- **验证**：用历史 5 次对话的专家 analysis 重跑仲裁，对比修复前后 key_conflicts 触发情况

### 2.2 L1 — 修复 `_detect_stance`（arbitration.py）

**修复点 1：关键词优先级调整为 sell > buy > hold**

原逻辑：hold > buy > sell（hold 最先匹配，压倒一切）
新逻辑：sell > buy > hold（先检测明确的操作意图，最后才 fallback 到 hold）

**理由**：
- "卖出/减仓/清仓/赎回"是明确操作意图，优先级最高
- "买入/加仓/建仓"是明确操作意图，次高
- "持有/观望"是被动立场，应作为 fallback
- "风险/谨慎"不是操作意图，是风险描述，不应作为 hold 判定依据

**修复点 2：从 hold 关键词中移除"风险""谨慎"**

原 hold 关键词：`["不建议", "观望", "持有", "风险", "谨慎", "先别", "暂停"]`
新 hold 关键词：`["不建议", "观望", "持有", "先别", "暂停"]`

**理由**："风险""谨慎"是风险描述，不是立场。任何负责任的专家都会提到风险，不能据此判 hold。

**修复点 3：sell 关键词移除"止盈"**

原 sell 关键词：`["卖出", "减仓", "止盈", "止损", "退出"]`
新 sell 关键词：`["卖出", "减仓", "清仓", "赎回", "退出"]`

**理由**："止盈"是盈利后的操作，说明标的在盈利，不应与"止损"混为 sell。保留"止损"（亏损保护性操作，确实是 sell 意图）。新增"清仓""赎回"覆盖更全。

**修复点 4：buy 关键词补充**

原 buy 关键词：`["建议买", "买入", "加仓", "建仓", "上车", "分批买"]`
新 buy 关键词：`["建议买", "买入", "加仓", "建仓", "上车", "分批买", "定投", "补仓"]`

**理由**：补充"定投""补仓"覆盖用户常见操作意图。

### 2.3 L1 — 修复 hold 门槛（arbitration.py）

**修复点 5：hold 需要多数专家支持，而非 1 个**

原逻辑：
```python
if counts["hold"] > 0:
    final_stance = "hold"
```

新逻辑：
```python
# hold 需要"多数"（≥half）专家支持，而非 1 个
total_stanced = sum(counts.values())
half = total_stanced / 2
if counts["hold"] > half:
    final_stance = "hold"
    arbitration_mode = "conflict" if (counts["buy"] or counts["sell"]) else "consensus"
elif counts["sell"] > counts["buy"]:
    final_stance = "sell"
    arbitration_mode = "consensus" if counts["sell"] > 1 else "conflict"
elif counts["buy"] > 0:
    final_stance = "buy"
    arbitration_mode = "consensus" if counts["buy"] > 1 else "conflict"
else:
    final_stance = "watch"
    arbitration_mode = "consensus"
```

**理由**：原逻辑"1 个 hold 就压倒一切"违反多数决原则。新逻辑要求 hold 需过半才生效，否则 buy/sell 的明确意图优先。

### 2.4 L1 — 修复 disagreements 提取（arbitration.py）

**修复点 6：disagreements 提取不再依赖 _detect_stance 的 hold 误判**

原逻辑：disagreements 提取所有 stance ∈ (buy/sell/hold) 的专家。由于 hold 误判严重，disagreements 会包含大量"伪 hold"。

新逻辑：disagreements 只提取 stance ∈ (buy/sell) 的专家（明确操作意图），hold 不作为分歧方。key_conflicts 的检测逻辑同步调整：

```python
# key_conflicts 检测：只关心 buy vs sell 的真实分歧
key_conflicts = []
if stance_groups["buy"] and stance_groups["sell"]:
    key_conflicts.append({
        "type": "buy_vs_sell",
        "buy_side": stance_groups["buy"],
        "sell_side": stance_groups["sell"],
        "note": "看多与看空观点存在分歧",
    })
# 移除 buy_vs_hold 和 sell_vs_hold（hold 是被动立场，不是真实分歧方）
```

**理由**：hold 是"不动"的被动立场，不是与 buy/sell 对立的操作意图。真实的分歧只有"买 vs 卖"。这避免"伪 hold"污染 key_conflicts。

### 2.5 L2 — 专家显式输出 stance（结构化升级，延后实施）

**暂不实施**：需要改所有专家 prompt + 解析逻辑，成本高、风险大。L1 修复后先用历史数据验证效果，再决定是否需要 L2。

## 四、风险评估

### 4.1 兼容性

- **`_detect_stance` 返回值不变**：仍是 "buy"/"sell"/"hold"/"unknown"，下游无需改动
- **`arbitrate_results` 返回字段不变**：verdict/confidence/key_conflicts/reasoning/disagreements 全保留
- **`_build_arbitration_reasoning` 输入不变**：仍接收 key_conflicts 列表
- **`_save_final` 和 metadata 结构不变**：仲裁 JSON 结构完全兼容

### 4.2 回归风险

- **风险点 1**：移除"风险""谨慎"作为 hold 关键词后，某些专家可能被识别为 unknown（如果全文没出现明确操作意图）
  - **缓解**：unknown 立场不计入 counts，不影响多数决；且 buy/sell 关键词补充后覆盖更全
- **风险点 2**：hold 门槛改为"过半"后，可能出现 final_stance 从"hold"变成"buy"/"sell"
  - **这是期望行为**：原逻辑 hold 压倒一切是 bug，不是 feature
- **风险点 3**：key_conflicts 只检测 buy_vs_sell 后，综合报告的"分歧反驳"段内容会变少
  - **缓解**：原 buy_vs_hold/sell_vs_hold 分歧大多是"伪 hold"导致的假冲突，移除反而更准确

### 4.3 与 conv#131 P0-A 仲裁守卫的关系

**修复前**：仲裁 stance 永远 hold → P0-A 守卫只能检测"hold vs sell"冲突 → 但仲裁永远是 hold，所以只要综合报告不说"买入"就不会触发
**修复后**：仲裁 stance 可能是 buy/sell/hold → P0-A 守卫能检测真实冲突 → conv#131 的"持有→卖出"冲突能被检测到
