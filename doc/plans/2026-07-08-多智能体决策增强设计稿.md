# 多智能体决策增强设计稿

**日期**: 2026-07-08
**目标**: 提升 AI 多智能体的决策质量，让专家协同从"汇报式"升级为"对抗辩论 + 风险否决 + 持仓感知 + 智能规划"，参考 TradingAgents / HedgeAgents / LangGraph 等成熟项目。
**原则**: 每条改动都必须可配置（默认行为不破坏现有流程）、可观测（日志 + trace_id）、可回退（开关关闭即恢复旧行为）。

---

## 一、当前架构回顾

```
用户消息
  ↓
orchestrator.orchestrate_stream()
  ├─【主路径】Pipeline 6 阶段
  │   Phase 0: 预处理 → Phase 1: 信息收集 → Phase 2: 计划生成
  │   → Phase 3: 专家执行（共享黑板）→ Phase 3.5: Reflection（可选）
  │   → Phase 4: 综合（LLM 综合 + 冲突检测）→ Phase 5: 记忆持久化
  └─【降级路径】ReAct 循环
```

### 核心痛点
1. **冲突处理弱**: 检测到冲突后只是传给 LLM 综合，容易"和稀泥"
2. **风险专家无否决权**: 风险意见可被配置专家无视
3. **专家结论格式不统一**: 缺少"对持仓影响"结构化字段
4. **模型未分级**: 所有专家可能用同一模型，token 成本高
5. **Phase 2 计划简单**: 专家调用顺序不够智能，缺少依赖与子问题拆解

---

## 二、5 条增强方案

### P0-A: 风险专家硬约束否决权

**问题**: 风险专家只是给意见，配置专家可以无视。
**方案**: 在 BlackboardEntry 增加 `risk_veto` 字段，综合阶段遇到 veto 强制降级建议。

**改动**:
1. `blackboard.py` BlackboardEntry 增加字段 `risk_veto: Optional[dict] = None`
   ```python
   # risk_veto 结构
   {
       "vetoed_action": "BUY",           # 被否决的动作
       "target": "中证500",              # 标的
       "reason": "最大回撤超30%",        # 否决理由
       "severity": "high",               # high/medium/low
       "suggested_action": "HOLD"        # 建议替代动作
   }
   ```
2. `blackboard.py` `extract_entry_from_result` 增加风险否决提取：
   - 仅 `risk_assessor` 专家触发（agent_key == "risk_assessor"）
   - 从分析文本中识别"不建议"/"禁止"/"否决"/"风险过高"等关键词
   - 提取被否决的动作和理由
3. `blackboard.py` 新增 `get_vetoes()` 方法返回所有风险否决
4. `pipeline.py` `_synthesize_multiple_specialists` 注入否决约束：
   - 如果存在 risk_veto，prompt 追加"⚠️ 风险否决约束"段落
   - 强制要求 LLM 综合时把被否决的 BUY 降级为 HOLD
5. `pipeline.py` `_phase_synthesis` 返回值增加 `risk_vetoes` 字段供前端展示

**配置开关**: `agent.risk_veto_enabled` 默认 `true`
**成本**: 零 LLM 调用增加（仅文本提取 + prompt 追加）
**参考**: HedgeAgents 风控硬约束

---

### P0-B: 专家结论统一标注「对持仓影响」

**问题**: 专家结论格式不统一，配置专家难以做组合层面决策。
**方案**: 在 BlackboardEntry 增加 `portfolio_impact` 字段，统一标注每个建议对持仓的影响。

**改动**:
1. `blackboard.py` BlackboardEntry 增加字段 `portfolio_impact: Optional[dict] = None`
   ```python
   # portfolio_impact 结构
   {
       "affected_holding": "中证500ETF",     # 受影响的持仓
       "action": "加仓",                     # 加仓/减仓/新建/清仓/无影响
       "suggested_change": "10%",            # 建议变动比例
       "current_position_pct": 15.0,         # 当前占比
       "post_change_pct": 25.0,              # 变动后占比
       "risk_check": "未超25%上限"           # 风险检查结果
   }
   ```
2. `blackboard.py` `extract_entry_from_result` 增加持仓影响提取：
   - 从分析文本识别"加仓 X%"/"减仓 Y%"/"新建仓位"等
   - 结合 `build_portfolio_summary_line` 的当前持仓数据计算 post_change_pct
   - 超过 25% 上限自动在 risk_check 标注"超限"
3. `blackboard.py` `_format_entry` 展示 portfolio_impact 摘要
4. `blackboard.py` 新增 `get_portfolio_impacts()` 方法返回所有影响标注
5. `pipeline.py` `_synthesize_multiple_specialists` 注入组合影响汇总：
   - 把所有专家的 portfolio_impact 聚合成表格注入 prompt
   - 让仲裁 LLM 看到全局组合影响

**配置开关**: `agent.portfolio_impact_enabled` 默认 `true`
**成本**: 零 LLM 调用增加
**参考**: HedgeAgents 组合影响标注

---

### P1: 对抗式辩论 + 仲裁节点（Phase 3.7）

**问题**: 专家各说各话，LLM 综合时容易"和稀泥"。
**方案**: 在 Phase 3 与 Phase 4 之间插入辩论节点（仅冲突时触发），参考 TradingAgents 的 Bull/Bear Debate。

**流程**:
```
Phase 3 专家执行完成
  ↓
检测冲突（blackboard.find_conflicts()）
  ├─ 无冲突 → 直接进 Phase 4
  └─ 有冲突 → Phase 3.7 辩论
      ├─ 看多方：从黑板找 BUY 信号的专家，生成看多论证
      ├─ 看空方：从黑板找 SELL 信号的专家，生成看空论证
      └─ 仲裁 Agent：综合两方论证 + 持仓约束，给出明确方向
      ↓
Phase 4 综合（用辩论结论替代原始冲突）
```

**改动**:
1. `pipeline.py` 新增 `_phase_debate(state, query, execution_result, blackboard, trace_id)` 函数
2. `pipeline.py` 在 Phase 3 后、Phase 4 前插入辩论触发逻辑：
   ```python
   conflicts = blackboard.find_conflicts()
   debate_result = None
   if conflicts and get_config_bool("pipeline.debate_enabled", True):
       state.transition_to(PipelinePhase.DEBATE)
       yield {"type": EVENT_PHASE_START, "phase": "debate"}
       debate_result = _phase_debate(state, query, phase3_result, blackboard, trace_id)
       yield {"type": EVENT_DEBATE_DONE, "result": debate_result}
       yield {"type": EVENT_PHASE_END, "phase": "debate"}
   ```
3. `pipeline.py` `_phase_synthesis` 接收 `debate_result` 参数，注入到综合 prompt
4. `pipeline_state.py` PipelinePhase 枚举增加 `DEBATE = "debate"`
5. 辩论 prompt 设计：
   - 看多方：基于估值分位/历史低位/政策利好
   - 看空方：基于波动率/资金流出/宏观风险
   - 仲裁：综合两方 + 持仓现状 + 风险否决，给出 BUY/HOLD/SELL 明确方向 + 理由
6. 新增事件类型 `EVENT_DEBATE_DONE = "debate_done"`

**配置开关**: `pipeline.debate_enabled` 默认 `true`（冲突时才触发，非冲突零成本）
**成本**: 冲突时增加 1 次 LLM 调用（看多方+看空方合并为一次 prompt，仲裁一次 prompt，共 2 次）
**参考**: TradingAgents Bull/Bear Debate

### P2: 分专家模型分级

**问题**: 当前 MIMO 模式下所有专家都用 `mimo-v2.5-pro`，未按任务复杂度分级。
**方案**: 强化 `cost_routing` 配置，按专家任务特性分配不同模型档位。

**改动**:
1. `db/config.py` 增加 P2 模型分级配置项：
   ```python
   ('cost_routing.macro_strategist_model', 'deepseek-v4-flash', '宏观策略师模型（趋势判断，用便宜模型）', 'cost_routing'),
   ('cost_routing.article_expert_model', 'deepseek-v4-flash', '文章解读专家模型（文本摘要，用便宜模型）', 'cost_routing'),
   ('cost_routing.arbitrator_model', 'mimo-v2.5-pro', '仲裁专家模型（需强推理）', 'cost_routing'),
   ('cost_routing.debate_arbitrator_model', 'mimo-v2.5-pro', '辩论仲裁模型（需强推理）', 'cost_routing'),
   ```
2. `orchestrator.py` `_AGENT_MODEL_MAP_MIMO` 增加新专家映射：
   ```python
   _AGENT_MODEL_MAP_MIMO = {
       "valuation_expert": "mimo-v2.5-pro",       # 需精确数值推理
       "allocation_advisor": "mimo-v2.5-pro",     # 需组合优化
       "fund_analyst": "mimo-v2.5-pro",           # 需穿透分析
       "risk_assessor": "mimo-v2.5-pro",          # 需风险判断
       "market_analyst": "deepseek-v4-flash",     # 趋势判断，便宜模型
       "macro_strategist": "deepseek-v4-flash",   # 趋势判断，便宜模型
       "article_expert": "deepseek-v4-flash",     # 文本摘要，便宜模型
       "orchestrator": "mimo-v2.5-pro",           # 编排器，强模型
       "cross_review": "mimo-v2.5-pro",           # 交叉审阅，强模型
       "arbitrator": "mimo-v2.5-pro",             # 仲裁，强模型
       "debate_arbitrator": "mimo-v2.5-pro",      # 辩论仲裁，强模型
   }
   ```
3. `pipeline.py` 辩论仲裁使用 `_get_model_for_agent("debate_arbitrator")`

**配置开关**: 已有 `cost_routing.enabled`
**成本**: 宏观/文章专家改用便宜模型，预计降本 15-20%
**参考**: TradingAgents 分级模型策略

### P3: Plan-and-Execute 显式计划（Phase 2 强化）

**问题**: 当前 Phase 2 计划生成较简单，专家调用顺序不够智能。
**方案**: 强化 `plan_executor.generate_plan`，生成结构化执行计划含子问题拆解 + 工具白名单 + 依赖关系。

**改动**:
1. `plan_executor.py` `PlanStep` 增加字段：
   ```python
   sub_query: str = ""           # 拆解后的子问题（比 query 更具体）
   allowed_tools: list = []      # 工具白名单
   needs_debate: bool = False    # 该步骤后是否需要辩论
   ```
2. `plan_executor.py` `_PLAN_GENERATION_PROMPT` 强化：
   - 要求 LLM 输出 `sub_query`（具体子问题）和 `allowed_tools`
   - 要求标注 `needs_debate`（涉及买卖分歧时）
   - 增加持仓感知："用户当前持仓: {portfolio_summary}"
3. `plan_executor.py` `generate_plan` 增加 `portfolio_summary` 参数
4. `pipeline.py` `_phase_planning` 传入 `portfolio_summary`
5. `multi_agent.py` `run_specialist` 支持 `allowed_tools` 参数过滤工具
6. `pipeline.py` Phase 3 执行后检查 `needs_debate` 标志，决定是否触发辩论

**配置开关**: `pipeline.enhanced_plan_enabled` 默认 `true`，关闭则退化为原 plan 逻辑
**成本**: Phase 2 增加 1 次 LLM 调用（但省下后续冗余专家调用，净降本）
**参考**: LangGraph Plan-and-Execute 范式

## 三、实施顺序与依赖

```
P0-A 风险否决 ──┐
P0-B 持仓影响 ──┤
               ↓
P1 辩论节点 ────┤（依赖 P0-A 的 risk_veto 和 P0-B 的 portfolio_impact）
               ↓
P2 模型分级 ────┤（独立，可与 P1 并行）
               ↓
P3 Plan-Execute ┘（依赖 P1 的 needs_debate 标志）
```

**实施顺序**:
1. P0-A + P0-B（blackboard.py 改动，可一起做）
2. P1 辩论节点（pipeline.py + pipeline_state.py）
3. P2 模型分级（config.py + orchestrator.py）
4. P3 Plan-Execute（plan_executor.py + pipeline.py + multi_agent.py）

3. **P1 验证**: 构造估值 BUY + 风险 SELL 冲突，检查辩论节点触发 + 仲裁输出明确方向

## 六、风险与回退

- 所有改动都有开关，关闭即恢复旧行为
- 辩论节点失败自动降级到原 Phase 4 综合
- Plan-Execute 失败降级到原 plan 逻辑
- 模型分级失败降级到默认 MODEL
