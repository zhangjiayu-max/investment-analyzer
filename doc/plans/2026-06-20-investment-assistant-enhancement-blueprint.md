# 投资分析助手增强蓝图：从工具到个人理财决策伙伴

> 日期：2026-06-20  
> 目标：不局限于当前功能，结合成熟理财工具、自动化投顾、多 Agent 应用和 RAG/评测实践，梳理项目下一阶段可以增强、补充和蒸馏的数据能力。  
> 核心方向：让系统越用越懂我，并能为未来理财提供更充分、更正确、更可复盘的决策支持。

---

## 1. 一句话结论

当前项目已经具备很好的底座：多 Agent 编排、专家 Agent、RAG 检索、知识蒸馏、KYC 画像、记忆治理、评测、持仓、估值、文章解读、决策档案都已经有雏形。下一阶段最值得做的不是再堆更多聊天能力，而是把这些能力串成一个闭环：

**问题进入 -> 意图澄清 -> 个性化上下文 -> 证据召回 -> 场景化 Agent 分析 -> 决策记录 -> 行动追踪 -> 结果复盘 -> 用户画像/知识库/评测集更新。**

换句话说，项目要从“会分析的投资助手”升级成“有记忆、有原则、有证据、有复盘习惯的个人理财决策伙伴”。

## 2. 当前能力盘点

### 2.1 已有优势

| 能力 | 当前基础 | 下一阶段机会 |
|------|----------|--------------|
| 多 Agent | `backend/agent/orchestrator.py` 负责编排，`multi_agent.py` 执行专家，支持交叉审阅、仲裁、异步和 SSE | 把专家调用从“回答问题”升级为“决策工作流”，引入场景模板、typed output、handoff 状态机 |
| 单 Agent | 已支持专家独立运行和工具调用 | 建立高频轻量场景，避免所有问题都走复杂多 Agent |
| RAG | FTS5 + ChromaDB 双路召回，RRF 融合，知识类型过滤，日志和命中测试 | 增加个人化重排、时间有效性、证据等级、反例召回、知识图谱和决策结果回流 |
| 知识蒸馏 | 书籍、文章、技能文档、估值数据已有入库流程 | 从“内容摘要”升级为“可执行知识原子”：原则、指标、规则、反例、适用边界、失效条件 |
| KYC/记忆 | `kyc.py`、`kyc_learner.py`、`memory_lifecycle.py` 已有画像和记忆生命周期 | 扩展为完整个人财务画像：现金流、目标、约束、税务、行为偏差、家庭责任、决策风格 |
| 评测 | 对话评估、LLM 评估、RAG 评测、坏例转评测集已具备 | 建立“质量飞轮”：每次低分回答、用户纠错、错误决策都自动变成回归用例 |
| 决策档案 | `db/decisions.py` 已有决策记录、行动、复盘表 | 加入投资假设、触发条件、退出条件、预期区间、证据快照和复盘评分 |

### 2.2 主要短板

1. **用户画像仍偏静态 KYC**  
   当前画像主要覆盖风险偏好、期限、资金体量、经验、亏损承受度、关注品种。真正影响理财建议的还有：收入稳定性、现金流、负债、家庭责任、保险保障、流动性需求、税务账户、目标优先级、行为偏差、历史决策表现。

2. **决策闭环还没有完全跑起来**  
   已有决策档案表，但每次建议是否进入决策记录、后续是否执行、结果是否复盘、复盘是否反哺画像和评测，还需要产品化。

3. **RAG 仍偏“找资料”，不够“找证据”**  
   投资决策需要的不只是相关文本，还需要证据类型：数据、历史案例、原则、反例、适用边界、时间有效性、和用户当前情况的匹配度。

4. **多 Agent 还缺少场景化工作流**  
   现在的编排能调专家，但“买入决策”“卖出决策”“资产配置调整”“文章观点验证”“持仓体检”等场景，应该有固定步骤、固定输出和固定质检。

5. **评测还可以更贴近投资结果**  
   当前评测偏回答质量。投资助手还需要评测：是否过度自信、是否忽略用户约束、是否错用旧数据、是否缺少反方观点、是否给出不可执行建议、是否导致不合适交易。

## 3. 成熟工具和多 Agent 应用的启发

### 3.1 自动化投顾：目标、再平衡、税务和约束

Betterment 和 Wealthfront 这类自动化投顾的核心启发不是“自动交易”，而是它们把投资建议放进一个长期管理系统：目标组合、现金流进入后的再平衡、税务优化、持仓偏离检查、执行约束。Betterment 的公开材料强调 tax-loss harvesting 与组合再平衡的结合；Wealthfront 也把税损收割描述为在保持资产类别分布不变的前提下进行替换。  

对本项目的借鉴：

- 建立“目标账户”概念：养老、教育、备用金、买房、长期增值、短期现金管理。
- 建立“目标组合 vs 当前组合”的偏离度。
- 把新增资金、分红、卖出回款作为再平衡触发点。
- 中国市场不应照搬美国税损收割，但可以做“费用、赎回费、持有期、税费、流动性、交易摩擦”的本地化执行约束。

### 3.2 个人财务仪表盘：净值、现金流、目标和共同决策

Empower Personal Dashboard、Monarch 这类工具强调把账户、预算、投资、净值、目标放在同一视图。它们的启发是：理财建议不能只看某只基金或某个指数，而要看家庭资产负债表、现金流和目标进度。

对本项目的借鉴：

- 增加净值曲线、现金流、资产负债、目标进度。
- 把投资组合放进“整体财务健康”里，而不是孤立看收益率。
- 支持“资金用途”和“不可动用期限”，避免 AI 建议把短期资金放进高波动资产。
- 增加“家庭/共同决策”备注能力：哪些目标是刚性，哪些是可调整。

### 3.3 多 Agent 工程：handoff、guardrail、trace、memory、observability

LangGraph、AutoGen、CrewAI、OpenAI Agents SDK 的共同方向是：Agent 不只是多个 prompt，而是有路由、handoff、工具、状态、guardrail、trace 和可观测性。OpenAI Agents SDK 文档强调 handoff、guardrails、tracing；CrewAI 文档强调 crews、flows、guardrails、memory、knowledge、observability；LangGraph 文档区分 supervisor/subagent 和 handoff 模式。

对本项目的借鉴：

- 对复杂决策使用“主管 Agent + 专家 Agent + 质检 Agent”的显式工作流。
- 对高风险建议增加 guardrail：数据新鲜度检查、适当性检查、仓位上限检查、风险提示检查。
- 所有 Agent 输出结构化：结论、证据、假设、风险、反方观点、行动项、复盘日期。
- trace 不只是调试日志，也要成为用户信任机制：为什么这么建议，调用了哪些数据，哪些证据最关键。

## 4. 北极星闭环：越用越懂我的系统

建议把项目的核心闭环设计成 8 个环节：

1. **采集事实**  
   持仓、交易、现金、估值、文章、关注列表、历史问题、用户手动补充的目标和约束。

2. **识别意图**  
   区分“问知识”“看市场”“要不要买”“要不要卖”“复盘决策”“配置调整”“情绪安抚”“文章观点验证”。

3. **注入个人上下文**  
   KYC、目标、现金流、风险承受、持仓、历史偏好、近期决策、曾经踩过的坑。

4. **召回证据**  
   RAG 不只召回相关文本，还召回历史决策、用户记忆、估值快照、反例、风险规则。

5. **生成建议**  
   按场景选择单 Agent 或多 Agent，输出结构化结论。

6. **记录决策**  
   把建议、证据、假设、风险、行动、执行条件和复盘时间写入决策档案。

7. **追踪结果**  
   价格、估值、收益、回撤、执行状态、用户反馈。

8. **反哺学习**  
   更新用户画像、知识库、Agent prompt、评测集、策略规则。

## 5. 建议增强的功能模块

### 5.1 个人财务画像 2.0

当前 KYC 是很好的起点，但要成为“懂我”的助手，需要扩展为多层画像。

#### 新增画像维度

| 类别 | 维度 | 用途 |
|------|------|------|
| 目标 | 目标名称、金额、期限、优先级、可调整性 | 决定风险预算和资产期限 |
| 现金流 | 月收入、月支出、结余率、收入稳定性 | 判断定投能力和流动性风险 |
| 负债 | 房贷、消费贷、利率、月供 | 判断风险资产上限 |
| 备用金 | 月支出倍数、存放位置 | 防止误把应急资金投入高波动资产 |
| 行为偏差 | 追涨、恐慌卖出、过度集中、频繁交易 | 给建议时增加行为提醒 |
| 决策风格 | 喜欢简短结论/详细推导、保守/进取表达 | 个性化回答方式 |
| 交易约束 | 平台、费率、赎回限制、最小交易单位 | 让行动建议可执行 |
| 税务/账户 | A 股、基金、港股、美股、养老金账户等 | 给出账户层面的约束提醒 |

#### 数据结构建议

把画像分成三类，避免 AI 把推测当事实：

- `facts`：用户明确填写或交易记录可验证的事实。
- `inferences`：从对话/行为推断出的画像，带置信度和证据。
- `preferences`：表达风格、关注点、厌恶点。

每条画像都应有：

- `value`
- `confidence`
- `evidence`
- `source`
- `last_verified_at`
- `expires_at`
- `contradicts`

### 5.2 理财目标与资金桶

新增“资金桶”模型，把资产按用途分层：

| 资金桶 | 示例 | 建议能力 |
|--------|------|----------|
| 备用金桶 | 3-12 个月生活费 | 流动性和安全性优先，禁止高波动建议 |
| 稳健增值桶 | 1-3 年资金 | 债券、固收+、低波动组合，关注回撤 |
| 长期权益桶 | 3-10 年资金 | 指数、基金、股票，关注估值和再平衡 |
| 机会资金桶 | 低估资产、主题机会 | 控制比例，明确退出条件 |
| 学习/试错桶 | 小仓位实验 | 防止影响主组合 |

这会显著提升建议质量，因为同样是“沪深300能买吗”，备用金桶和长期权益桶的答案应该完全不同。

### 5.3 决策中枢升级

已有 `decision_records`、`decision_actions`、`decision_reviews`，建议扩展成完整决策账本。

#### 决策记录应包含

- 决策类型：买入、卖出、加仓、减仓、持有、观察、换仓、再平衡。
- 投资假设：为什么现在值得做。
- 证据快照：估值、持仓、RAG 来源、市场数据、文章观点。
- 反方观点：什么情况下这个决策是错的。
- 风险预算：最大仓位、最大亏损、可承受回撤。
- 执行条件：一次性/分批、触发价、估值区间、时间窗口。
- 退出条件：止盈、止损、基本面恶化、估值回归、目标变化。
- 复盘日期：7 天、30 天、90 天或自定义。

#### 新增功能

1. **一键把 AI 建议转为决策草案**  
   对话回答下方出现“保存为决策”，自动填入证据、行动项和复盘日期。

2. **决策前检查清单**  
   在执行前检查：是否符合目标、是否超过仓位、是否有现金需求、是否数据过期、是否有反方观点。

3. **决策复盘卡片**  
   到期后自动提醒：结果如何、假设是否成立、执行是否偏离、下次要改什么。

4. **个人决策统计**  
   统计哪些建议最有帮助、哪些场景容易出错、用户是否经常追涨杀跌、是否过度集中。

### 5.4 多 Agent 对话增强

#### 推荐的多 Agent 场景

| 场景 | 参与 Agent | 输出 |
|------|------------|------|
| 买入/加仓决策 | 估值、资产配置、风险、行为教练、执行审查 | 是否行动、分批计划、最大仓位、风险条件 |
| 卖出/减仓决策 | 估值、基本面/趋势、税费/成本、行为教练 | 卖出理由是否成立、替代方案、退出计划 |
| 持仓体检 | 组合医生、风险、相关性、费用、目标匹配 | 问题清单、优先级、调整建议 |
| 文章观点验证 | 文章解读、数据验证、反方观点、用户适配 | 作者观点、证据强弱、是否适合我 |
| 极端市场应对 | 风险、行为教练、流动性、估值 | 是否需要行动、不要做什么、观察指标 |
| 年度理财规划 | 目标规划、资产配置、现金流、风险、复盘 | 年度目标组合、行动计划、检查频率 |

#### 编排增强

1. **场景模板化**  
   `clarification` 不只输出专家列表，还输出 `scenario_type`。不同场景绑定固定流程和输出 schema。

2. **handoff 状态显式化**  
   记录每次专家接力：谁把问题交给谁、理由是什么、带了哪些上下文。

3. **专家输出结构化**  
   每个专家输出 JSON 字段：`claim`、`evidence`、`risk`、`confidence`、`missing_data`、`action_items`。

4. **风险 Agent 拥有 veto 权**  
   当建议违反用户约束、数据过期、仓位超限、缺少关键证据时，风险 Agent 可以要求降级为“观察/补数据”。

5. **仲裁从总结升级为裁决**  
   仲裁 Agent 不只汇总，还要明确：采纳哪些专家、否决哪些观点、关键分歧是什么、需要用户确认什么。

### 5.5 单 Agent 高频场景

不是所有问题都要多 Agent。建议建立轻量单 Agent 场景，提高速度、降低 token 成本。

| 场景 | Agent | 说明 |
|------|-------|------|
| 快速知识问答 | 投资教师 | 解释概念、术语、策略，不做个性化交易建议 |
| 今日持仓提醒 | 理财顾问 | 查看持仓相关估值、新闻、预警 |
| 单标的速查 | 估值助手 | 返回估值、趋势、历史分位、观察点 |
| 文章摘要 | 文章解读助手 | 结构化拆解，不必调用全套专家 |
| 决策复盘 | 复盘教练 | 对已有决策进行复盘，不重新生成大段分析 |
| 情绪安抚 | 行为教练 | 市场大跌时帮助用户避免冲动操作 |

单 Agent 输出也应结构化，至少包含：结论、依据、风险、下一步。

## 6. RAG 与知识库增强

### 6.1 从“知识条目”升级为“知识原子”

建议知识库不再只存 `category/title/content/source`，而是将蒸馏结果拆成多种知识原子。

| 知识原子 | 示例 | 用途 |
|----------|------|------|
| principle | 安全边际、长期主义、分散配置 | 给建议提供原则依据 |
| rule | PE 百分位低于某区间才考虑分批 | 转成可执行检查 |
| indicator | PE/PB/ROE/股债利差/实际利率 | 连接市场数据 |
| case | 历史牛熊、某类资产表现 | 做类比和风险教育 |
| counterexample | 低估但继续下跌、盈利陷阱 | 防止单一证据过度自信 |
| checklist | 买入前检查、卖出前检查 | 决策 guardrail |
| user_lesson | 用户历史复盘得到的经验 | 个性化建议 |
| source_view | 某作者/文章观点 | 保留观点来源和偏见 |
| data_snapshot | 某日估值、持仓、价格 | 决策复盘需要 |

### 6.2 RAG 缺少的关键能力

1. **时间有效性**  
   对估值、新闻、政策、基金经理、费率、持仓等内容设置 `valid_until` 或 `as_of_date`。过期数据可召回，但必须标注“仅作历史参考”。

2. **证据等级**  
   区分数据、原文、观点、用户记忆、模型推断。回答时优先使用高等级证据。

3. **反例召回**  
   当用户问“能买吗”，RAG 应主动召回“买入失败案例”“低估陷阱”“估值失效条件”。

4. **个人化重排**  
   根据用户关注资产、风险偏好、持仓、历史问题，对召回结果加权。

5. **跨资产知识图谱**  
   黄金、美元、实际利率、美债、A 股估值、人民币汇率、北向资金、商品周期之间需要显式关系。

6. **决策结果回流**  
   历史决策及其复盘应进入 RAG。以后遇到类似问题，系统能说：“你上次在类似场景下因为追涨导致回撤，建议这次先做仓位上限检查。”

7. **引用可追溯**  
   每段关键建议最好能追溯到来源：书籍、文章、估值快照、用户画像、持仓数据或决策复盘。

### 6.3 需要优先蒸馏的数据

#### P0：立刻能提升决策质量

| 数据 | 蒸馏目标 |
|------|----------|
| 用户历史对话 | 用户目标、偏好、禁忌、反复关注的问题、风险承受表达 |
| 用户历史决策 | 决策理由、执行结果、复盘教训、行为偏差 |
| 持仓与交易 | 仓位变化、盈亏来源、集中风险、交易频率 |
| 估值历史 | 指数估值区间、历史极端区间、均值回归特征 |
| 已有书籍高质量条目 | 买入/卖出/配置/风险/行为检查清单 |

#### P1：提升专业度和覆盖面

| 数据 | 蒸馏目标 |
|------|----------|
| 基金定期报告 | 重仓变化、风格漂移、基金经理观点、费率 |
| 指数编制规则 | 行业权重、成分规则、调仓规则、风险暴露 |
| 宏观数据 | 利率、通胀、汇率、社融、PMI、股债利差 |
| 监管和政策 | 与行业/资产相关的政策事件和影响路径 |
| 经典失败案例 | 低估陷阱、主题泡沫、流动性危机、风格切换 |

#### P2：长期个性化

| 数据 | 蒸馏目标 |
|------|----------|
| 用户阅读行为 | 哪类内容真正影响用户决策 |
| 用户反馈语料 | 喜欢的解释风格、讨厌的回答方式 |
| 家庭财务目标 | 目标变化、资金用途、优先级 |
| 现金流记录 | 定投能力、风险资产上限、备用金是否充足 |

### 6.4 蒸馏输出模板

建议所有蒸馏结果使用统一 schema：


## 7. 评测体系增强

### 7.1 从回答质量到决策质量

当前评测已经覆盖数据准确性、逻辑一致性、可执行性、用户理解等维度。建议增加投资决策特有维度：

| 维度 | 检查点 |
|------|--------|
| 适当性 | 是否符合用户风险、期限、目标、现金流 |
| 数据新鲜度 | 是否使用过期估值、旧新闻、旧持仓 |
| 证据完整性 | 是否同时有支持证据和反方证据 |
| 仓位纪律 | 是否给出仓位上限、分批计划、退出条件 |
| 不确定性表达 | 是否明确假设和置信度，避免过度确定 |
| 行为风险 | 是否识别追涨、恐慌、锚定、损失厌恶 |
| 可复盘性 | 是否能在未来判断建议对错 |

### 7.2 评测集来源

1. 用户点踩或低分回答。
2. LLM 评估低于阈值的回答。
3. 被用户手动纠错的回答。
4. 决策复盘结果差的建议。
5. RAG 召回失败或证据错配的查询。
6. 高风险场景：加仓、卖出、重仓、追热点、极端行情。

### 7.3 自动回归

建议建立每日/每周评测任务：

- 跑固定 eval cases。
- 跑最近 7 天新增坏例。
- 比较 Agent prompt 修改前后的分数。
- 输出“退步项”：哪个 Agent、哪个场景、哪个指标变差。
- 将失败案例自动归类到：路由问题、RAG 问题、工具问题、prompt 问题、数据缺失、用户画像缺失。

## 8. 产品功能建议清单

### P0：最优先，直接形成闭环

1. **AI 建议保存为决策草案**  
   从对话、持仓分析、指数分析一键生成 `decision_records`。

2. **决策前检查清单**  
   对买入/卖出/加仓/减仓建议做适当性、数据新鲜度、仓位、风险、反方观点检查。

3. **复盘提醒和复盘卡片**  
   到 `review_at` 后提醒用户复盘，并让复盘结果进入记忆和评测。

4. **个人画像 2.0 基础字段**  
   增加目标、现金流、备用金、负债、资金用途、不可动用期限。

5. **RAG 证据类型和时间字段**  
   为知识条目增加 `atom_type`、`evidence_level`、`as_of_date`、`valid_until`。

6. **场景路由**  
   `clarification` 输出 `scenario_type`，至少区分：知识问答、买入决策、卖出决策、持仓体检、文章验证、复盘。

### P1：增强专业度和个性化

1. **目标账户/资金桶**  
   将持仓和现金按用途归属到不同目标。

2. **组合偏离和再平衡建议**  
   展示目标配置、当前配置、偏离、建议调仓路径。

3. **行为教练 Agent**  
   在大跌、追热点、重仓、频繁交易时主动提醒。

4. **反方观点 Agent**  
   所有高风险决策必须给出“为什么不该做”。

5. **知识库个人化重排**  
   根据用户画像和持仓调整 RAG 排序。

6. **坏例自动入评测集**  
   低质量回答、用户纠错、失败复盘进入 eval cases。

### P2：长期高级能力

1. **情景推演/压力测试**  
   模拟：市场下跌 20%、利率上行、人民币贬值、行业政策冲击、失业/现金流下降。

2. **基金/指数穿透分析**  
   穿透重仓、行业、风格、估值、相关性、费用。

3. **家庭财务仪表盘**  
   净值、现金流、负债、目标进度、投资组合统一视图。

4. **策略沙盒和回测**  
   对定投、再平衡、估值分位买卖规则做历史模拟。

5. **多模型评审**  
   关键决策由不同模型/不同 prompt 交叉验证，降低单模型偏差。

## 9. 推荐实施路线

### Phase 1：决策闭环 MVP（1-2 周）

目标：让每次重要建议都能沉淀、追踪和复盘。

- 对话回答增加“保存为决策草案”。
- 决策记录补充：假设、反方观点、触发条件、退出条件、证据快照。
- 新增决策前检查清单。
- 新增复盘提醒和复盘入口。
- 复盘结果写入 `decision_reviews`，并生成 `user_lesson` 知识原子。

验收标准：

- 用户可以从一次 AI 建议生成完整决策草案。
- 每条决策都有复盘日期和至少一个行动项。
- 复盘后能在后续对话中被 RAG 或记忆召回。

### Phase 2：画像和场景路由（2-3 周）

目标：让系统先判断“这是什么场景”和“这个建议适不适合我”。

- 扩展用户画像表或新增画像扩展表。
- 增加目标、现金流、备用金、负债、资金用途字段。
- `clarification` 输出 `scenario_type`。
- 建立单 Agent/多 Agent 场景模板。
- 风险 Agent 对高风险建议做 veto/降级。

验收标准：

- 同一个投资问题，在不同资金用途下给出不同建议。
- 买入/卖出/持仓体检/文章验证能走不同流程。
- 高风险建议必须出现适当性检查。

### Phase 3：RAG 证据化和蒸馏升级（2-4 周）

目标：让知识库从“相关内容库”升级为“证据库”。

- 扩展知识 schema：`atom_type`、`evidence_level`、`as_of_date`、`valid_until`、`limitations`、`counterpoints`。
- 优先蒸馏用户历史决策、复盘、已有高质量书籍条目。
- 新增反例召回和个人化重排。
- 低质量知识条目清理和重蒸馏。

验收标准：

- 关键建议能标出证据类型和时间。
- “能买吗”类问题能自动召回反方证据。
- 复盘教训可被后续相似问题召回。

### Phase 4：评测飞轮（长期持续）

目标：每次失败都让系统变强。

- 坏例自动生成 eval case。
- 每周回归评测 Agent/RAG/场景模板。
- 评测结果反向生成优化建议。
- 建立质量趋势页：哪个 Agent 变好/变差，哪些场景最容易出错。

验收标准：

- 低分回答能一键转评测。
- prompt 或 RAG 修改后可看到回归对比。
- 决策复盘差的案例能成为后续 guardrail。

## 10. 关键设计原则

1. **先适当性，后收益率**  
   投资建议首先要适合用户目标、期限和风险承受，而不是只追求收益。

2. **先证据，后观点**  
   每个关键建议都应能追溯证据来源。

3. **先仓位，后方向**  
   “看对方向但仓位错误”仍然会导致坏结果。

4. **先反方，后行动**  
   所有买入/卖出建议都要主动说明反方观点和失效条件。

5. **先复盘，后进化**  
   不复盘的系统不会真正懂用户，只会记住更多噪声。

6. **AI 不替用户做决定，只帮助用户更充分地决策**  
   系统应提供证据、选项、风险、检查清单和复盘，而不是制造“确定性幻觉”。
