> 日期：2026-07-09

## 一、问题诊断

### 1.1 RAG 检索问题
- **rag_logs 日志缺失**：Pipeline 主路径 `_phase_info_gather` 调 `build_rag_context_with_details` 但未调 `log_rag_search`，默认路径检索质量不可观测
- **子查询展开是死代码**：`query_rewriter.expand_query()` 已实现 7 条意图规则+多角度展开，完全未接入检索
- **intent/needed_info 浪费**：`query_understander` 已提取但仅 targets 被简单拼接，intent/needed_info 未用于检索范围控制

### 1.2 澄清链路问题
- **链路断裂**：Pipeline 触发澄清后 return，orchestrator 判定 `pipeline_completed=False` 降级 ReAct 重跑，澄清形同虚设
- **前端看不到**：clarification 事件被转译成 status，只在状态条闪过，不落库不交互
- **无续答机制**：用户无法回答澄清问题，只能发新消息重走全流程

## 二、检索提升方案

### 2.1 补全 rag_logs（[pipeline.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py)）

在 `_phase_info_gather` 的 RAG 检索后调用 `log_rag_search`：
```python
from services.rag import build_rag_context_with_details, log_rag_search
rag_data = build_rag_context_with_details(enhanced_query, limit=8, content_types=content_types)
# 日志记录
log_rag_search(
    conversation_id=state.conversation_id,
    message_id=state.message_id,
    query=enhanced_query,
    keywords=...,  # 从 rag_data 提取
    results=results["rag"],
    content_types=content_types,
    fts_count=rag_data.get("fts_count", 0),
    chroma_count=rag_data.get("chroma_count", 0),
    trace_id=trace_id,
)
```

需要 `build_rag_context_with_details` 返回 fts_count/chroma_count（检查现状，如未返回则从 results 推断）。

### 2.2 接入子查询展开（[pipeline.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py)）

在 `_phase_info_gather` 中，RAG 检索前调 `expand_query` 生成子查询，多路检索后 RRF 融合：

```python
from agent.query_rewriter import expand_query
sub_queries = expand_query(enhanced_query)  # [原始, 估值视角, 风险视角, ...]
all_results = []
for sq in sub_queries[:3]:  # 最多 3 个子查询
    data = build_rag_context_with_details(sq, limit=5, content_types=content_types)
    all_results.extend(data.get("results", []))
# 去重 + 按分数排序（复用已有 RRF 逻辑或简单去重）
```

**成本控制**：子查询最多 3 个，每个 limit=5，总检索量 ≤15 条（与当前 limit=8 相当）。
**开关**：`rag.subquery_expansion_enabled`（默认 true）。

### 2.3 intent 驱动 content_types（[pipeline.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py)）

用 `query_info` 的 `needed_info` 映射到检索 `content_types`：

```python
_INTENT_TO_CONTENT_TYPES = {
    "valuation": ["valuation", "analysis"],
    "portfolio": ["book", "analysis"],
    "risk": ["analysis", "article"],
    "strategy": ["book", "analysis", "article"],
    "article": ["article", "author_article"],
}
def _map_content_types(needed_info: list) -> list | None:
    types = set()
    for info in needed_info:
        types.update(_INTENT_TO_CONTENT_TYPES.get(info, []))
    return list(types) if types else None  # None 表示全类型
```

**收益**：避免全类型检索的噪音，提升精度。
**开关**：`rag.intent_driven_types_enabled`（默认 true）。

## 三、澄清链路修复方案

### 3.1 后端：修复链路断裂（[orchestrator.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/orchestrator.py)）

修改 clarification 事件处理：不再转译为 status，而是**直接转发**并标记 `pipeline_completed=True`：

```python
elif evt_type == "clarification":
    pipeline_completed = True  # 关键：不再降级 ReAct
    yield {
        "type": "clarification",
        "question": evt.get("question", ""),
        "options": evt.get("options", []),
        "checkpoint": evt.get("checkpoint"),  # 用于续答
    }
    return
```

### 3.2 后端：结构化澄清（[query_understander.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/query_understander.py)）

修改 LLM prompt，让澄清问题带选项：

```json
{
  "needs_clarification": true,
  "clarification_question": "您想了解哪方面？",
  "clarification_options": ["估值分析", "买卖建议", "风险评估"]
}
```

规则降级路径也补充常见澄清模板（如空查询时给选项）。

### 3.3 后端：Pipeline 澄清 yield checkpoint（[pipeline.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/pipeline.py)）

澄清时 yield checkpoint state（序列化 PipelineState），供续答恢复：

```python
if need_clarify:
    checkpoint = state.to_dict()
    yield {
        "type": EVENT_CLARIFICATION,
        "question": clarify_q,
        "options": query_info.get("clarification_options", []),
        "checkpoint": checkpoint,
    }
    state.transition_to(PipelinePhase.CANCELLED)
    return
```

### 3.4 后端：续答接口（[routers/conversations.py](file:///Users/xiaoyuer/projects/investment-analyzer/backend/routers/conversations.py)）

新增 `POST /api/conversations/{conv_id}/clarify-answer`：
- 接收：`{message_id, answer, checkpoint}`
- 逻辑：用 answer 改写 query（`refined_query = f"{original_query} {answer}"`），从 checkpoint 恢复 PipelineState，从 Phase 1 开始执行（跳过 Phase 0 的澄清检查）
- 返回：SSE 流（与正常对话一致）

### 3.5 前端：澄清展示 + 续答（[useStreamingState.js](file:///Users/xiaoyuer/projects/investment-analyzer/frontend/src/composables/useStreamingState.js) + [ChatView.vue](file:///Users/xiaoyuer/projects/investment-analyzer/frontend/src/components/ChatView.vue)）

新增 `case 'clarification'`：
- 把澄清问题作为**正式消息**添加到消息列表（role=assistant，特殊类型 clarification）
- 渲染可点击选项按钮
- 用户点击选项或手动输入 → 调 `/api/conversations/{id}/clarify-answer` 接口
- 续答后的 SSE 流复用现有 handleStreamEvent 处理
