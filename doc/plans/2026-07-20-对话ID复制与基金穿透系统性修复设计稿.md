## 一、问题清单与根因

### 任务 1+2：AI 对话 convid/messageid 复制功能

#### 用户原话
> 1. AI 对话的时候 convid 和 messageid 现在支持复制。但是移动端好像不能复制呀
> 2. 复制的时候不要只复制 id，要补充清楚，是 convid 还是 messageid，方便我们粘贴排查问题

#### 当前实现

| 复制项 | 文件:行号 | 复制内容 | 反馈方式 |
|---|---|---|---|
| conv ID | `frontend/src/components/ChatView.vue:1021-1033` | `对话ID: 123`（带前缀） | CSS class 切换按钮颜色 1.5s |
| message ID（气泡） | `frontend/src/components/ChatView.vue:1513-1526` | `123`（纯 ID，**无前缀**） | 全局 toast |
| message ID（评估卡） | `frontend/src/components/ui/ConversationEvalCard.vue:331-347` | `123`（纯 ID） | toast（但组件是孤儿未挂载） |

#### 根因
- **根因 1.1（不一致）**：conv id 复制带前缀 `对话ID:`，message id 复制不带前缀，格式不统一，粘贴排查时容易混淆
- **根因 1.2（移动端 UI 不适配）**：移动端通过 `MobileApp.vue:145` 直接复用 PC 端 `ChatView`，但 `.btn-conv-id` 和 `.message-id-badge` 没有移动端专属样式，按钮偏小点击区域不足
- **根因 1.3（移动端 clipboard 兼容性）**：`navigator.clipboard.writeText()` 在部分移动端浏览器（HTTP 协议、微信内置浏览器、低版本 iOS WebView）下不可用或返回 reject，当前 `copyMessageId` 有 execCommand 兜底但 `copyConvId` 只有 `window.prompt` 降级（不友好）
- **根因 1.4（DOM 查询不规范）**：`copyConvId` 用 `document.querySelector('.btn-conv-id')` 找 DOM 添加 `.copied` class，多对话切换时可能找错按钮
- **根因 1.5（孤儿组件）**：`ConversationEvalCard.vue` 未在任何地方 import 使用，其复制功能实际不可达

### 任务 5：conv 129 穿透查基金持仓失败

#### 用户原话
> 排查一下 convid 129 回答的不好，没有穿透查做这个基金持仓重仓哪些呢，只是看名字是债券，其实中仓了很多锂金属等 所有大跌亏损

#### conv 129 实际执行情况

- 用户问题（msg id=414）："深度分析我持有的博时恒乐两个债券。亏损严重，穿透看下亏损原因。已经到底了吗，最近有什么政策新闻等等利好吗"
- 路由结果：3 专家（`allocation_advisor`、`valuation_expert`、`risk_assessor`），**`fund_analyst` 未路由**
- 综合报告结论：**"博时恒乐债券 A 和 C 目前均为正收益，并非亏损"**，直接否定用户"亏损严重"感知
- 工具调用 9 次，**0 次 `query_fund_info`**

#### 5 环全断根因

**根因 5.1（路由截断）**：`fund_analyst` 在 `_KEYWORD_ROUTES` 优先级第 6 位（priority=8），被 `max_specialists.complex=4` 截断
- 文件：`backend/agent/core/router.py:57-119`
- 匹配到关键词 `债券/穿透/亏损/原因` 等，但被优先级更高的 4 个专家挤掉

**根因 5.2（工具未分配）**：被路由的 3 个专家（`valuation_expert`、`risk_assessor`、`allocation_advisor`）的 tools 列表都没有 `query_fund_info`
- 文件：`backend/db/agents.py`
- 即使路由截断，这些专家想穿透查重仓也没有工具可调用

**根因 5.3（akshare 系统性失效）**：`get_fund_holdings()` 调用的 4 个 akshare 接口中，`fund_portfolio_hold_em` 和 `fund_portfolio_bond_hold_em` 对所有基金都报 `JSONDecodeError: Can not decode value starting with character ';'`
- 文件：`backend/db/portfolio.py:2148-2246`
- 测试 4 个基金代码（014846、161725、110011、012410）全部失败，是 akshare 接口被东财页面变更破坏
- 实际 `query_fund_info('014846')` 返回 `top_stocks=[]`、`bond_holdings=[]`，但 `asset_allocation` 和 `industry_allocation` 正常

**根因 5.4（ttfund 兜底未暴露）**：`backend/mcp/ttfund_client.py:314-316` 的 `fund_holding` 方法已实现，但 `tools/__init__.py` 的 TOOLS 数组未暴露 `ttfund_fund_holding`
- ttfund MCP 当前未登录（`RuntimeError: 需要登录`），需先登录才能用
- 即使不能立刻用，至少需要把工具入口先建好

**根因 5.5（基金类型元数据误导）**：`portfolio_holdings.fund_category='bond'` 是过度简化字段
- 博时恒乐实际是"二级债基"（`fund_metadata.fund_type='债券型-混合二级'`），可持有最多 20% 股票
- AI 看到 `bond` 就当纯债处理，直接否定用户"亏损严重"
- `fund_analysis.py:2364` 的债基降级逻辑 `if fund_category in ("bond", "纯债", "混合债") and not top_stocks` 会跳过基本面维度，但"混合二级债基"不应跳过

#### 核心结论
"路由失败 + 工具失效 + 兜底缺失 + 类型误导"四重故障叠加，导致 AI 完全没有看到穿透数据，仅凭"债券"二字下结论。

#### P0-6：修复 `get_fund_holdings` akshare 失败的兜底链路

**文件**：`backend/db/portfolio.py:2148-2246`

修改 `get_fund_holdings(fund_code)` 函数，建立三级兜底：

```python
def get_fund_holdings(fund_code: str) -> dict:
    """获取基金重仓股/债券持仓（三级兜底：akshare → ttfund MCP → 本地表快照）。"""
    result = {"top_stocks": [], "bond_holdings": [], "asset_allocation": [], "industry_allocation": [], "report_date": None}

    # 第 1 级：akshare（当前主路径，部分失效）
    try:
        ak_data = _akshare_fund_holdings(fund_code)
        if ak_data.get("top_stocks"):
            result.update(ak_data)
            _save_fund_holdings_snapshot(fund_code, ak_data)  # 同步写本地快照
            return result
        # akshare 失效或为空，继续兜底
        result.update({k: v for k, v in ak_data.items() if v})
    except Exception as e:
        logger.warning(f"[fund_holdings] akshare 失败: {e}")

    # 第 2 级：ttfund MCP（需先登录）
    try:
        ttfund_data = _ttfund_fund_holding(fund_code)
        if ttfund_data.get("top_stocks"):
            result.update(ttfund_data)
            _save_fund_holdings_snapshot(fund_code, ttfund_data)
            return result
    except Exception as e:
        logger.warning(f"[fund_holdings] ttfund 失败: {e}")

    # 第 3 级：本地快照表（最近一次成功的缓存）
    try:
        snapshot = _get_fund_holdings_snapshot(fund_code)
        if snapshot:
            result.update(snapshot)
            result["_data_source"] = "local_snapshot_stale"  # 标记为陈旧数据
    except Exception as e:
        logger.warning(f"[fund_holdings] 本地快照读取失败: {e}")

    return result
```

新增本地快照表 `fund_holdings_snapshots`：
```sql
CREATE TABLE IF NOT EXISTS fund_holdings_snapshots (
    fund_code TEXT PRIMARY KEY,
    top_stocks_json TEXT,
    bond_holdings_json TEXT,
    asset_allocation_json TEXT,
    industry_allocation_json TEXT,
    report_date TEXT,
    data_source TEXT,  -- akshare / ttfund
    updated_at TEXT DEFAULT (datetime('now','localtime'))
)
```

## 五、风险与回滚

### 风险
1. **路由优先级提升导致 `fund_analyst` 被过度路由**：可能影响简单估值查询场景的成本（多一个专家的 LLM 调用）。缓解：仅对含穿透关键词的问题强制路由，普通估值查询仍走 `valuation_expert`
2. **akshare 升级风险**：本方案不升级 akshare，仅增加兜底，避免引入新接口变更风险
3. **ttfund MCP 登录依赖**：若 ttfund 未登录，第 2 级兜底会失败，但第 3 级本地快照仍可用
4. **数据库迁移**：扩展 tools 字段需 UPDATE 已有专家记录，注意不要覆盖现有 tools

### 回滚
- 所有修改可通过 git revert 回滚
- 数据库迁移：新增的 `fund_holdings_snapshots` 表可保留不删（不影响现有功能）
- tools 字段更新：可通过备份的旧 tools 字段值恢复
