## 二、🔴 严重问题（影响稳定性/数据正确性/安全性）

### 2.1 数据库连接泄漏风险

**位置：** `db/portfolio.py` 多处  
**现状：** `_get_conn()` 调用后缺少 `try/finally conn.close()`，异常时连接不释放：

```python
# db/portfolio.py 多处：
conn = _get_conn()
rows = conn.execute("SELECT ...").fetchall()
conn.close()  # ← 如果中间抛异常，close 不执行
return [dict(r) for r in rows]
```

**影响：** 高并发或长时间运行时 SQLite 连接池耗尽 → 应用不可用。  
**修复：** 统一使用 `with` 上下文管理器或者 `try/finally` 包装所有 `_get_conn()` 调用。

**影响文件：** `db/portfolio.py` (30+ 处)、`db/decisions.py`、`db/eval.py`、`routers/analysis/*.py`

### 2.3 全系统无输入校验

**位置：** 所有 API 端点  
**现状：** FastAPI 路由接收 `data: dict` 后直接使用，不做校验：

```python
# routers/portfolio.py
@router.post("/holdings")
async def create_holding(data: dict):  # ← 无 Pydantic model
    create_holding(
        fund_code=data.get("fund_code", ""),
        shares=float(data.get("shares", 0)),  # ← 可能 ValueError
    )
```

**影响：**
- 用户可以传入负数份额、非法基金代码、超长字符串
- 可能导致数据库写入垃圾数据
- SQLite 无数据类型强制，错误数据会静默存储

**修复：** 全部使用 Pydantic model + Field 校验（项目中 `models/portfolio.py` 已定义但未被所有端点使用）。

---

### 2.4 Python 3.11+ 异步事件循环管理问题

**位置：** `app.py`, `orchestrator.py`, 多个 router 文件  
**现状：** 使用 `asyncio.new_event_loop()` 在 `to_thread` 中创建新事件循环：

```python
# orchestrator.py
loop = asyncio.new_event_loop()
try:
    result = loop.run_until_complete(fetch_article(url))
finally:
    loop.close()
```

**影响：**
- Python 3.11+ 中 `new_event_loop` 后未设置为当前循环，某些库（aiohttp, httpx）可能报错
- `loop.close()` 后如果有未完成的异步生成器，会触发 ResourceWarning
- 多线程共用事件循环存在线程安全问题

**修复：** 使用 `asyncio.run()` 替代手动创建循环（Python 3.11+ 推荐方式）。

### 2.5 前端 API 无请求去重/防抖

**位置：** `frontend/src/api/index.js`  
**现状：** 无请求去重，用户快速点击"分析"按钮会发起多个相同请求：

```javascript
// 无防抖的 AI 分析触发
export function runPortfolioAiAnalysis(question) {
  return api.post('/portfolio/ai-analysis', { question }, { timeout: 300000 })
}
```

**影响：** 重复调用 LLM → 浪费 Token 和费用。一个 5 分钟的 AI 分析如果用户点了 3 次，会同时跑 3 个。

**修复：** 对高成本 API（AI 分析、LLM 调用）加请求去重（pending state + disabled button）。

## 五、🟢 数据精度问题

### 5.1 基金净值日期不是 T-1

**位置：** `market_data.py`, `db/portfolio.py` 的 `fetch_fund_nav()`  
**现状：** akshare 返回的净值日期可能是 T-2 甚至更旧，没有做新鲜度校验。QDII 基金净值延迟 2-3 个交易日。

**建议：** 在 `refresh_all_fund_prices()` 中标注净值日期，前端展示时提示"数据截至 X 月 X 日"。

---

### 5.2 交易记录和持仓盈亏不同步

**位置：** `db/portfolio.py`  
**问题：** 手动创建的买入交易 (`create_transaction`) 和 `_recalculate_holding` 的计算可能存在浮点舍入误差导致不匹配。

**建议：** 加一个 `reconcile` 函数，定期检查 `portfolio_holdings.total_cost` 和 `portfolio_transactions.amount` 之和是否一致。

---

### 5.3 估值数据去重逻辑不足

**位置：** `db/valuations.py`  
**问题：** `list_valuation_indexes()` 返回所有记录，但同一天同一指数可能有 PE 和 PB 两条记录。前端多次调用可能展示不一致的数据。

**建议：** 在 SQL 查询层直接 GROUP BY 去重，取最新日期的 PE+PB 合并返回。

## 七、性能审查

### 7.1 N+1 查询问题

**位置：** `health_score.py` 的 `_get_holding_valuations()`  
**现状：**
```python
for h in holdings:
    for vn, vv in val_map.items():
        if vn in name or name in vn:  # O(n*m) 双重循环
```

当持仓 20 只、估值 50 条时，1000 次字符串匹配。

**优化：** 预建索引字典，O(n+m)。

---

### 7.2 前端大量 computed 重计算

**位置：** `PortfolioManagement.vue`  
**问题：** 8649 行组件内有几十个 `computed`，每个都可能在任意响应式依赖变化时重算。

**建议：** 拆分组件后，每个小组件的 computed 只依赖自己的 props，不交叉污染。

---

### 7.3 SQLite 无连接池

**现状：** 每次查询都 `_get_conn()` → `conn.close()`，SQLite 文件频繁打开关闭。

**建议：** 使用单例连接 + WAL 模式（Write-Ahead Logging），提升并发读取性能：

```python
# db/_conn.py
conn = sqlite3.connect(DB_PATH, check_same_thread=False)
conn.execute("PRAGMA journal_mode=WAL")
```
