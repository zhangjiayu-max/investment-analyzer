# 对话流水线与上下文管理增强设计稿

> 日期：2026-07-07  
> 目标：解决死循环/重复执行/token浪费，引入分阶段对话状态机，增强上下文管理，提升决策质量

---

## 一、现状问题诊断

### 1.1 死循环/重复调用风险

| 问题 | 现状 | 风险 |
|------|------|------|
| Orchestrator ReAct 最大 3-6 轮 | ✅ 有 MAX_TURNS 保护 | 但同 turn 内如果 LLM 反复输出空 tool_call 或无意义思考，仍浪费 token |
| Specialist 内部 ReAct 最大 3 轮 | ✅ 有限制 | 但 tool_call 结果解析失败时可能进入无意义重试 |
| already_called 集合 | ✅ 防止同一 agent_key 重复调用 | 只按 agent_key 去重，同 agent 不同 query 仍可重复触发 |
| 相同专家不同 query 重复调用 | ❌ 无检测 | 如 valuation_expert 被 wealth_advisor 和 fund_analyst 各自调一次，重复拉数据 |
| 收敛检测 | ❌ 无 | 连续两轮没有新增 agent/新信息时，应直接进入综合阶段 |

### 1.2 Token 浪费点

| 浪费点 | 现状 | 估算 |
|--------|------|------|
| 工具结果重复获取 | 每个 specialist 独立调 query_valuation/search_knowledge | 估值查询可能重复 2-3 次，每次约 500-1000 tokens |
| RAG 只在开始检索一次 | ✅ 有 prebuilt_context 共享 | 但 RAG 结果可能不覆盖后续专家的特定问题 |
| Tool 历史未跨轮压缩 | _compress_prior_tool_messages 压缩同轮，但跨轮 tool 结果累积 | 多轮后 tool 结果占上下文 50%+ |
| 语义摘要默认关闭 | compress_history_semantic 的 LLM 摘要开关默认 false | 长对话历史简单截断，丢失关键信息 |
| 专家结果全量注入 | specialist 返回的 analysis 全文（可能 2000+字）全部给后续专家 | 实际只需要结论+关键数据，长文浪费 token |
| Shared Blackboard 只在 2-3 专家时启用 | 4+ 专家并行，后执行专家看不到前序结论 | 交叉审阅时需要重复解释观点 |

### 1.3 上下文管理问题

| 问题 | 现状 |
|------|------|
| 共享/私密上下文无区分 | RAG、持仓、用户记忆是共享的，每个专家 tool 结果是私密的，但当前全部混在 llm_messages 中 |
| 滑动窗口 | history 有 token 预算压缩，但 orchestration 过程中消息无滑动窗口 |
| 永久记忆 | user_memories 表有写有读，但只在对话开始时注入一次，执行中不更新 |
| 多轮压缩摘要 | conversation_summaries 表存在，但默认不用 LLM 生成高质量摘要 |
| 上下文分层 | 无 tier 概念：系统 prompt → 用户画像 → 当前对话 → RAG → 工具结果 → 专家结论 全混在一起 |

## 二、设计方案：对话流水线 (Conversation Pipeline)

### 2.1 核心思想：从 ReAct Loop 到确定性 Pipeline

将当前 Orchestrator 的隐式 ReAct 循环替换为**显式分阶段流水线**，每个阶段有明确的输入/输出、token 预算和终止条件，消除隐式规划带来的不确定性和循环风险。

```
┌─────────────────────────────────────────────────────────────┐
│                    Conversation Pipeline                      │
│                                                              │
│  Phase 0: 预处理 (Preprocess)                                │
│  ├─ 意图识别 + 澄清检测                                       │
│  ├─ Query 改写/拆分                                           │
│  └─ 复杂度评估 → 决定预算                                     │
│                         ↓                                    │
│  Phase 1: 信息收集 (Information Gathering)                    │
│  ├─ RAG 检索（混合检索，一次到位）                              │
│  ├─ 持仓/估值数据预取（去重）                                  │
│  ├─ 用户记忆加载（跨对话永久记忆 + 本对话摘要）                  │
│  └─ 信息缺口检测 → 决定是否需要补充工具调用                      │
│                         ↓                                    │
│  Phase 2: 计划生成 (Planning)                                 │
│  ├─ 选择专家（基于信息需求匹配）                                │
│  ├─ 确定执行顺序（串行依赖/并行无依赖）                          │
│  ├─ 每个专家的精确 query + 分配的 token 预算                    │
│  └─ 产出 AnalysisPlan JSON（升级3已有，扩展）                   │
│                         ↓                                    │
│  Phase 3: 专家执行 (Execution)                                │
│  ├─ 并行执行无依赖专家                                         │
│  ├─ Shared Blackboard（结论摘要同步）                           │
│  ├─ 收敛检测（新信息阈值）                                     │
│  └─ 每步 checkpoint（中断恢复）                                │
│                         ↓                                    │
│  Phase 4: 综合与质检 (Synthesis)                               │
│  ├─ 交叉审阅（基于结构化结论，而非全文）                         │
│  ├─ 冲突检测 + 仲裁                                           │
│  ├─ Light Validator 质量检查                                   │
│  └─ 生成最终回答                                              │
│                         ↓                                    │
│  Phase 5: 记忆持久化 (Memory Persistence)                      │
│  ├─ 保存分析结论到 analysis_conclusions                        │
│  ├─ 更新对话摘要                                              │
│  ├─ 提取用户偏好 → 更新 user_memories                          │
│  └─ 记录 token 消耗审计                                        │
└─────────────────────────────────────────────────────────────┘
```

### 3.1 死循环/重复执行防护（P0）

#### 3.1.1 收敛检测器

```python
# backend/agent/convergence.py（新建）

class ConvergenceDetector:
    """检测执行是否已收敛（无新信息），防止无效循环。"""
    
    def __init__(self, min_new_info_ratio: float = 0.2):
        self.history = []  # 每轮新信息摘要
        self.min_new_info_ratio = min_new_info_ratio
    
    def record_round(self, new_conclusions: list[str], new_data_refs: list[str]):
        """记录一轮结果。"""
        self.history.append({
            "conclusions": set(new_conclusions),
            "data_refs": set(new_data_refs),
        })
    
    def has_converged(self) -> tuple[bool, str]:
        """判断是否收敛，返回 (是否收敛, 原因)。"""
        if len(self.history) < 2:
            return False, ""
        
        latest = self.history[-1]
        prev = self.history[-2]
        
        new_conclusions = latest["conclusions"] - prev["conclusions"]
        new_data = latest["data_refs"] - prev["data_refs"]
        
        total = len(latest["conclusions"]) + len(latest["data_refs"])
        if total == 0:
            return True, "本轮无新结论和数据引用"
        
        new_ratio = (len(new_conclusions) + len(new_data)) / max(total, 1)
        if new_ratio < self.min_new_info_ratio:
            return True, f"新信息比例 {new_ratio:.0%} < 阈值 {self.min_new_info_ratio:.0%}"
        
        return False, ""
    
    def should_skip_agent(self, agent_key: str, query: str) -> bool:
        """判断是否应该跳过该专家（已调用过相同 query）。"""
        normalized = self._normalize_query(query)
        key = (agent_key, normalized)
        if key in self.called_queries:
            return True
        # 检查相似 query（简单编辑距离）
        for (ak, nq) in self.called_queries:
            if ak == agent_key and self._similarity(normalized, nq) > 0.85:
                return True
        return False
    
    def record_call(self, agent_key: str, query: str):
        normalized = self._normalize_query(query)
        self.called_queries[(agent_key, normalized)] = True
```

#### 3.1.2 工具调用去重缓存

```python
# backend/agent/tool_dedup.py（新建）

class ToolCallCache:
    """工具调用结果缓存 — 同参数调用直接返回缓存结果，避免重复 LLM 调用。"""
    
    def __init__(self, ttl_seconds: int = 300):
        self._cache: dict[str, tuple[float, any]] = {}
        self._ttl = ttl_seconds
    
    def _key(self, tool_name: str, args: dict) -> str:
        # 对参数排序后哈希，避免参数顺序影响缓存命中
        normalized = json.dumps(args, sort_keys=True, ensure_ascii=False)
        return f"{tool_name}:{hashlib.md5(normalized.encode()).hexdigest()[:12]}"
    
    def get(self, tool_name: str, args: dict) -> any:
        key = self._key(tool_name, args)
        if key in self._cache:
            ts, result = self._cache[key]
            if time.time() - ts < self._ttl:
                return result
            del self._cache[key]
        return None
    
    def set(self, tool_name: str, args: dict, result: any):
        key = self._key(tool_name, args)
        self._cache[key] = (time.time(), result)
```

**设计要点**：
- Phase 1 预取常用数据（持仓、关注指数估值）存入 cache
- Specialist 内调工具时先查 cache，命中则直接返回
- TTL 5 分钟，同一轮对话内的重复查询直接复用
- 对 query_valuation / search_knowledge 效果最显著（估计减少 30-50% 重复调用）

        # 1. 硬限制
        if state.tokens_used > state.token_budget.get("max_total", 12000):

### 3.3 上下文分层管理（P1）

#### 3.3.1 上下文分层架构

```
┌─────────────────────────────────────────────┐
│ Layer 0: System Prompts（永久，~500 tokens）  │
│  - 角色定义、输出格式、行为约束                   │
├─────────────────────────────────────────────┤
│ Layer 1: 用户画像（跨对话，~200 tokens）        │
│  - 风险偏好、投资风格、反馈学习结果               │
│  - 永久记忆 user_memories（最近10条）            │
├─────────────────────────────────────────────┤
│ Layer 2: 对话记忆（本对话，动态）               │
│  - 对话摘要（conversation_summaries）          │
│  - 最近 N 轮消息原文（滑动窗口）                 │
│  - 滑动窗口大小：simple=5轮, medium=8轮, complex=12轮│
├─────────────────────────────────────────────┤
│ Layer 3: 共享工作上下文（本次分析，~1500 tokens）│
│  - RAG 检索结果（已压缩）                       │
│  - 持仓摘要                                    │
│  - 预取的估值数据                               │
│  - 宏观/债市温度                               │
├─────────────────────────────────────────────┤
│ Layer 4: 专家私有上下文（每个专家独立）          │
│  - 该专家调用的工具结果                          │
│  - 分配额度：每个专家 2000-3000 tokens          │
├─────────────────────────────────────────────┤
│ Layer 5: 黑板共享（专家间，动态增长）           │
│  - 已完成专家的结构化结论（不是全文分析）          │
│  - 格式: [{agent, conclusion, signals, confidence}]│
└─────────────────────────────────────────────┘
```

#### 3.3.2 上下文构建策略

每个专家看到的上下文 = Layer 0 + Layer 1 + Layer 2 + Layer 3 + Layer 5（黑板结论）+ Layer 4（自己的工具结果）

**关键优化**：
- **专家间不互传全文分析**，只传结构化结论（conclusion + signals + confidence + data_references）
- **Layer 4 不跨专家共享**，减少 token 传递
- **Layer 5 黑板有大小限制**（最多 5 条结论摘要，每条 200 字以内）
- **Layer 2 滑动窗口**只保留最近 N 轮原文，更早的用摘要替代

#### 3.3.4 多轮对话压缩策略增强

```python
# 增强 memory.py 的压缩策略

def build_conversation_context(history: list, conversation_id: int, 
                                max_tokens: int) -> list:
    """
    三层压缩策略：
    1. 最近 K 轮保留原文（K 根据预算动态调整）
    2. 次近消息合并为结构化摘要（LLM 生成，包含意图+结论+关键数据）
    3. 最早消息用一句话总结
    """
    # 策略1：如果不超限，直接返回
    if estimate_tokens_for_messages(history) <= max_tokens:
        return history
    
    # 策略2：三层压缩
    # - recent: 原文保留（占预算60%）
    # - mid: 结构化摘要（占预算30%）
    # - early: 一句话总结（占预算10%）
    
    recent_budget = int(max_tokens * 0.6)
    mid_budget = int(max_tokens * 0.3)
    early_budget = max_tokens - recent_budget - mid_budget
    
    recent, mid, early = _split_history_by_budget(history, recent_budget, mid_budget)
    
    result = []
    if early:
        result.append(_build_early_summary(early, conversation_id))
    if mid:
        result.append(_build_structured_summary(mid, conversation_id))
    result.extend(recent)
    return result
```

### 3.4 共享黑板增强（P1）

将共享黑板从"2-3专家串行时启用"扩展为"所有专家执行时都使用"：

```python
# backend/agent/blackboard.py（新建/增强）

@dataclass
class BlackboardEntry:
    agent_key: str
    agent_name: str
    conclusion: str          # 一句话结论（100字内）
    action_signals: list     # 结构化信号
    confidence: float        # 置信度
    key_data: dict           # 关键数据点 {metric: value}
    timestamp: float

class Blackboard:
    """共享黑板 — 专家执行时实时写入，后续专家可见。"""
    
    def __init__(self, max_entries: int = 6):
        self.entries: list[BlackboardEntry] = []
        self.max_entries = max_entries
    
    def write(self, entry: BlackboardEntry):
        self.entries.append(entry)
        if len(self.entries) > self.max_entries:
            self.entries = self.entries[-self.max_entries:]
    
    def to_context_text(self, max_chars: int = 800) -> str:
        """生成注入专家上下文的黑板摘要。"""
        if not self.entries:
            return ""
        lines = ["## 已完成专家的结论（Blackboard）"]
        total = 0
        for e in self.entries:
            line = f"- [{e.agent_name}] (置信度{e.confidence:.0%}) {e.conclusion}"
            if e.action_signals:
                signals = ", ".join(f"{s.get('type','')} {s.get('target','')}" for s in e.action_signals[:2])
                line += f" → {signals}"
            if total + len(line) > max_chars:
                break
            lines.append(line)
            total += len(line)
        return "\n".join(lines)
    
    def get_key_data(self) -> dict:
        """收集所有专家引用的关键数据，供综合阶段使用。"""
        data = {}
        for e in self.entries:
            data.update(e.key_data)
        return data
```

**与现有 shared_blackboard 的区别**：
- 现有：只在 2-3 专家串行时启用，传递全文前 N 字
- 新方案：所有专家都用，传递结构化结论（每条约 100 字），并行动态更新

### 3.5 简单对话快速路径（P0）

对简单问候/感谢等不需要分析的对话，直接走快速路径：

```python
def _is_simple_chat(query_info: dict) -> bool:
    """判断是否为简单闲聊，走快速路径不调专家。"""
    return query_info.get("intent") in ("greeting", "thanks", "farewell", "capability_query")

def _handle_simple_chat(query: str) -> str:
    """简单对话直接回答，不经过专家分析。"""
    # 用轻量 prompt 直接回答，max_tokens=200
    return _call_llm_simple(
        system="你是一个友好的投资助手。简短回答用户的问候或感谢，不超过2句话。",
        user=query,
        max_tokens=200,
    )
```

**预期效果**：像"你好""谢谢""这个系统能做什么"这类对话，token 消耗从 3000-5000 降到 200-400。

## 六、降级策略

- Pipeline 模式通过 `orchestration_config.pipeline_enabled` 控制，默认 false（先灰度）
- Pipeline 任何阶段失败，立即降级到现有 ReAct 模式
- 降级时记录失败原因，不影响用户体验
- Query 理解失败 → 跳过，直接进入 Plan/ReAct
- 工具缓存失效 → 正常调用工具
