# 分析 Agent 统一记录体系设计稿

> 日期：2026-07-14
> 目标：所有分析 agent 的执行记录统一归集，可按 agent / 时间 / 类型快速定位分析数据与结果，方便排查与优化

## 一、现状问题

### 1.1 记录分散在 3 张表，字段不一致

| 表 | 用途 | 关键字段 |
|----|------|---------|
| `analysis_history` | 指数分析/日报/热点/情报/增强策略 | index_code, agent_id, agent_name, prompt_used, result |
| `portfolio_analysis_records` | 全景/深度/复盘/情景/基金/AI/债券/分散度 | analysis_type, agent_id(常NULL), result_data |
| `health_scores` | 健康分（独立体系） | 无 agent_id |

问题：
- 两张主表字段不统一（`result` vs `result_data`，`agent_name` 仅前者有）
- `portfolio_analysis_records` 的 `agent_id` 在 `bond_recommend`/`portfolio_ai` 写入时为 NULL
- `hotspots.py` 写入 `agent_id=13` 但该 agent 不存在（seed 只有 1-11）
- 无法跨表统一查询"最近所有分析记录"

### 1.2 trace_id 生成但未持久化
6 个路由（diversification/fund_analysis/deep_dive/panorama/health_score/trade_review）生成 trace_id 仅传给 `_call_llm`，不写入任何表，事后无法通过 trace_id 定位分析记录。

### 1.3 质量评估覆盖不足
仅 `index_analysis.py:252` 和 `daily_report.py:388` 接入 `eval_scorer.evaluate_llm_output`，其余 10+ 路由无质量评分。

### 1.4 无统一查看入口
前端记录散落在各业务页面（全景/深度/复盘/基金分析各有独立 records 接口），无统一"分析记录"页面。

## 二、方案设计

### 2.1 核心思路：新增 `agent_analysis_log` 统一索引表

**不迁移历史数据，不动现有表**。新增一张轻量索引表，所有分析路由执行时写入一条统一格式的记录，作为"分析执行的统一入口"。

```sql
CREATE TABLE IF NOT EXISTS agent_analysis_log (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    trace_id TEXT UNIQUE NOT NULL,          -- 统一 trace_id，格式 log_xxxxx
    agent_id INTEGER,                        -- analysis_agents.id（无则 NULL）
    agent_name TEXT,                         -- 冗余存储，便于查询
    analysis_type TEXT NOT NULL,             -- 业务类型（见 2.2 枚举）
    source_table TEXT NOT NULL,              -- 原始记录表（analysis_history/portfolio_analysis_records/health_scores/none）
    source_id INTEGER,                       -- 原始记录 id（可跳转查详情）
    query TEXT,                              -- 分析请求摘要（前300字）
    input_summary TEXT,                      -- 输入摘要（基金代码/指数代码等关键参数）
    status TEXT DEFAULT 'running',           -- running/done/error
    duration_ms INTEGER,                     -- 耗时
    token_usage INTEGER,                     -- token 消耗
    has_eval INTEGER DEFAULT 0,              -- 是否已质量评估（0/1）
    eval_score REAL,                         -- 质量评分（1-10）
    error_msg TEXT,                          -- 错误信息
    created_at TEXT DEFAULT (datetime('now','localtime')),
    completed_at TEXT
);
CREATE INDEX IF NOT EXISTS idx_aal_agent ON agent_analysis_log(agent_id);
CREATE INDEX IF NOT EXISTS idx_aal_type ON agent_analysis_log(analysis_type);
CREATE INDEX IF NOT EXISTS idx_aal_created ON agent_analysis_log(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_aal_trace ON agent_analysis_log(trace_id);
```

**设计要点**：
- `trace_id` 全局唯一，格式 `log_` + 12位 hex，所有路由统一生成
- `source_table` + `source_id` 保留与原表的关联，可跳转查完整 result
- `agent_name` 冗余存储，避免每次 JOIN
- `has_eval` + `eval_score` 支持质量评估闭环
- `input_summary` 存关键参数（如 `基金:014847` / `指数:000300`），便于快速识别

### 2.2 analysis_type 业务类型枚举

统一字符串枚举，覆盖所有分析路由：

| analysis_type | 路由 | agent_id | 说明 |
|---------------|------|----------|------|
| `daily_report` | daily_report.py | 1 | 市场日报 |
| `diversification` | diversification.py | 2 | 分散度分析 |
| `panorama` | panorama.py | 3 | 全景诊断 |
| `deep_dive` | deep_dive.py | 4 | 基金深度分析 |
| `trade_review` | trade_review.py | 5 | 交易复盘 |
| `what_if` | what_if.py | 6 | 情景推演 |
| `fund_analysis` | fund_analysis.py | 6 | 指定基金分析（与 what_if 区分） |
| `hotspots` | hotspots.py | 7 | 热点分析 |
| `bond_recommend` | bond_recommend.py | 8 | 债券推荐 |
| `index_analysis` | index_analysis.py | 9 | 指数深度分析 |
| `market_intel` | market_intel.py | 10 | 市场情报 |
| `enhanced_strategy` | valuation.py | 11 | 增强策略 |
| `portfolio_ai` | portfolio_ai.py | NULL | AI持仓分析 |
| `health_score` | health_score.py | NULL | 健康分 |

**注意**：
- `fund_analysis` 与 `what_if` 原本共用 agent_id=6 和 analysis_type=what_if，此处拆分以区分业务
- `hotspots.py` 原硬编码 agent_id=13 修正为 7（热点分析专家）
- `portfolio_ai` 和 `health_score` 无 analysis_agent，agent_id 存 NULL

### 2.3 写入时机：轻量埋点

每个分析路由在现有流程中增加 2 处埋点（不改原表写入逻辑）：

**a. 创建时（running）**：
```python
from db.agent_analysis_log import create_analysis_log
trace_id = f"log_{uuid.uuid4().hex[:12]}"
log_id = create_analysis_log(
    trace_id=trace_id,
    agent_id=6,
    agent_name="情景推演分析师",
    analysis_type="fund_analysis",
    source_table="portfolio_analysis_records",
    source_id=record_id,  # 原表记录 id
    query=user_content[:300],
    input_summary=f"基金:{fund_code}({fund_name})",
    status="running",
)
```

**b. 完成时（done/error）**：
```python
from db.agent_analysis_log import complete_analysis_log
complete_analysis_log(
    trace_id=trace_id,
    status="done",
    duration_ms=elapsed_ms,
    token_usage=tokens,
)
```

**设计要点**：
- 埋点代码极简（2 行），对现有路由侵入最小
- 原表写入逻辑完全不动，`agent_analysis_log` 是旁路记录
- trace_id 同时传给 `_call_llm`，token_usage 表可关联

### 2.4 质量评估接入

`complete_analysis_log` 完成 status=done 后，异步触发质量评估（受配置开关控制，默认关闭）：

```python
# complete_analysis_log 内部
if status == "done" and get_config_bool("analysis.auto_eval", False):
    asyncio.create_task(_auto_evaluate(trace_id, ...))
```

评估结果回写 `has_eval=1` + `eval_score`。复用现有 `eval_scorer.evaluate_llm_output`，target_type=`analysis_log`，target_id=log_id。

### 2.5 查询 API

新增 3 个统一查询接口：

| API | 说明 |
|-----|------|
| `GET /api/analysis/log/list` | 统一列表，支持 agent_id / analysis_type / status / date 过滤，分页 |
| `GET /api/analysis/log/{log_id}` | 单条详情，含原始记录 result（按 source_table + source_id 查回） |
| `POST /api/analysis/log/{log_id}/evaluate` | 手动触发质量评估 |

`GET /api/analysis/log/{log_id}` 返回结构：
```json
{
  "log": { "trace_id": "...", "agent_name": "...", "analysis_type": "...", "status": "done", "duration_ms": 3200, "token_usage": 1500, "eval_score": 8.5, "input_summary": "基金:014847", "created_at": "..." },
  "source": { "table": "portfolio_analysis_records", "id": 123, "result": "完整结果文本..." }
}
```

## 四、前端页面设计

### 4.1 分析记录列表页

**布局**：顶部筛选栏 + 统计卡片 + 记录列表

**筛选栏**：
- agent 下拉（全部 + 11 个分析 agent）
- 类型下拉（全部 + 14 种 analysis_type）
- 状态下拉（全部 / 完成 / 运行中 / 失败）
- 日期范围

**统计卡片（4个）**：
- 今日总数 / 平均耗时 / 平均 token / 已评估数

**记录列表**（表格）：
| 列 | 内容 |
|----|------|
| 时间 | created_at（MM-DD HH:mm） |
| Agent | agent_name |
| 类型 | analysis_type（中文标签） |
| 输入摘要 | input_summary |
| 耗时 | duration_ms（秒） |
| Token | token_usage |
| 状态 | done/running/error（彩色标签） |
| 评分 | eval_score（星级或数字，未评估显示"-"） |
| 操作 | 查看详情 / 手动评分 |

**行点击**：展开抽屉显示完整 result（从 source_table + source_id 查回）

### 4.2 详情抽屉

- 顶部：trace_id、agent、类型、状态、耗时、token、评分
- 中部：输入摘要 + 完整分析结果（markdown 渲染）
- 底部：质量评估详情（如已评估，显示各维度评分 + 评语）
