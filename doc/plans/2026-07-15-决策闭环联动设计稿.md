## 二、P0 修复：预警→决策候选 自动桥接

### 问题
`alert_scanner` 估值预警/持仓预警仅创建 alert 记录，不自动生成决策候选，用户需手动查看预警后到决策页操作。

### 修复方案
在 `run_periodic_scan()` 的 `scan_valuation_thresholds()` 和 `scan_portfolio_risk()` 执行后，将高优先级预警（估值低估/持仓亏损）自动转为 `recommendation_candidate`。

### 改动文件
- `backend/services/advisor/alert_scanner.py` — 新增 `_alert_to_candidate()` 函数

### 核心逻辑
```python
def _alert_to_candidate(title, content, alert_type, fund_code, fund_name, severity, source_snapshot):
    """将高优先级预警自动转为决策候选。"""
    action_map = {
        "valuation_low": "add",
        "valuation_high": "reduce",
        "concentration_high": "rebalance",
        "loss_large": "add",
    }
    from db.decisions import create_candidate_from_structured_recommendation
    create_candidate_from_structured_recommendation({
        "source_type": "alert",
        "scenario_type": alert_type,
        "action_type": action_map.get(alert_type, "watch"),
        "target_type": "fund",
        "target_code": fund_code,
        "target_name": fund_name,
        "summary": title[:120],
        "rationale": content,
        "confidence": "medium",
        "dedupe_key": f"alert_{alert_type}_{fund_code}",
        "priority": 2 if severity == "warning" else 3,
        "source_snapshot": source_snapshot,
    })
```

### 去重策略
- `dedupe_key` = `alert_{alert_type}_{fund_code}`，同类型同标的14天内不重复创建
- 在 `create_recommendation_candidate` 中检查（已有 `dedupe_key` 字段支持）

### 开关控制
- 新增配置项 `alerts.auto_candidate_enabled`，默认 `false`（遵循项目规范：新LLM相关开关默认关闭）

## 三、P0 修复：smart_add 补仓建议→决策候选

### 问题
`smart_add_planner` 生成补仓计划后，用户需手动去持仓页面操作，没有进入决策流。

### 修复方案
在 `generate_smart_add_plan()` 汇总 plans 后，对触发补仓建议的标的（`pyramid.released_amount > 0` 或 `engine1.monthly_dca > 0`），自动创建 `recommendation_candidate`。

### 改动文件
- `backend/services/advisor/smart_add_planner.py` — 新增 `_plans_to_candidates()` 函数

### 核心逻辑
```python
def _plans_to_candidates(plans, total_assets, pool_total):
    """将补仓建议自动转为决策候选。"""
    from db.decisions import create_candidate_from_structured_recommendation
    for plan in plans:
        pyramid = plan.get("pyramid") or {}
        engine1 = plan.get("engine1") or {}
        released = pyramid.get("released_amount", 0)
        monthly = engine1.get("monthly_dca", 0)
        if released <= 0 and monthly <= 0:
            continue
        # 安全阀未通过的不创建
        if not plan.get("safety", {}).get("can_add", True):
            continue
        amount = released if released > 0 else monthly
        create_candidate_from_structured_recommendation({
            "source_type": "smart_add",
            "action_type": "add",
            "target_type": "fund",
            "target_code": plan["fund_code"],
            "target_name": plan["fund_name"],
            "summary": f"{plan['fund_name']} 智能补仓建议：{amount:.0f}元",
            "rationale": f"当前亏损 {plan.get('profit_rate_pct', 0):.1f}%，估值分位 {plan.get('valuation', {}).get('percentile', 'N/A')}%。月定投 {monthly:.0f}元" + (f"，金字塔释放 {released:.0f}元" if released > 0 else ""),
            "suggested_amount": amount,
            "confidence": "high" if plan.get("confidence_mult", 1) > 1 else "medium",
            "dedupe_key": f"smart_add_{plan['fund_code']}",
            "priority": 1 if plan.get("profit_rate_pct", 0) < -10 else 2,
            "source_snapshot": {
                "profit_rate": plan.get("profit_rate"),
                "valuation": plan.get("valuation"),
                "safety": plan.get("safety"),
            },
        })
```

### 去重策略
- `dedupe_key` = `smart_add_{fund_code}`，每标的每天最多1条候选（通过 `create_recommendation_candidate` 的 `dedupe_key` 去重，需在 `create_recommendation_candidate` 中增加去重逻辑）

### 开关控制
- 复用 `smart_add.enabled` 配置项，无需新增开关

## 四、P1 修复：交易行为分析→决策预检查

### 问题
`trade_pattern` 分析出行为偏差，但 `build_decision_precheck` 没有引用行为分析结果。

### 修复方案
在 `build_decision_precheck` 中增加行为偏差检测：当用户创建 `add` 或 `reduce` 类型决策时，查询 `trade_pattern` 分析结果，如果有同类操作的负面模式，在 warnings 中提示。

### 改动文件
- `backend/db/decisions.py` — `build_decision_precheck()` 函数

### 核心逻辑
```python
# 在 build_decision_precheck 末尾，lesson_rows 之后增加：
if decision.get("action_type") in ("add", "reduce"):
    try:
        from db._conn import _get_conn
        conn = _get_conn()
        # 查询最近30天交易行为分析中的负面模式
        pattern_rows = conn.execute("""
            SELECT behavior_type, behavior_pattern, confidence
            FROM trade_patterns
            WHERE user_id = ? AND is_negative = 1 AND confidence >= 0.6
            ORDER BY analyzed_at DESC LIMIT 3
        """, (user_id,)).fetchall()
        conn.close()
        if pattern_rows:
            patterns = [f"{r['behavior_pattern']}（置信度{r['confidence']:.0%}）" for r in pattern_rows]
            warnings.append(f"行为分析发现以下负面模式，请谨慎决策：" + "；".join(patterns))
    except Exception:
        pass
```

### 开关控制
- 无需新增开关，行为分析结果已存在，仅读取不产生新成本

## 五、P2 修复：事件雷达→决策触发

### 问题
`event_radar` 监控财报/分红/换经理等事件，发现后只通知不触发决策。

### 修复方案
在 `event_radar` 扫描到高影响事件（如重仓基金换经理、重大分红、监管处罚）时，自动创建决策候选，标记为 `action_type: "watch"`，引导用户关注。

### 改动文件
- `backend/routers/market/event_radar.py` — 事件扫描结果处理

### 核心逻辑
```python
# 在事件扫描完成后，对高影响事件自动创建候选
HIGH_IMPACT_TYPES = {"manager_change", "regulatory_penalty", "major_dividend", "delisting_risk"}
for event in high_impact_events:
    if event.get("event_type") in HIGH_IMPACT_TYPES:
        create_candidate_from_structured_recommendation({
            "source_type": "event_radar",
            "action_type": "watch",
            "target_type": "fund",
            "target_code": event["fund_code"],
            "target_name": event.get("fund_name", ""),
            "summary": f"事件预警：{event.get('event_desc', '')}",
            "rationale": f"事件类型：{event.get('event_type')}，影响评估：{event.get('impact', '')}",
            "confidence": "high",
            "dedupe_key": f"event_{event['fund_code']}_{event.get('event_type', '')}",
            "priority": 1,
        })
```

### 开关控制
- 新增配置项 `alerts.event_auto_candidate_enabled`，默认 `false`

## 八、风险与注意事项

1. **去重机制**：`create_recommendation_candidate` 需增加 `dedupe_key` 去重逻辑（14天内同key不重复创建），否则每次扫描都生成新候选
2. **开关默认关闭**：遵循项目规范，新 LLM 相关开关默认 `false`，用户手动开启
3. **候选池容量**：每日最多生成约 10-15 条候选（5个估值预警 + 3个补仓建议 + 2个事件），在 `list_recommendation_candidates` 的 50 条分页内，无需扩容
4. **不破坏现有功能**：所有新增逻辑在已有函数末尾 append，不修改现有流程
