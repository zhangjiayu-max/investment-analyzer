## 二、修复方案

### P0-1 交叉审阅强制反驳

**当前现状**：
- `should_run_cross_review`（orchestrator.py:4752-4762）检查开关 `cross_review_enabled` / `cross_review_min_specialists` / `cross_review_min_severity`
- `run_cross_review_opinion`（multi_agent.py:1236-1464）已实现"disagreements 为空时二次提示强制反驳"
- 但 conv#130 的 metadata 显示 `cross_review_results: []`，说明交叉审阅整体没触发

**修复**：
1. 检查 `should_run_cross_review` 触发条件，降低 `cross_review_min_specialists` 默认值从 3 → 2（2 个专家即可触发交叉审阅）
2. 在 `_stream_handle_no_tool_calls`（orchestrator.py:4730）中，若交叉审阅未触发但专家数 >=2，记录原因到 metadata 便于排查
3. 确保 `run_cross_review_opinion` 的强制反驳逻辑（multi_agent.py:1345-1412）始终生效（开关 `agent.force_devil_advocate_enabled` 默认 true）

### P0-2 仲裁 Agent 重构

**当前现状**：
- 仲裁 prompt 硬编码在 orchestrator.py:1514-1543（违反 AGENTS.md "所有 prompt 通过 analysis_agents 表配置管理"）
- `arbitration.py:34, L117` 把每个专家 `analysis[:120]` 拼接成 reasoning — 这就是 conv#130 看到"原文拼接"的原因

**修复**：
1. `arbitration.py:19-134` 的 `arbitrate_results` 重构：reasoning 从"拼接原文"改为"识别分歧+解决冲突+独立置信度评估"
2. 仲裁 prompt 改为从 `analysis_agents` 表查询 `agent_key='arbitrator'`（若不存在则 seed）
3. 仲裁输出必须包含：`key_conflicts`（关键分歧）/ `resolution`（如何解决）/ `confidence_basis`（置信度依据）

### P0-3 工具调用重复

**当前现状**：
- `tool_dedup.py:43` 白名单已有 `yingmi_latest_quotations`
- `tools/__init__.py:1112-1127` 缓存命中检查代码正常
- 但 conv#130 仍重复调用 2 次

**根因假设**：
- 缓存 key 生成可能不稳定（参数排序问题）
- 或两次调用的 arguments 略有差异导致 cache miss

**修复**：
1. 检查 `tool_dedup.py:74-83` 的 key 生成逻辑，确保参数排序稳定
2. 在 `tools/__init__.py:1112` 缓存命中检查前增加日志，记录 cache miss 原因
3. 修复后 conv#130 重试验证

### P1-1 数据冲突标注

**修复**：
- 在 `analysis_agents` 表的专家 prompt 中增加引导："若多个数据源返回的同一指标值不一致，必须明确说明差异原因（如数据时间点不同、数据源口径不同），并标注采用哪个值及理由"
- 影响专家：估值分析师、市场分析师、风险管理师

### P1-2 query_policy_news 降级

**修复**：
- 在 `services/market/policy_news.py:180-217` 的 `get_policy_news` 函数中，当三源全失败返回空时，自动调用 `web_search` 工具查政策新闻
- 实现：import web_search 工具函数，用 `"政策 利好 {query}"` 作为搜索词
- 返回结构保持一致，但标注 `data_source: "web_search_fallback"`

### P1-3 行为金融学专家路由

**修复**：
- `router.py:100-101` 关键词列表扩展：增加 "暴涨/暴跌/大涨/大跌/爆炒/追涨/杀跌/量化/资金注入/主力/游资/炒作"
- `router_config.yaml:65-67` 同步扩展 patterns
- 命中这些关键词时强制路由 `behavioral_advisor` 参与协作

### P2-1 耗时优化（轻量）

**修复**：
- 交叉审阅 max_workers 从 `min(len(original_specialists), 3)` 提升到 `min(len(original_specialists), 5)`（项目已默认专家最大 5 个）
- 仲裁 LLM max_tokens 从 800 调整为 1200（避免截断导致二次调用）
- 不改串行黑板架构（保留最后位专家质疑角色）
