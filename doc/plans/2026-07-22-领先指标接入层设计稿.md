# 领先指标接入层设计稿

> **版本**：v1.0  
> **日期**：2026-07-22  
> **背景**：用户反馈"只看新闻无法预估未来利好哪些行业"。新闻是滞后信号（事件发生→媒体报道→扩散，股价已反映），需要接入**领先指标**（政策草案、资本开支、产业协会月报、海关数据等）才能将"猜未来"变为"算概率"。  
> **定位**：本设计稿是对 2026-07-22-三大核心功能深度优化设计稿的**补充扩展**，聚焦"数据源层"升级，不改变现有事件雷达/机会雷达的核心评分逻辑。

---

## 一、问题诊断

### 1.1 当前系统的信号层级

| 信号类型 | 领先性 | 当前系统覆盖 | 问题 |
|---------|-------|------------|------|
| 新闻 | 滞后（已发生） | ✅ 三源融合（盈米/akshare/东财） | 主力信号，但本质是"已发生" |
| 研报 | 中性（已有共识） | ❌ 未接入 | — |
| 龙头公司资本开支 | 领先 3-12 月 | ❌ 未接入 | 产业资本动向比研报准 |
| 政策草案/征求意见稿 | 领先 6-24 月 | ❌ 仅靠新闻二手传播 | 政策落地前有征求意见期 |
| 产业协会月报 | 领先 1-3 月 | ❌ 未接入 | 中汽协/光伏协会/TrendForce |
| 海关进出口数据 | 领先 1-3 月 | ❌ 未接入 | HS编码细分看芯片/设备出口 |
| PMI 分项 | 领先 1-3 月 | ❌ 未接入 | 新订单/新出口订单/价格 |
| 专利/论文 | 领先 1-3 年 | ❌ 未接入 | 技术方向预判 |

### 1.2 核心痛点

1. **无统一数据源抽象层**：3 处分散调用（盈米 MCP / 东财 MCP / akshare），每个模块直接 import，无统一接口
2. **配置项未驱动代码**：`alerts.event_radar_news_sources` 配置存在但未被读取，3 个数据源被硬编码调用
3. **akshare 调用无统一超时保护**：仅 event_radar 有 `_call_akshare_with_timeout`，opportunity_engine 直接调用存在卡死风险（项目记忆中 conv#133 曾因 akshare zombie 线程卡死 6 分钟）
4. **板块映射两套并存**：event_radar 有 18 板块完整体系，opportunity_engine 仅 5 主题，通过 `theme_rules.sector` 字段对齐
5. **回测不区分信号来源**：L3 基准化后用沪深300超额收益 ≥2% 判定 hit，但无法区分"新闻驱动的机会"和"领先指标驱动的机会"谁更准

### 1.3 设计目标

将"领先指标"作为**第一类公民**接入系统，实现：
- 统一数据源抽象层（Provider 接口），新数据源可插拔
- 领先性分级（强/中/弱领先），融入事件雷达和机会雷达评分
- 配置真正驱动数据源选择
- akshare 统一超时保护
- 回测按信号来源分组统计命中率，反哺评分权重

## 二、领先指标分类与数据源

### 2.1 指标分级体系

按**领先性**和**可获取性**分三级：

#### L1 - 强领先（领先 6-24 月，优先接入）

| 指标 | 数据源 | akshare 接口 | 更新频率 | 覆盖板块 |
|------|--------|-------------|---------|---------|
| 政策征求意见稿 | 发改委/工信部官网 | ❌ 无 akshare 接口，需 web 抓取 | 不定期 | 全板块 |
| 五年规划中期评估 | 国务院官网 | ❌ 同上 | 2.5 年一次 | 全板块 |
| 龙头公司资本开支公告 | 巨潮资讯网（巨潮公告） | `ak.stock_notice_report` 间接 | 季度 | 半导体/新能源/AI |
| 产业资本增减持 | 东财 | `ak.stock_hold_management_detail` | 不定期 | 全板块 |

**接入策略**：
- 政策类：web 抓取发改委/工信部"政策征求意见"栏目，关键词过滤（芯片/新能源/AI/储能等）
- 资本开支：`ak.stock_notice_report` 拉取"重大项目/投资协议/扩产公告"，LLM 提取金额和方向
- 增减持：`ak.stock_hold_management_detail`，统计近 30 日净增持行业排名

#### L2 - 中领先（领先 1-3 月，第二批接入）

| 指标 | 数据源 | akshare 接口 | 更新频率 | 覆盖板块 |
|------|--------|-------------|---------|---------|
| 产业协会月报 | 中汽协/光伏协会/TrendForce | `ak.stock_car_agent` / `ak.energy_oil_hist` 部分 | 月度 | 汽车/光伏/半导体 |
| 海关进出口数据 | 海关总署 | `ak.china_imports_export` | 月度 | 全板块（HS编码细分） |
| PMI 分项 | 国家统计局 | `ak.macro_china_pmi` | 月度 | 全板块 |
| 用电量 | 国家能源局 | `ak.energy_get_country_report` | 月度 | 全板块 |

**接入策略**：
- 产业协会：直接调 akshare 月度数据，与历史同期对比计算同比/环比
- 海关：`ak.china_imports_export` 拉取 HS 编码细分，关键词映射到板块（如"8542 集成电路"→半导体）
- PMI：`ak.macro_china_pmi` 取分项（新订单/新出口订单/价格），>55 为强扩张

#### L3 - 弱领先（领先 1-3 年，探索性，暂不接入）

| 指标 | 数据源 | 接入难度 |
|------|--------|---------|
| 顶会论文 | NeurIPS/CVPR | 高，需 NLP 处理 |
| 专利申请 | 国家知识产权局 | 高，需专利分类映射 |
| 技术白皮书 | 厂商研究院 | 中，需人工筛选 |

**决策**：L3 暂不接入，作为未来探索方向。

## 三、统一数据源抽象层设计

### 3.1 Provider 接口设计

新增 `backend/services/market/leading_indicators/` 目录，定义统一接口：

```python
# backend/services/market/leading_indicators/base.py
"""领先指标数据源统一抽象层。"""
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from datetime import datetime
from typing import Optional


@dataclass
class LeadingSignal:
    """领先指标信号统一数据结构。"""
    signal_type: str          # policy_draft / capex_announcement / insider_trading / customs_data / pmi_subitem
    leading_level: str        # strong (6-24月) / medium (1-3月) / weak (1-3年)
    title: str
    summary: str
    source_url: str
    publish_date: str         # YYYY-MM-DD
    affected_sectors: list[str] = field(default_factory=list)   # 对齐 event_radar.SECTOR_TO_INDEX
    affected_themes: list[str] = field(default_factory=list)
    direction: str = "neutral"  # positive / negative / neutral
    confidence: float = 0.5     # 0-1
    raw_data: dict = field(default_factory=dict)   # 原始数据，供 LLM 二次分析
    # 量化字段（可选，L2 指标填充）
    metric_value: Optional[float] = None    # 如 PMI=56.3、出口同比+12.5%
    metric_unit: str = ""                   # % / 亿元 / 万辆
    metric_yoy: Optional[float] = None       # 同比
    metric_mom: Optional[float] = None      # 环比


class LeadingIndicatorProvider(ABC):
    """领先指标数据源统一接口。所有数据源实现此接口。"""

    @property
    @abstractmethod
    def provider_name(self) -> str:
        """数据源标识（如 policy_draft / capex / insider_trading）。"""

    @property
    @abstractmethod
    def leading_level(self) -> str:
        """领先级别：strong / medium / weak。"""

    @abstractmethod
    def fetch(self, lookback_days: int = 7) -> list[LeadingSignal]:
        """拉取近 N 天的领先指标信号。子类实现具体抓取逻辑。"""

    def map_to_sectors(self, text: str) -> list[str]:
        """文本 → 板块映射。复用 event_radar._match_sectors 逻辑。"""
        from services.market.event_radar import _match_sectors
        return _match_sectors(text)
```

### 3.3 聚合层（aggregator.py）

```python
# backend/services/market/leading_indicators/aggregator.py
"""领先指标聚合层 — 多 Provider 合并 + 去重 + 配置驱动。"""
import logging
from datetime import datetime, timedelta
from services.market.leading_indicators.base import LeadingSignal

logger = logging.getLogger(__name__)

# Provider 注册表（provider_name → Provider 实例）
_PROVIDER_REGISTRY: dict[str, "LeadingIndicatorProvider"] = {}

def register_provider(provider: "LeadingIndicatorProvider"):
    _PROVIDER_REGISTRY[provider.provider_name] = provider

def _get_enabled_providers() -> list["LeadingIndicatorProvider"]:
    """根据配置启用的 Provider 列表。真正驱动数据源选择。"""
    from db import get_config
    enabled_str = get_config("alerts.leading_indicator_sources", "policy_draft,capex,insider_trading")
    enabled_set = {s.strip() for s in enabled_str.split(",") if s.strip()}
    return [p for name, p in _PROVIDER_REGISTRY.items() if name in enabled_set]

def collect_leading_signals(lookback_days: int = 7) -> list[LeadingSignal]:
    """聚合所有启用的 Provider，返回去重后的领先指标信号列表。"""
    providers = _get_enabled_providers()
    all_signals: list[LeadingSignal] = []
    seen_keys: set[str] = set()

    for provider in providers:
        try:
            signals = provider.fetch(lookback_days=lookback_days)
            for sig in signals:
                # 去重 key = provider_name + title + publish_date
                dedupe_key = f"{provider.provider_name}:{sig.title}:{sig.publish_date}"
                if dedupe_key in seen_keys:
                    continue
                seen_keys.add(dedupe_key)
                all_signals.append(sig)
        except Exception as e:
            logger.warning(f"[leading_indicators] {provider.provider_name} 抓取失败: {e}")

    # 按发布日期降序
    all_signals.sort(key=lambda s: s.publish_date, reverse=True)
    logger.info(f"[leading_indicators] 聚合 {len(providers)} 个 Provider，共 {len(all_signals)} 条信号")
    return all_signals
```

### 3.4 akshare 统一超时保护

提取 event_radar 的 `_call_akshare_with_timeout` 到公共位置，所有 Provider 复用：

```python
# backend/services/market/leading_indicators/akshare_utils.py
"""akshare 统一超时保护 — 防止 zombie 线程卡死。"""
import logging
from concurrent.futures import ThreadPoolExecutor, TimeoutError as FuturesTimeoutError

logger = logging.getLogger(__name__)
_AKSHARE_TIMEOUT = 15  # 秒，与 event_radar 一致

def call_akshare_with_timeout(fn, *args, timeout: int = _AKSHARE_TIMEOUT, **kwargs):
    """在 ThreadPoolExecutor 中调用 akshare 函数，超时返回 None。"""
    with ThreadPoolExecutor(max_workers=1) as executor:
        future = executor.submit(fn, *args, **kwargs)
        try:
            return future.result(timeout=timeout)
        except FuturesTimeoutError:
            logger.warning(f"[akshare] 超时 {timeout}s: {fn.__name__}")
            return None
        except Exception as e:
            logger.warning(f"[akshare] 调用失败 {fn.__name__}: {e}")
            return None
```

**后续可让 event_radar 和 opportunity_engine 也复用此工具**，消除重复代码。

### 4.1 事件雷达集成

#### 4.1.1 新增信号类型

- `policy_draft` — 政策征求意见稿（L1 强领先）
- `capex_announcement` — 资本开支公告（L1 强领先）
- `insider_trading` — 产业资本增减持（L1 强领先）
- `customs_data` — 海关进出口数据（L2 中领先）
- `pmi_subitem` — PMI 分项（L2 中领先）

#### 4.1.2 扫描流程增强

`event_radar.scan_forward_events()` 主流程新增"领先指标采集"分支：

```
原流程：_collect_news() → LLM 提取事件 → 校准 → 写表
新流程：
  ├─ _collect_news() → LLM 提取事件（保持不变，滞后信号）
  └─ _collect_leading_signals() → 规则映射事件（新增，领先信号）
       ↓
  合并去重（标题相似度 + 日期差）
       ↓
  校准时对领先信号加权（leading_level=strong 置信度 +0.1）
```

#### 4.1.3 新增方法

```python
# event_radar.py 新增
def _collect_leading_signals(trace_id: str = "") -> list[dict]:
    """采集领先指标信号，转换为 market_events 格式。"""
    from services.market.leading_indicators import collect_leading_signals
    signals = collect_leading_signals(lookback_days=7)
    events = []
    for sig in signals:
        events.append({
            "title": sig.title,
            "summary": sig.summary,
            "event_type": sig.signal_type,
            "direction": sig.direction,
            "confidence": min(sig.confidence + _leading_bonus(sig.leading_level), 0.95),
            "expected_date": datetime.now().strftime("%Y-%m-%d"),  # 领先指标"预期日期"=今天
            "affected_sectors": sig.affected_sectors,
            "affected_themes": sig.affected_themes,
            "sources": [{"title": sig.title, "url": sig.source_url, "publish_date": sig.publish_date}],
            "time_frame": "short" if sig.leading_level == "medium" else "medium",
            "evidence": f"领先指标({sig.leading_level}): {sig.metric_value or ''} {sig.metric_unit}",
        })
    return events

def _leading_bonus(level: str) -> float:
    """领先级别置信度加成。"""
    return {"strong": 0.1, "medium": 0.05, "weak": 0.0}.get(level, 0.0)
```

### 4.2 机会雷达集成

#### 4.2.1 评分维度增强

`opportunity_engine._score_theme()` 新增第 10 维度"领先指标得分"：

| 条件 | 得分 |
|------|------|
| 近 7 天有 strong 领先指标命中该主题 | +15 |
| 近 7 天有 medium 领先指标命中 | +8 |
| 近 7 天有领先指标但方向 negative | -10 |
| 无领先指标命中 | 0 |

#### 4.2.2 新增评分方法

```python
# opportunity_engine.py 新增
def _get_leading_indicator_score(theme_rule: dict, trade_date: str) -> tuple[int, str]:
    """计算领先指标得分。返回 (score, reason)。"""
    from db import get_config
    if get_config("opportunity.leading_indicator_score_enabled", "false") != "true":
        return 0, ""
    
    from db.market_events import list_market_events_by_type_and_date
    theme = theme_rule.get("theme", "")
    sectors = [theme_rule.get("sector", "")] if theme_rule.get("sector") else []
    
    # 查近 7 天领先指标事件
    leading_types = ("policy_draft", "capex_announcement", "insider_trading", "customs_data", "pmi_subitem")
    events = list_market_events_by_type_and_date(
        event_types=leading_types,
        days=7,
        end_date=trade_date,
    )
    
    matched = []
    for ev in events:
        ev_sectors = json.loads(ev.get("affected_sectors", "[]"))
        ev_themes = json.loads(ev.get("affected_themes", "[]"))
        if theme in ev_themes or set(sectors) & set(ev_sectors):
            matched.append(ev)
    
    if not matched:
        return 0, ""
    
    # 按领先级别评分
    strong_count = sum(1 for e in matched if e.get("event_type") in ("policy_draft", "capex_announcement", "insider_trading"))
    medium_count = sum(1 for e in matched if e.get("event_type") in ("customs_data", "pmi_subitem"))
    negative_count = sum(1 for e in matched if e.get("direction") == "negative")
    
    score = 0
    if strong_count > 0:
        score += 15
    if medium_count > 0:
        score += 8
    if negative_count > len(matched) / 2:  # 过半利空
        score = -10
    
    reason = f"领先指标命中: strong={strong_count}, medium={medium_count}, negative={negative_count}"
    return score, reason
```

```python
# db/opportunities.py 新增
def get_backtest_stats_by_source() -> dict:
    """按 signal_source 分组统计命中率。"""
    conn = _get_conn()
    rows = conn.execute("""
        SELECT signal_source,
               COUNT(*) as total,
               SUM(CASE WHEN hit=1 THEN 1 ELSE 0 END) as hits
        FROM theme_opportunity_backtests
        WHERE reviewed_at IS NOT NULL
        GROUP BY signal_source
    """).fetchall()
    conn.close()
    return {r["signal_source"]: {"total": r["total"], "hits": r["hits"], 
            "hit_rate": r["hits"]/r["total"] if r["total"] else 0} for r in rows}
```

## 八、风险与约束

### 8.1 技术风险

1. **akshare 接口稳定性**：`ak.stock_notice_report` 可能因东财反爬失效 → 超时保护 + 降级跳过
2. **政策草案 web 抓取**：发改委/工信部页面结构可能变更 → 选择性抓取 + 异常容错
3. **板块映射准确性**：领先指标文本可能不包含明确板块关键词 → LLM 辅助提取（但需开关控制）

### 8.2 成本控制

- 领先指标抓取为**纯数据采集**，无 LLM 调用（符合硬约束）
- LLM 仅在"LLM 辅助板块映射"时调用，且需独立开关 `alerts.leading_indicator_llm_mapping_enabled`（默认 false）
- 回测命中率反哺为纯规则，无 LLM

### 8.3 兼容性

- 领先指标接入层为**新增模块**，不修改 event_radar/opportunity_engine 现有核心逻辑
- 仅在 `scan_forward_events()` 末尾新增 `_collect_leading_signals()` 调用，开关控制
- 回测表新增字段用 `ALTER TABLE`，向后兼容
