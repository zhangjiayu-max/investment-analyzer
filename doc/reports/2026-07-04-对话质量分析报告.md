## 三、多智能体流程健康度（核心问题）

从 10 条助手消息的 metadata 提取流程数据：

| 流程环节 | 触发率 | 状态 |
|----------|:---:|:---:|
| 专家协作（阶段一） | 100% | 正常 |
| 工具调用 | 90% | 正常 |
| RAG 知识检索 | 0% | 异常 |
| 交叉审阅（阶段二） | **0%** | 严重异常 |
| 仲裁阶段（阶段三） | **0%** | 严重异常 |

### 3.1 交叉审阅为何 0% 触发

日志明确记录了根因（`backend/logs/backend.log.2026-07-*`）：

```
2026-07-02 09:30:32 交叉审阅: 预判=False → 跳过
2026-07-02 18:41:33 交叉审阅: 预判=False → 跳过
2026-07-03 08:32:43 交叉审阅: 预判=False → 跳过
2026-07-03 08:47:52 交叉审阅: 预判=False → 跳过
```

**所有对话的 `predicted_need_cross_review` 都被需求澄清 LLM 预判为 False**，导致 `should_run_cross_review()` 直接返回 False（`orchestrator.py:759-770`）。

当前逻辑（`orchestrator.py:765`）：
- `predicted_need_cross_review=True` + `conflicts.detected=True` → 执行
- `predicted_need_cross_review=True` + 无冲突 → 跳过
- `predicted_need_cross_review=False` → **直接跳过**

问题在于：**预判一票否决过于激进**。即使路由结果 `needs_arbitration=True`（日志显示多个 complex 对话都是 True），只要预判 False，交叉审阅就不执行。

### 3.2 仲裁为何 0% 触发

仲裁依赖交叉审阅的输出。交叉审阅跳过 → 无冲突报告 → 仲裁无输入 → 不触发。

### 3.3 "伪仲裁"现象

回答内容里出现了"仲裁法官""反方审查员""六大专家"等角色，但这些是通过 `consult_arbitrator`、`consult_counter_argument` 工具作为**普通专家并行调用**实现的，而非三阶段流程的真正仲裁。

这意味着：
- 仲裁法官和反方审查员变成了"第 N 个专家"，与估值/风险/配置专家**并行发言**
- 失去了"交叉审阅发现冲突 → 仲裁裁决冲突"的**递进纠错**价值
- 各专家观点可能互相矛盾，但没有真正的裁决环节统一

### 3.4 RAG 检索 0%

10 个对话没有任何一个触发 RAG 知识库检索。用户问"7月政策利好""金价暴涨原因""AI基金暴跌原因"等问题，本应检索知识库中的文章/研报，但全部走专家直答。

可能原因：需求澄清路由未将这些问题判定为需要 `search_knowledge` 工具，或专家 prompt 里没有强制"先检索后回答"。

## 五、提升建议（按优先级）

### P0：修复交叉审阅预判一票否决（影响：质量纠错能力）

**问题**：`predicted_need_cross_review=False` 直接跳过交叉审阅，过于激进。

**方案**：修改 `should_run_cross_review()`（`orchestrator.py:741`），当路由 `needs_arbitration=True`（即 complexity=complex）时，**即使预判 False 也应基于实际冲突检测结果决定**，而非被预判一票否决。

建议逻辑：
```
if complexity == 'complex' and needs_arbitration:
    # complex 场景强制做冲突检测，由检测结果决定是否审阅
    run_conflict_detection = True
    if conflicts.detected:  # 实际检测到冲突才审阅
        return True
    return False
elif predicted_need_cross_review and conflicts.detected:
    return True
else:
    return False
```

### P0：恢复质量评估自动化（影响：质量可追踪性）

**问题**：6/28 后 0 条评估记录。

**方案**：将评估从"用户手动点击"改为"**助手回答完成后自动触发轻量评估**"。在 `orchestrator.py` 最终答案生成后，异步触发 `auto_score` 评估（已有 `conversation_evaluations` 表和评估逻辑），不阻塞用户看到答案。用户仍可手动触发 LLM 深度评估。

### P1：激活 RAG 检索（影响：回答可信度）

**问题**：0% RAG 触发率，专家"凭空回答"政策/行情问题。

**方案**：在需求澄清路由中，对"政策""原因""分析""新闻""文章"类问题强制注入 `search_knowledge` 工具到相关专家的 tools 列表；或在专家 prompt 里加"回答前先检索知识库"约束。

### P1：收敛回答 emoji（影响：专业度）

**问题**：平均每条回答 22.2 个 emoji。

**方案**：在专家 system_prompt 里加约束："输出使用 Markdown 表格和标题结构，**禁止使用 emoji 作为图标或装饰**，专业术语直接用文字"。配合当前 UI 美化的 emoji 清理，内外一致。

### P2：真正的三阶段仲裁（影响：深度分析质量）

**问题**：仲裁被降级为并行专家，失去递进纠错价值。

**方案**：修复 P0 后，确保 complex 对话走完"并行专家 → 交叉审阅（发现冲突）→ 仲裁（裁决冲突）"完整链路。可考虑对 complex 对话**强制执行交叉审阅**（跳过预判），用实际冲突检测过滤。

### P2：评估数据驱动 prompt 迭代（影响：持续优化闭环）

**方案**：恢复评估自动化后，建立"评估低分 → 自动归因（哪个维度扣分）→ 反馈到对应专家 prompt"的闭环。利用已有的 `eval_suggestions` 表和 `llm_evaluations` 维度分解。
