# 仲裁法官裁决质量增强设计稿

> 基于 AI 对话 ID 84 的实证分析，定位仲裁法官在"加仓建议"场景下的三类质量缺陷，给出可落地的 prompt + 代码增强方案。

## 一、问题背景

### 用户场景
用户提问："结合近期市场情绪走向，看下有哪些周一可以加减仓的基金。目前零钱较多，还未有个好的收益。"

### 用户反馈的三个问题
1. **加仓金额依据不清**：仲裁裁决给出"本次用 6.5 万"，但明细加起来只有 4.5 万，且没说明为什么是 6.5 万而非 5 万或 8 万
2. **白酒 60000 限制与现状矛盾**：硬性约束"单只基金仓位上限 60000 元"，但用户当前白酒成本 75000 元（已超上限），裁决却建议"继续加仓白酒 1 万"
3. **其他信息质量不佳**：恒生科技估值数据冲突被忽略、卖出债券建议被否决但理由弱

### 实证数据（DB 查询）
| 项目 | 估值分析师 | 风险管理师 | 资产配置师 | 仲裁裁决 |
|---|---|---|---|---|
| 建议金额 | 7.8-9.1 万 | 8 万 | 6.5 万 | 4.5 万（说成 6.5 万） |
| 恒生科技百分位 | 7%（低估） | 64%（合理） | - | 64%（采纳） |
| 卖出债券建议 | - | - | 卖出 20 万 | 拒绝（理由弱） |

**白酒持仓现状**：成本 75000 元，当前市值 48635 元（亏 -35.15%）

## 二、根因分析

### 缺陷 1：run_arbitration 未注入用户持仓数据

[backend/agent/multi_agent.py:879-947](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/multi_agent.py) 中 `run_arbitration` 函数签名：

```python
def run_arbitration(query, specialist_results, rag_context="", trace_id="") -> dict:
```

**只接收 query + specialist_results + rag_context，没有用户持仓数据**。仲裁法官无法看到：
- 用户当前白酒成本 75000 元（已超 60000 上限）
- 用户当前白酒市值 48635 元
- 用户现金 13 万的来源（是零钱还是已规划资金）

**后果**：仲裁法官定的"上限 60000"是凭空设的，无法与现状对比；建议"加仓白酒 1 万"时不知道用户已经超上限。

### 缺陷 2：仲裁 prompt 缺少"金额自洽性"约束

[agents 表 id=5772 的 system_prompt](file:///Users/xiaoyuer/projects/investment-analyzer/backend/db/agents.py) 中：
- 有"输出格式"约束（专家可信度表格、分歧点、裁决结果）
- 有"交易记录硬约束"（禁止对已卖出基金建议卖出）
- **但没有任何关于"金额推导"的约束**

仲裁法官可以说"本次用 6.5 万"但明细加起来 4.5 万，prompt 没有强制要求"金额加总 = 总额"。

### 缺陷 3：仲裁 prompt"数据冲突"定义过窄

prompt 第三步说：
> 当专家引用**同一指标但数值不同**：
> 1. 优先采用最近 7 天内发布的数据
> 2. 优先采用一手来源

**问题**：
- 估值分析师说恒生科技 7%、风险管理师说 64%，仲裁法官在"数据冲突"部分写"**无实质性数据冲突**"
- 原因：仲裁法官可能认为 7% 和 64% 是"不同维度"（实际是同一指标），或认为差异"不实质性"
- prompt 没有给出"差异多少算冲突"的量化标准（如 >2 倍即冲突）

### 缺陷 4：仲裁 prompt 缺少"专家建议被否决时强制解释"约束

资产配置师建议"卖出 20 万博时恒乐债券"（止盈 + 降低集中度），仲裁法官否决但理由只有"提供组合稳定性，调整为时尚早"。

prompt 没有要求：
- 否决专家建议时必须给出"数据理由"（不能用"稳定性"这种主观词）
- 必须给出"替代方案"（不卖债券→那如何降集中度？）

### 缺陷 5：仲裁 prompt 缺少"持仓约束与现状一致性"检查

仲裁法官引用任何持仓约束（如"上限 X 万"）前，没有要求：
- 必须查询用户当前持仓
- 若已超上限，必须明确说"不再加仓"而非"加 X 万"
- 必须区分"市值上限"和"成本上限"

## 三、修复方案

采用**方案 B：prompt 强化 + 持仓数据注入**。理由：
- 仅改 prompt 无法解决"仲裁法官看不到持仓"的根本问题
- 注入持仓数据后，配合 prompt 约束，能让 LLM 自主避免上述问题
- 不引入后处理校验（避免二次 LLM 调用增加成本）

### 改动点 1：run_arbitration 注入用户持仓摘要

**文件**：[backend/agent/multi_agent.py:879](file:///Users/xiaoyuer/projects/investment-analyzer/backend/agent/multi_agent.py)

**改动**：
- `run_arbitration` 新增参数 `portfolio_summary: str = ""`
- 在 user_content 中追加"## 用户当前持仓摘要"段
- 调用方（orchestrator.py 的 `_run_arbitration_stream`）传入持仓摘要

**持仓摘要内容**（从 portfolio_holdings 表聚合，控制 < 800 字符）：
```
- 总资产：约 50 万
- 现金：约 13 万
- 重仓基金（按市值排序）：
  1. 招商中证白酒指数(LOF)A：成本 75000 元，市值 48635 元，盈亏 -35.15%
  2. 博时恒乐债券 A+C：合计市值约 30 万
  3. ...
```

### 改动点 3：仲裁法官 prompt 增强

**文件**：[agents 表 id=5772 的 system_prompt](file:///Users/xiaoyuer/projects/investment-analyzer/backend/db/agents.py) 字段

**新增 4 段约束**：

#### 3.1 持仓现状对照（新增段，位于"裁决流程"前）

```markdown
## 持仓现状对照（必须遵守）

输入会包含"## 用户当前持仓摘要"。在给出任何加仓/减仓建议前，必须：
1. 核对建议标的的当前持仓（成本、市值、盈亏）
2. 若标的已超"建议上限"（按成本计），明确说"已超上限，不再加仓"
3. 若标的接近上限（>80%），明确说"接近上限，最多加 X 元"
4. 区分"市值"和"成本"：上限约束默认按"成本"计（用户实际投入）
```

#### 3.2 金额自洽性约束（新增段，位于"输出格式约束"前）

```markdown
## 金额自洽性约束（必须遵守）

给出加仓/减仓建议时：
1. 必须给出"金额推导公式"：现金总额 × 配置比例 × 标的权重 = 加仓金额
2. 明细加总必须 = 总额（差额需在 ±100 元内，否则视为错误）
3. 必须说明每个标的金额的依据（为什么是 1 万而非 2 万）
4. 保留现金必须单独列出，不算入加仓总额
```

#### 3.3 数据冲突量化标准（修改第三步）

```markdown
### 第三步：数据冲突处理

**冲突判定标准**（量化）：
- 同一指标数值相差 > 2 倍 → 必须显式列为冲突
- 同一指标数值相差 > 5 倍 → 必须标注"严重冲突，需用户核实"
- 同一指标解读方向相反（一正一负）→ 必须显式列为冲突

**禁止**：在"数据冲突"部分写"无实质性数据冲突"，除非所有指标差异都在 2 倍内。
```

#### 3.4 专家建议否决约束（新增段）

```markdown
## 专家建议否决约束（必须遵守）

当否决某专家的具体建议时（如"卖出 X 基金""加仓 Y 标的"）：
1. 必须给出"数据理由"（不能用"稳定性""调整为时尚早"等主观词）
2. 必须给出"替代方案"（不卖债券→那如何降集中度？不加白酒→那加什么？）
3. 必须说明"否决后该专家指出的风险如何缓解"
```

## 七、风险与回退

### 风险
1. **持仓摘要过长**：用 800 字符上限控制，超长时只保留前 5 大持仓
2. **prompt 过载**：4 段约束总计约 800 字符，相对原 2383 字符增加 34%，可接受
3. **LLM 不遵守约束**：靠 prompt 约束非 100% 可靠，但比无约束强；后续可考虑后处理校验

### 回退方案
- prompt 修改通过 SQL UPDATE，可随时回退（保留原 prompt 备份）
- run_arbitration 新增参数有默认值，向后兼容
- 调用方改动小，可快速回退
