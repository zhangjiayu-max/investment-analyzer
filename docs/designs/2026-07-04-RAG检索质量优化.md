## 一、问题分析

### 1.1 现状实证（基于 rag_logs 近期数据）

| 指标 | 实测值 | 说明 |
|------|--------|------|
| jieba 模块状态 | 未安装 | `ModuleNotFoundError: No module named 'jieba'` |
| 中文 query 分词策略 | 单字分割 | `_tokenize` 在 jieba 不可用时回退到 `" ".join(text)` |
| 样本相关率 | 1/8 = 12.5% | 8 条最近 RAG 命中中仅 1 条与对话问题相关 |
| content_types 字段空值率 | 93% | rag_logs 中 content_types 为 null/空 |
| URL 类 query 走 RAG | 频繁 | 抓文章的 URL 也会被当成检索 query |
| freshness_filtered 平均 | 5.7 条/次 | 时效性过滤有效但前置检索噪声大 |

### 1.2 根因链

```
jieba 未安装
   ↓
_tokenize 回退到单字分割（"白酒估值" → "白 酒 估 值"）
   ↓
入库时 title/body 都按单字存入 FTS5
查询时也按单字 MATCH
   ↓
FTS5 rank 排序失效（每个单字 BM25 权重接近）
   ↓
长查询 OR 降级返回大量弱相关结果
   ↓
RRF 融合后被 score_threshold=0.018 漏过
   ↓
最终注入 LLM 的上下文与问题不相关，反而干扰回答
```

### 1.3 现有代码定位

| 函数 | 行号 | 问题 |
|------|------|------|
| `_get_jieba` | rag.py:432 | `_jieba = None` → 首次 import 失败后永久返回 `False`，无重试 |
| `_tokenize` | rag.py:443 | jieba 不可用时 `" ".join(text)` 单字分割 |
| `_index_document` | rag.py:480 | 入库 title/body 都用 `_tokenize`，单字索引无法被多字 query 匹配 |
| `_build_fts_query` | rag.py:692 | 分词后构建 AND/OR 查询，单字 token 严重稀释精度 |
| `search_knowledge` | rag.py:787 | 三级降级策略，但底层分词已坏，降级无意义 |
| `build_rag_context_with_details` | rag.py:1664 | 仅靠 RRF + score_threshold 过滤，无相关性硬过滤 |

### 1.4 影响范围

- 所有 Agent 的 `prompt_context` 注入质量
- `search_knowledge` 工具调用返回结果质量
- LLM token 浪费（不相关上下文仍占 token 预算）
- 用户感知：Agent 回答"答非所问"或忽略知识库

## 二、P0 方案：安装 jieba + 重建 FTS5 索引

### 2.1 目标

让中文分词恢复正常，FTS5 入库和查询一致使用 jieba 分词，恢复 rank 排序能力。

### 2.2 实施

#### 2.2.1 安装 jieba

```bash
pip install jieba
```

并在 `requirements.txt` 中补全依赖（若存在）。

#### 2.2.2 重建 FTS5 索引

由于历史索引是按单字分割存储的，jieba 分词后的 query 无法匹配（query 是 "白酒 估值"，索引里是 "白 酒 估 值"）。必须重建。

**重建策略**：新增 `rebuild_fts_index()` 函数，遍历各内容源表，重新调用对应的 `index_*` 函数。

```python
def rebuild_fts_index() -> dict:
    """重建 FTS5 索引：清空 knowledge_fts 后从各源表重新索引。
    
    Returns:
        {"article": N, "author_article": N, "valuation": N, ...}
    """
    conn = _get_conn()
    # 清空 FTS5
    conn.execute("DELETE FROM knowledge_fts")
    conn.commit()
    conn.close()
    
    stats = {}
    # 1. articles
    # 2. author_articles
    # 3. index_valuations (valuation)
    # 4. analysis_records
    # 5. skill_documents
    # 6. book_knowledge / notes
    # 各自调用 index_* 函数
    return stats
```

#### 2.2.3 复位 jieba 缓存

`_get_jieba` 在 import 失败后会永久返回 `False`，安装 jieba 后需要复位：

```python
def _reset_jieba_cache():
    """复位 jieba 缓存，允许重新 import。"""
    global _jieba
    _jieba = None
```

在 `rebuild_fts_index` 入口调用一次。

### 2.3 验证

1. `python -c "import jieba; print('ok')"` 成功
2. `rebuild_fts_index()` 返回非零计数
3. 测试样本 query：`"白酒现在估值高吗"` → FTS5 命中包含 "白酒" 的索引项
4. 对比重建前后 rag_logs 中 results 与 query 的相关性

## 三、P1 方案：相关性评分过滤

### 3.1 目标

在 RAG 结果注入 LLM 之前，用 query 与 result 的词重叠率做硬过滤，剔除明显不相关的结果。

### 3.2 设计

#### 3.2.1 相关性评分函数

```python
def _relevance_score(query: str, result: dict) -> tuple[float, dict]:
    """计算 query 与检索结果的词重叠相关性分数。
    
    策略：
    1. jieba 分词 query，提取多字核心词（≥2 字）
    2. 在 result 的 title + body 中匹配
    3. 命中率 = 命中词数 / 核心 query 词数
    4. 标题命中加权 1.5x，正文命中加权 1.0x
    
    Returns:
        (score, debug_info)
    """
```

#### 3.2.2 过滤阈值

- `relevance_min_score`：硬过滤阈值，默认 0.15（即至少 15% 的核心词命中）
- 阈值过低 → 仍注入噪声；过高 → 误杀相关结果
- 通过 `rag_config` 表配置，可动态调整

#### 3.2.3 注入点

在 `build_rag_context_with_details` 中，**score_threshold 过滤之后、最终排序之前**注入：

```python
# 相关性硬过滤：剔除 query 词重叠率过低的结果
RELEVANCE_MIN = get_rag_config_float("relevance_min_score", 0.15)
if all_results:
    before = len(all_results)
    scored = []
    for r in all_results:
        score, info = _relevance_score(query, r)
        r["_relevance"] = score
        if score >= RELEVANCE_MIN or r.get("_direct_inject"):
            scored.append(r)
        else:
            logger.info(f"相关性过滤: 移除 '{r.get('title','')[:30]}' (score={score:.2f})")
    all_results = scored
    logger.info(f"相关性过滤: {before} → {len(all_results)}条 (阈值={RELEVANCE_MIN})")
```

### 3.3 边界处理

| 场景 | 处理 |
|------|------|
| 直接注入的估值数据 `_direct_inject=True` | 跳过过滤（已通过指数名严格匹配） |
| query 是 URL 或纯英文 | 跳过相关性过滤（交由 FTS5 自身处理） |
| query 核心词数 ≤ 1 | 跳过过滤（无法计算重叠率） |
| 过滤后结果为空 | 保留原结果（避免空上下文，让 LLM 自己判断） |
| valuation/analysis 类型已通过实体过滤 | 仍参与相关性过滤（双保险） |

### 3.4 防退化

- 阈值通过 `rag_config` 表配置，可灰度调整
- 过滤前后数量记录到 logger，便于事后审计
- 不影响 `rag_logs` 表的 `results_count` 字段（记录的是过滤前数量）

## 六、风险与回滚

### 6.1 风险

| 风险 | 概率 | 缓解 |
|------|------|------|
| jieba 安装失败（网络/权限） | 低 | 用国内镜像源 `pip install jieba -i https://pypi.tuna.tsinghua.edu.cn/simple` |
| 重建索引耗时过长 | 低 | 知识库规模 < 1万条，预计 < 30s |
| 相关性阈值过高误杀 | 中 | 默认 0.15 偏保守，可通过 rag_config 动态下调 |
| jieba 分词结果与预期不符 | 低 | jieba 默认词典对投资术语支持良好，必要时可加自定义词典 |

### 6.2 回滚

- 代码层面：相关性过滤是新增逻辑，注释掉即回滚
- 索引层面：FTS5 重建是幂等的，再次重建即可；最坏情况是单字索引继续使用（与现状一致）
- 配置层面：`rag_config.relevance_min_score` 设为 0 即关闭过滤
