## 一、现状梳理

### 1.1 Agent 架构现状

| 阶段 | 执行方式 | 数据可见性 | 割裂点 |
|------|---------|------------|--------|
| Phase A 专家分析 | 并行（ThreadPoolExecutor） | 每个专家只看 `prebuilt_context`，互不可见 | ❌ 专家间无实时信息流 |
| Phase B 交叉审阅 | 串行 | 通过 `<round_table>` 注入其他专家分析 | ✅ 已缓解，但串行慢 |
| Phase C 仲裁 | 串行 | 拿到全量专家分析 | ✅ 已解决 |
| 上下文过滤 | `filter_context_for_agent` 白名单 | 按职责裁剪（如 behavior_coach 看不到估值） | ⚠️ 过严可能丢信息 |

### 1.2 数据孤岛现状

| 关联 | 现状 | 问题 |
|------|------|------|
| 持仓↔预警 | `portfolio_alerts.related_fund_code` 字符串 | 无 FK，删持仓留孤儿预警 |
| 决策↔知识库 | `knowledge_base.source = "decision_review:123"` 字符串 | 无 FK，LIKE 查询走不了索引 |
| 文章↔估值 | 仅通过图片解析间接关联 | 无法反向查询"文章涉及的指数" |
| 历史结论→决策 | `analysis_conclusions` 表已建 | `build_decision_precheck` 未调用 `get_conflicting_conclusions` |
| 复盘教训→决策 | 教训写入 `knowledge_base` | precheck 不检索同标的历史教训 |
| 交易→估值快照 | `portfolio_transactions.valuation_snapshot` JSON | 覆盖率不全（依赖 holdings.index_code） |

### 1.3 决策链路断点

```
估值查询 → 持仓分析 → 风险评估 → 建议候选 → 决策草案 → 执行前检查 → 交易草稿 → 交易确认 → 决策复盘
   ✅         ✅        ⚠️         ⚠️         ✅         ⚠️         ✅         ✅         ⚠️
```

| 断点 | 描述 |
|------|------|
| 风险评估→候选 | `generate_portfolio_alert` 生成预警，但不自动入 `recommendation_candidates` |
| 候选→决策 | 转化率不可见，无统计 API |
| 执行前检查 | 不读 peer review、不读历史结论、不读历史教训 |
| 决策无过期 | accepted 状态超 7 天不确认交易，永远停留 |
| 复盘教训 | 写入 KB 但不参与下次决策生成 |

## 二、Agent 协同增强（解决割裂）

### 2.1 共享黑板架构（P0）

**目标**：Phase A 专家执行时，能读到同批次已完成专家的中间结论摘要。

**设计**：在 `orchestrate()` Phase A 阶段引入 `shared_blackboard` 字典：

```python
# orchestrator.py Phase A 改造
shared_blackboard = {}  # {agent_key: {"rating": "买入", "confidence": 0.8, "key_evidence": "..."}}

def _run_specialist_with_blackboard(agent_key, query, context, blackboard):
    """专家执行前，注入黑板中已有专家结论摘要。"""
    if blackboard:
        peer_summary = "\n".join([
            f"- {k}: {v.get('rating','')} (置信度{v.get('confidence','')})"
            f" | 关键证据: {v.get('key_evidence','')[:100]}"
            for k, v in blackboard.items()
        ])
        context += f"\n\n## 同批次专家结论（供参考，可质疑）\n{peer_summary}"
    result = run_specialist(agent_key, query, context)
    # 结构化提取后写入黑板
    blackboard[agent_key] = _extract_structured_conclusion(result)
    return result
```

**执行策略**：
- 单专家：直接执行（无黑板）
- 2-3 专家：改为**串行**（牺牲速度换协同），后执行的能看到前面的结论
- 4+ 专家：分两批，批内并行，批间串行（黑板传递）

**避免循环依赖**：黑板只读，专家不能修改前序结论，只能引用或质疑。

### 2.2 结构化专家输出（P1）

**目标**：专家输出从纯文本升级为结构化 JSON，便于仲裁精确裁决。

**设计**：在 `analysis_agents` 表的 `system_prompt` 末尾追加输出格式约束：

```json
{
  "rating": "买入|加仓|持有|减仓|卖出|观望",
  "confidence": 0.0-1.0,
  "key_evidence": ["证据1", "证据2"],
  "target_codes": ["161725", "510300"],
  "actions": [{"action": "buy", "target": "161725", "amount": "5000"}],
  "risks": ["风险1", "风险2"],
  "analysis": "详细分析文本（给用户看的自然语言）"
}
```

**兜底**：`_parse_structured_conclusion()` 函数做三级解析（JSON 块 → 正则提取 → 纯文本降级），保证旧 prompt 也能工作。

**收益**：
- 仲裁 agent 能基于 `rating`/`confidence` 字段精确判断分歧
- 冲突检测 `detect_conflicts` 用结构化字段替代关键词匹配，准确率提升
- 候选抽取 `extract_recommendation_candidates_from_analysis` 直接读 `actions` 字段

### 2.3 关键信号广播（P1）

**目标**：解决 `filter_context_for_agent` 白名单过严问题，关键信号跨专家广播。

**设计**：定义 `_CRITICAL_SIGNALS` 规则，当某专家发现关键信号时，强制注入到后续专家的上下文：

```python
_CRITICAL_SIGNALS = {
    "valuation_expert": {
        "高估": ["百分位>80", "高估"],
        "低估": ["百分位<20", "低估"],
    },
    "risk_assessor": {
        "集中度过高": ["集中度", "超过30%"],
        "现金不足": ["备用金", "不足"],
    },
}

def _broadcast_critical_signals(blackboard, agent_key, context):
    """从黑板中提取关键信号，注入到当前专家上下文。"""
    signals = []
    for src_key, rules in _CRITICAL_SIGNALS.items():
        if src_key == agent_key or src_key not in blackboard:
            continue
        peer = blackboard[src_key]
        for signal_name, keywords in rules.items():
            if any(kw in peer.get("analysis", "") for kw in keywords):
                signals.append(f"[{src_key} 发现: {signal_name}]")
    if signals:
        context += "\n\n## ⚠️ 关键信号广播\n" + "\n".join(signals)
    return context
```

**收益**：behavior_coach 能看到"估值专家发现高估"，进而分析用户是否处于追涨状态。

### 2.4 冲突预防而非仅检测（P2）

**目标**：当前是"冲突发生后仲裁"，增强为"冲突前提示专家协调"。

**设计**：Phase A 结束后，若 `detect_conflicts_smart` 检测到 high severity 冲突，**在 Phase B 交叉审阅前**，给每个专家注入"冲突摘要"：

```python
if conflict_severity == "high":
    conflict_summary = detect_conflicts_smart(specialist_results, query)
    for agent in original_specialists:
        # 注入："估值专家建议卖出（高估），风险专家建议加仓（分散度不足）"
        # 请你重新评估，要么给出一致建议，要么明确说明分歧原因
```

**收益**：减少 Phase B 交叉审阅的轮次，专家在审阅前就 aware 冲突。

## 三、数据共享增强（解决数据孤岛）

### 3.1 持仓↔预警 FK 强关联（P0）

**现状**：`portfolio_alerts` 只有 `related_fund_code` 字符串字段。

**改造**：

```sql
-- db/__init__.py init_db 中 _add_column_if_not_exists
ALTER TABLE portfolio_alerts ADD COLUMN holding_id INTEGER REFERENCES portfolio_holdings(id) ON DELETE CASCADE;

-- 创建索引加速关联查询
CREATE INDEX IF NOT EXISTS idx_alerts_holding_id ON portfolio_alerts(holding_id);
```

**数据回填**：`backfill_alert_holding_id()` 函数，用 `related_fund_code` 匹配 `portfolio_holdings.fund_code` 填充 `holding_id`。

**收益**：删除持仓时级联删除预警；`build_decision_precheck` 能高效 JOIN 查询"该持仓的所有预警历史"。

### 3.2 决策↔知识库 FK 强关联（P0）

**现状**：`knowledge_base.source = "decision_review:123"` 字符串。

**改造**：

```sql
ALTER TABLE knowledge_base ADD COLUMN source_decision_id INTEGER REFERENCES decision_records(id) ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS idx_kb_source_decision ON knowledge_base(source_decision_id);
```

**数据回填**：`backfill_kb_decision_id()` 解析 `source` 字段填充 `source_decision_id`。

**precheck 增强**：`build_decision_precheck` 新增"同标的历史教训"检查：

```python
# db/decisions.py build_decision_precheck 末尾追加
lessons = conn.execute("""
    SELECT title, content, importance FROM knowledge_base
    WHERE source_decision_id IN (
        SELECT id FROM decision_records
        WHERE target_code = ? AND target_type = ? AND status = 'reviewed'
    ) AND category = 'user_lesson'
    ORDER BY importance DESC LIMIT 3
""", (target_code, target_type)).fetchall()
if lessons:
    checks.append({
        "type": "historical_lessons",
        "severity": "warning",
        "message": f"该标的已有 {len(lessons)} 条历史教训",
        "details": [{"title": l["title"], "content": l["content"][:200]} for l in lessons],
    })
```

### 3.3 历史结论桥接层激活（P0）

**现状**：`analysis_conclusions` 表已建，`get_conflicting_conclusions` 已实现，但 `build_decision_precheck` 未调用。

**改造**：在 `build_decision_precheck` 中新增"历史结论冲突"检查：

```python
# db/decisions.py build_decision_precheck
from db.analysis_conclusions import get_conflicting_conclusions

# 查询同标的 24h 内的冲突结论
conflicts = get_conflicting_conclusions(target_code, action=proposed_action)
if conflicts:
    checks.append({
        "type": "historical_conflicts",
        "severity": "high",
        "message": f"24h 内有 {len(conflicts)} 条相反结论",
        "details": [{"source": c["source_system"], "action": c["action"], "summary": c["summary"]} for c in conflicts],
    })
```

**定期清理**：在 `app.py` 的定时任务中新增 `cleanup_expired_conclusions()`，清理 `expires_at` 过期的记录。

### 3.4 文章↔估值反向关联（P1）

**现状**：文章与估值无直接关联，无法查询"这篇文章涉及的指数当前估值如何"。

**设计**：新增 `article_index_mentions` 表：

```sql
CREATE TABLE IF NOT EXISTS article_index_mentions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    article_id INTEGER NOT NULL REFERENCES articles(id) ON DELETE CASCADE,
    article_type TEXT NOT NULL,  -- article / author_article / linked_doc
    index_code TEXT NOT NULL,
    index_name TEXT,
    mention_context TEXT,  -- 提及该指数的上下文片段
    created_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(article_id, article_type, index_code)
);
CREATE INDEX IF NOT EXISTS idx_aim_index ON article_index_mentions(index_code);
```

**抽取时机**：
- 文章入库时（`index_article`），调用 `_extract_index_mentions(content)` 用 `_get_known_index_names()` 匹配
- 已有文章批量回填：`backfill_article_mentions()`

**反向查询 API**：`GET /api/valuations/{index_code}/articles` 返回提及该指数的文章列表。

**收益**：估值页面展示"相关文章"，文章页面展示"涉及的指数当前估值"。

## 四、决策链路增强

### 4.1 决策过期机制（P0）

**现状**：accepted 状态超 7 天不确认交易，永远停留。

**改造**：在 `app.py` 定时任务中新增 `expire_stale_decisions()`：

```python
def expire_stale_decisions():
    """accepted 状态超 7 天的决策自动转 deferred。"""
    conn.execute("""
        UPDATE decision_records
        SET status = 'deferred', notes = COALESCE(notes,'') || '\n[系统] 7天未执行交易自动转延后'
        WHERE status = 'accepted'
        AND updated_at < datetime('now','localtime','-7 days')
    """)
```

### 4.2 多模型评审反哺 precheck（P0）

**现状**：`decision_peer_reviews` 表存评审结果，但 `build_decision_precheck` 未读取。

**改造**：

```python
# build_decision_precheck 新增
peer_reviews = conn.execute("""
    SELECT reviewer_agent, recommendation, confidence, reasoning
    FROM decision_peer_reviews
    WHERE decision_id = ?
""", (decision_id,)).fetchall()
```

## 五、功能节点增强（更好理财决策）

### 5.1 估值查询节点增强（P1）

| 增强点 | 实现 |
|--------|------|
| 低估指数→持仓匹配 | 估值列表 API 返回 `matched_holdings` 字段，前端展示"你持有的XX基金跟踪此指数" |
| 估值历史+持仓买卖点叠加 | `ValuationHistory.vue` 在 PE 曲线上叠加 `portfolio_transactions` 的买卖点 |
| 估值变动提醒 | 指数百分位跨过 30%/70% 阈值时，自动生成 `portfolio_alerts` |

### 5.2 持仓分析节点增强（P1）

| 增强点 | 实现 |
|--------|------|
| 持仓健康度综合评分 | 新增 `/api/portfolio/health-score`，综合估值（40%）+ 集中度（30%）+ 行为偏差（30%） |
| 持仓↔决策历史关联 | 持仓详情 API 返回 `decision_history`，展示"该基金 3 次决策，胜率 67%" |
| 跨持仓相关性 | `analyze_portfolio_diversification` 调用 `analysis/correlation` 子路由，返回相关性矩阵 |

### 5.3 风险评估节点增强（P1）

| 增强点 | 实现 |
|--------|------|
| 风险预警→候选自动转化 | 见 4.3 |
| 预警→历史对比 | 预警 API 返回 `historical_alerts`，展示"该持仓历史 N 次同类预警" |
| 跨持仓相关性预警 | 集中度超 30% 时，检查持仓间相关性，高相关追加预警 |

### 5.4 决策草案节点增强（P2）

| 增强点 | 实现 |
|--------|------|
| 决策证据卡片化 | 决策详情 API 返回结构化 `evidence_cards`：估值快照/持仓状态/历史教训/反方观点 |
| 决策置信度评分 | `decision_confidence = 0.3*证据数量 + 0.3*时效性 + 0.2*反方充分性 + 0.2*历史胜率` |
| 多模型评审集成 | 决策草案自动触发 peer review，结果作为 precheck 输入（见 4.2） |

### 5.5 决策复盘节点增强（P2）

| 增强点 | 实现 |
|--------|------|
| 实际收益 vs 预期 | `record_decision_review` 自动计算 `actual_return` vs `expected_return` |
| 教训标签化 | 教训写入 KB 时打标签：`behavior_bias`/`valuation_misjudge`/`timing_error` |
| 教训参与决策 | 见 3.2，precheck 检索同标的历史教训 |

## 八、风险与回滚

### 8.1 风险

| 风险 | 概率 | 缓解 |
|------|------|------|
| 共享黑板导致专家趋同（失去多样性） | 中 | 黑板只读，prompt 强调"可质疑" |
| 结构化输出解析失败 | 中 | 三级兜底解析，纯文本降级 |
| FK 级联删除误删 | 低 | ON DELETE SET NULL（知识库）vs CASCADE（预警） |
| precheck 过严阻断正常决策 | 中 | severity 分级，warning 不阻断，block 才阻断 |
| 定时任务性能 | 低 | 过期清理在凌晨执行 |

### 8.2 回滚

| 改动 | 回滚方式 |
|------|----------|
| 共享黑板 | 注释 `_run_specialist_with_blackboard`，恢复纯并行 |
| 结构化输出 | prompt 改回纯文本，`_parse_structured_conclusion` 降级返回 `{"analysis": text}` |
| FK 字段 | 字段保留不删，回填脚本不执行，precheck 不调用 |
| 决策过期 | 定时任务停用 |
| precheck 检查 | `rag_config.precheck_history_lessons = false` 等开关控制 |
