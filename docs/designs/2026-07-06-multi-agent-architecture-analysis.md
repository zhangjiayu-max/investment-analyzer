# 多智能体编排架构问题分析 — 为什么建议是割裂的

**日期**: 2026-07-06
**版本**: v1.0
**问题**: 用户反馈 AI 回答不结合持仓盈亏，只看估值推荐加减仓

---

## 一、当前架构流

```
用户提问
    │
    ▼
Orchestrator（协调者，不做深度分析）
    │  Smart Router（规则+LLM 选择专家）
    ▼
专家并行/串行执行（每个只看自己领域）
    ├── valuation_expert → "估值低，建议加仓"
    ├── risk_assessor → "风险适中，可以接受"
    ├── allocation_advisor → "配置合理"
    │
    ▼
交叉审阅（已输出的专家互评，难改变结论）
    │
    ▼
仲裁（缝合所有结论，但回不去）
    │
    ▼
输出："估值低，建议加仓"
```

## 二、架构层面 6 个问题

### 问题 1：没有"全局扫描"角色（最核心）

```
当前架构：

Orchestrator（协调者）          →  只做路由，不做分析
↓
valuation_expert（估值专家）     →  只看估值
risk_assessor（风控专家）        →  只看风险
allocation_advisor（配置专家）   →  只看配置

没有任何一个角色被要求做"全局综合分析"。
```

**问题本质**：每个专家都被刻意训练成"在自己的领域最专业"，但投资决策需要**跨领域综合判断**。估值专家的 prompt 要求他分析估值水位——他就分析估值水位。他**没有义务**去看持仓盈亏、看历史操作。

**类比**：像医院会诊——眼科只看眼睛、骨科只看骨头。但患者是来看"全身不舒服"的。需要一个全科医生先做**整体评估**，再决定让谁深入检查。

**当前系统缺少这个"全科医生"角色。**

### 问题 2：专家一旦输出结论，就很难回头

```
时间线：
Step 1: valuation_expert 输出 "估值低，建议加仓"    ✅ 已得出结论
Step 2: risk_assessor 输出 "风险可接受"              ✅ 已得出结论
Step 3: orchestrator 发现 "用户实际已亏损15%"       ❌ 但valuation_expert已经分析完了
Step 4: 仲裁者想修正 → 只能拼接，不能重跑
```

**语言模型的认知固化问题**：LLM 一旦在某个上下文中输出了一组结论，它就有了"我认同这个结论"的倾向（类似人类的确认偏误）。交叉审阅让另一个专家来看结论——但另一个专家也有自己的偏见（他站在自己的领域）。

**正确做法**：在专家**开始分析之前**，就告诉他全貌。而不是等他分析完了再告诉他"还有重要信息你没考虑"。

### 问题 3：Orchestrator 的 prompt 角色定位窄了

当前 orchestrator prompt（~2024 行）的核心职责：
```
1. 理解用户问题
2. 决定需要哪些专家
3. 收集专家结果
4. 综合给出建议
```

注意第 4 点："综合"——但 orchestrator 并不做深度分析。它只是在专家输出基础上"拼接"。如果专家们都没有看 P&L，orchestrator 拼接的结果也不会有 P&L 的考量。

### 问题 4：共享黑板（P0-2.1 新增）的局限性

近期加的共享黑板让后执行的专家能看到前序结论，但：

```
局限性 1：顺序依赖
第一个执行的专家仍然没有全局视角
→ 如果 valuation_expert 第一个执行，它还是只看估值

局限性 2：只能引用，无法反攻
后续专家可以"引用"前序结论，但不能"因为前序专家没看P&L，我要求重跑"

局限性 3：没有强制"必须质疑"
共享黑板的措辞是"供参考，可质疑"——这是可选的，不是必须的
```

### 问题 5：没有"迭代闭环"

```
当前：Route → Execute → Arbitrate → Output
需要：Route → Execute → Review → **Re-execute** → Arbitrate → Output

缺少"再执行"这个环节。
```

如果仲裁者发现"valuation_expert 忘看 P&L 了"，当前无法让它重跑。这是一个**单向流程**，不是**迭代流程**。

### 问题 6：反方观点（counter_argument）没有被充分利用

counter_argument（反方观点审查员）只在"高风险行动"时才触发——"清仓、满仓、梭哈"等关键词。

但它应该是一个**标准流程组件**——每个建议都应该有一个"站在对立面"的检查：

```
"你建议加仓XX基金，但从用户持仓看：
- 该基金已占组合32%（超过25%上限）
- 用户已亏损12%（继续加仓可能加剧焦虑）
- 近30天已有3次加仓记录
你还坚持建议加仓吗？"
```

## 三、修复方案：架构调整

### 核心改动：从"领域专家"到"情景专家"

```
改动前：
valuation_expert(query) → 只看估值 → 输出分析

改动后：
valuation_expert(query + portfolio_snapshot + recent_trades) 
→ 先看全局 → 再在自己的领域输出 → 输出包含"对当前持仓的影响"
```

### 改动 1：新增"全景扫描"步骤（P0）

在 route 之后、specialist 执行之前，增加一个"全景扫描"步骤：

```python
def _run_panorama_scan(query, portfolio, recent_trades, decisions) -> dict:
    """
    在全方面扫描：
    1. 当前持仓概况（总资产/总盈亏/各大类占比）
    2. 每个持仓的盈亏率
    3. 近30天交易记录（哪些基金加仓/赎回了）
    4. 待执行/待决策的事项
    5. 历史上针对同类问题的建议
    
    产出：一个"全景上下文"，注入到所有 specialist 的 prebuilt_context
    """
```

**关键**：这个扫描**不调 LLM**，纯 SQL 查询。结构化数据（JSON），不是文本段落，让 LLM 更容易解析。

### 改动 2：Specialist prompt 增加"全局视角"指令（P0）

在每个 specialist 的 system prompt 末尾强制追加：

```python
UNIVERSAL_CONTEXT_INSTRUCTION = """
## ⚠️ 你必须先看全局数据，再看自己的领域

在执行分析前，你必须先查看以下数据（在上下文中）：
1. 「用户当前持仓」— 哪些基金、各盈亏率、各占比
2. 「交易记录」— 近30天用户对相关基金的操作
3. 「决策记录」— 之前针对该标的的决策和建议
4. 「债市数据」— 当前债市温度

### 你的分析不能仅基于估值/风险/配置单一维度
估值低 → 先看用户是否已持有、已亏赚、已买过几次 → 再给出综合建议
"""
```

### 改动 3：改变共享黑板格式（P0）

当前共享黑板是"前序专家的结论文本"。改为结构化的"关键信息提取"：

```python
# 不传完整结论，只传"关键发现"
BLACKBOARD_FORMAT = """
## 已分析专家关键发现
| 专家 | 核心结论 | 关键数据引用 | 置信度 |
|------|---------|------------|--------|
| valuation | 沪深300PE分位42% | 数据来源: 估值表 | 高 |
| risk | 组合整体风险R3 | 数据来源: 持仓分析 | 中 |

### 待确认事项
- 估值专家未考虑用户持仓盈亏
- 风控专家未考虑近30天交易频率
"""
```

### 改动 4：增加"再执行"环节（P1）

```python
# 仲裁者检查完整性
def _check_analysis_completeness(specialist_results, portfolio_context) -> list[str]:
    """
    检查专家分析是否覆盖了所有必要维度。
    
    检查清单：
    □ 估值分析是否提及了该基金在持仓中的占比？
    □ 风控分析是否考虑了用户的实际亏损？
    □ 配置建议是否引用了最近的操作记录？
    
    返回缺失项列表 → 触发再执行
    """
```

### 改动 5：反方观点作为标准环节（P1）

把 `counter_argument` 从"高风险才触发"改为"所有建议的标准检查环节"：

```python
# 在所有分析完成后，自动触发反方检查
def _run_counter_check(answer: str, portfolio_context: str) -> str:
    """
    对最终建议做"反方检查"：
    "这个建议在什么情况下是错的？"
    "用户如果按建议操作，最坏的结果是什么？"
    "有没有更保守的替代方案？"
    """
```

```
之前找到的（配置级问题）：
1. reuse_recent_conclusions = false ← 改1行配置
2. prompt 没提"必须看P&L" ← 改10行prompt
3. 缺少交易记录注入 ← 加30行代码

这次找到的（架构级问题）：
1. 没有全局扫描角色 ← 架构调整
2. 专家隔离导致视野窄 ← prompt架构调整
3. 单向流程无迭代 ← 流程架构调整
4. 反方未充分利用 ← 角色调整
```

**配置级问题修了，能改善 30%。架构级问题修了，能改善 70%。**

最简单的第一步是先加第 1 个（全景扫描），能让所有专家在分析前就看到全局数据。
