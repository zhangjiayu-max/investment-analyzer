# 2026-07-05 多智能体结论价值最大化与高频 Bug 修复

## 一、问题诊断（基于真实数据）

### 1.1 数据采集结果

| 维度 | 现状 | 问题 |
|------|------|------|
| rebalancing 异步任务 | 近 7 天 12 次失败（仅 1 次成功） | unsupported operand type(s) for +: float and str |
| market_intelligence 异步任务 | 近 7 天 5 次失败 | Unknown format code f for object of type str |
| index_valuations.percentile 类型 | 873 real + 12 text + 13 null | 12 条历史数据未规范化 |
| analysis_conclusions 表 | 仅 5 条（全部来自独立分析路径） | ai_dialogue 路径从未持久化结论 |
| cross_system_references 表 | 0 条 | 表存在但从未写入 |
| conv 82 chat_turn | 485928ms（8 分钟） | 长对话无超时保护，用户感知差 |
| conv 84 cross_review | run#739/#740 仅产出函数调用文本，无分析 | token 被消耗但无可用结论 |

### 1.2 5 个高价值增强点

**Bug A — percentile 字符串致任务失败**
- 根因：DB 中 12 条历史 percentile 是文本型，部分消费点未受保护
- 影响：每天触发 rebalancing/market_intel 任务都失败

**Bug B — cross_review 仅产出 tool_call 文本**
- 根因：multi_agent.py L478-484 清理逻辑有缺陷，cleaned 过短时保留原 answer
- 影响：cross_review 标记 success 但无分析内容

**Bug C — ai_dialogue 路径不持久化结论**
- 根因：save_analysis_conclusion 仅被 5 个独立分析路由调用，orchestrator/multi_agent 从未调用
- 影响：用户花 50 万 token/天做多智能体对话，结论只活在内存中

**Bug D — cross_system_references 表空置**
- 根因：建表但无任何 insert 调用
- 影响：跨系统决策无法追溯

**Bug E — 长对话无超时保护**
- 根因：orchestrate() 主循环没有总时长限制
- 影响：单个 chat_turn 可达 8 分钟，用户体验差、token 失控

## 二、设计目标

| 目标 | 度量 |
|------|------|
| 修复高频 Bug | rebalancing/market_intel 失败率从 92%/100% 降到 0% |
| cross_review 输出有效性 | 不再出现仅含 tool_call 文本的 agent_run |
| 多智能体结论沉淀 | ai_dialogue 路径每日新增 ≥3 条 analysis_conclusions |
| 跨对话结论复用 | 24h 内同标的查询命中率 ≥30%（节省 token） |
| 长对话保护 | chat_turn 大于 5min 警告、大于 8min 自动收尾 |

## 三、技术方案

### P0-1：修复 percentile 文本型致任务失败（Bug A）

策略：写入防御 + 消费防御 + 数据迁移 三层兜底

1. 数据迁移（幂等）：在 db/_conn.py 的 init_db() 末尾追加迁移逻辑，遍历 typeof(percentile)=text 的记录，把 "97.25%" 转为 97.25 后 UPDATE 回写。

2. 消费点防御：新增 _safe_float(v, default=None) 工具函数。
   - 应用到 services/rebalancer.py L99 和 L218
   - 应用到 routers/analysis/market_intel.py L350

3. 验证：触发一次 rebalancing/market_intel 任务应成功。

### P0-2：修复 cross_review 仅产出 tool_call 文本（Bug B）

策略：清理后若内容过短 → 强制重新生成

修改 agent/multi_agent.py L478-484（run_specialist 收尾）和 L683-687（run_specialist_with_context 收尾）。

新逻辑：
- 检测 answer 是否含函数调用标签起始子串
- 用正则 DOTALL 模式剔除整段函数调用块，得到 cleaned
- 若 cleaned 长度大于等于 200 字 → 采纳 cleaned
- 否则 → 追加一条 user 消息要求给出分析结论（不要调用工具），再调用一次 LLM
  - 若重新生成的文本长度大于等于 200 字 → 采纳
  - 否则 → 兜底设为"分析过程遇到问题，请重试"
- 最后走 _clean_dsml_from_content(answer) 收尾

### P1-1：多智能体结论持久化（Bug C）

策略：在 orchestrator 收尾时把每个专家 + 仲裁的结论写入 analysis_conclusions

1. 表结构调整（幂等 ALTER）：在 db/analysis_conclusions.py 的 init_conclusions_tables 末尾追加，给 analysis_conclusions 加 conversation_id / message_id 字段。

2. 扩展 save_analysis_conclusion 签名：增加可选参数 conversation_id 和 message_id。

3. 新增 _persist_agent_conclusions() 工具函数（在 agent/orchestrator.py）：
   - 幂等检查：同 conversation_id+message_id 已有记录则直接 return
   - 提取 target_subject：用正则匹配 6 位基金代码，否则匹配中文板块关键词，最后兜底"整体组合"
   - 遍历 specialist_results，每个非 cross_review 的专家写一条
   - 仲裁结论单独写一条：source_type=arbitrator, confidence=0.9, urgent=1

4. 在 orchestrator 主循环收尾时调用，失败仅 warning 不影响主流程。

### P1-2：跨对话结论复用（结论缓存）

策略：在 orchestrate 入口检查 24h 内同标的结论，命中则注入到 specialist 的 prebuilt_context

新增 _load_recent_conclusions(target_subject, hours=24, limit=3)：
- SQL 查询 analysis_conclusions 中 target_subject LIKE 且 source_system=ai_dialogue 且 24h 内的记录
- 格式化为 markdown 列表
- 在 run_specialist 调用前注入到 prebuilt_context（仅给初始 specialist）

注意：此功能默认关闭（agent.reuse_recent_conclusions=false）。

### P1-3：激活 cross_system_references 桥接（Bug D）

策略：当 analysis_conclusion 写入时，自动链接相关 decision_records

新增 link_conclusion_to_decisions(conclusion_id, target_subject)：
- 查询 decision_records 中 target_code LIKE target_subject AND status=accepted
- 对每条匹配的 decision_record，INSERT OR IGNORE 到 cross_system_references

在 save_analysis_conclusion 写入成功后调用。

### P2-1：长对话超时保护（Bug E）

策略：5min 警告 SSE + 8min 硬收尾

修改 routers/conversations.py 的 _producer_thread：
- 在每个 agent_run 启动前检查 elapsed = time.time() - producer_started
- elapsed 大于等于 8min → queue.put warning + break
- elapsed 大于等于 5min 且未警告过 → queue.put info + 标记 warned

## 六、配置开关（遵守硬约束）

| key | 默认值 | 说明 |
|-----|--------|------|
| agent.persist_conclusions | true | 是否持久化多智能体结论到 analysis_conclusions |
| agent.reuse_recent_conclusions | false | 是否跨对话复用 24h 内同标的结论 |
| agent.reuse_conclusions_hours | 24 | 复用窗口（小时） |
| agent.link_cross_system_refs | true | 是否激活 cross_system_references 桥接 |
| conversation.warn_at_minutes | 5 | 长对话警告阈值 |
| conversation.abort_at_minutes | 8 | 长对话硬收尾阈值 |

注：reuse_recent_conclusions 默认 false，因为它会改变 agent 看到的 prompt；其余 3 个开关纯 DB 操作，默认开启以直接修复用户痛点。

## 七、回滚方案

- 所有 ALTER TABLE 都是 ADD COLUMN（不破坏现有数据）
- 所有新函数都包 try/except，失败不影响主流程
- 迁移函数幂等，可重复执行
- 若 conclusion 持久化导致性能问题，可通过 agent.persist_conclusions=false 关闭
