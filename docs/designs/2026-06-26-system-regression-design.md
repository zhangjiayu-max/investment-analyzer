## 二、逐功能点分析与优化建议

### 1. AI 对话（核心功能）

**现状：** 16个Agent协作，含仲裁法官、交叉审阅、人在回路、成本路由。对话54条，但评测仅2条。

**问题：**

| # | 问题 | 严重度 | 分析 |
|---|------|--------|------|
| 1.1 | 评测仅2条，自我增强闭环形同虚设 | P0 | 自动评测胶水已写，但 `has_evaluation` 去重 + ThreadPoolExecutor 可能静默失败。145万token/天跑了多少评测？需查日志 |
| 1.2 | 对话摘要全空 | P1 | `conversation_summaries` 表0条。对话超过12条消息后，旧消息被摘要压缩，但摘要从未生成。长对话上下文丢失风险 |
| 1.3 | 交易记录刚加入上下文，但未加入"已卖出基金不可再建议卖出"的硬约束 | P1 | 当前只是 ⚠️ 提示文字，Agent可能忽略 |
| 1.4 | cost_routing 全部指向 `deepseek-chat`，失去差异化 | P2 | 原本设计是计算类用便宜模型、推理类用贵模型。现在全一样，成本路由名存实亡 |
| 1.5 | 6月25日消耗145万token | P2 | 按当前MIMO套餐估算，单日成本不低。缺少按Agent维度的成本分析 |

**优化建议：**

- **1.1** 检查 `_schedule_auto_evaluation` 的 ThreadPoolExecutor 是否有异常吞没。加日志：评测启动时 log，完成时 log，失败时 log + error traceback。手动触发一次评测验证端到端链路
- **1.2** 检查摘要生成触发条件。对话≥8条消息时自动生成摘要。当前可能是触发逻辑有bug或从未达到触发条件
- **1.3** 在仲裁法官prompt中加入硬规则：「禁止对用户近30天内已卖出的基金建议继续卖出。如果该基金仍有持仓，只能建议持有或加仓」
- **1.4** 重新配置cost_routing：计算类（fund_analyst/cross_review/conservative）用 `deepseek-chat`，推理类（arbitrator/market_analyst/orchestrator）用 `mimo-v2.5-pro`
- **1.5** 在 `token_usage` 表增加 `agent_key` 维度统计，Dashboard增加成本热力图

### 2. 每日持仓提示

**现状：** 1次运行，24条信号，含定投建议+板块轮动+行业扫描。

**问题：**

| # | 问题 | 严重度 | 分析 |
|---|------|--------|------|
| 2.1 | 仅运行过1次（6-25），没有形成每日习惯 | P0 | 定时任务可能未配置或未触发 |
| 2.2 | 4条pending的buy交易amount=0，shares=None | P1 | 信号转交易时金额未传递，需检查 `create_candidate` 到交易的映射逻辑 |
| 2.3 | 板块轮动信号只给"watch"级别，没有具体的基金推荐 | P2 | 用户看到"中证中药极度低估"但不知道买什么基金 |
| 2.4 | 信号缺少"已持有检查" | P1 | 持仓中已有医疗器械ETF（-20%），但信号仍建议关注医疗相关。信号应标注"你已持有XX" |

**优化建议：**

- **2.1** 配置cron每日8:30自动运行 `daily_advice/run`。加日志确认执行
- **2.2** 检查 `suggested_amount` 到 `portfolio_transactions.amount` 的映射。pending交易amount=0说明映射断了
- **2.3** 板块轮动信号增加"推荐基金"字段。维护一个"指数→推荐基金"映射表（如中证中药→中药ETF 501011）
- **2.4** 信号生成时检查持仓表，如果用户已持有该行业基金，在summary中标注"你已持有XX，当前盈亏-20%"

### 3. 持仓管理

**现状：** 19只基金，总资产约80万。债券占62%，指数型20.7%，现金16.1%。

**问题：**

| # | 问题 | 严重度 | 分析 |
|---|------|--------|------|
| 3.1 | 债券占比62%，其中博时恒乐A+C合计41万（51%）| P0 | 单一基金过度集中。虽然用户本周在止盈，但系统未主动预警此风险 |
| 3.2 | 4条pending交易amount=0，状态卡住 | P1 | 用户前端操作可能未完成或后端处理异常 |
| 3.3 | 4个pending交易未自动过期 | P2 | 基金交易T+1确认，pending超过2天应自动标记为failed |
| 3.4 | fund_name很多是None | P2 | 交易记录创建时未关联基金名称 |

**优化建议：**

- **3.1** 健康分应反映集中度风险。当前健康分已有集中度维度，检查是否正确计算
- **3.2** 提供API手动清理pending交易，或前端增加"取消"按钮
- **3.3** 加定时任务：pending超过48小时自动标记failed，并通知用户
- **3.4** 创建交易时自动从 `portfolio_holdings` 或基金信息表补全 `fund_name`

### 6. 评测系统

**现状：** 72条用例，137次运行，但对话评测仅2条。

**问题：**

| # | 问题 | 严重度 | 分析 |
|---|------|--------|------|
| 6.1 | 对话评测覆盖率极低（2/54 = 3.7%） | P0 | 自我增强闭环的核心输入，没评测就没有改进 |
| 6.2 | LLM-as-Judge 五维度评分，但评分模型用的是 `deepseek-chat` | P1 | 评分质量取决于模型能力，用便宜模型评分可能不够准确 |
| 6.3 | 评测结果未反馈到Agent改进 | P1 | 评测跑完了但没有触发prompt优化 |

**优化建议：**

- **6.1** 手动触发一批历史对话评测（至少10条），验证自动评测链路。然后开启每次对话结束自动评测
- **6.2** 评测模型改用 `mimo-v2.5-pro`（推理能力强）。评测不频繁，成本可控
- **6.3** 评测均分 < 70 的对话，自动提取Bad Case写入 `eval_cases`，作为prompt优化的输入

## 四、核心洞察

### 最关键的结构性问题

**自我增强闭环是整个系统的"灵魂"，但从未跑通过。** 这个闭环设计得非常完整：

```
对话 → 自动评测 → 低分 → 根因分析 → prompt改进 → Shadow → 晋升 → 评测基准更新
```

但如果这个闭环从未运行，意味着：
- Agent prompt从不改进（9个版本都是手动创建的）
- 评测用例不增长（72条都是手写的）
- Shadow模式是空架子
- 系统在"原地踏步"而非"自我进化"

**建议：花一天时间端到端跑通一次完整闭环，修复所有断点。这比做10个P2功能价值大得多。**

### 数据质量 > 功能数量

系统已经有375条路由、73个组件、90张表。功能足够多了。当前最大的问题不是"缺功能"，而是"已有功能没有跑起来"：

- 评测系统：有，但只评了2条
- 每日提示：有，但只跑了1次
- Shadow模式：有，但从未运行
- 对话摘要：有，但0条记录
- 决策追踪：有，但没有闭环

**建议：暂停新功能开发，花一周时间把已有功能全部跑通。**
